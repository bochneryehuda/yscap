# E2 — Verification Gates & Rule Ideas: the Encompass-Gated Rule Layer

Research agent E2. Scope: ideate the rule/verification layer that sits on top of the read-only
Encompass integration — the catalog of Encompass-gated rules, severity/staleness/override design,
and the mismatch dashboard. Grounded in `docs/condition-center.md`, `docs/SYNC-GUARD-MATRIX.md`,
the live gate code in `src/routes/staff.js`, and the Encompass Postman endpoint index.

---

## 1. Executive summary

The portal already has **every structural pattern this layer needs** — it just needs an
"Encompass evidence" input plugged into them:

| Existing pattern | Where | What the Encompass layer reuses it for |
|---|---|---|
| Plain-language sign-off gate returning a block reason (422) | `signOffGate()`, `src/routes/staff.js:2400`, enforced at `:2600` | Condition-parity gates (Family A) |
| Status-advancement gate with blockers + admin `force` + `forced` flag in history | `advancementBlockers()` `src/routes/staff.js:3895`; PATCH `/applications/:id` `:4247–4266` | CTC / funded gates (Family B) — add an `encompass` blocker group |
| DB belt-and-suspenders trigger on condition completion | `db/069_sow_budget_guard.sql` precedent | Optional second layer for the CTC gate |
| Human review queue, two-sided values, winner resolution, sticky dismissals, LO ownership | `sync_review_queue` (`db/108`, `db/110`), `SyncReviews.jsx` contract (CLAUDE.md §5d) | Override/decision flows & stuck states |
| Drift auditor, staleness heartbeat, value-aware dismissals, notification caps, high-stakes-per-row | Guards #7/#37/#73, #16, #67, #66/#79, #64 in `docs/SYNC-GUARD-MATRIX.md` | The mismatch dashboard & quiet-by-default design |
| Registry-over-code philosophy | SYNC-GUARD-MATRIX "Reading the matrix": *"a guard that can be a registry row should be a registry row"* | Rules ship as config rows with per-rule severity/tolerance/kill-switch |

Because Phase 1 is strictly read-only, **every rule consumes a local "Encompass evidence cache",
never a live write scope** — the rule layer itself reinforces the architecturally-read-only goal.

Design center: **quiet by default**. Only ~5 rules hard-block. Everything else is a badge, a
dashboard row, or silence. Rules must *help* close loans (including "Encompass already cleared
this — want to mark it here?" suggestions), not nag.

## 2. Grounding — the portal machinery these rules attach to

  (snapshots). Types: `document`, `info_field`, `tool`, `esign`, `internal_task`,

  `prior_to_approval, prior_to_docs, prior_to_closing, at_closing, prior_to_funding, funding,
  post_closing` (`db/037_condition_center.sql:56`).
- The rule engine evaluates `rule_logic` jsonb against a typed field registry
  (`src/lib/conditions/field-registry.js`, ~45 fields) at defined re-evaluation hooks

### 2.2 The advancement gate that CTC rules extend

`PATCH /api/staff/applications/:id` (staff.js:4247): moving to `clear_to_close` or `funded`
calls `advancementBlockers()`; blockers ⇒ 409 `{error:'blocked', target, blockers}`; an admin
may `force:true`, which stamps `forced` into `application_status_history` and the audit log
(staff.js:4260–4276). **This is exactly the shape an Encompass gate needs — add a third blocker
group `encompass:[...]` beside `conditions` and `gates`, and the readiness widget
(`GET /applications/:id/gating`, staff.js:3926) surfaces it for free.**

Note the asymmetry: a status arriving *inbound from ClickUp* is already a fact in another
system — it must be **flagged, not blocked** (create a mismatch row). Only portal-initiated
transitions can be blocked.

### 2.3 What the portal knows today about Encompass: hearsay

`applications.encompass_status` is a free-text column **pulled from a hand-typed ClickUp
dropdown** (`db/047_clickup_extra_fields.sql:25`; mapper `src/clickup/mapper.js:122`,
`dir:'pull'`). Today "what Encompass says" reaches the portal second-hand through a human
re-typing it into ClickUp. The integration replaces hearsay with evidence — and comparing the
two is itself a useful rule (F4 below).

## 3. Architecture of the verification layer

### 3.1 Evidence cache, never live calls in the request path (by default)

New tables (names illustrative): `encompass_loans` (one row per linked loan: loan_guid,
loan_number, amount, rate, closing dates, current milestone, folder, lock summary,
`fetched_at`, `source poll|on_demand`), `encompass_conditions` (per condition: title,
category, status, status dates, tracking tail, `fetched_at`), `encompass_milestones`.
Applications gain `encompass_loan_guid` (+ exclusivity, rule I3).

Rules read the cache. Two exceptions ("refresh-then-decide", §3.4): the hard gates B1 and E1
may make **one bounded on-demand GET** when the cache is older than the rule's freshness
budget. The rule layer needs zero write scopes — read-only by construction.

### 3.2 Verdict model

Every rule evaluation returns one of:

`PASS` · `FAIL` · `STALE` (cache older than budget, refresh failed) · `UNLINKED` (file has no
Encompass loan bound) · `UNKNOWN` (linked, but the mapped Encompass object doesn't exist — e.g.
no matching condition).

`UNLINKED`/`UNKNOWN` are first-class: they route to linkage rules (Family I) instead of
producing false FAILs.

### 3.3 Severity tiers (quiet by default)

| Tier | UX | Default share of catalog |
|---|---|---|
| **BLOCK** | 422/409 with a plain-language reason in `signOffGate` style (state both values, say how to fix, name the override) + override path | ~5 rules |
| **CONFIRM** | One modal: "Encompass disagrees — here's what it shows. Proceed anyway?" Proceeding is logged with reason; no email | small |
| **NOTICE** | Quiet badge/chip on the file + dashboard row; at most one consolidated banner per file screen ("Encompass: 3 things to look at"); no email | most rules |
| **SILENT** | Dashboard row only | drift metrics, low-stakes parity |

Severity is **per-rule config, not code** (`encompass_rules` registry row: `rule_key, enabled,
severity, tolerance, freshness_budget_sec, override_capability`), so the owner can promote or
demote a rule without a deploy. Global mode switch `ENCOMPASS_GATES=off|shadow|notice|enforce`
for staged rollout (§8).

### 3.4 Staleness handling ("what if the data is 2 hours old?")

Per-rule **freshness budget**; behavior depends on tier:

| Tier | Budget (suggested default) | If cache is older |
|---|---|---|
| BLOCK gates (B1, B2, E1, A1) | 15 min | Attempt ONE on-demand refresh (≤5s, single loan GET). Fresh ⇒ decide. Refresh fails ⇒ downgrade to CONFIRM: "Encompass couldn't be reached; its last known answer (as of 9:40 AM) was X. Proceed / wait / retry." Proceeding on stale data is journaled as `stale_proceed` with the age. |
| CONFIRM | 1 h | Show the age inside the modal ("as of 2 hours ago"). Never block on Encompass being down. |
| NOTICE / SILENT | 6 h / 24 h sweep | Rows carry `as of` timestamps; a single staleness banner appears portal-wide only when the whole feed is dead (heartbeat, guard #16 style: staleness is a number, shown once — not per-row nagging). |

Key principle: **Encompass being unreachable must never freeze the portal.** Hard gates degrade
to informed human confirmation; soft rules degrade to silence + heartbeat.

### 3.5 Override path (every BLOCK has one)

- **Who:** new capability `override_encompass_gate` layered on the existing capability system
  (`src/lib/permissions.js`; default: underwriter, admin, super_admin — grantable per user like
  every other capability, condition-center.md §Roles).
- **How:** the block response carries `overridable:true`; the UI collects a **required reason**
  (mirrors the `pushBack` reason requirement, staff.js:2636) and re-submits with
  `override:true`. Server re-verifies capability, applies, and:
  - writes `audit_log` (`encompass_gate_override`: rule_key, portal value, Encompass value,
    snapshot age, reason) — same pattern as `forced` status changes (staff.js:4276);
  - creates a dashboard row in state `overridden` that stays visible until Encompass agrees or
    the file closes;
  - is **value-aware sticky** (guard #67): the override binds to the exact (portal value,
    Encompass value) pair. If Encompass's value *changes again*, the gate re-arms — an old
    override never suppresses a new disagreement.
- **Reporting:** weekly override digest to admin; override count is a dashboard KPI. Overrides
  are legitimate (Encompass lags reality sometimes) — but they must be visible, so the pattern
  "everyone overrides rule X daily" reads as "fix rule X or fix the process", not silence.
- High-stakes overrides are **per-row only, never bulk** (guard #64).

### 3.6 Condition mapping model (prereq for Family A)

`encompass_condition_map`: `template_code / template_id → matcher` (Encompass condition **title
pattern + category**, per condition set). Managed in a Condition Studio extension (the
`manage_conditions` surface already exists, condition-center.md §surfaces). Rules:

- Unmapped portal conditions are **not gated** (quiet by default; gating is opt-in per mapping).
- Matching is confirm-first: a fuzzy title match below exact confidence creates a one-time
  "confirm this mapping" review row instead of silently binding (the G-D4 first-contact
  precedent). Confirmed matches pin the Encompass `conditionId` on the `checklist_item`
  (`origin_detail` jsonb or a new column) so later renames can't re-target.
- Encompass Enhanced Conditions carry a richer status vocabulary (typically
  Added/Requested/Received/Reviewed/Cleared/Rejected/Waived + tracking log — confirm for this
  instance). Define once, in one module, a normalization to portal semantics:
  `cleared = Cleared|Waived`, `received = Received|Reviewed`, `open = everything else`.

## 5. The Mismatch Dashboard

### 5.1 Two surfaces

1. **Per-file "Encompass agreement" panel** (staff file screen): one chip in the header —
   green "In agreement (as of 10:42)", or amber "3 things to look at" expanding to the rows.
   Never more than one banner; CONFIRM/BLOCK surfaces only appear at their trigger moments.
2. **Global `/internal/encompass-drift`** (pattern-match `/internal/sync-reviews`): every open
   row portfolio-wide, filterable by rule family / severity / LO / milestone / age.

### 5.2 Row model

`encompass_drift` table (separate from `sync_review_queue` — different lifecycle: drift rows
**auto-close on agreement**, review rows resolve by decision):

```
file · rule_key · severity · portal_value · encompass_value · first_seen · last_seen ·
evidence_fetched_at · state(open|acknowledged|overridden|auto_closed) · owner (file LO) ·
dismissed_value_pair (value-aware sticky, guard #67)
```

- **Dedupe:** one open row per (application, rule, value-pair) — the partial-unique-index
  pattern of `uq_sync_review_open` (db/108:44).
- **Auto-close on agreement** (guard #10 precedent): when a refresh shows the values agree, the
  row closes itself silently. Nobody ever cleans up a stale card.
- **Sticky dismissals are value-aware:** dismissing "amount differs by $500" stays dismissed
  for *that pair*; a new $2,000 divergence respawns (guard #67 / FMEA R7).
- **High-stakes rows (B, E1, I3) are per-row only — excluded from any bulk action** (guard #64).

### 5.3 Notifications: state-transition only, capped, digested

- A row **appearing** at NOTICE tier notifies no one; it badges.
- Only G2 (CTC revoked), G4 (new condition post-CTC), D4 (closing moved), E3 (lock died) send a
  targeted LO notification — one per state transition, deduped, through `src/lib/notify.js`.
- Daily LO digest ("Your files: 4 Encompass notes") + weekly admin digest (drift totals,
  override list) — per-recipient cap + roll-up per guards #66/#79.

### 5.4 KPIs on the dashboard header

Open mismatches by severity · oldest open row · evidence-feed heartbeat (minutes since last
successful sweep — guard #16 "staleness is a number") · overrides last 30d · files at
`underwriting`+ unlinked · rules in shadow mode with would-have-fired counts.

## 6. User-friendliness: help, not nag

1. **Quiet by default.** New rules enter SHADOW (evaluate + count, zero UI), get promoted with
   data. Only ~5 rules ever hard-block: A1 (mapped conditions), B1/B2 (CTC), E1 (expired lock,
   if locks are used), I3/I0-at-CTC (linkage). Everything else is a badge or dashboard row.
2. **One surface per screen.** All NOTICE rows collapse into a single chip; CONFIRMs only
   interrupt at the moment of the action they're about; no toast storms, no per-row emails.
3. **Every message is a signOffGate-style sentence:** what disagrees, both values, data age,
   the concrete fix, and the override path — matching the house plain-language style
   (staff.js:2436, 2451-2462 are the templates to copy).
4. **The layer gives before it takes:** A5 ("Encompass cleared this — mark it here?"), H2
   ("appraisal already in Encompass"), F2 ("Encompass shows this file further along") make the
   integration visibly save clicks in week one, building trust before enforcement mode.
5. **Positive confirmation is visible:** the green "In agreement" chip means staff *see* the
   verification working when everything's fine — the layer isn't only present when scolding.
6. **Severity budget as policy:** if shadow data shows a rule would fire on >20% of active
   files, it does not get promoted past NOTICE until the tolerance/mapping is fixed. Nagging is
   treated as a rule defect, not a user problem (the review-queue lesson from guard #10: floods
   train staff that cards are noise).
7. **Snooze, with teeth:** NOTICE rows can be snoozed 7 days (value-aware); BLOCK cannot be
   snoozed, only overridden with a reason.

## 8. Rollout phasing (safety + trust)

1. **Shadow (2–4 weeks):** cache + dashboard live; all rules SILENT; collect would-have-fired
   counts; tune tolerances and the condition map. No staff-facing change except the per-file
   chip (arguably ship the chip here — it's pure value).
2. **Notice:** promote A5/H2/F2 (the helpful ones) and the NOTICE set; LO digests on.
3. **Enforce:** promote A1 → CONFIRM → BLOCK; then B1/B2; E1 only if locks are real; I3/I0 at
   link/CTC. Each promotion is a registry flip, reversible per rule (kill switch), no deploy.
