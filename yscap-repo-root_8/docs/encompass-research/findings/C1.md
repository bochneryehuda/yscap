## 1. Identity platform at a glance

Encompass Developer Connect (EDC) uses OAuth 2.0 issued by ICE's identity platform. Key endpoints (all confirmed by both the official Authorization doc and the Postman collection):

| Purpose | Endpoint |
|---|---|
| Token issuance (all grants) | `POST https://api.elliemae.com/oauth2/v1/token` |
| Token introspection | `POST https://api.elliemae.com/oauth2/v1/token/introspection` |
| Token revocation | `POST https://api.elliemae.com/oauth2/v1/token/revocation` |
| Interactive authorization (auth-code flow) | `https://idp.elliemae.com/authorize` |

- Scope for all Lending Platform calls is **`lp`** ("The token scope is 'lp' for Lending Platform"). Confirmed in every Postman auth request (`scope = lp`).
- Tokens are opaque bearer tokens sent as `Authorization: Bearer <token>`.
- Per ICE's platform overview: "entitlements and rate limits are linked to the Encompass client ID," while "fine-grained permissions are derived from the username" — i.e., *what a token can do is the Encompass user's rights*, and *how fast you can call is per instance/client*.
- ICE warns: "the OAuth client ID should not be confused with an Encompass client ID or instance ID." Three distinct identifiers exist: **OAuth client_id** (the API key), **Encompass Client ID / CID** (10 digits, usually starts with `30`), and **Instance ID** (starts with `BE` — YS Cap's is `BE11397907`).

Sources:
- https://developer.icemortgagetechnology.com/developer-connect/docs/authentication (the "Authorization" page)
- https://developer.icemortgagetechnology.com/developer-connect/docs/authentication-and-authorization-in-the-encompass-lending-platform
- https://knowledge.himaxwell.com/creating-an-encompass-api-user ("The Instance ID starts with BE and the Client ID usually starts with 30")

## 2. OAuth2 flows — which one applies to YS Capital

### 2.1 Resource Owner Password Credentials (ROPC) — **the lender path; this is ours**

Official doc: "Grant type client_credentials is for ISV partners only. **Lenders must use grant type = password**."
(https://developer.icemortgagetechnology.com/developer-connect/docs/authentication)

Exact request shape, from the Postman collection (`Authentication / Get Access Token / Resource Owner Password Credentials`, `POST {{API_SERVER}}/oauth2/v1/token`, body mode `urlencoded`):

| Form field | Value |
|---|---|
| `grant_type` | `password` |
| `username` | `{{smart_client_user}}@encompass:{{instance}}` |
| `password` | `{{smart_client_password}}` |
| `client_id` | `{{client_id}}` |
| `client_secret` | `{{client_secret}}` |
| `scope` | `lp` |

**Fully-qualified username format** (official wording): "The fully qualified username. The format is: Encompass `user_name@encompass:{{encompass_instance_id}}`", e.g. `user1@encompass:BE1111234`. For YS Capital's service user this will be `serviceuser@encompass:BE11397907`.

The documented sample token response contains `access_token` and `token_type`; **no `expires_in` is documented** — lifetime is governed by the activity-window rules in §3, so the client must track its own timestamps (verify in sandbox whether `expires_in` is actually returned; see Open Questions).

### 2.2 Client Credentials — **ISV-partner-only; architecturally not our path**

- Official wording again: "Grant type client_credentials is for ISV partners only."
- Mechanics visible in Postman (`Authentication / Get Access Token / Client Credentials (API User)`): HTTP **Basic auth** with a *partner's* `api_user_client_id`/`api_user_client_secret`, form body `grant_type=client_credentials`, `scope=lp`, and `instance_id={{instance}}` — the partner names the lender instance it is acting against.
- The lender-side counterpart is the **"API User" checkbox** in Organization/Users: the admin creates a user, checks *API User*, and enters the **partner's API Client ID** — pairing that ISV's key to a user in the lender's instance. Checking the box "hides the password fields" entirely (no password ⇒ no ROPC; that user exists solely for a partner's client_credentials tokens). Sources: https://help.lendertoolkit.com/admin/enable-api-access ("The 'API Client ID' is different from your 'Encompass Client ID'"), https://knowledge.himaxwell.com/creating-an-encompass-api-user, https://support.capacity.com/article/545315/encompass.
- **Why it's not for YS Cap:** YS Cap is the lender integrating its own portal, not an ISV registered in ICE's partner program. Lenders don't get client_credentials keys for their own instance; the lender-native equivalent of a "service account" is ROPC with a dedicated Encompass user (§7). (If YS Cap ever productized the portal for other lenders, *that* would be Partner Connect / client_credentials territory.)

### 2.3 Authorization Code — interactive/browser flow (and the only SSO-compatible flow)

- "Allows an application to obtain an access token through a two-step process: obtain an authorization code by authenticating the user (using the authorization endpoint) and then exchange the authorization code for an access token."
- `/authorize` request params (official reference): `client_id` (required), `redirect_uri` (required, "must be an HTTPS URL"), `response_type=code` (required), `scope=lp` (required), `is_sso` ("Required. Indicates whether SSO is to be used. When set to true, the instance_id must be specified"), `instance_id` ("Required if is_sso is true. The instance_id directs the browser to the Encompass Developer Connect SSO login page").
- Relevant to YS Cap only if the portal ever wants *staff* to log in to Encompass through the portal UI with their own identities. **Not needed for the Phase-1 headless read-only sync.**

Source: https://developer.icemortgagetechnology.com/developer-connect/docs/authentication

### 2.4 Flow comparison

| Flow | Who | Credential used | SSO-compatible | Fit for Phase 1 |
|---|---|---|---|---|
| ROPC (`password`) | **Lenders** | API key + Encompass user+password | No | **Yes — chosen** |
| `client_credentials` | ISV partners only | Partner API key (Basic auth) + `instance_id` | n/a | No (not eligible) |
| `authorization_code` | Lenders (interactive) | API key + browser login | Yes (`is_sso=true`) | No (headless service) |
| Token exchange (impersonation) | Lender API users / super admins | An existing actor token | n/a | No (write-oriented; see §5) |

## 3. Token lifetime rules

Official wording (Authorization page): "**The access token is active for 30 minutes, up to a maximum of 24 hours. To avoid expiration, the access token must be used (by calling an API) at least once within 15 minutes.**"

The Access Token Best Practices page (https://developer.icemortgagetechnology.com/developer-connect/docs/access-tokens-bp) refines this:

- "Access tokens are valid for up to 24 hours."
- Tokens should be *used* at least every **15 minutes** — "50% of the 30-minute inactivity timeout."
- "If a token is not refreshed within 30 minutes of its last use, it will expire and require authentication."
- Using the token within the 15-minute half-window extends it another 30 minutes; a call made *after* 15 minutes but before 30 may succeed but **does not extend** the window — the token still dies at the 30-minute mark.
- Continuously-used tokens hard-expire at **24 hours** regardless of activity.
- ICE's own caching advice: cache "the bearer token (encrypted) and the current Date Timestamp"; before each call check age/idleness and re-authenticate when stale — or, alternatively, "If it has a Status of 401 'Unauthorized', get a new token."

**Discrepancy to flag:** the newer "Authentication and Authorization in the Encompass Lending Platform" overview describes "a sliding-window session with an idle timeout of 15 minutes and a total duration of up to 2 hours." That page describes the broader/newer Lending Platform identity stack; the EDC-specific pages say 30-min idle / 24-h max. Design conservatively: assume the **tighter** rule may apply (idle ≤ 15 min, max ≤ 2 h) and always self-heal on 401 — then either regime works. (Open question §10.)

There is **no OAuth refresh token** in this model: "refreshing" simply means calling `/oauth2/v1/token` again with the same grant. Password-grant re-auth is cheap and idempotent.

## 4. Introspection and revocation

Both take HTTP **Basic auth** with `client_id:client_secret` and a form field `token` (confirmed in Postman requests `01b`/`02b - Introspect ... token`, which carry `"type": "basic"` auth using `{{client_id}}`/`{{client_secret}}`).

- **Introspection** — `POST /oauth2/v1/token/introspection`. Response fields (official): `active`, `scope`, `client_id`, `username`, `token_type`, `exp`, `sub`, `encompass_instance_id`, `user_name`, `user_key`, `identity_type`. Useful at startup to *prove* which user/instance a token is bound to (e.g., assert `encompass_instance_id == BE11397907` and the expected service username before syncing a single row) and to distinguish user vs partner identities (`identity_type`).
- **Revocation** — `POST /oauth2/v1/token/revocation`, same auth + `token` field. "If the token is successfully revoked, the API returns a 204 status; otherwise, an error is returned." Use on graceful shutdown and in the credential-leak runbook (§9).

Source: https://developer.icemortgagetechnology.com/developer-connect/docs/authentication

## 5. User impersonation via token exchange (know it exists; do not use it)

The Postman `Authentication / User Impersonation` folder shows the full sequence:

1. **01a** — get an *actor token* via normal ROPC (`grant_type=password`, `username={{smart_client_user}}@encompass:{{instance}}`).
2. **01b** — introspect the actor token (Basic `client_id:client_secret`).
3. **02a** — exchange it: `POST /oauth2/v1/token` with `grant_type=urn:ietf:params:oauth:grant-type:token-exchange`, `actor_token_type=urn:ietf:params:oauth:token-type:access_token`, `actor_token={{TOKEN1}}`, `subject_user_id={{subjectuserId}}`, `scope=lp`, plus `client_id`/`client_secret`. The result is a token that acts **as the subject user**.
4. **03** — the sample then *creates a loan* as the subject — a reminder that impersonation tokens are fully write-capable.

Official constraints: "Super Administrators can impersonate all users"; lender API users (password grant) can impersonate users "at the same level or below them in the organization hierarchy." (https://developer.icemortgagetechnology.com/developer-connect/docs/authentication)

**Phase-1 position:** no impersonation. A read-only service account has no reason to act as anyone else, and impersonation widens both audit ambiguity and blast radius. If a future phase needs per-staff attribution, prefer the authorization-code flow (real user logins) over impersonation.

## 6. API key management and secret regeneration (rotation)

From the official API Keys page (https://developer.icemortgagetechnology.com/developer-connect/docs/get-an-api-key):

- **Who:** "Only an Encompass user with the **super administrator** persona can access and distribute the API key to other users."
- **Where:** the Developer Connect portal — "In the top navigation menu, click **My Account**, then select **API Key**." Sign-in requires Instance Name (or SmartClient ID), User ID, Password of the super admin.
- **What you get:** "the API Key screen opens and displays the **Client ID (OAuth)** and **Client Secret**." These are the `client_id`/`client_secret` used in every token, introspection, and revocation call.
- **Scoping:** "An API key is linked to an Encompass Client ID, which means that it cannot be used across all Encompass instances." One instance (BE11397907) ⇒ one API key. There is no documented facility for two concurrently-valid secrets on the same key.
- **Rotation:** the portal has a **Regenerate Secret** action with an explicit warning: "Clicking **Regenerate Secret** breaks any current Developer Connect applications using this API key." Regeneration is therefore (a) the *only* remediation for a leaked secret and (b) a coordinated-cutover event, not a zero-downtime dual-key rotation (see design §8 and Open Questions).
- **Handling:** "When you share the API key with your developers, please do so securely and in compliance with your organization's data security policy."

## 7. What the YS Cap super admin must do to provision API access for BE11397907

### 7.1 Checklist

1. **Obtain the API key** (super admin only): Developer Connect portal → My Account → API Key → sign in with Instance/User ID/Password → record Client ID (OAuth) + Client Secret **directly into the secret store** (Render env vars, matching the repo's existing pattern — `render.yaml` + `src/config.js:142-146` pull `MS_*` secrets from env). Never transmit them over chat/email; if that ever happens, regenerate (§9).
2. **Create a dedicated service user** in Encompass: Settings → Company/User Setup → **Organization/Users** → select an org folder → New. Give it a self-describing User ID (e.g., `ysportal.svc`), a real mailbox the team monitors, and a strong password. **Do NOT check the "API User" box** — that pairing is for ISV client_credentials keys and *removes the password* ("checking the API User checkbox hides the password fields"), which would break the lender password grant. (Sources: Maxwell, LenderToolkit, Capacity guides above.)
3. **Create a least-privilege persona** (e.g., `Portal ReadOnly API`) rather than reusing Super Administrator (vendor guides often lazily demand Super Admin — reject that; official docs confirm persona limits bind the API, §7.2). Assign only this persona to the service user. Note the persona must grant *platform access*: "at least one of the personas must be set up with permission to access LO Connect in order for this user to access the Lender's Encompass instance via the APIs" — on the Personas → **Access** tab, enable the web/LO Connect access option (Capacity guide, corroborated by LenderToolkit).
4. **Constrain loan access:** in the persona/org setup give access to the loan folders the portal needs; then add a **"Persona Access to Loans"** business rule for the persona with access level **View Only** ("View Only – All areas of the loan file can be viewed, but not edited"; options are View Only / Edit / Custom). This makes the account read-only *server-side*, independent of anything the portal does. (ICE documentation library: https://help.icemortgagetechnology.com/DocumentationLibrary/360/PersonaAccesstoEncompassSettings-Banker.pdf and Persona Access to Loans documentation surfaced via ICE help.)
5. **Review "Persona Access to Fields"** rules for the persona: since EDC 25.3 the V3 Get Loan and Field Reader APIs *honor* field-deny rules — restricted fields come back hidden or as "User doesn't have access." Ensure no rule hides fields the portal legitimately needs (e.g., status/CTC/condition fields), and conversely consider deny rules for fields the portal must never see. (Changelog: https://developer.icemortgagetechnology.com/developer-connect/changelog/253-major-release.)
6. **Licensing:** the service user consumes a seat — "New users indicated as an 'API User'…count against your enabled user license limits" (LenderToolkit); a normal enabled user does too. Budget one license.
7. **Password policy:** exempt or calendar the service user's password expiry — an unnoticed forced password change is the #1 way headless ROPC integrations die. Rotate the password deliberately on a schedule instead (§8).
8. **Verify before go-live:** with the new token, call introspection and assert `encompass_instance_id`, `user_name`, `active:true`; then `GET` one known loan and confirm all required fields are visible (not "User doesn't have access"); then confirm a write attempt (e.g., PATCH in a sandbox) is **rejected** by the View Only rule.

### 7.2 Why personas bind the API (the load-bearing fact)

- Official position (surfaced from ICE docs via search): "The features and data users can access with the APIs is determined by their assigned Encompass persona, and **the API key does not grant access to any additional features or data outside of the user's Encompass persona settings**."
- Enforcement is real and actively patched: EDC 25.3 fixed field-level leakage ("The following API was allowing users to retrieve field ID values when 'Persona Access to Field' business rules were set to deny access…" — now the API "hides the values … or provides … 'User doesn't have access'"), and EDC 26.1 fixed EDC-1252: "the V3 Get Loan API was not honoring the 'Persona Access to Loan' business rule and was allowing access to loan 'Edit Custom Access' objects the rule was restricting." (https://developer.icemortgagetechnology.com/developer-connect/changelog/253-major-release, https://developer.icemortgagetechnology.com/developer-connect/changelog/261-major-release)
- Caveat: the `admin` user "has complete access … ignoring all configured Business Rules" — one more reason the service account must be a normal user, never `admin`/Super Administrator.
- Corollary for a read-only integration: server-side persona restriction (View Only) is the *primary* control; the portal's own GET-only HTTP guard (§8) is defense-in-depth, exactly like the repo's existing ClickUp delete guard (`src/clickup/client.js:15-40`).

### 7.3 SSO caveats

- "SSO authentication with Encompass Developer Connect API integrations is supported for lenders, not ISV partners", and it is "meant to be used with **authorization code** UI login flow (grant_type = authorization_code)" with `is_sso=true` + `instance_id` on `/authorize`. (https://developer.icemortgagetechnology.com/developer-connect/docs/authentication)
- **Implication:** the ROPC password grant authenticates against native Encompass credentials. If YS Cap federates staff logins to an external IdP, the *service user must remain a native-credential Encompass user* (excluded from SSO enforcement), or the headless grant will fail. Document this exemption so a future SSO rollout doesn't silently break the sync.
- Federated SSO (SAML 2.0 / OIDC) is on ICE's roadmap per the platform overview page; re-check this constraint at that time.

## 8. Recommended design: service account + token caching + secret rotation

The repo already contains the exact pattern to copy: the Microsoft Graph client in `src/lib/sharepoint.js` — in-memory token cache with expiry (`_tok = { value, exp }`, line 38; reuse while `Date.now() < _tok.exp - 60000`, line 78), **single-flight** so bursts share one fetch (`_tokenInflight`, lines 39, 76-81), one re-auth retry on 401 (comment at line 116, fetch at 124-125), and Retry-After-honoring backoff. Secrets come only from env via `src/config.js` (lines 142-146), mirroring `src/clickup/client.js:1-13` ("Token comes from the environment … NEVER hardcoded, never sent from the browser").

**Proposed `src/encompass/auth.js` (new module, same shape):**

1. **Config (env only):** `ENCOMPASS_CLIENT_ID`, `ENCOMPASS_CLIENT_SECRET`, `ENCOMPASS_API_USER` (bare username), `ENCOMPASS_INSTANCE_ID=BE11397907`, `ENCOMPASS_API_PASSWORD`, `ENCOMPASS_BASE_URL=https://api.elliemae.com`. Compose the login as `${user}@encompass:${instanceId}` in code so the instance binding is explicit and greppable.
2. **Token cache:** in-memory only (never DB, never disk, never logs). Track `{ value, obtainedAt, lastUsedAt }`. Reuse the token when `now - lastUsedAt < 12–14 min` **and** `now - obtainedAt < ~1.5 h` (safe under both the 30-min/24-h EDC rule and the stricter 15-min/2-h platform rule); otherwise fetch fresh. Update `lastUsedAt` only on a *successful* API response. Since there's no refresh token, "refresh" = new ROPC call — cheap, so bias toward re-authenticating early rather than racing the idle window.
3. **Single-flight + 401 self-heal:** concurrent callers await one token fetch; on any 401 from an API call, drop the cache, re-authenticate once, retry once, then surface the error. This makes the client correct even if ICE changes window semantics.
4. **Read-only choke point:** every Encompass request funnels through one `request(method, path)` that **throws unless method is GET** (allow POST only for the two documented read-helper endpoints if adopted later, e.g. `fieldReader`, pipeline query — each individually allowlisted). Model it on the ClickUp deletion guard (`src/clickup/client.js:31-40`) — same "single choke point, no code path can bypass" philosophy. This plus the View Only persona gives two independent write-prevention layers.
5. **Startup assertion:** on boot (and after every re-auth at most once/hour), call introspection and assert `active`, expected `user_name`, and `encompass_instance_id === 'BE11397907'`; log the assertion result (never the token). Refuse to sync on mismatch.
6. **Shutdown hygiene:** best-effort `POST /oauth2/v1/token/revocation` for the cached token on graceful shutdown.
7. **Secret rotation runbook (quarterly, and immediately on suspicion of exposure):**
   - *Client secret:* super admin opens Developer Connect portal → API Key → Regenerate Secret → paste new secret into Render env → redeploy. Expect a brief outage ("Regenerate Secret breaks any current Developer Connect applications using this API key") — schedule off-hours; the sync queue's retry behavior (existing `sync_queue` design, `CLAUDE.md:45`) absorbs the gap.
   - *Service-user password:* rotate on the same cadence via Encompass admin, update env, redeploy. Alternate the two rotations so a bad rotation of one credential is diagnosable.
   - After any rotation, run the §7.1-step-8 verification.
8. **Observability without leakage:** log token *events* (obtained/renewed/revoked/401-heal) with timestamps and the introspected `user_name` — never token values, never passwords, and never the Authorization header (add a redaction guard to any request logger, in the spirit of `src/lib/redact.js` for SSNs).

Token demand estimate: a polling sync every ≤ 10 minutes keeps one token alive all day within the idle window; a cron-style sync every ≥ 15 minutes should simply fetch a fresh token per run. Either is fine; the cache design above handles both.

## 9. Explicit warning list

1. **A pasted secret is a burned secret.** Any client secret or service-account password that transits chat, email, a ticket, a doc, or a commit must be regenerated/changed immediately — regeneration is the only remediation, and it breaks running apps, so treat paste-incidents as scheduled-outage events.
2. **Never log tokens.** No access token, password, or `Authorization` header in application logs, error messages, crash dumps, audit rows, or `sync_queue` payloads. Introspection metadata (username, instance, exp) is the loggable surrogate.
3. **Never persist tokens** to Postgres, disk, or browser storage; memory only. Tokens are bearer credentials for the whole persona.
4. **Never ship credentials to the client side.** All Encompass calls originate server-side (same rule as `src/clickup/client.js:1-4`).
5. **Do not commit** the Postman environment file (or any filled-in variant) into the repo; the uploaded one is placeholder-only today — keep it that way.
6. **Don't use `admin`/Super Administrator as the service account** — `admin` ignores business rules entirely, which would silently void the read-only guarantee.
7. **Don't check "API User"** on the service user — it deletes the password and is meaningless without an ISV partner key.
8. **Watch for password expiry** on the service user; a forced change = integration outage with a misleading 401 signature.
9. **Field-deny rules can silently blank data** ("User doesn't have access", EDC 25.3+): validate field visibility after every persona/business-rule change, or portal rules (e.g., condition-clearing checks) may evaluate against missing values. Fail closed: treat an inaccessible field as "Encompass does not agree."
10. **A 200 response after 15 idle minutes may not extend the token** — never infer "still valid for 30 more minutes" from call success; trust only your own timestamps + 401 self-heal.
11. **Impersonation tokens are write-capable** (the Postman sample literally creates a loan) — keep the token-exchange grant out of the codebase entirely in Phase 1.
12. **SSO rollout risk:** enabling SSO for staff must exempt the native-credential service user or ROPC breaks.
13. **Rate limits are per Encompass client ID** — a runaway poller can throttle the whole instance's integrations, not just the portal; honor 429/Retry-After exactly as `src/lib/sharepoint.js` already does for Graph.

## 10. Open questions

1. **Does the token response include `expires_in`?** The official sample omits it; verify empirically in the sandbox and prefer own-clock tracking regardless.
2. **Which lifetime regime applies today** — EDC-documented 30-min idle/24-h max vs. the Lending Platform overview's 15-min idle/2-h max? The recommended cache is safe under both, but confirm with ICE support for capacity planning.
3. **Does Regenerate Secret also invalidate outstanding access tokens**, or only future token grants? Affects the rotation runbook's outage window. Not documented; ask ICE.
4. **Can a lender hold two API keys for one instance** (for true zero-downtime rotation)? Docs tie one key to one Encompass Client ID and describe only Regenerate; ask ICE whether a second key can be issued.
5. **Sandbox availability:** confirm YS Cap's contract includes a test instance (separate `BE…` id) so provisioning and the read-only persona can be validated before touching production BE11397907.
6. **Exact persona template:** which stock persona (if any) is closest to "read-everything-write-nothing," vs. building `Portal ReadOnly API` from scratch; and whether "Persona Access to Loans = View Only" blocks *all* v3 write verbs at the API layer (validate per §7.1 step 8).
7. **Concurrent token policy:** whether issuing a new ROPC token invalidates prior tokens for the same user (relevant if the portal ever runs multiple workers with separate caches; a shared single-flight cache sidesteps it).
