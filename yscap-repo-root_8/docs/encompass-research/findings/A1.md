Research agent: A1 ("ClickUp write/delete guards"). Repo: `/home/user/yscap/yscap-repo-root_8` (all paths below relative to that root). All claims cite file:line from the current working tree (read-only inspection, 2026-07-17).

## 1. Architecture in one paragraph

Every portal→ClickUp API call funnels through **one HTTP chokepoint** — `call()` in `src/clickup/client.js:173-214` — and every guard that matters is enforced either *inside* that chokepoint or in the single push orchestrator that feeds it (`src/clickup/orchestrator.js`). The design philosophy, stated repeatedly in code comments and `docs/CLICKUP-DATA-SAFETY.md`, is **structural incapability**: destructive operations are not merely "not called," they *throw* at the chokepoint so that no present or future code path, refactor slip, or copy-paste can perform them. Guards are layered (chokepoint → orchestrator → mapper → queue → config flags → DB journal/review queue), each layer independently testable with no DB or network (`scripts/test-clickup-write-guards.js`, `scripts/test-clickup-delete-guard.js`).

## 2. The three HTTP-chokepoint hard stops (`src/clickup/client.js`)

| Guard | Location | What it blocks | Error code |
|---|---|---|---|
| `guardNoTaskDeletion` | client.js:29-39, enforced in `call()` at :174 | **Any HTTP DELETE addressed to a task path** — `DELETE /task/{id}` (task delete) AND `DELETE /list/{id}/task/{id}` (remove-from-list). Regex `/(^|\/)task\/[^/?]+/` catches subtasks and query-string variants. Webhook plumbing (`DELETE /webhook/{id}`) allowed. | `CLICKUP_DELETE_FORBIDDEN` |
| `guardNoFieldClearing` | client.js:51-78, enforced in `setField()` at :227-230 | **Any write that would blank/clear a field**: null, undefined, empty string, whitespace-only string, empty array, empty `{add:[]}` users list — plus the subtle class where `JSON.stringify` silently converts values to null: nested `null`, `NaN`, `Infinity`, `undefined` anywhere in the value tree (recursive `findJsonUnsafe`, :51-64). Rationale: a null latitude or NaN amount reaches ClickUp as a field-clearing/garbage write. | `CLICKUP_EMPTY_WRITE_FORBIDDEN` |
| `guardTaskUpdatePayload` | client.js:84-99, enforced in `updateTask()` at :220-223 | **Task updates carrying anything except `status`** — an ALLOWLIST (`TASK_UPDATE_ALLOWED_KEYS = {'status'}`), not a blocklist, so any future payload key (name, description, markdown_description, priority…) is refused by default. Empty status also refused. The sync never renames a task; names are set only at creation. | `CLICKUP_RENAME_FORBIDDEN` / `CLICKUP_EMPTY_WRITE_FORBIDDEN` |

Notable: `removeTaskFromList` is exported (client.js:283) but **intentionally self-blocking** — it is a DELETE on a task path, so invoking it throws through `guardNoTaskDeletion`; the delete-guard test asserts exactly that (`scripts/test-clickup-delete-guard.js:38-42`).

Also in the client: rate-limit manners (token bucket pacing under ClickUp's 100 req/min — `CLICKUP_MAX_RPM` default 70, client.js:115, 150-161), short in-call retry with Retry-After honoring (:177-213), per-request timeout, and **PII-free error messages by construction** (`httpError`, :139-145 — "ClickUp POST /task/x/field/y -> 429", never the value).

## 3. Orchestrator-level guards (`src/clickup/orchestrator.js`)

### 3.1 Gates before any write
- **Master switch**: `pushApplication` no-ops unless `cfg.clickupSyncEnabled` (orchestrator.js:214).
- **Deleted-file guard**: a portal-archived/deleted file is *never pushed* — `if (ctx._row.deleted_at) return { skipped: 'portal-deleted (ClickUp left untouched)' }` (:217-220). Combined with the client delete-block, portal deletion structurally cannot propagate.
- **Scoped pushes never CREATE tasks** (:229-233): a queue job carrying `opts.only` (a field-scoped edit) on an unlinked file is skipped and audited (`push_skipped_unlinked`), so a lost binding cannot spawn a near-duplicate task. Creation is reserved for `createForNewFile` (:567-579) and the admin full repush.

### 3.2 Read-before-write + no-op suppression
For updates, the task is read ONCE before writing (:282-300); each field whose current ClickUp value is **equivalent** to the outgoing value is skipped (`fieldValueEquivalent`, mapper.js:482-529 — type-aware comparison through the same transforms the sync uses: dates via `fromEpochMs`, dropdowns orderindex→UUID, coordinates with ~10 m tolerance, phones by last-10-digits, users add-lists). Consequence: the sync never rewrites a field it isn't changing, so a conversion bug can't silently sweep values, and repushes produce no journal churn.

- **Fail closed**: if the pre-write read fails on a *queue-driven scoped push*, the push throws (`CLICKup_PREREAD_FAILED` class) and the queue retries — it never writes blind (:288-298). Only the human-watched admin full repush proceeds warn-only (:299).

### 3.3 DOB gates (the 2026-07-15 incident class)
- **Any change to an existing DOB is a human decision.** `isDobChange` (mapper.js:539-543) detects any-magnitude change; `isSuspectDobShift` (mapper.js:545-551) detects the exact ±1-day corruption signature. The push gate (orchestrator.js:340-368) consults one shared decision engine (`src/lib/sync-autoresolve.js decideDob()`): provable verdicts settle silently (possibly healing the *portal* instead of writing to ClickUp, :346-352); genuine ambiguity is **blocked, journaled (`blocked:true`), and queued to `sync_review_queue`** (:354-364). Blank DOBs may be filled.
- **Blind-write guard** (:329-339): a full repush whose pre-read failed cannot write a DOB at all (no before-image = the gate can't evaluate) — blocked + review row.
- **Human-edit bypass**: `opts.humanEditKeys` (carried through the queue payload, set only by authenticated portal form routes) marks a staff-typed DOB as the human decision; it writes through — still journaled and breaker-limited (enqueue.js:19-26, orchestrator.js:321, clickup-sync.js:73-76).

### 3.4 PII overwrite shield (:370-389, set defined :457-469)
A FULL repush may **fill a blank** identity field but never **rewrite a differing** one: `PII_OVERWRITE_SHIELD = {borrowerName, borrowerSSN, borrowerEmail, borrowerCell, borrowerAddress}`. Blocked writes are journaled + queued (`pii_overwrite_blocked`) with SSN-masked previews (`reviewPreview`, :472-477). Deliberate edits still flow because they arrive as *scoped* pushes of exactly that field; an approved review re-pushes scoped via `resolveOnly` keys (`ssn`, `current_address`, `email`, `cell_phone`, `first_name` — mapper.js:454-463). DOB is deliberately excluded (it has its own gate).

### 3.5 Volume circuit breaker (:479-507)
The **ultimate anti-mass-write guard**: more than `CLICKUP_MAX_FIELD_WRITES_10MIN` (default 300, floor 50) field writes in a rolling 10-minute window opens the circuit — every further push **throws** `CLICKUP_CIRCUIT_OPEN` (queue retries later, admin repush surfaces the error) and the opening is audited once/minute (`outbound_circuit_open`). Task creates count into the same breaker (:260). Known limitation (SYNC-GUARD-MATRIX #4/#54): per-process, deploy-resets.

### 3.6 Overwrite-storm alarm (:404-411)
One push rewriting >10 *existing* values → loud console + audit row (`push_overwrite_storm`). Detect-not-block (a legitimate admin repush after big edits exists), so the incident signature can never happen silently.

### 3.7 Failed writes are never swallowed (WO-1; :163-210, :423-428)
`recordFieldFailure` + `assertPushComplete`: if any intended field write failed, the push **throws** (PII-free message: field ids + HTTP status only) so the queue retries instead of marking a lossy push 'done'; already-landed fields are no-op-suppressed on retry. All-transient failures mark the whole push retryable (outage class).

## 4. The write journal (`db/107_clickup_write_journal.sql` + orchestrator.js:509-532)

**`clickup_write_log`** — append-only journal of EVERY outbound field write:

| Column | Meaning |
|---|---|
| `application_id`, `task_id`, `field_id`, `field_key` | what was touched (`field_key` for status writes) |
| `old_value` jsonb | ClickUp's value **immediately before** the write (from the pre-read; NULL = unknown/new task) |
| `new_value` jsonb | the value written (or attempted) |
| `changed` / `blocked` | `blocked=true`, `changed=false` = a guardrail refused the write (also journaled!) |
| `source` | `'create' \| 'scoped_push' \| 'full_repush'` |

Properties: **masking at the app layer before insert** — SSN via `maskSSN`, card via parse-then-mask-number-only so exp/CVV never land (orchestrator.js:514-523); cleartext PII never reaches the table. Indexed by task, app, and blocked-only partial index (107:29-31). Best-effort (a journal failure never blocks the push, :531). Purpose stated in the migration header: the incident's only trail was a field *count*; this makes damage assessment a query, and (per `docs/CLICKUP-DATA-SAFETY.md:20-27`) provides attribution — "if an event is not in the journal, the portal did not do it."

## 5. Enqueue-layer guards (`src/clickup/enqueue.js`)

- **There is NO delete op** — the module only ever enqueues `op='update'` push jobs (enqueue.js:6-9, INSERT at :63-66); a portal deletion never propagates (enforced belt-and-suspenders with the client hard stop + orchestrator deleted_at guard).
- **Scoped-only**: jobs carry the exact changed logical keys (`payload.only`); an **empty key set enqueues nothing** (:29-30). The drainer (`src/sync/clickup-sync.js:66-76`) skips any job with no field set — a legacy/empty job can **never fall back to a full-payload overwrite**.
- **Union-merge into queued jobs only** (:34-62): rapid edits coalesce into one scoped push, but never merged into a `'processing'` job (its field snapshot was already taken — merging would silently lose the push).
- `enqueueChecklistStatusPush` (:75-87) is self-gating: no-ops unless the checklist item is mapped AND the file is linked and not deleted; routes through `checklist:<fieldId>` scoped keys.
- Master switch honored at enqueue time (`cfg.clickupSyncEnabled`, :28, :76).

Queue drain guards (`src/sync/clickup-sync.js`): crash-orphan reclaim after 5 min (:49-62); outage-class patient retries (breaker-open / pre-read-failed / retryable → 10-min spacing, dead after ~7 h, vs 8 attempts for genuinely bad jobs, :89-106); **dead-letter = a visible review row** with a "Retry push" action that auto-closes on a later success (:107-123, :80-88).

## 6. Mapper / field-level protections (`src/clickup/mapper.js`)

- **Direction discipline** (`FIELD_MAP` `dir`, :55-133): `pull`-only fields (ClickUp-owned: lender, channel, appraisals, title/insurance, actual closing, `encompass_status`…) are **never pushed** (:207); `push`-only fields (portal-owned pricing: LTV, rate, desired rate, date submitted) are **never pulled** (:318). The portal cannot echo-clobber ClickUp-owned data and vice versa.
- **Empty values never emitted**: the `put()` helper drops null/undefined/'' (:203); `writeValue` returns undefined for empty inputs (:152-153). Combined with the client guard this is two independent layers against clears.
- **Synthetic-placeholder scrub** (:194-201): shadow-profile values (`*@clickup.local`, first/last "Unknown") are blanked before build so portal-internal junk can never overwrite real ClickUp contact data.
- **Location hygiene** (`addressField`, :172-182): a location field is emitted only with **finite, non-null** lat/lng (explicit `Number.isFinite` — `Number(null)` is 0 and NaN JSON-serializes to null, either would be a location-clearing/garbage write).
- **Date writes** only via `dateOnlyToClickUpEpoch` (4 AM America/New_York convention; refuses out-of-range years) (:161-165).
- **Dropdown writes resolve label → live option UUID** via the registry (`src/clickup/registry.js` — TTL-cached `getListFields` pull); an unresolvable label yields undefined → field skipped, never a garbage write (:155-159). (`src/clickup/mapping.js` holds the hardcoded fallback option-UUID maps with the same skip-on-null discipline, mapping.js:36-49.)
- **Checklist dropdowns are structurally isolated** (:263-271): checklist condition statuses go in a SEPARATE `checklistFields` array, never merged into `customFields` — a task create or full repush *cannot* touch a ClickUp checklist dropdown; they reach ClickUp only via explicitly scoped `checklist:<fieldId>` keys (orchestrator.js:247-254).
- **Scoped-push resolver** (`resolveOnly`, :429-472): changed logical keys map to exactly their ClickUp field ids; **unknown keys resolve to nothing** (safe default — the field simply isn't pushed).
- **Task name set only at creation** (:255-261); updates are status-only by the client allowlist.
- Placeholder YS loan numbers ("TBD"/"0"/"N/A") dropped on read so they can't become match keys (:323-330).

## 7. Kill switches, control flags, staged rollout (`src/config.js:208-232`)

| Flag | Default | Effect |
|---|---|---|
| `CLICKUP_SYNC_ENABLED` | **off** | Master switch. Orchestrator, enqueue, worker loops all no-op when off (orchestrator.js:214, enqueue.js:28, clickup-sync.js:1191). |
| `CLICKUP_OUTBOUND_ENABLED` | **off** | Separate gate for portal→ClickUp *writes*; inbound/reconcile can run and be validated with outbound fully dark (clickup-sync.js:1283-1297). Also consulted by autoresolve appliers (`sync-autoresolve.js:119,254,281,302`) and unlinked-file recovery (clickup-sync.js:140). |
| `CLICKUP_OUTBOUND_SINCE` | empty | Go-live cutoff: never bulk-push the pre-existing backlog (would create duplicate tasks) (config.js:220-224). |
| `CLICKUP_INBOUND_CREATE_FILES` | **off** | Inbound may update linked files but never *materialize* new portal files until identity matching is trusted (config.js:225-231, clickup-sync.js:681-683). |
| `CLICKUP_DRYRUN` | off | Boot **dry-run mode**: read-only validation only, "do not start the live loops" (clickup-sync.js:1177-1189). |
| `CLICKUP_RUN_BACKFILL` (`'data'`/`'full'`) | empty | One-shot inbound backfill — "writes to the portal, never to ClickUp" (clickup-sync.js:1206-1214). |
| `CLICKUP_MAX_FIELD_WRITES_10MIN` | 300 | Circuit-breaker cap (orchestrator.js:489). |
| `CLICKUP_MAX_RPM` / `CLICKUP_MAX_TRIES` / `CLICKUP_TIMEOUT_MS` | 70 / 3 / 20000 | Client pacing/retry/timeout (client.js:115-117). |

Additionally: the **no-dirty-sweep rule** — even with outbound on, only the enqueue-on-write queue drains; the old sweep that auto-pushed any touched file (including ones just re-ingested FROM ClickUp — a clobber loop) is intentionally not started (clickup-sync.js:1286-1294). And `db/042_clickup_control.sql` provides the runtime control plane: `clickup_field_mappings` (admin-editable per-field overrides with `direction` push/pull/both, `source_of_record`, `is_active` — i.e., per-field direction can be changed without a deploy) and `clickup_webhook_inbox` (durable, event-id-deduped inbound queue with status/attempts/last_error).

## 8. Dry-run modes

1. **Boot dry-run** (`CLICKUP_DRYRUN=1`): `dryRunBackfill()` (clickup-sync.js:1102-1138) fetches sample tasks per folder, runs the real mapper, and reports what *would* happen (programs, materializable counts, SSN/LLC coverage, samples) — "READ-ONLY validation, zero DB writes"; runs even with the master switch off; live loops never start (:1188).
2. **Admin on-demand dry-run**: `POST` via the Control Center calls the same `dryRunBackfill` (`src/routes/admin-clickup.js:97`).
3. **Repair tooling is dry-run by default**: `scripts/clickup-date-restore.js` writes NOTHING without `--apply` (script header :24, `APPLY = args.includes('--apply')` :68); with `--apply` every write is verified by re-read and journaled.

## 9. How deletes are prevented (complete chain)

1. No delete op exists in the enqueue layer (enqueue.js:6-9) — nothing can even *ask* for one.
2. `pushApplication` skips portal-deleted files entirely (orchestrator.js:217-220).
3. The HTTP chokepoint throws on any DELETE to a task path (client.js:29-39) — including remove-from-list and subtasks; `removeTaskFromList` is kept only for interface symmetry and is blocked by its own client (:280-283).
4. Field-level "deletes" (clears) are equally impossible: `guardNoFieldClearing` + mapper empty-skip + inbound COALESCE (a ClickUp blank never clears the portal: `src/clickup/ingest.js:293-300, 496-498`) — wipes are structurally impossible in both directions.
5. Regression-locked by `scripts/test-clickup-delete-guard.js` (blocked/allowed matrix, 12 assertions) and `scripts/test-clickup-write-guards.js` (10 sections covering every guard above: clearing shapes incl. NaN/null-latitude, status-only allowlist, delete regression, mapper payload hygiene, `resolveOnly` exactness, PII-shield membership + review-key round-trip, date/DOB sanitizers, object-value equivalence, `isDobChange`). Both run with **no DB / no network** — they exercise the exported pure guards.
6. `docs/CLICKUP-DATA-SAFETY.md:305-318` pins the invariants as an audit checklist: no ClickUp API call outside client.js; the three client guards must keep throwing; scoped enqueue-on-write only; journal + review queue wired into any new push path; the guard suites stay green.

## 11. Which guard patterns translate directly to a READ-ONLY Encompass integration

Phase 1 Encompass is strictly pull-only, so the *outbound-content* guards (PII shield, DOB gate, scoped pushes, no-op suppression) mostly evaporate — what survives is the **chokepoint pattern**, the **inbound-protection half**, and the **observability half**. Ranked:

### Translate directly (the write-guard becomes absolute)
1. **Single HTTP chokepoint with a method/path guard** (client.js `call()` + `guardNoTaskDeletion` pattern). For Encompass: one `src/encompass/client.js` through which every request funnels, with a `guardReadOnly(method, path)` that **throws on anything not on an explicit allowlist**. Critical nuance from the Postman index (`postman-endpoint-index.md`): Encompass read operations are *not* all GETs — the auth token grant is `POST /oauth2/v1/token`, and read workhorses like `POST /encompass/v3/loans/{id}/fieldReader` and the loan-pipeline query are POSTs. So the guard must be `guardTaskUpdatePayload`-style: an **ALLOWLIST of exact known-read endpoints** (token, introspection, fieldReader, loanPipeline query, plus all GETs), refusing every PUT/PATCH/DELETE and any unlisted POST by default. Additionally: simply **do not implement** any write function (the strongest structural guarantee — there is nothing to call), the way enqueue.js has no delete op.
2. **Default-off staged kill switches** (config.js:208-232 pattern): `ENCOMPASS_SYNC_ENABLED` master (default off) + no outbound flag at all — the absence of `ENCOMPASS_OUTBOUND_ENABLED` should be permanent and documented as such. Boot log should state "READ-ONLY — zero write paths exist" the way clickup-sync.js:1296 states its mode.
3. **Dry-run boot mode** (`dryRunBackfill` pattern): fetch sample loans, run the mapper, log what WOULD land in the portal, start no loops — ideal for validating the Encompass field mapping against instance BE11397907 before go-live.
4. **Inbound never-clears + review queue**: COALESCE writes (ingest.js:293-300), out-of-range/implausible-value quarantine to `sync_review_queue`, and the "portal condition cannot clear unless Encompass agrees" rule is exactly the existing checklist no-downgrade/gate philosophy. The two-sided review row (db/108/110) ports as-is — with the simplification that "winner=encompass" can only heal the portal, never write back.
5. **Pull journal** (db/107 pattern, mirrored on the pull side like `clickup_pull_field_change`): an append-only `encompass_pull_log` (loan GUID, field, old portal value, new value, source pass, masked PII) so every portal change driven by Encompass data is queryable — attribution was the hard lesson of §1 of CLICKUP-DATA-SAFETY.
6. **Rate pacing + retry contract** (client.js:101-213): token bucket under the vendor limit, Retry-After honoring, PII-free value-free error strings, transient/permanent classification — ports verbatim; Encompass rate limits make the boot-storm gap (#18/#53) worth fixing *before* launch, not after.
7. **No-DB/no-network guard test suite** (`test-clickup-write-guards.js` pattern): a `test-encompass-readonly-guard.js` asserting the allowlist blocks every PUT/PATCH/DELETE and unlisted POST, and that no exported function of the client can produce a write — the cheapest and highest-value port of all.
8. **Guarded durable queue** semantics: `sync_queue` rows for Encompass should only ever exist as `direction='pull'`; a **DB CHECK constraint** (or trigger) refusing `target='encompass' AND direction='push'` would give a belt-and-suspenders layer the ClickUp side achieves only in code.

### Translate with inversion
- **Circuit breaker** → becomes a *read*-volume breaker (API-call budget per window) protecting the rate limit and the co-tenant, not data. Fix the deploy-reset weakness (journal-backed) this time.
- **Master-switch-at-every-layer** (enqueue + orchestrator + worker): for Encompass, gate at client + scheduler + ingest — same defense-in-depth.
- **Dedicated token identity** (matrix #80 lesson): give the portal its own Encompass API client/user so its reads are attributable and its credentials can be scoped.

### Does not translate (nothing to guard)
Scoped pushes, PII overwrite shield, DOB outbound gate, no-op suppression, overwrite-storm alarm — all exist to make *writes* safe; with zero writes they are unnecessary. Their *review-queue* halves survive as inbound-conflict machinery.
