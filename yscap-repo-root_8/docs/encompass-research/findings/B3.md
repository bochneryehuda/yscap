## 1. The core object graph

```
borrowers (PII)  1──*  applications (loan files)   *──1  llcs (vesting entity)
     │ 1──1 borrower_auth (login, separate table)
     │ 1──*  track_records (past deals, PER BORROWER not per file)
     │ 1──*  llcs (entity library, borrower-owned)
     │ *──*  llcs via llc_borrowers (co-owned vesting entities)
     │ 1──*  borrower_contacts (every email/phone ever seen)
     │ *──*  borrowers via borrower_profile_links (shared-email visibility links)
     │ 1──*  borrower_dedup_candidates / borrower_link_candidates (review queues)
```

### 1.1 `borrowers` — identity & PII (`db/schema.sql:49-82`)
- Identity: `first_name`, `last_name`, `email citext UNIQUE NOT NULL`, `cell_phone`, `date_of_birth`, `ssn_encrypted bytea` (never plaintext), `ssn_last4`, `fico`.
- Residence/employment: `current_address jsonb`, `years_at_residence`, `prior_address`, `citizenship`, `marital_status`, `employment_type`, `employer`. Residence duration is now an anchored **move-in date** recomputed live (`src/lib/residence.js:14-45`, `moveInFrom`/`durationSince`; `residence_since` wins over the stored count, `residence.js:50-59`).
- Book-keeping: `tier` ("derived from verified track records", `schema.sql:74`), `primary_officer_id` (LO of record), `clickup_crm_task_id`.
- Login lives in a **separate** `borrower_auth` table (`schema.sql:85-94`) for blast-radius reduction — a borrower row with no auth row is a **"shadow profile"** (internal CRM contact, docs/BORROWER-HISTORY-BACKFILL.md §3).
- Identity-graph columns from `db/044_borrower_history_backfill.sql:10-12`: `ssn_hash` = **HMAC-SHA256(last-9-digits-of-SSN, SSN_MATCH_KEY)** (`src/clickup/identity.js:26-30`) and `origin` (`'portal' | 'clickup_backfill'`), plus an index on `ssn_hash`.
- Multi-contact: `borrower_contacts` (`db/044:15-26`) keeps *every* email/phone ever seen per borrower (`kind`, `value`, `source` e.g. `'clickup:<task_id>'`, `is_primary`, `UNIQUE (borrower_id, kind, value)`).

### 1.2 `applications` — the loan file (`db/schema.sql:155-163`)
- `borrower_id` (NOT NULL, RESTRICT), optional `co_borrower_id` (a second `borrowers` row), `llc_id` (the **vesting entity**, SET NULL), `loan_officer_id`, `processor_id`.
- One borrower : many applications (one per property). Claimed experience lives on the file: `requested_exp_flips/holds/ground` (`db/029_application_rehab_experience_fields.sql:12-14`).
- **LO stickiness** (`db/105_borrower_owning_officer.sql`): trigger `trg_set_owning_officer` (lines 21-36) stamps `borrowers.primary_officer_id` the first time any of the borrower's files gets an officer — *never overwrites an established owner* — plus a deterministic backfill from the most-recent officered application (lines 41-52). New files inherit that owner so a returning borrower's 2nd file keeps the relationship.

### 1.3 Co-borrowers (`src/lib/co-borrower.js`)
- The co-borrower is a full `borrowers` row referenced by `applications.co_borrower_id`.
- Linking a co-borrower creates a real document condition "Government-issued ID — <name> (co-borrower)" tagged `field_key='cob_gov_id'` (idempotent marker, `co-borrower.js:21,73-87`) and renames the primary's gov-ID condition to carry the primary's name (`co-borrower.js:57-65`); unlinking removes the condition and reverts the label (`co-borrower.js:45-54`).
- Co-borrower personal info is enforced as required completeness fields, deliberately *not* a condition (`co-borrower.js:16-17`).

## 2. LLC / entity model

### 2.1 Base entity (`db/schema.sql:102-117`, `src/lib/llc.js`)
- `llcs`: `borrower_id` (the primary owning borrower), `llc_name`, `ein`, `formation_state`, `formation_date`, `ownership_pct` (the borrower's own stake), `is_verified`/`verified_at`/`verified_by`. Provenance columns `origin` + `source_task_id` from `db/044:29-30`.
- Each LLC carries **three required document slots** — state formation docs, EIN letter, operating agreement (`LLC_SLOT_CODES`, `llc.js:20`) — plus an **optional** Certificate of Good Standing that *never* gates verification and **expires 30 days after upload** (slot reopens; LLC stays verified; `llc.js:21-29,148-166`).
- Verification requirements (`missingForVerification`, `llc.js:181-213`): EIN present + `XX-XXXXXXX` shape, formation state/date, ownership totals exactly 100% across borrower + members, all three required slots uploaded *and accepted*, no rejected doc anywhere, and — for layered entities — every owning entity itself verified (bottom-up, `llc.js:206-211`).
- The file-side `rtl_p1_llc` umbrella condition is fulfilled *by the linked LLC's state, never by uploads of its own*: verified → `satisfied` (with sign-off), any rejected slot → `issue`, all slots in → `received`, else `outstanding` (`syncLlcConditions`, `llc.js:287-362`); chain-aware across ancestor entities (`llc.js:307-313`) and cascading down to owned entities (`llc.js:368-380`).

### 2.2 Layered entities (`db/094_layered_entities.sql`, `llc.js`)
- `llc_members` grows `member_kind ('person'|'entity')` + `owner_llc_id → llcs(id)` (`db/094:17-32`): an LLC's member can be **another entity**, which is itself a first-class `llcs` row of the same borrower with its own info, slots, SharePoint folder, and verification — recursively.
- Depth cap `MAX_ENTITY_DEPTH = 5` measured as the longest ownership **path** (not node count), with BFS walkers both directions and cycle detection (`walkChain`/`ownershipLinkError`, `llc.js:72-123`).

### 2.3 Multi-borrower ownership (`db/061_llc_borrowers.sql`, `src/lib/llc-borrowers.js`)
- `llc_borrowers (llc_id, borrower_id, ownership_pct)` PK-pair join table: which of *our* borrowers own an LLC. Additive over the single-owner `llcs.borrower_id` (which remains primary; `db/061:10-12`). Backfilled from existing llcs + co-borrower vesting files (`db/061:27-37`).
- `syncVestingLlcBorrowers(appId)` links the file's primary + co-borrower to the file's vesting LLC whenever either is set (`llc-borrowers.js:50-56`).

### 2.4 Entity creation chokepoint & vesting authority
- `findOrCreateLlc(borrowerId, {llcName…})` (`llc.js:511-532`) is THE single creation path from anywhere (entity screens, new file, track-record typed name, ClickUp task): a name the borrower already has is **reused, never duplicated and never rejected** — matched case/whitespace-insensitively (`findLlcByName`, `llc.js:493-500`) against the unique index `uq_llcs_borrower_name` on `(borrower_id, lower(btrim(llc_name)))` (`db/082:110`), with a 23505 race fallback.
- `setVestingLlc` (`src/lib/vesting.js:32-57`) is the single authority for the file↔entity link, used by both ClickUp ingest and HTTP routes. Guards: never changes a CTC/funded/declined/withdrawn file (`vesting.js:30,45`); an automated `'clickup'` source never overwrites a verified or human-linked entity (`vesting.js:48-52`, `llcs.origin <> 'clickup_backfill'` = human intent).
- `db/082_dedup_and_unique_llc_track_record.sql` retroactively **collapsed duplicate LLCs onto a keeper** (verified first, then oldest), re-pointing applications, track_records, clickup_task_index, documents, llc_members, llc_borrowers, then deleting losers (`db/082:33-104`) — the model for any future entity dedup.

## 3. Track record — past deals / experience

### 3.1 Row shape
`track_records` is **per borrower, not per application** (`db/schema.sql:119-149`):
| Field group | Columns | Source |
|---|---|---|
| Ownership | `borrower_id` (CASCADE), `llc_id` (SET NULL — "experience under this entity"), `entity_name` (db/031:54), `owned_personally` (db/093:15-16 — held under personal name, tool hides entity input, server nulls entity) | schema + 031 + 093 |
| Deal | `property_address jsonb`, `property_type`, `deal_type` (flip / fix-and-hold / ground-up / rental), `purchase_price`, `purchase_date`, `rehab_amount` | schema + 031 |
| Exit | `sale_price`, `sale_date` (flips), `rent_amount`, `rent_date`, `refi_amount`, `refi_date` (holds), `current_value` | schema + 029 |
| Verification | `is_verified`, `verified_at/by`, `verification_status` (`pending`/`docs`/`verified`/`limited`, db/031:55-60), `docs_status` (outstanding→satisfied), `lo_notes` | schema + 031 |
| Provenance/dedup | `origin` (`portal`\|`clickup_backfill`), `source_task_id`, `inferred` (deal type guessed), `address_key` (normalized addr) — db/044:32-36; `client_row_id` idempotency key for tool autosaves with partial unique `(borrower_id, client_row_id)` — db/087:16-19 | 044 + 087 |
Unique guards: one auto record per `source_task_id` (`uq_track_records_source_task`, `db/082:135-136`).

### 3.2 Experience math (`src/lib/experience.js`)
- **Frozen 36-month exit window**: a deal counts only if its exit — `sale_date` for flips, `COALESCE(rent_date, refi_date)` for holds — is non-null, not future, and within 36 months (`EXIT_DATE_SQL`/`RECENT_EXIT_SQL`, `experience.js:13-18`; mirrors the frozen tool `web/tools/track-record.js:18,88-97`).
- Buckets by `deal_type` substring: ground/construction → `ground`, flip → `flips`, else `holds` (`bucketOf`, `experience.js:25-30`).
- A file's experience = primary + co-borrower **summed** (`countBorrowersExperience`/`fileBorrowerIds`, `experience.js:59-84`), with a per-borrower breakdown payload on co-borrower files (`experience.js:99-118`). Comment at `experience.js:62-64`: "Each track_records row already belongs to exactly one borrower, so summing the per-borrower counts never double-counts a deal" — **an assumption an Encompass ingest must not break** (see §6).
- The `track_record` condition (tool_key) auto-syncs: claim (`requested_exp_*`) vs counts → `received` when met / `outstanding` when not / auto-satisfied-not-applicable when nothing claimed (`experience.js:165-220`); a requirement growing past sign-off reopens it (`experience.js:197-204`); an experience **drop** below what the current product registration priced with marks the registration stale and reopens Products & Pricing + the signed term sheet (`experience.js:119-163`).
- **Tier** = count of verified rows in the window: `UPDATE borrowers SET tier=(SELECT count(*) FROM track_records WHERE … is_verified=true AND RECENT_EXIT_SQL)` (`src/routes/staff.js:3358,3787`). Pricing counts verified-only unless the file carries a claim (`staff.js:1404-1414`; CLAUDE.md "sizes on claimed experience" rule).
- **Verification is a human sign-off**: `POST /track-records/:id/verify` (`staff.js:3729-3800`) — counting statuses (`verified`/`limited`) are processor-only, gated on a completed in-window exit (422 for no-exit / future / >36mo, `staff.js:3759-3775`), and revoking requires a reason + notifies the borrower.
- Documents attach per line (`documents.track_record_id`, `db/031:63-65`); staff can raise a per-line document-request condition (`checklist_items.track_record_id`, `db/093:21-25`, ON DELETE SET NULL so a borrower deleting a line never deletes a file condition). SharePoint mirrors line docs to `REO/<project address>` at borrower-profile level (CLAUDE.md SharePoint rules).
- The borrower's live builder state is snapshotted as ONE current self-contained HTML document per borrower (`doc_kind='track_record_html'`, `src/lib/track-record-snapshot.js:13,31-74`), with same-session autosave coalescing (10-min window) to prevent snapshot-row/mirror churn.

### 3.3 Auto-derivation from ClickUp — the existing "history enrichment" precedent
`docs/BORROWER-HISTORY-BACKFILL.md` (design, decisions §11 locked) + `src/clickup/ingest.js`:
- Every **closed/funded** task becomes an unverified `track_records` line: `is_verified=false, origin='clickup_backfill', source_task_id, inferred, address_key`, note "Auto-derived from ClickUp; unverified" (`upsertTrackRecord`, `ingest.js:476-520`).
- Dedup: `WHERE borrower_id=$1 AND (source_task_id=$2 OR address_key=$3)` (`ingest.js:480-482`); `address_key = lowercase(alnum-only formatted address)` (`ingest.js:470-473`).
- Deal-type inference (BORROWER-HISTORY-BACKFILL.md §5/§11.3): program Fix&Flip → `flip`; purchase + refi on the same address → `fix-and-hold`; else default `fix-and-hold`, flagged `inferred=true` (`ingest.js:483-491`).
- Every `*LLC Name` on any task → `upsertLlc` into the borrower's library, unverified with provenance (`ingest.js:447-468`).

## 4. Identity resolution & dedup-candidate logic

### 4.1 Matching engine (`src/clickup/identity.js`)
- 8 identity fields: address, loanNumber, borrowerName, dob, email, ssn, phone, purchasePrice (`identity.js:14-16`); normalized comparably (`identity.js:33-63`). **≥2 fields must agree** to call two records the same (`isMatch`, `identity.js:79`); one field is never enough.
- `ssnHash` (`identity.js:26-30`) is the strong key (design: "Primary key = SSN", BORROWER-HISTORY-BACKFILL.md §2).
- `emailMatchCorroborated` (`identity.js:113-141`): a shared email merges only with a second agreeing field — phone (last 10), DOB, or the FULL name (first token AND last name, placeholders excluded). **Last name alone is explicitly NOT corroboration** (2026-07-15 incident: family email + family surname merged an LO's lead with a different real borrower and leaked PII — `identity.js:100-111`).
- `nameConflict` (`identity.js:153-161`) blocks same-email adoption when the typed name belongs to someone else (single-initial tolerant).

### 4.2 `resolveBorrower` (`src/clickup/ingest.js:310-444`) — the decision ladder
1. **Strong**: exact `ssn_hash` match → reuse, heal blank fields (COALESCE-only, `healBorrowerFields` `ingest.js:288-307`), record contacts.
2. **Email exact**: merge only if no SSN conflict AND (SSN agrees OR corroborated). A differing SSN-hash = definitely two people.
3. **Else create a DISTINCT profile** — with the email replaced by a synthetic `noemail+<taskId>@clickup.local` when the real email matched someone we declined to merge (otherwise `ON CONFLICT (email)` would silently re-merge them, `ingest.js:377-382`).
4. **No-identity guard**: no real name, no SSN, no DOB → no profile at all (no "Unknown Unknown" husks, `ingest.js:364-375`).

### 4.3 The three review/link tables
| Table | Migration | Purpose |
|---|---|---|
| `borrower_dedup_candidates` | `db/081` | "Possible duplicate — please check": recorded when an email matched but couldn't be corroborated so a distinct profile was safely created (`ingest.js:420-426`, also public intake `src/routes/intake.js:138`). One open row per pair (`uq_borrower_dedup_pair`), statuses `open/distinct/duplicate/reviewed`. Resolution records a verdict only — the actual merge is a deliberate manual step (`db/081:9-12`). Surfaced in admin Control Center (`src/routes/admin-clickup.js:238-256`) + as a two-sided sync-review card ("give the email to ONE of them", `ingest.js:427-441`, boot sweep `src/sync/clickup-sync.js:525-569`). |
| `borrower_link_candidates` | `db/044:54-67` | Registration-time weak-match confirmation queue: a registering borrower with a 2-piece (non-SSN) match to a shadow profile is asked to confirm before linking; a full SSN match links silently (BORROWER-HISTORY-BACKFILL.md §11.1). Carries `match_fields jsonb` + `score`. |
| `borrower_profile_links` | `db/114` | Staff-granted "Allow — same email for both" outcome (spouse pair / same-person duplicate): written both directions; the borrower portal's `OWN_FILE_SQL` then lets either login see BOTH people's files (`src/routes/borrower.js:36-46`). **Never merges** — each person keeps their own identity; LLCs and track record stay per-profile (`db/114:12-14`). Also suppresses future dedup cards for the pair (`ingest.js:414-419`). |

### 4.4 Address canonicalization (`src/lib/address-canon.js`)
Formatting variants of one property ("Ave"/"Avenue", "Unit 114"/"114") resolve once via Google Geocoding to a stable `place_id` cached forever (`address_canon_cache`, db/124); fully degradable to heuristics; "canonicalization only ever ADDS matches" (`address-canon.js:4-13`). Relevant to any Encompass address matching.

## 6. How Encompass loan history would enrich profiles & auto-build track records

Encompass instance BE11397907 is YS Capital's own LOS, so its closed-loan history is **first-party, lender-verified data** — a materially stronger source than the ClickUp backfill. Phase 1 is strictly read-only, which fits perfectly: everything below is portal-side writes only.

### 6.1 Read path (from the Postman index, `scratchpad/research/postman-endpoint-index.md`)
- **Enumerate history**: `POST /encompass/v3/loanPipeline?start=0&limit=1000` (index lines 343-350; v1 cursor variant lines 351-356) filtered on borrower TaxIdentificationNumber / name / folder / milestone; `GET /encompass/v3/loanPipeline/canonicalFields` to discover filterable fields. Include archived loans (line 347) for full history.
- **Per-loan detail**: `GET /encompass/v3/loans/{loanId}` (line 107) or targeted `POST /encompass/v3/loans/{loanId}/fieldReader` (line 15) for exact field IDs (loan amount, purchase price, appraised value, property address, loan purpose, funds/disbursement dates).
- **Lifecycle dates**: `GET /encompass/v3/loans/{loanId}/milestones` (line 158) — Funded/Completion milestones give the closed date without parsing the whole loan.
- Optionally `borrowerContacts` (index lines 76-88) for contact enrichment into `borrower_contacts`.

### 6.2 Enrichment mapping (mirror the proven ClickUp ingest, don't reinvent)
1. **Borrower matching**: compute `identity.ssnHash(encompassTaxId, cfg.ssnMatchKey)` — the HMAC infrastructure (`identity.js:26-30`, `borrowers.ssn_hash` + index `db/044:10-12`) works unchanged for Encompass SSNs. Strong SSN match → attach; email match → require `emailMatchCorroborated`; weak/none → **do not create** (unlike ClickUp, Phase 1 need not mint shadow profiles — start attach-only-to-known-borrowers, then extend). Record uncorroborated near-matches into `borrower_dedup_candidates` with a new `reason` (e.g. `'encompass_weak_match'`) — the review UI, unique-pair idempotency, and sweep already exist.
2. **Track-record rows**: for each borrower-matched loan whose milestone shows funded/closed, upsert a `track_records` line exactly like `upsertTrackRecord` (`ingest.js:476-520`): `is_verified=false` initially, a new `origin='encompass'`, `inferred` as appropriate, `address_key` from the subject property, and a **new provenance column** `encompass_loan_guid` with a partial unique index mirroring `uq_track_records_source_task` (`db/082:135-136`) so re-syncs are idempotent and concurrent runs collapse (23505 → re-select, the established pattern). New numbered idempotent migration per repo rules.
3. **Exit semantics — the key subtlety**: the experience engine counts **exits** (sale/lease/refi within 36 months, `experience.js:13-18`), but an Encompass origination proves an **entry** (YS funded the purchase/rehab loan). What Encompass *does* prove directly:
   - a **refinance loan** (loan purpose = refi) on a property = the exit event of a prior hold → sets `refi_date`/`refi_amount` on the matching `address_key` line (this is exactly the purchase+refi inference already locked in BORROWER-HISTORY-BACKFILL.md §11.3);
   - a **paid-off YS loan** (payoff/paid-in-full date) is a strong proxy that the flip exited ≈ then — usable to *pre-fill* `sale_date` as a suggestion, but it should stay `inferred=true` and unverified until a human confirms, because payoff ≠ sale price/date.
4. **Deal-type inference**: Encompass loan program/purpose fields map to `bucketOf` buckets (`experience.js:25-30`) far more reliably than ClickUp labels — fix&flip program → `flip`, ground-up/construction → `ground`, refi-out or DSCR → `fix-and-hold`.
5. **LLC library**: the Encompass vesting entity name (borrower vesting / title fields) feeds `findOrCreateLlc` (`llc.js:511`) — name-normalized reuse means no duplicates; new entities land unverified with Encompass provenance (add `origin='encompass'` on `llcs` too).
6. **Verification posture**: because these are YS's own funded loans, staff could fast-track them to `'limited'` (public-record-equivalent, counts toward tier per `staff.js:3727-3728`) — but keep the **processor-only human sign-off** (`staff.js:3747-3749`) and the exit-date gate (`staff.js:3759-3775`); never auto-set `is_verified=true` from the sync, or the tier/pricing chain (`tier` recompute, product staleness `experience.js:119-163`) is driven by machine matches.
7. **Read-only alignment**: all of the above is pull-then-write-portal-side; nothing needs an Encompass write, matching the "architecturally incapable of writing" mandate.

### 6.3 Matching / dedup risks (ranked)
1. **Same name, different person**: the codebase already treats a bare name (even full-name) as insufficient on its own — merges need SSN or email+corroboration (`identity.js:94-141`). Keep that bar for Encompass; a name-only pipeline filter must feed a review queue, never an auto-attach. The 2026-07-15 family-email/surname incident (`identity.js:100-110`) is the cautionary precedent.
2. **Same person, different/missing SSN**: typos or ITIN/EIN-in-SSN-field in Encompass produce hash misses → a legitimately experienced borrower shows zero Encompass history, or worse a *different* hash creates a dedup pair. Mitigate: fall through to email+corroboration, and surface hash-miss-but-3-field-match into `borrower_dedup_candidates`.
3. **Entity-vested loans**: hard-money loans often close in the LLC's name; the Encompass "borrower" may be the entity while the person is the guarantor. Attaching by entity name alone risks crediting the wrong member of a multi-member LLC. Use guarantor SSN when present; otherwise route through `llc_borrowers` (an LLC known to be co-owned by two borrowers must not silently credit only `llcs.borrower_id`).
4. **Double-counting across sources**: the same historical deal can arrive from ClickUp (`source_task_id`) AND Encompass (`encompass_loan_guid`). The only cross-source key is `address_key`, whose normalization (`ingest.js:470-473`, strip non-alphanumerics) treats "123 Main St" vs "123 Main Street" as *different* keys → duplicate lines → inflated experience/tier. Mitigate: canonicalize both sides through `address-canon.js` place_id before keying, and extend the `upsertTrackRecord` dedup probe to `(source_task_id OR encompass_loan_guid OR address_key)`.
5. **Co-borrower double-count**: an Encompass loan with two borrowers, ingested as one line per borrower, breaks the `experience.js:62-64` invariant ("summing per-borrower counts never double-counts") on files where both are present — one deal would count twice toward the file. Decide policy up front: one line on the primary + a linked flag, or per-borrower lines with a file-level dedup by loan GUID.
6. **Shared emails**: family/broker-shared emails are common; `borrower_profile_links` pairs must suppress dedup cards for Encompass matches the same way `ingest.js:414-419` already does.
7. **Stale/withdrawn loans**: pipeline queries must filter to genuinely funded loans (milestone Funded / archived-funded), or dead files become fake track record — mirror the CLOSED_STATUSES gate (`ingest.js:967`).
8. **Tier inflation blast radius**: any bad auto-built line flows into `borrowers.tier`, experience conditions, and product-registration staleness. The unverified-by-default + human-verify posture (§6.2.6) is the containment; keep it.

### 6.4 Recommended build order
1. Migration: `track_records.encompass_loan_guid` (+ partial unique), `origin` value `'encompass'` on `track_records`/`llcs`, optionally `borrowers.encompass_borrower_ref`.
2. An `encompass/ingest.js` mirroring `resolveBorrower`/`upsertLlc`/`upsertTrackRecord` shapes — attach-only (no shadow-profile creation) in Phase 1.
3. Reuse `borrower_dedup_candidates` (+ new reasons) and the sync-review card machinery for every non-strong match; never auto-merge.
4. Address canonicalization on both sides before `address_key` comparison.
5. Keep all Encompass-derived lines `is_verified=false`/`inferred` until processor sign-off; consider a one-click "confirm from Encompass" staff action that carries the loan GUID as evidence.
