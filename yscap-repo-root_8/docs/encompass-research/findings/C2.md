## 1. The three read paths for loan data

| Path | Verb/URL | Best for | Payload shape |
|---|---|---|---|
| **V3 Get Loan (entity query)** | `GET /encompass/v3/loans/{loanId}` + `view`/`entities` params | Structured objects: borrower pairs, property, residences, employment, logs | Nested JSON matching V3 loan schema |
| **Field Reader** | `POST /encompass/v3/loans/{loanId}/fieldReader` (also V1) | Point reads of specific classic field IDs incl. virtual roll-ups (milestone, lock status, condition counts) | Flat array of `{fieldId, value}` strings |
| **Sub-resource GETs** | `GET /encompass/v3/loans/{loanId}/<collection>` (milestones, conditions, applications/{appId}/vods, …) | One collection at a time, with stable IDs | Array of entity objects |

(Contrast: the Loan Pipeline API — `POST /encompass/v3/loanPipeline` — reads *across* loans using canonical field names like `Loan.LoanNumber`, `Loan.CurrentMilestoneName`; that is a different naming system from classic field IDs and is covered by another agent. Note: fieldReader explicitly does **not** accept canonical names.)

## 2. V3 Get Loan — entities and view

`GET /encompass/v3/loans/{loanId}` — reference: https://developer.icemortgagetechnology.com/developer-connect/reference/loan-management (V3 Get Loan page confirmed at `/reference/get-loan-1`).

Query parameters (from the official reference):

| Param | Values / behavior |
|---|---|
| `view` | `entity` = "everything in the loan file other than log entries"; `logs` = only log entries; `full` = both; (`id` exists on create/update responses) |
| `entities` | "Comma-separated list of loan entities to retrieve from the loan" — the payload-trimming mechanism |
| `includeEmpty` | `true` to include empty fields/objects (default: empty values omitted) |
| `includeRemoved` | include records removed from entities |

Key facts:
- Entities are included with `view=entity|full` only when populated; `includeEmpty=true` forces empty objects out.
- Field-level permissions apply silently: "If you do not have permission to a requested field, the field will not be returned."
- Collection example of a trimmed read (Postman: *Encompass Docs / Send Encompass Docs / Initial Disclosures / 12*):
  `GET /encompass/v3/loans/{{loanId}}?view=full&entities=disclosureTracking2015Logs,documents` — i.e. log-type entities require `view=full` (or `logs`).
- There is **no published enumeration** of legal `entities` values; the authoritative list is the set of top-level property names of the V3 loan schema (`GET /encompass/v3/schemas/loan`, itself filterable by `entities`). V1 has an explicit helper: `GET /encompass/v1/loans/supportedEntities` (Postman: *Encompass Loan / Loan Schema / V1 Loan Schema / 05*).
- V1 Get Loan (`GET /encompass/v1/loans/{loanId}`) also takes `entities`, plus a `format` param that can export ULDD / URLA / MISMO / UCD / CD / LE datasets (reference: https://developer.icemortgagetechnology.com/developer-connect/reference/get-loan).
- Change detection helper: `GET /encompass/v1/loans/{loanId}/metadata` (Postman: *V1 Manage Loan / 04*) returns loan metadata (e.g. last-modified) without pulling the loan.

An unfiltered `view=full` loan is very large (thousands of fields + logs). Always pass `entities`.

## 3. Field Reader — POST verb, read-only semantics

Two generations, both **pure reads** despite the POST verb (POST is used only to carry the request body listing field IDs):

- **V3**: `POST /encompass/v3/loans/{loanId}/fieldReader?invalidFieldBehavior=Fail|Exclude|Include`
  Reference: https://developer.icemortgagetechnology.com/developer-connect/reference/v3-field-reader
  - Body: JSON array of field-ID strings. Collection sample (*Encompass Loan / Loan Management / V3 Manage Loan / 05*): `["4002", "4000#2", "1109", "CUST02FV", "LR.1500", "BE0139x"]`
  - `invalidFieldBehavior`: `Include` returns invalid IDs with empty values; `Exclude` omits them; `Fail` (default since 24.2) errors the request.
  - Accepts "standard, custom and virtual field IDs". **Canonical field IDs (pipeline names) are explicitly unsupported** (passing them currently "works" as a known issue slated to be corrected — do not rely on it).
  - Response: HTTP 200 with values as **strings**; dates `MM/DD/YYYY`; datetimes with timezone as UTC ISO (`2022-08-27 16:57:04Z`); datetimes without timezone as `MM/DD/YYYY HH:MM:SS AM/PM`. Response objects carry `fieldId` + `value` (confirmed by the EncompassRest client docs: https://github.com/EncompassRest/EncompassRest/wiki/Retrieving-Loan-Data).
  - Field permissions again apply silently — unpermitted fields simply don't come back.
- **V1**: `POST /encompass/v1/loans/{loanId}/fieldReader` (Postman: *V1 Manage Loan / 03 - Loan Field Reader (Retrieve values for field IDs)*). Same body shape; this is where the collection's virtual-field showcase lives (see §6).

Field-ID syntax observed in collection samples:
- Plain numeric: `1109`, `4002`, `2025`
- **Borrower-pair index suffix**: `4000#2` = field 4000 for borrower pair #2 (used in both fieldReader and fieldWriter samples). Caveat: the EncompassRest wiki claims per-pair reads are "not possible through the field reader API" — that note predates V3; verify `#N` behavior in the YS sandbox.
- Multi-instance fields with embedded index: `BE0002` / `BE0102` / `BE0202` ("Multi-instance fields can be queried with any index" — V3 Get Field Schema reference)
- Custom: `CUST01FV`–`CUST99FV` (predefined) and `CX.*` (admin-defined)
- Prefixed families: `LR.1500`, `TQL.x27`, `NEWHUD.X961`, `FICC.X11`
- Virtual: `Log.MS.CurrentMilestone`, `PCC.ALL`, `LOCKRATE.RATESTATUS`, …

The **write twin** is `POST /encompass/v3/loans/{loanId}/fieldWriter` (*V3 Manage Loan / 04*). A read-only client must allowlist POST for the exact path suffix `/fieldReader` and never `/fieldWriter` — method-level filtering is insufficient because the two share verb and base path.

## 4. Loan schema APIs (the data dictionary toolchain)

Guide: https://developer.icemortgagetechnology.com/developer-connect/docs/encompass-loan-data-dictionary-guide — "This solution guide supports standard and custom fields only. Virtual fields are not included."

| API | Endpoint | Purpose |
|---|---|---|
| V3 Loan Schema | `GET /encompass/v3/schemas/loan` | JSON Schema of the loan object; filterable by `entities` and `includeFieldExtensions`. Top-level properties double as the `entities` vocabulary for Get Loan. |
| V3 Field Schema (standard fields) | `GET /encompass/v3/schemas/loan/standardFields?start=0&limit=10000` or `?ids=4000,4002,2025,TQL.x27` | Maps classic field ID → metadata incl. **`jsonPath`** ("a JSON path that can be used to locate the specified fields in the responses returned by loan APIs like Get Loan"), type, description, options. `X-Total-Count` header; no fixed upper `limit`, only max-payload-size. Reference: https://developer.icemortgagetechnology.com/developer-connect/reference/get-field-schema-1 |
| V3 Virtual Fields | `GET /encompass/v3/schemas/loan/virtualFields` | Enumerates virtual (computed, UI-invisible) fields — the only systematic way to discover `Log.MS.*`, `LOCKRATE.*`, `PCC.*`, etc. on the instance. |
| V3 Custom Field Settings | `GET /encompass/v3/settings/loan/customFields[?ids=CUST01FV,…][&start&limit]` | All custom field definitions (id, description, format, maxLength, calculation). The PATCH variants (`action=add|update|delete`) are **writes — excluded**. |
| V1 Loan Schema | `GET /encompass/v1/schema/loan` | Legacy schema. |
| V1 Field Schema | `GET /encompass/v1/schema/loan/{fieldId}` | Per-field legacy definition. |
| V1 **pathGenerator** | `POST /encompass/v1/schema/loan/pathGenerator?ignoreInvalidFields=true&fieldNamePattern=loanType` with body `["4000","4002"]` | Returns JSON paths for field IDs / name patterns. Officially: "This API's response is used for specifying the filters attributes when subscribing for a change event webhook." Compute-only, no loan touched. |
| V1 **contractGenerator** | `POST /encompass/v1/schema/loan/contractGenerator` with body `{"NEWHUD.X961":200, "4000":"fname", "4002":"lname"}` | Converts fieldID→value pairs into the nested loan-JSON contract fragment (where those values would live). Compute-only; mainly useful for writers, so optional for Phase 1. |
| V1 Supported Entities | `GET /encompass/v1/loans/supportedEntities` | Explicit entity list (V1 contract). |

Data-dictionary maintenance: each Encompass release ships a Loan Schema Delta File whose FieldIds can be fed to V3 Get Field Schema; or diff successive `GET /v3/schemas/loan` payloads (https://developer.icemortgagetechnology.com/developer-connect/docs/update-data-dictionary-with-changes-to-the-encompass-loan-schema).

**Recommendation:** at integration startup (and on each Encompass major release), snapshot `standardFields`, `virtualFields`, `customFields`, and `/v3/schemas/loan` into Postgres. That snapshot is the translation table between classic field IDs (what ops/UW staff quote) and V3 JSON paths (what entity queries return) — and it documents exactly which CX.* fields the YS instance defines.

## 5. V1 vs V3 — differences that matter

Source: https://developer.icemortgagetechnology.com/developer-connect/docs/v1-vs-v3-encompass-apis-whats-the-difference-1

| Aspect | V1 | V3 |
|---|---|---|
| Architecture | "RESTFUL wrappers on our legacy SOAP/WCF APIs" | "True REST operations", leaner codebase, better performance |
| Collections | Fixed-size (e.g. Assets) and variable-size (e.g. VODs) merged into one contract; caller must know which is which | Segregated contracts. Fixed collections are read-only after creation; variable collections support add/delete/reorder |
| IDs | Inconsistent — some entities have unique IDs, others positional indices | Fixed-size entities: no ID (identified by metadata). Variable-size entities: "system generated unique id" (typically GUID), **unique within the loan and immutable for the entity's lifetime** |
| Guidance | — | "Always use the V3 API when its available, if not, use V1." Both currently stable/supported; V1 will eventually deprecate |

Stable V3 IDs are exactly what the portal needs for durable joins (e.g., `applications[].id`, condition IDs) — persist them as foreign keys rather than array positions.

Where V1 is still required today: borrower-pair CRUD collection (`/v1/loans/{loanId}/applications`, §7), classic (non-enhanced) conditions (`/v1/loans/{loanId}/conditions/underwriting|preliminary|postclosing`), rate-lock requests (`/v1/loans/{loanId}/ratelockrequests`), loan metadata, supportedEntities, pathGenerator/contractGenerator.

All of the above are readable via fieldReader (V3 fieldReader supports virtual IDs per the official reference). This makes fieldReader the cheapest "status snapshot" primitive: one POST returns milestone + lock + condition-counts in a single flat payload.

### 6.3 Custom fields

- Two families: 100 predefined `CUST01FV`…`CUST99FV` slots, and admin-created `CX.*` fields (samples in collection: `CX.RATELOCKCUSTOMFIELD`, `CX.ZIPCODE…`, `CX.EBSP…` with formats string/integer/DECIMAL/dropdown, optional `calculation` expressions referencing classic IDs, e.g. `[4000] + [4001] + [4002]`).
- Definitions readable at `GET /encompass/v3/settings/loan/customFields` (filter `?ids=`, paginate `?start&limit`) and legacy `GET /encompass/v1/settings/loan/customFields[/{id}]`.
- Values readable per-loan via fieldReader (e.g. `CUST02FV`, any `CX.*` id) or in the loan JSON's custom-fields section via Get Loan.
- Hard-money-relevant note: Encompass's standard schema is consumer-mortgage-centric; YS concepts like ARV, rehab budget, flip experience, DSCR are very likely CX.* fields in BE11397907 — enumerate them on day 1 via the settings endpoint and map to `applications.arv`, `rehab_budget`, `dscr_ratio` etc. (`db/schema.sql:169-186`).

## 7. Borrower pairs model

Reference: https://developer.icemortgagetechnology.com/developer-connect/reference/loan-borrower-pair — "Borrower pairs are referred to as 'applications' within a loan object."

- A loan has 1..n applications (pairs); each pair = `borrower` + optional `coborrower` objects plus pair-scoped sub-collections (assets, income, employment, residences, REO, tax4506s…). Collection sample bodies (*Encompass Loan / Borrower Pairs / 01, 04*) show rich borrower attributes: names, `birthDate`, `taxIdentificationIdentifier` (SSN), `emailAddressText`, `homePhoneNumber`, `maritalStatusType`, `citizenshipResidencyType`, `middleCreditScore`, `minFicoScore`, HMDA fields, declarations, `borrowerType` ("Individual"), `vestingTrusteeOfType`.
- Read endpoints (V1, still authoritative for pair CRUD): `GET /encompass/v1/loans/{loanId}/applications` (all pairs) and `GET …/applications/{applicationId}` (one pair). (Write ops: POST create, PATCH update/move/swap, DELETE — excluded.)
- V3 exposure: pairs appear in Get Loan as the `applications` entity, and V3 sub-collection routes hang off `/encompass/v3/loans/{loanId}/applications/{applicationId}/…` including per-applicant paths `…/borrower/residences`, `…/borrower/employment`, `…/borrower/urlaAlternateNames` (and `coborrower/…`).
- Classic-field addressing across pairs uses the `#N` suffix (`4000#2`) or multi-instance IDs (`BE0002`/`BE0102`).
- Fixed vs variable: borrower/coborrower are fixed entities (no IDs); sub-collections like VODs/VOLs are variable with stable GUIDs (§5).
- Related but distinct: **non-borrowing owners** (`nonBorrowingOwners` in create-loan sample) and **borrower vesting** — `GET /encompass/v3/loans/{loanId}/closingDocument/vestingEntities?view=entity` returns vesting parties (alias, vestingType, linkage to an application via `{entityId, entityType:"Application"}`). Relevant to YS because loans vest in LLCs: the portal's `llcs`/`applications.llc_id` model (`db/schema.sql:102,158`) should reconcile against `closingDocument.vestingEntities`.

## 9. Recommended minimal pulls

### (a) Loan status snapshot → **fieldReader** (V3)

One `POST /v3/loans/{loanId}/fieldReader?invalidFieldBehavior=Exclude` per loan per poll. Body:

```json
["364", "1109", "353", "2025", "3142",
 "CoreMilestone", "Log.MS.CurrentMilestone", "Log.MS.Stage", "Log.MS.LastCompleted",
 "LOCKRATE.CURRENTSTATUS", "LOCKRATE.RATESTATUS", "LOCKRATE.REQUESTPENDING", "LOCKRATE.CURRENTDATETIME",
 "UWC.ALLCOUNT", "UWC.OPENCOUNT", "UWC.CLEAREDCOUNT", "UWC.WAIVEDCOUNT",
 "PRECON.ALLCOUNT", "PRECON.OPENCOUNT",
 "PCC.ALL", "PCC.NOTCLEARED"]
```

Rationale: flat, tiny, one call; virtual roll-ups come pre-computed. For the **clear-to-close gate**, do not rely on a single field: require (1) `GET /v3/loans/{loanId}/milestones` showing the CTC-equivalent milestone finished for BE11397907's milestone template, AND (2) condition-level verification per (c). Add YS-specific date/status fields (est. closing date, funded date) after confirming their IDs against the instance's `standardFields` snapshot rather than guessing.

### (b) Borrower profile enrichment → **V3 entity query**

`GET /v3/loans/{loanId}?view=entity&entities=applications,property,closingDocument` (confirm exact entity names against the top-level properties of `GET /v3/schemas/loan` before hard-coding; add the custom-fields entity for CX.* values in bulk). Map:
- `applications[].borrower/.coborrower` → `borrowers` columns (firstName/lastName/emailAddressText/homePhoneNumber/birthDate/taxIdentificationIdentifier→ssn, maritalStatusType, citizenshipResidencyType, middleCreditScore→fico)
- `…/borrower/residences`, `…/borrower/employment` sub-GETs (or the same entities inline) → `current_address`/`prior_address`/`employer`
- `property` → `applications.property_address`, `property_type`, `units`; `closingDocument.vestingEntities` → LLC/vesting reconciliation.

Rationale: structured nested objects with stable V3 IDs beat dozens of `#N`-indexed field reads; entity filtering keeps payloads small. Use fieldReader only for spot values not surfaced by the entity contract (e.g. specific CX.* singletons).

### (c) Condition matching → **conditions sub-resource GETs**, fieldReader roll-ups as cross-check

Primary: `GET /v3/loans/{loanId}/conditions` (+ `/tracking`, `/comments`, `/documents` per condition) if BE11397907 uses Enhanced Conditions; else V1 `GET /v1/loans/{loanId}/conditions/underwriting|preliminary|postclosing`. Persist condition IDs (stable in V3) against `checklist_items`. Secondary: the `UWC.*`/`PRECON.*`/`PCC.*` counts from snapshot (a) as a cheap consistency check — portal must refuse to clear a portal condition while the matched Encompass condition is not cleared/waived, and refuse portal CTC unless the milestone gate in (a) and `UWC.OPENCOUNT=0`-style evidence agree.

### Read-only enforcement note

Allowlist by method+path, not method alone: permit `GET` on `/encompass/v1|v3/loans/**`, `/v3/schemas/**`, `/v1/schema/**`, `/v1|v3/settings/loan/customFields`; permit `POST` **only** for `/fieldReader` (and, if used, `/loanPipeline`, `/schema/loan/pathGenerator`, `/schema/loan/contractGenerator` — all compute/read-only). Deny everything else (`fieldWriter`, all PATCH/PUT/DELETE, `loanBatch`, milestone PATCHes). Also ensure the Encompass API user/persona is provisioned without loan-edit rights so read-only is enforced server-side too, not just client-side.

## 10. Open questions

1. **Enhanced vs classic conditions** on BE11397907 — determines whether condition matching uses `/v3/loans/{id}/conditions` or the three V1 condition types. (Enhanced Conditions requires instance enablement.)
2. **Authoritative `entities` list** — not published; derive from `GET /v3/schemas/loan` top-level properties at runtime and pin in config.
3. **`#N` pair-index support in V3 fieldReader** — Postman samples show `4000#2`; older client docs say per-pair reads aren't possible (likely V1-era). Verify in sandbox.
4. **`PCC.ALL`/`PCC.NOTCLEARED` return format** (delimited list vs count) — confirm via `virtualFields` schema + sandbox read.
5. **Which CX.* fields the YS instance defines** for ARV / rehab / experience / DSCR — pull `GET /v3/settings/loan/customFields` first thing.
6. **CTC representation** in YS's milestone template (milestone name for clear-to-close, and whether `Log.MS.CurrentMilestone` string matching is stable across template edits).
7. **Field-level permissions of the integration user** — both Get Loan and fieldReader silently drop unpermitted fields; a missing field is indistinguishable from an empty one unless `includeEmpty`/`invalidFieldBehavior` are used deliberately and permissions are audited.
