## 1. Executive summary

The portal runs a **three-layer lifecycle**: (a) a borrower-facing **status bucket** on `applications.status` (11 values, `file_intake` → `funded`, plus `on_hold`/`declined`/`withdrawn`), (b) an **internal status** that is a verbatim mirror of the ClickUp Loan Pipeline's ~38-status workflow (`applications.internal_status`, mapped down to the 11 buckets by `src/clickup/status.js`), and (c) two **condition systems** (checklist items + first-class `conditions`) whose open items *gate* the decision transitions. "Clear to close" is simultaneously: a **status value** (`clear_to_close`) that only see-all roles can set and only when the blocker query is empty; a **gate checklist condition** (`rtl_f_ctc` — "CTC — Clear to Close received"); and a **structural freeze point** after which pricing, the rehab budget, and the vesting entity are locked. Funding (`funded`) seeds a separate **post-closing trailing-doc checklist**. Pricing "locks" through **product registration** (append-only, server-recomputed on frozen engines) rather than through any rate-lock concept — there is **no rate-lock object in the portal today**.

Encompass appears in the portal today only as: (1) five **manual staff checklist tasks** telling the LO/processor to do things *in* Encompass (`db/005_rtl_workflow.sql:74-84`), (2) a staff-only, pull-only `encompass_status` free-label column mirrored from a ClickUp dropdown (`src/clickup/mapper.js:122`, `src/clickup/fields.js:74`), hidden from borrowers (`src/routes/borrower.js:472`), and (3) a comment in the sync-queue that Encompass integration is deferred (`src/sync/queue.js:4`). Nothing reads Encompass directly yet — which is exactly what Phase 1 will change.

## 3. Status transitions — who moves a file forward

### 3.1 The two transition endpoints

**A. `PATCH /api/staff/applications/:id { status, force? }`** — `src/routes/staff.js:4247-4292`:

1. Status must be in `APP_STATUS` (`:4250`).
2. **Role gate (S3-05 fix):** `DECISION_STATUSES = {approved, clear_to_close, funded, declined}` require `seesAll` (admin / super_admin / underwriter / loan_coordinator via `see_all_files`) — `:4246,4256-4257`. LOs/processors can move a file through the working statuses only.
3. **Blocker gate:** `clear_to_close` and `funded` run `advancementBlockers()`; any open blocker → 409 with the blocker list, unless `force` **and** admin, which sets `forced=true` (`:4260-4266`).
4. Write status + `status_changed_at`; push ONLY the status to ClickUp (`:4267-4269`); insert history row (`:4271-4273`); **`funded` seeds post-closing** (`:4275`); audit (`:4276`); re-run the Condition Center rule engine (status is a rule field, `:4278`); notify borrowers (email only for `MAJOR_STATUSES = {approved, clear_to_close, funded, declined, withdrawn}` — `:3888,4281-4285`) and the whole staff team (`:4286-4288`).

**B. `POST /api/staff/applications/:id/internal-status { internalStatus }`** — `src/routes/staff.js:4298-4325`: sets the exact ClickUp status, **re-derives** the external bucket via `statusMap.externalFor`, and is gated the same way for decision-grade external results (`:4309-4310`, "a second door into the decision-grade buckets"). Records history and pushes to ClickUp. Note: this path does **not** run `advancementBlockers` — the decision-role gate applies but the condition gate does not (an underwriter picking ClickUp status "ctc (4-email)" lands the file in `clear_to_close` without the blocker check). Worth remembering when designing the Encompass CTC rule.

**C. Inbound ClickUp sync** also moves both statuses (crosswalk `src/clickup/crosswalk.js:100`, ingest) — the lifecycle is effectively **bidirectional with ClickUp** and ClickUp is where most real stage moves originate (the 38-status workflow is the ops team's working surface).

### 3.2 Role/capability model (`src/lib/permissions.js:49-59`)

| Role | Files visible | Lifecycle powers |
|---|---|---|
| `loan_officer` | Own/assigned only | `review_conditions` only — can "mark reviewed", can advance working statuses; **cannot** sign off conditions, cannot set approved/CTC/funded/declined |
| `processor` | Own/assigned only | `review_conditions` + `sign_off_conditions` (clears/waives-with-cap conditions); not decision statuses |
| `underwriter`, `loan_coordinator` | `see_all_files` | + `sign_off_conditions` (+ `waive_conditions` for underwriter) + decision statuses |
| `admin` / `super_admin` | all | everything, incl. **force** past blockers (`isAdmin`, `staff.js:114,4263`) |

Sign-off vs review is enforced at `src/routes/staff.js:2561-2572` (checklist) and `:2250-2261` (first-class clear — "Clearing (signing off) a first-class condition is the PROCESSOR/underwriter's call (audit S3-01) — a loan officer marks it REVIEWED instead"). Waive requires `waive_conditions` + a written reason (`:2283-2299`). The historical holes here (LO could clear conditions / move to funded) were documented in `docs/portal-audit/03-loan-officer-desk.md` (S3-01, S3-05) and are now fixed in code.

### 3.3 Multi-assignee model

A file's team is one **primary** LO + one primary processor (denormalized pointers `loan_officer_id`/`processor_id`) plus full-access **assistants** in `application_assignees`; the single access chokepoint is `VISIBLE_OFFICERS_SQL` (`src/routes/staff.js:126-140`). Officer↔ClickUp folder routing is in `docs/ROUTING.md` (real ClickUp folder IDs per officer; processors are pipeline-only; unassigned intake lands in "Lead Capture").

## 4. The condition machinery that feeds the gates

Two generations coexist:

1. **Checklist items** (`checklist_templates` / `checklist_items`): statuses `outstanding / requested / received / satisfied / issue` (`staff.js:2559`), audience `borrower/staff/both`, `item_kind` document/task/condition/tool, plus workflow columns from `db/005_rtl_workflow.sql:13-36`: `phase`, `role_scope`, **`is_gate`**, **`is_milestone`**, `signed_off_by/at`, per-item assignee.
2. **First-class `conditions`** (`db/022_conditions.sql`): severity `standard | prior_to_docs | prior_to_funding | post_closing`, status `open | borrower_responded | cleared | waived`, borrower-safe wording columns, linked-entity pointers, clear/waive workflow.
3. The **Condition Center** rule engine (`docs/condition-center.md`, `src/lib/conditions/engine.js`) auto-applies admin-authored conditions; it only evaluates files in `OPEN_STATUSES = [new … clear_to_close]` (`engine.js:33`) and re-runs on status change, details edit, and product registration.

### 4.1 The RTL 5-phase processing workflow (`db/005_rtl_workflow.sql:56-104`)

Seeded from the "RTL / Fix & Flip Loan Processing Guide"; LO runs intake/setup/numbers/attorney, processor runs verifications/orders:

- **Phase 1 — Borrower intake** (contract, ID, LLC docs, SSN, budget, ARV, title/insurance contacts…).
- **Phase 2 — File setup**: SharePoint folders, ClickUp task, **`rtl_p2_enc` "Encompass loan opened as Fix & Flip"** (`:74`) and **`rtl_p2_loprep` "LO Prep shows nothing missing in Encompass"** (`:75`) — today these are *manual* attestations.
- **Phase 3 — Verifications & orders**: **`rtl_p3_usps`** "USPS address verification completed" (hint: "Encompass → Services → Additional Services", `:77`); appraisal ordered/paid (milestone, `:78-79`); SOW; soft credit pull + **`rtl_p3_credit2` "Scores entered in Encompass + report saved"** (`:83`); **`rtl_p3_titleord`** title order (hint: "Encompass → Print → Custom Forms → Order Services", milestone, `:84`); insurance order; fraud report; liquidity/bank statements; REO/experience sheet.
- **Phase 4 — Appraisal back, check the numbers**: LTC/LTV/ARV vs guidelines; `rtl_p4_ts` "Term sheet generated — only after every number checks out" (**gate + milestone**, `:97`).
- **Phase 5 — Attorney & final review**: attorney email (milestone), **`rtl_f_review` "Final review requested"** (gate+milestone, hint "Last approval step before CTC", `:102`) and **`rtl_f_ctc` "CTC — Clear to Close received"** (gate+milestone, `:103`).

The `rtl_f_ctc` wording ("received") reflects the delegated/correspondent reality: CTC is *received from* the capital partner — mirrored in the ClickUp statuses `delegated ctc submission` → `ctc (4-email)`.

The vesting-entity LLC condition `rtl_p1_llc` was later promoted to a gate (`db/067_llc_condition_gate.sql:1-15`) so entity verification counts toward CTC.

### 4.2 Sign-off integrity gates (`signOffGate`, `src/routes/staff.js:2400-2540`)

Blocking a `satisfied`/sign-off with a 422 until reality agrees:

- A required **document** condition can never be signed off with zero non-rejected documents (super_admin-only override) (`:2428-2437`).
- **Insurance** needs binder + invoice; **title** needs a doc; **fraud** needs the background report, plus the criminal report on Gold files (`:2442-2466`).
- **Product** condition: a product must be registered (`:2486-2489`).
- **Rehab budget / SOW**: first-page budget = line-item total = file budget = registered product budget **to the cent**, plus Gold's ≥5% contingency (`:2490-2513`; DB trigger belt-and-suspenders `db/069`).
- **Experience**: claimed experience must be VERIFIED on the (3-year-window) track record before sign-off (`:2514-2539`) — "clear-to-close blocks until the claim is VERIFIED" (`staff.js:1425`).

Ad-hoc "raise an issue" conditions against a track-record line/LLC become real `both`-audience conditions idempotently (`src/lib/raise-issue.js:25-89`).

## 5. What "clear to close" means today — the precise definition

`clear_to_close` is reached only when **all four** of these align:

1. **Role**: actor has `see_all_files` (underwriter/coordinator/admin) — `staff.js:4256-4257`.
2. **Blockers empty** (`advancementBlockers(appId,'clear_to_close')`, `staff.js:3893-3923`):
   - no `conditions` row in `open`/`borrower_responded` with severity `standard` or `prior_to_docs` (`CTC_SEVERITIES`, `:3893`) — `funded` additionally adds `prior_to_funding` (`FUND_SEVERITIES`, `:3894`); `post_closing` never blocks;
   - no **required** checklist item of kind `document`/`condition` that isn't `satisfied`/signed-off (`:3909-3917`);
   - no **gate** checklist item (`is_gate=true`) unsatisfied (`:3918-3921`) — i.e. term sheet generated, LLC verified, final review, and the `rtl_f_ctc` item itself.
3. **Or admin force**, recorded as `forced=true` in the status history (`:4263-4264`, `db/027`).
4. The readiness widget (`GET /applications/:id/gating`, `:3926-3937`) surfaces this as "Clear-to-close ready" vs "N to clear before CTC" (`app-v2/src/screens/StaffApplication.jsx:2461-2465`, `app-v2/src/components/DealSnapshot.jsx:37,65`).

**Consequences of reaching CTC** (`STRUCTURE_LOCKED = ['clear_to_close','funded','declined','withdrawn']`):

- **Structural freeze (#84)** — `src/lib/file-lock.js:3-30`: product registration refused (staff `staff.js:1541-1542`; borrower `borrower.js:650-651`), rehab budget/SOW frozen (`staff.js:1795`, `borrower.js:959`), vesting entity locked (`staff.js:1371-1372`, `borrower.js:1231-1234`, `src/lib/vesting.js:23-30`). The only way back is deliberately moving the file to an earlier status (status changes themselves are not frozen).
- Borrower gets a **major-status email** (`staff.js:3888`).
- ClickUp inbound guards "never touch a Clear-to-Close-locked file" for vesting/economics (`src/clickup/ingest.js:1483`).

Caveat: the **internal-status door** (`staff.js:4298-4325`) re-derives `clear_to_close` from ClickUp statuses without running `advancementBlockers`, and the **inbound ClickUp sync** can set the status bucket too — so today "Encompass-style" ops movement (via ClickUp) can put a file in CTC even if portal conditions are open. The role gate holds; the blocker gate only guards endpoint A.

## 6. Where pricing locks

The portal has **no rate-lock request object** (nothing like lock expiry/extension). "Locking" is a chain:

1. **Frozen engines**: pricing math lives in the frozen browser engines, require()d verbatim server-side (`src/lib/pricing.js:16-23`) — the server "NEVER reimplements or alters engine math" and recomputes authoritatively "so a tampered client can never inject fabricated terms" (`:1-13`).
2. **Registration = commitment**: `POST /applications/:id/pricing/register` (staff `staff.js:1530-1682`; borrower `borrower.js:640-719`) recomputes the quote and calls `persistProductRegistration` (`src/lib/product-registration.js:74-162`), which (a) flips prior rows `is_current=false` and appends the new snapshot (append-only history per `db/025_product_registration.sql:1-27` — program, status ELIGIBLE/MANUAL/INELIGIBLE, note_rate, total_loan, full `inputs` + `quote` jsonb); (b) **writes the priced structure back onto the application** (loan_amount, rate_pct, ltv, rehab_budget, term, IR months/amount, ARV, assignment split, claimed-experience GREATEST — `:113-158`); (c) waives superseded product conditions and raises a staff `prior_to_docs` "Clear manual pricing exceptions" condition for MANUAL quotes (`:39-72`).
3. **Concurrency lock**: `econVersionFor(app)` fingerprints the file-owned pricing basis (`src/lib/pricing.js:401-424`); register 409s on mismatch so a stale studio session can never re-register old economics (staff `:1554-1559`, borrower `:657-662`).
4. **Re-open on drift**: DB triggers (`db/071`/`db/072`, per CLAUDE.md) reopen the `product_pricing` condition on ANY economics change and the SOW condition on budget change; every (re-)register itself reopens `product_pricing` to `received` for re-verification (`staff.js:1638-1644`) and re-syncs the dynamic liquidity condition (`:1652`) and the Gold SOW contingency (`:1656`).
5. **Privilege locks**: non-admin overrides of rate/leverage/experience are refused loudly rather than silently stripped (`:1565-1567`); non-see-all staff cannot raise ARV/as-is on a priced file (`:1574-1583`; also `PATCH /details`, `:3996-4004`); borrower register strips claimed-experience keys entirely (`borrower.js:665-670`); sticky per-file markup floors borrower self-repricing (`staff.js:1609-1619`, `pricing.js:131-140`).
6. **Terminal lock**: at CTC/funded/declined/withdrawn, registration is refused outright (`file-lock.js`), making the registered product the file's final economics unless the status is deliberately rolled back.

Borrower "economics" edits on a registered file become **change requests** approved/rejected by staff (`staff.js:2301-2333`).

## 7. Post-closing flow

- **Model**: `db/023_post_closing.sql` — `post_closing_items(application_id, code, label, status pending|ordered|received|accepted|exception, document_id, due_date, exception_note, assigned_staff_id)`, `UNIQUE(application_id, code)`. Deliberately separate from borrower intake ("trailing-doc tracking… Seeded when a file funds").
- **Seeding**: automatic on the transition to `funded` (`staff.js:4275`) or manual via `POST /applications/:id/post-closing/seed` (`:2091-2094`). The standard set (`staff.js:83-99`): `note` (Final executed note), `mortgage` (Recorded mortgage/deed of trust), `title_policy`, `settlement` (final CD/HUD), `closing_package`, `funding_confirmation`, `trailing_docs`.
- **Working it**: `GET /applications/:id/post-closing` (`:2084-2090`); `PATCH /post-closing/:pid` updates status/exception note/assignee with per-file auth + audit (`:2095-2112`). `post_closing_exceptions` is a dashboard KPI (`:75,706`).
- **Interplay with gating**: `post_closing`-severity conditions never block CTC or funding (`:3892`); the Condition Center engine stops evaluating funded files (`engine.js:33`).
- **ClickUp**: the whole post-closing tail (`in purchase review`, `purchase conditions`, `pa issued-post closing.`, `waiting for final docs`, `closed reconciled`, …) maps borrower-facing to `funded` (`status.js:56-63`) — the borrower simply sees "Funded" while ops reconciles.
- **Draws** are deliberately sandboxed: only a "Request a draw" email on funded files; the full draw workflow lives on a separate portal (CLAUDE.md session rule).

**Recommended architecture notes (Phase 1):**
- Enforce read-only structurally: an Encompass client module that only exposes GET/`loanPipeline` POST-query verbs (the one POST that is semantically a read), no PATCH/PUT/DELETE code paths at all — mirroring how `sharepoint.remove()` is a throwing no-op.
