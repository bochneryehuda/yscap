## 1. The Encompass milestone model

### 1.1 What a milestone is

A milestone is a step in the loan workflow that "defines loan activities and the role that carries out those activities." Encompass ships **13 predefined milestones**: *Started, Qualification, Processing, Submittal, Cond. Approval, Resubmittal, Approval, Doc Preparation, Docs Signing, Funding, Post Closing, Shipping, Completion* — but **"Your Encompass system administrator can configure the behavior (and the names) of these milestones, create new custom milestones as needed, and set up milestone templates."** ([Associates & Milestones reference](https://developer.icemortgagetechnology.com/developer-connect/reference/loan-associates-milestones))

Two consequences that drive the whole portal design:

1. **Milestone names are lender-configured, per instance.** Instance BE11397907's actual list must be *discovered*, never assumed. A typical retail sequence looks like Started → Processing → Submittal → Approval → Docs Signing → Funding → Completion, but YS Cap (hard money) may have renamed, added ("Clear to Close" is a common custom milestone), or archived entries.
2. **Encompass has its own built-in normalization layer** — every custom milestone maps to a small fixed "core" set (see §1.4). The portal can use this as a fallback bucket, but must not rely on it for a lender-specific signal like clear-to-close.

### 1.2 Milestone log entries (per-loan)

Each loan carries an ordered list of **milestone logs** — one entry per milestone in the template applied to that loan. As work completes, "the milestone is marked as finished, and work begins on the next milestone."

Read endpoints (Postman index lines 158–177):

| Purpose | V3 | V1 |
|---|---|---|
| List all milestone logs of a loan | `GET /encompass/v3/loans/{loanId}/milestones` | `GET /encompass/v1/loans/{loanId}/milestones` |
| One milestone log | `GET /encompass/v3/loans/{loanId}/milestones/{milestoneId}` | `GET /encompass/v1/loans/{loanId}/milestones/{milestoneLogId}` |
| Milestone-free role logs | `GET /encompass/v3/loans/{loanId}/milestoneFreeRoles` | `GET /encompass/v1/loans/{loanId}/milestoneFreeRoles` |
| All loan associates | — | `GET /encompass/v1/loans/{loanId}/associates` |

Milestone log fields (V1 API contract, confirmed via [V3 Update Milestone Log reference](https://developer.icemortgagetechnology.com/developer-connect/reference/v3-update-milestone-log) + EncompassRest `Loans/Milestones/LoanMilestone.cs`):

| Field | Type | Meaning |
|---|---|---|
| `id` | string (GUID) | Milestone log id (stable per loan) |
| `milestoneName` (v1) / `name` (v3) | string | **"Name of the milestone set by the Encompass administrator"** — retrieve-only |
| `startDate` | date-time | When the milestone started / is expected to start |
| `doneIndicator` | bool | **Finished flag** — the single most important lifecycle bit |
| `reviewedIndicator` | bool | Activities/documents reviewed |
| `expectedDays` / `actualDays` (v1); `days`, `duration` (v3) | int | Planned vs actual duration |
| `roleRequired` | bool/string | Whether a team member must be assigned before finishing |
| `loanAssociate` | object | Assigned user/group for the milestone's role (`loanAssociateType`, user `entityId`, name, phone, email) |
| `comments` | string | Free text grouped by milestone |
| `milestoneSetting` (v3) | entity ref | Points at the instance-level milestone setting (joins to §1.5 discovery) |

The loan entity itself also embeds `milestoneLogs`, plus `milestoneFreeRoleLog`, `milestoneHistoryLog` (audit of milestone changes), and per-log `stage` and `tpoConnectStatus` properties (EncompassRest `Loans/MilestoneLog.cs`).

**Finished vs unfinished:** finishing = `PATCH …/milestones/{id}` with `{"doneIndicator": true}`; unfinishing = `PATCH …?action=unfinish` (Postman index lines 162, 173–174). *Read-only takeaway:* `doneIndicator` can go **backwards** (staff roll a file back), so the portal sync must treat milestone state as fully replaceable on every pull, not append-only.

### 1.3 The `Log.MS.*` virtual fields

The collection's Loan Field Reader example (`POST /encompass/v1/loans/{loanId}/fieldReader`, folder *Loan Management / V1 Manage Loan*, request "03 - Loan Field Reader") reads exactly the loan-level milestone rollup fields we care about:

```
"CoreMilestone", "Log.MS.CurrentMilestone", "Log.MS.Stage", "Log.MS.LastCompleted"
```

(also present in the collection: `Log.MS.CurrentTPOConnectStatus`, `Log.MS.Comments.Approval` — the latter shows the `Log.MS.Comments.<MilestoneName>` per-milestone comment pattern used in the Audit Trail example.)

| Field ID | Meaning |
|---|---|
| `Log.MS.CurrentMilestone` | Name of the **current** (in-progress) milestone — lender-configured string |
| `Log.MS.LastCompleted` | Name of the last milestone whose `doneIndicator` flipped true |
| `Log.MS.Stage` | The loan's current **stage** (Encompass Web groups milestones into stages; also surfaced per-log as `stage`) |
| `CoreMilestone` | The instance-independent core bucket (see §1.4) |
| `Log.MS.CurrentTPOConnectStatus` | TPO Connect-facing status label (only relevant if TPO channel is used) |

These same rollups exist as **pipeline canonical fields** for bulk queries: the V3 Loan Pipeline report example filters on `Loan.CurrentMilestoneName` (value `"Started"`, matchType `MultiValue`) — Postman index lines 349–350 — and EncompassRest's `CanonicalLoanField.cs` confirms `Loan.CurrentMilestoneId` / `Loan.CurrentMilestoneName`. This is the cheap way to sweep the whole pipeline's milestone position in one POST (`/encompass/v3/loanPipeline`).

### 1.4 Core-milestone normalization (built into Encompass)

The V3 loan entity carries fixed-schema milestone rollups (EncompassRest `Loans/Loan.cs` + `Loans/Enums/MilestoneCurrentName.cs`):

- `milestoneCurrentName` — enum: **`Started`, `Sent to processing`, `Submitted`, `Approved`, `Doc signed`, `Funded`, `Completed`** (the "core milestone" buckets every custom milestone maps to)
- `milestoneCurrentDateUtc`, `milestoneStage`
- Per-core-milestone date pairs: `milestoneFileStartedDate`, `milestoneProcessedDate`, `milestoneSubmittedDate`/`…DueDate`, `milestoneApprovedDate`/`…DueDate`, `milestoneDocSignedDate`/`…DueDate`, `milestoneFundedDate`/`…DueDate`, `milestoneCompletedDate`/`…DueDate`, `milestoneDuration`

**Design significance:** even with fully custom milestone names, Encompass always exposes a normalized 7-bucket progression *with dates*. The portal should store BOTH the raw lender-configured names and the core bucket/dates — the core dates give durable "approved/doc-signed/funded/completed" timestamps that survive milestone renames. But note "Clear to Close" is NOT a core bucket, which is exactly why CTC detection must be config-driven (§8).

### 1.5 Discovering the instance's milestone list (Settings API)

`GET /encompass/v3/settings/milestones?includeArchived=True&view=Detail` and `GET /encompass/v3/settings/milestones/{milestoneId}` (Postman index lines 732–733; [Settings: Milestones reference](https://developer.icemortgagetechnology.com/developer-connect/reference/settings-milestones) — "retrieving lists of milestone settings currently configured in the system"). This is the **per-instance discovery call**: run it at integration bootstrap (and on a daily refresh) to learn BE11397907's ordered milestone list, then have an admin map portal statuses onto it (§8). The per-loan milestone log's `milestoneSetting` reference joins back to these ids.

### 1.6 Associates and milestone-free roles

- Every milestone can have a **loan associate** (the user/group responsible). `GET /v1/loans/{loanId}/associates` returns all of them in one call — the Encompass-side LO/processor/closer/funder roster for the loan.
- **Milestone-free roles** are roles "not associated with any milestone" ([Associates & Milestones reference](https://developer.icemortgagetechnology.com/developer-connect/reference/loan-associates-milestones)) — read via `GET …/milestoneFreeRoles`. Hard-money shops often run funder/servicer/draw-admin as milestone-free roles; pull these too so the portal can display the Encompass-side team.
- A user can hold multiple roles on the same loan; role assignment varies per loan.

## 2. Rate lock read model

### 2.1 Endpoints (Postman index lines 366–381)

Read-only subset (all V1):

- `GET /encompass/v1/loans/{loanId}/ratelockrequests` — list all lock requests (the lock history)
- `GET /encompass/v1/loans/{loanId}/ratelockRequests/{LockId}?view=detailed` — one request, detailed
- `GET /encompass/v1/loans/{loanId}/ratelockrequests/{LockId}/snapshot` — loan-data snapshot frozen at lock time

Write endpoints that define the lifecycle (never to be called; listed to model states): submit request (`POST …/RatelockRequests`), confirm (`PUT …/{LockId}/Confirmation`), deny (`…/denial`), cancel (`…/cancellation` or `?action=cancel`), extend (`?action=extend`), re-lock (`?action=relock`), revise (`?action=revise`), void (`…/{LockId}/void`), sell-comparison update (`?action=SellComparisonUpdate`).

### 2.2 Lock request lifecycle

Per the [Rate Lock Requests overview](https://developer.icemortgagetechnology.com/developer-connect/reference/rate-lock-management): "The Rate Lock Management API allows you to submit a rate lock request for a loan and view rate lock information, extend, re-lock and cancel an existing lock." A request "can remain in *requested* state until it is confirmed or denied by a user who is assigned the **Secondary Registration persona**"; locks are "subject to the Lock Desk hours and settings." Voiding "voids a pricing event (lock, relock, extension, denial, or cancellation) and returns the lock to its previous state."

State machine (read side):

```
requested ──confirm──▶ locked ──time──▶ expired
    │                    │   ├─extend──▶ locked (new expiration; prior row becomes "old lock")
    │                    │   ├─relock──▶ locked (new request row)
    ├─deny──▶ denied     │   └─cancel─▶ cancelled
    └─void (event rolled back to previous state)
```

### 2.3 Field model

From the [V1 Submit/Get Rate Lock Request reference](https://developer.icemortgagetechnology.com/developer-connect/reference/submit-a-rate-lock-request):

| Field | Values / meaning |
|---|---|
| `id` | Lock request id |
| `requestType` | `"lock"`, `"re-lock"`, `"extension"`, `"trade extension"`, `"cancellation"` |
| `requestStatus` | loan lock state before the action: `"not locked"`, `"active lock"`, `"cancelled lock"`, `"expired lock"` |
| `lockStatus` | per-request status: `"requested"`, `"locked"`, `"old lock"`, `"old request"`, `"expired"`, `"cancelled"`, `"extension requested"`, `"denied"`, `"registered"`, `"old registration"` |
| `requestedDate` / `fulfilledDate` | request created / actioned timestamps |
| `buySide` / `sellSide` / `comparison` | pricing objects; buySide carries `lockDate`, `lockNumberOfDays`, `lockExpirationDate`, rate/price/margin + adjustments (Postman bodies, index lines 526–593) |
| `lockExpirationDate`, `lockNumberOfDays` | expiration inputs |
| `requestIndicators` | flags: `lockExtension`, `lockCancellation`, `reLock` |
| `isCancellable` / `isExtendable` / `isVoidable` | capability flags |

**Loan-level rollup:** the loan entity's `rateLock` object exposes `rateStatus` with enum **`notLocked` / `locked` / `expired` / `cancelled`** (EncompassRest `Loans/RateLocks/RateStatus.cs`, `LockRequest.cs`; the collection's sample loan JSON shows `"rateStatus": "NotLocked"`), plus pending-flags `requestPending`, `extensionRequestPending`, `cancellationRequestPending`, `isCancelled`.

**LOCKRATE.\* virtual fields** (all present in the collection's field-reader example — one `fieldReader` POST returns the whole lock posture without parsing the request list): `LOCKRATE.RATESTATUS`, `LOCKRATE.REQUESTSTATUS`, `LOCKRATE.RATEREQUESTSTATUS`, `LOCKRATE.CURRENTSTATUS`, `LOCKRATE.CURRENTDATETIME`, `LOCKRATE.REQUESTED`, `LOCKRATE.REQUESTPENDING`, `LOCKRATE.EXTENSIONREQUESTPENDING`, `LOCKRATE.CANCELLATIONREQUESTPENDING`, `LOCKRATE.RELOCKREQUESTPENDING`, `LOCKRATE.ISCANCELLED`, `LOCKRATE.REQUESTTYPE`, `LOCKRATE.REQUESTCOUNT`, `LOCKRATE.BUYLOCKCOUNT`, `LOCKRATE.SELLLOCKCOUNT`, `LOCKRATE.CONFIRMATIONCOUNT`, `LOCKRATE.DENIALCOUNT`.

**Lock-expiry alerts (portal feature):** store `rateStatus`, current `lockExpirationDate` (buy side), and `lockNumberOfDays`; compute days-to-expiry locally and alert at configurable thresholds (e.g. 7/3/1 days) plus on observed transitions `locked→expired`, `locked→cancelled`, and on any `*RequestPending` flag appearing. Do NOT recompute expiry from `lockDate + days` — trust Encompass's `lockExpirationDate` (extensions/re-locks change it).

**Disambiguation:** *Resource Locks* (`/encompass/v3/loans/{loanId}/locks`, "Resource Lock" Postman folder) are **concurrency locks** (exclusive/NGSharedLock editing locks), not rate locks — irrelevant to lifecycle except as a reminder that a read-only integration never needs one: "a session-less lock is automatically applied, locking the loan only for the duration of the single API call" ([Loan Locks best practices](https://developer.icemortgagetechnology.com/developer-connect/docs/loan-locks-bp)).

## 3. Registration logs

`GET /encompass/v3/loans/{loanId}/registrationlogs` (Postman index lines 382–384). The registration log records the loan being **registered with an investor** (secondary marketing). Entity fields: `registrationDate`, `expirationDate`, `investorName`, `referenceNumber` (Postman bodies); the loan-schema log adds `currentIndicator` (the active registration), `registeredBy(Id/Name)`, `registeredDate`, `expiredDate`, `alerts`, `comments` (EncompassRest `Loans/RegistrationLog.cs`). The rate-lock `lockStatus` enum's `"registered"` / `"old registration"` values tie registration into the same secondary lifecycle. For YS Cap (note-sale to capital partners), registration expiration is a useful staff-facing signal — but **investor names must never surface to borrowers** (repo rule: capital-partner names are staff-only, `CLAUDE.md` "Never expose a note buyer").

## 5. Loan alerts

- **Postman collection bug:** the *Loan Alerts* folder's only request, "Get Good Faith Fee Variance Violations," points at `…/fundingFees` — a copy/paste error (verified in the raw JSON; same URL and description as the Loan Funding request). The real endpoint is `GET /encompass/v3/loans/{loanId}/goodFaithFeeVarianceViolations` (fee-tolerance/GFE variance alerts; supports `filter`/`sort` per the [changelog](https://developer.icemortgagetechnology.com/developer-connect/changelog)).
- Encompass loan alerts (rate-lock-expired, milestone-expected, compliance alerts…) are visible in the log; via API, **alert webhooks** exist but are gated: "Only Encompass Loan Alerts that are identified as a Compliance Alert Type can generate Webhooks," event `alertchange` under the loan resource, and the feature must be enabled by an ICE **support ticket** ([Loan Alert Webhooks doc](https://developer.icemortgagetechnology.com/developer-connect/docs/encompass-loan-alert-webhooks)). Non-compliance/custom alert types are "a future release."
- Log-entry `alerts` arrays (conversation logs, registration logs, milestone logs all carry `LogAlert` lists with `dueDate` + assigned `role`) are the in-file follow-up alerts — readable via the v3 loan entity views.
- **Recommendation:** don't build phase 1 on alert webhooks. Derive lock-expiry and milestone-stall alerts locally from pulled data (portal already has a notifications engine, `src/lib/notify.js`).

## 6. Conversation log & Consumer Engagement (minor for phase 1)

- **Conversation logs** (Postman index lines 202–204): `GET /encompass/v1/loans/{loanId}/conversationLogs` (+ per-id). Entries: `name`, `company`, `phone`, `comments`, `inLogIndicator`, `isEmailIndicator`, and `alerts` (dueDate + role). Read-only value: a staff-facing "last contact" feed; low priority.
- **Consumer Engagement** (index lines 21–61): Prospect Engagement (`POST /consumers/v1/invitations`, `/reminders`) and Loan Opportunities (`/loanopportunity/v1/...`) are the pre-application/POS layer — almost entirely POST/PATCH (writes) and not part of the loan lifecycle read model. **Out of scope for phase 1**; the portal is itself the POS.

## 7. Change detection (polling + webhooks)

- Webhook loan resource supports events "Create, Update, Submit, Change, Move, Document, Milestone" ([Webhook overview](https://developer.icemortgagetechnology.com/developer-connect/reference/webhook)); the `change` event takes JSON-path filter attributes — the documented milestone example is exactly `"filters": { "attributes": ["/milestoneLogs/*/doneIndicator"] }` ([Say Goodbye to Polling, Part 2](https://developer.icemortgagetechnology.com/developer-connect/docs/say-goodbye-to-polling-in-the-encompass-lending-platform-part-2)). Rate-lock request events also exist on the loan resource. Creating a webhook subscription (`POST /webhook/v1/subscriptions`) configures the integration, not loan data — it is compatible with "never writes TO loans," but strictest phase 1 can start with **pipeline polling**: one `POST /encompass/v3/loanPipeline` sweep on `Loan.CurrentMilestoneName` + lock canonical fields (`include=LockInfo` appears in the collection, index line 345), then per-loan detail pulls for changed files.
- Whether polling or webhooks: treat every pull as a **full snapshot replace** with a history append (milestones can unfinish, locks can void back to a prior state).

## 10. RECOMMENDED: lender-configurable clear-to-close detection (never hardcoded)

CTC is **not** a core milestone and has no fixed field — it is whatever milestone/state BE11397907's admin configured. Detection must be a config-driven mapping, exactly like `EXTERNAL_FOR` in `src/clickup/status.js`:

1. **Bootstrap discovery:** pull `GET /encompass/v3/settings/milestones?includeArchived=True&view=Detail` → present the instance's ordered milestone list in an admin UI.
2. **Admin mapping table** (`encompass_milestone_map`): `encompass_milestone_setting_id / name → portal_status` (one of the portal's existing `APP_STATUS` buckets, `src/routes/staff.js:3883`), with flags `is_ctc_signal`, `is_funded_signal`. Ship a *suggested* default mapping (fuzzy keyword: `/ctc|clear.?to.?close/i`, `/fund/i`, like `fallbackExternal` at `src/clickup/status.js:80-92`) that an admin must confirm — suggestions may prefill, only humans approve.
3. **Runtime rule:** portal CTC is allowed **only if** the mapped CTC milestone's log entry has `doneIndicator=true` (or it is `Log.MS.CurrentMilestone`/`LastCompleted`, per admin's choice of "reached vs completed" semantics), AND (belt-and-suspenders) the core rollup is at/after `Approved`. Enforce in `signOffGate`/`advancementBlockers` (`src/routes/staff.js:3889-3934`) as an additional gate: *portal cannot issue clear-to-close unless Encompass agrees*.
4. **Drift safety:** on every settings refresh, if a mapped milestone id disappears or a new unmapped milestone appears, open a review item (reuse the `sync_review_queue` pattern) and **fail closed** — an unmapped state never satisfies a CTC gate.
5. **Unfinish handling:** if the CTC milestone's `doneIndicator` reverts, the portal must surface a review alert (not auto-demote a decided file) — consistent with the portal's "suspicious changes go to review, never silently applied" doctrine.

## 11. Open questions for instance BE11397907

1. What is the actual configured milestone list (names, order, stages, archived) — and is there an explicit "Clear to Close" milestone, or is CTC represented by a custom field/alert instead?
2. Does YS Cap's hard-money workflow use rate locks/lock desk at all (business-purpose lending often doesn't) — if not, the lock panel should be feature-flagged off rather than showing "Not Locked" everywhere?
3. Which channel(s) are in use (Retail vs TPO) — determines whether `Log.MS.CurrentTPOConnectStatus` matters.
4. Are milestone-free roles used for funder/servicer, and should Encompass associates be displayed alongside portal assignees?
5. Webhooks vs polling for phase 1 — is creating webhook subscriptions acceptable under "architecturally incapable of writing" (it writes integration config, not loan data)?
