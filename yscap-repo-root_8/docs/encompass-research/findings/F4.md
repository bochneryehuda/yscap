# F4 — Rules Engine Integration Design: Encompass-Gated Rules in the Existing Conditions Engine

**Agent:** F4 · **Date:** 2026-07-19 · **Status:** DESIGN ONLY — nothing here is implemented; no repo files were modified.
**Inputs:** B2 (portal conditions engine), C5 (Encompass conditions/eFolder read contract), C6 (milestones/lifecycle), E2 (rule catalog & guard patterns) — all in `scratchpad/research/findings/`. Repo evidence re-verified read-only against `/home/user/yscap/yscap-repo-root_8`.

**Design center (from the assignment, stricter than E2 in one respect):**
1. Rules evaluate **at action time against the latest local Encompass snapshot** — *never* a live blocking API call in the request path. Stale snapshot ⇒ queue a refresh and apply the rule's staleness policy. (E2 proposed a bounded "refresh-then-decide" live call at hard gates; this design rejects that for Phase 1 — see §3.4 — and notes it as a possible Phase-2 relaxation.)
2. **Encompass agreement is a gate, not an actor.** No auto-clearing in either direction, ever: Encompass showing "Cleared" never clears a portal condition (it *unlocks* a human clear); a portal clear never writes to Encompass (structurally impossible — pull-only client per B2 §7.1). Phase 1 also does **not** auto-reopen portal conditions on Encompass drift (divergence from B2 §7.3.3, discussed in §5.5).
3. Phase 1 ships **4 rules** (+1 embedded linkage clause); the registry, predicate vocabulary, and trigger-point enum are designed for the full ~35-rule catalog in E2 §4.

## 2. Rule definition model (declarative registry)

### 2.1 The `encompass_rules` registry table

One row per rule; severity/tolerance/staleness are **config, not code** (SYNC-GUARD-MATRIX doctrine per E2: "a guard that can be a registry row should be a registry row"). Promoting or demoting a rule is a row update, not a deploy.

| Column | Type | Meaning |
|---|---|---|
| `rule_key` | text PK | Stable key, e.g. `condition_parity_signoff` |
| `label`, `description` | text | Admin-facing; `description` is also the template for block copy |
| `trigger_points` | text[] | Which portal chokepoints evaluate this rule (enum, §2.2) |
| `predicate` | jsonb | Declarative Encompass predicate (whitelisted vocabulary, §2.3) |
| `severity` | text CHECK | `hard_block` \| `soft_warn` \| `review` (§2.4) |
| `tolerance` | jsonb | Comparison tolerances, e.g. `{"money": 1.00, "percent": 0.001, "days": 3}`; consumed only by `field_within_tolerance` predicates |
| `staleness_max_age_sec` | int | Max acceptable age of the newest relevant snapshot for this rule (defaults §3.4) |
| `stale_behavior` | text CHECK | `block` \| `warn` \| `pass_with_note` — what a too-old snapshot does at this rule's severity (§3.4) |
| `override_capability` | text | Capability required to override (default `override_encompass_gate`); NULL ⇒ not overridable below super_admin |
| `mode` | text CHECK | `off` \| `shadow` \| `warn` \| `enforce` — per-rule rollout dial (shadow = evaluate + count, zero UI) |
| `scope` | jsonb NULL | Optional applicability filter (loan program, status range, category) — Phase 1 leaves NULL |
| `version`, `updated_by`, `updated_at`, `created_by`, `created_at` | — | Audit; every evaluation records the rule `version` it ran (§3.5) |

Global kill switch: env/config `ENCOMPASS_GATES = off | shadow | notice | enforce` caps every rule's effective mode (E2 §3.3) — one lever to stand the whole layer down if the feed misbehaves.

### 2.2 Trigger-point enum (where rules attach)

Small closed vocabulary; each value corresponds to exactly one code seam:

| Trigger point | Seam | Phase 1? |
|---|---|---|
| `condition_sign_off` | `signOffGate` branch (checklist PATCH `staff.js:2599`) | ✅ |
| `condition_clear` | `POST /loan-conditions/:cid/clear` (`staff.js:2250`) | ✅ |
| `condition_waive` | checklist waive branch + `/loan-conditions/:cid/waive` — **advisory only**; waive is the human escape hatch and is never hard-blocked | ✅ (soft) |
| `status_advance:clear_to_close` | `advancementBlockers` via PATCH `/applications/:id` (`staff.js:4261`) **and** `POST /internal-status` (`staff.js:4298` — gap closed) | ✅ |
| `status_advance:funded` | same seams, target `funded` | Phase 2 |
| `borrower_notify:ctc` | `notifyAppBorrowers` major-email chokepoint (`staff.js:4283`) | ✅ (inherits CTC rule) |
| `snapshot_refresh` | sync worker completion hook — drives drift rows + positive suggestions; **never blocks anything** (no user is present) | ✅ |
| `nightly_sweep` | scheduled job over all open linked files | Phase 2 |

Rule of the enum: **blocking triggers are always portal-initiated actions.** Inbound facts (ClickUp status arriving, Encompass drift observed) are flagged (`review`), never blocked — you cannot 409 a fact that already happened in another system (E2 §2.2).

### 2.3 Predicate vocabulary (whitelisted, like `rule_logic`)

The predicate is a jsonb tree with the same discipline as `src/lib/conditions/rules.js` (`checklist_templates.rule_logic`): a **closed set of predicate kinds** evaluated by pure code over the mirror tables — nothing eval'd, nothing interpolated; an unknown kind or malformed tree evaluates to `UNKNOWN` (never `PASS`). One combinator level: `{"all": [...]}` / `{"any": [...]}`.

Phase-1 kinds (five are enough for the whole Phase-1 set; the vocabulary grows additively for E2's families C–K):

| Kind | Shape | Semantics (evaluated against mirror only) |
|---|---|---|
| `loan_linked` | `{}` | `applications.encompass_loan_guid` set AND an `encompass_loans` snapshot row exists |
| `mapped_condition_terminal` | `{"match_mode":"all"}` | For the portal condition in trigger context: ≥1 **confirmed** mapping exists; every (match_mode=all) mapped Encompass condition has `is_removed=false` AND is terminal — V1: `status ∈ {Cleared, Waived}`; V3: `status_open=false` AND status ∈ the instance's configured terminal set (C5 §13.2); `status_date` ≥ portal condition `created_at` (stale-clear guard) |
| `ctc_milestone_done` | `{"semantics":"completed"}` | The admin-mapped CTC milestone (`encompass_milestone_map.is_ctc_signal`) shows `done_indicator=true` (`semantics:"completed"`) or is current/last-completed (`"reached"`) per C6 §10; **fail-closed** (`UNKNOWN`) if no milestone is mapped or a mapped id vanished from the last settings sync |
| `no_open_prior_to_closing_conditions` | `{"priorto_set":"ctc"}` | Zero non-removed mirrored Encompass conditions whose `prior_to` maps (via `encompass_priorto_map`) into the named bucket set are non-terminal — covers **Encompass-only** conditions the portal never materialized (§5.4) |
| `field_within_tolerance` | `{"portal_field":"loan_amount","encompass_field":"loan_amount","tolerance_key":"money"}` | Both values present and \|Δ\| ≤ tolerance; either side missing/unparseable ⇒ `UNKNOWN`, not `FAIL` (E2 C2 lesson: `actual_rate` is free text) |

### 2.4 Severity tiers (assignment's three, mapped to E2's four)

| Severity | Behavior | E2 equivalent |
|---|---|---|
| `hard_block` | 422 (`signOffGate` style) / 409 (`advancementBlockers` style) with plain-language reason + exact evidence + override affordance | BLOCK |
| `soft_warn` | Non-blocking modal/inline warning at the action ("Encompass disagrees — here's what it shows. Proceed?"); proceeding is journaled with the shown evidence; no email | CONFIRM |
| `review` | No interruption; creates/updates a mismatch-dashboard row + per-file chip; auto-closes on agreement | NOTICE + SILENT (dashboard `visibility: badge\|dashboard_only` sub-flag distinguishes them) |

Budget: **at most ~5 `hard_block` rules ever** (E2 §6.1). Phase 1 ships two.

### 2.5 Override policy (every hard block has exactly one door)

- **Capability:** new `override_encompass_gate` in `src/lib/permissions.js` CAPABILITIES (defaults: underwriter, admin, super_admin — grantable per user like every other capability at `permissions.js:30-62`).
- **Mechanics:** the 422/409 body carries `overridable: true` + `rule_key`. UI collects a **required reason** (mirrors the pushBack-reason and waive-reason precedents, `staff.js:2285-2299`); client re-submits with `encompass_override: {rule_key, reason}`. Server re-checks capability, re-evaluates (the state may have changed), then proceeds.
- **Audit (three writes):** (1) `audit_log` action `encompass_gate_override` with `{rule_key, portal_value, encompass_value, snapshot_age_sec, reason, rule_version}`; (2) an `encompass_gate_decisions` evidence row (§3.5); (3) for status advances, the existing `application_status_history.forced` flag plus detail `forced_over_encompass: true` (extends the pattern at `staff.js:4275`).
- **Value-aware stickiness (guard #67 per E2 §3.5):** the override binds to the exact `(portal_value, encompass_value)` pair captured at override time. If a later snapshot shows Encompass changed *again*, the gate re-arms — an old override never suppresses a new disagreement. For condition-parity overrides the pair is `(portal item id + intended status, encompass condition id + status + status_date)`.
- **Waive ≠ override.** Waiving remains the audited business decision "we accept this without Encompass agreeing" (`waive_conditions` + reason); override is "sign off as satisfied despite Encompass" — rarer, and reported. Weekly admin digest lists both; overrides are a dashboard KPI (a rule everyone overrides daily is a broken rule, not a broken team).
- Overrides are **per-row only, never bulk** (guard #64).

## 3. Evaluation flow

### 3.1 The evaluator

```
evaluateTrigger(trigger_point, ctx) -> [{rule_key, verdict, severity, message,
                                         evidence, snapshot_age_sec, overridable}]
  ctx = { application_id, checklist_item_id | condition_id, actor, target_status? }
```

Pure function over: rule registry rows whose `trigger_points` contain the trigger and `mode != off` → for each, resolve the predicate against **mirror tables only** (one batched query set per application, not per rule). Verdict per rule:

| Verdict | Meaning | Produced when |
|---|---|---|
| `PASS` | Encompass agrees | predicate true on fresh snapshot |
| `FAIL` | Encompass disagrees | predicate false on fresh snapshot |
| `STALE` | Snapshot older than `staleness_max_age_sec` | age check fails **before** predicate evaluation |
| `UNLINKED` | No `encompass_loan_guid` / no snapshot rows | routes to the linkage clause, not a false FAIL |
| `UNKNOWN` | Linked, but the mapped object/field is missing, unmapped, or unparseable | e.g. no confirmed condition mapping; unmapped CTC milestone |

`UNKNOWN` handling is per-rule: for the CTC rule it fails closed (an unmapped milestone can never satisfy a CTC gate — C6 §10.4); for the parity rule an unmapped condition means **not gated** (gating is opt-in via mapping, §5.3); for tolerance rules it means silence.

### 3.2 Request-path sequence (e.g. condition sign-off)

```
PATCH /api/staff/checklist/:itemId {signedOff:true}
  └─ signOffGate(itemId, actor)                    [staff.js:2400]
       ├─ existing branches (zero-doc, slots, product, budget, experience)
       └─ NEW: encompassGates.evaluateTrigger('condition_sign_off', ctx)
            ├─ all PASS (or rule mode=shadow/warn) → gate returns null → proceed
            ├─ any FAIL @ hard_block → 422 { error: "<plain sentence>",
            │      encompass: { rule_key, portal_value, encompass_value,
            │                   as_of, overridable: true } }
            ├─ any STALE @ hard_block, stale_behavior=block →
            │      enqueueEncompassPull(application_id, reason:'gate_stale')   ← queue, don't call
            │      422 "Encompass data for this loan is 47 min old (limit 30).
            │           A refresh has been queued — retry in about a minute,
            │           or override with a reason."
            └─ soft_warn FAIL/STALE → returned as warnings[] (not a 422);
                   UI shows confirm modal; confirmed proceed journaled
```

**No network I/O anywhere in this path.** `enqueueEncompassPull` inserts (idempotently — one open row per application, the `uq_sync_review_open` partial-index pattern from `db/108:42-45`) a high-priority `sync_queue` row with `target='encompass'`, `direction='pull'` (the queue already admits this, `db/schema.sql:306-307`; Phase 1 adds the CHECK that target='encompass' ⇒ direction='pull' per B2 §6). The background worker performs the GET, updates the mirror, and the user retries — typically <60s. This is the assignment's mandated posture; E2's "one bounded refresh-then-decide GET at hard gates" is noted as a Phase-2 UX relaxation to consider only if retry friction proves real, since it reintroduces Encompass latency/availability into the request path.

### 3.3 Non-blocking path (`snapshot_refresh`)

After each successful pull, the worker calls `evaluateTrigger('snapshot_refresh', {application_id})` for `review`-severity rules and reconciles the mismatch-row table (§6.3): open rows whose values now agree auto-close; new disagreements insert (deduped per value-pair); positive suggestions (ready-to-clear, §5.5) upsert. **This path never mutates conditions, statuses, or checklist items** — it only writes drift/suggestion rows and (rate-capped) notifications.

### 3.4 Staleness policy defaults

| Severity | Default `staleness_max_age_sec` | Default `stale_behavior` |
|---|---|---|
| `hard_block` (parity) | 1800 (30 min) | `block` (+ queue refresh) |
| `hard_block` (CTC) | 900 (15 min) | `block` (+ queue refresh) |
| `soft_warn` | 3600 (1 h) | `warn` — modal shows "as of HH:MM" |
| `review` | 21600 (6 h) | `pass_with_note` — rows carry `as_of`; a single portal-wide heartbeat banner (guard #16: staleness is one number, shown once) appears only when the whole feed is dead |

Freshness is measured against the **newest snapshot of the specific evidence family** the rule reads (`encompass_conditions.snapshot_at` for parity; `encompass_loans`/`encompass_milestones.snapshot_at` for CTC), not a global timestamp. Key principle preserved from E2 §3.4: Encompass being down **never freezes the portal** — hard blocks on stale data are overridable with a reason (`stale_proceed` journaled with age), and everything below hard_block degrades to silence + heartbeat.

### 3.5 Evidence & decision audit

Append-only `encompass_gate_decisions`: `(id, application_id, checklist_item_id|condition_id NULL, rule_key, rule_version, trigger_point, verdict, decision — proceeded|blocked|overridden|warned_proceeded, actor_id, evidence jsonb, snapshot_at, created_at)`. `evidence` is the **raw mirrored Encompass object(s)** consulted (condition JSON incl. `status/statusDate/isRemoved/tracking tail`, milestone log entry, field values) — immutable, so every historical block/pass/override can be replayed. The CTC pass stores the full evidence **bundle** (C5 §13.3.5): all conditions enumerated, milestone log, config versions.

## 4. Condition-matching model (portal condition ↔ Encompass condition)

### 4.1 Two-level mapping: template matchers + per-file bindings

**Level 1 — template-level matcher specs** (`encompass_condition_matchers`), authored by admins in the Condition Studio extension: `(id, template_code | conditions_source, matcher jsonb — {title_pattern, category, condition_type, api_family}, active, created_by/at)`. Matchers never gate anything directly — they only generate *suggestions*.

**Level 2 — per-file bindings** (`encompass_condition_map`), the rows the gate actually reads:

| Column | Notes |
|---|---|
| `id` | PK |
| `application_id` | FK |
| `checklist_item_id` NULL / `condition_id` NULL | exactly one set (CHECK, mirroring `chk_one_owner` at `db/schema.sql:252-256`) — supports both portal condition stores (B2 §1) |
| `encompass_condition_id` | Encompass GUID |
| `encompass_condition_api` | `v1_underwriting` \| `v1_preliminary` \| `v1_postclosing` \| `v3_enhanced` (per-loan discriminator `useEnhancedConditionIndicator`, C5 §1) |
| `encompass_title_at_link` | title snapshot — rename/drift detection; the binding pins the **id**, so later renames can't re-target |
| `match_mode` | `all` (default) \| `any` — one portal condition may bind several Encompass conditions; `all` requires every one terminal |
| `state` | `suggested` \| `confirmed` \| `rejected` |
| `suggested_by` | `matcher:<id>` \| `fuzzy` \| NULL (manual) |
| `confirmed_by / confirmed_at` | the human who approved |

Constraints: partial unique index — one **confirmed** binding per Encompass condition per application (an Encompass condition backs at most one portal condition; exceptions are a re-map, not a second row). Multiple Encompass conditions per portal condition allowed (match_mode governs).

### 4.2 Seeding & suggestion flow (admin-curated, confirm-first)

1. On each conditions pull, the matcher pass runs Level-1 specs + a conservative fuzzy pass (normalized title similarity + category equality) against unbound Encompass conditions.
2. Hits create `state='suggested'` rows — **never auto-confirm**, even at 100% title match (titles are free text; C5 §13.1: "Never auto-match on title alone"). Suggested rows render as a per-file "Suggested matches" list and a global review queue (G-D4 confirm-first precedent per E2 §3.6).
3. Staff with `sign_off_conditions` (per-file) or `manage_conditions` (global) confirm/reject; confirm requires `canTouchApp` like every per-file action.
4. Unbound in either direction is a first-class, non-error state: an unmapped **portal** condition simply isn't Encompass-gated (§4.4); an unmapped **Encompass** condition still counts in the CTC no-open-conditions clause (§5) and shows in the per-file Encompass panel.

### 4.3 NO auto-clearing in either direction — the gate/actor boundary

| Event | What happens | What NEVER happens |
|---|---|---|
| Encompass condition goes terminal (Cleared/Waived) | Portal condition gets a green **"Ready to clear — Encompass agrees"** chip (suggestion row); one click opens the *normal* sign-off flow, same capability checks, same `signOffGate`, audited as a human action | Portal condition status is auto-set; sign-off stamp auto-applied |
| Portal condition signed off | Nothing flows to Encompass (no write path exists) | Any Encompass write |
| Encompass condition reopens after portal sign-off | `review` drift row + one-click **pushBack** offer (existing verb, reason prefilled "Encompass reopened <title> on <date>") | Auto-reopen of the portal item (Phase 1 — see §5.5) |
| Encompass condition removed (`isRemoved=true`) | Binding flagged `review` ("mapped condition was removed in Encompass"); gate treats it as non-terminal (fail) | Binding silently dropped |

### 4.4 Which portal conditions are gated

Gating is **opt-in per condition**: a portal condition is Encompass-gated iff it has ≥1 `confirmed` binding **or** its template carries a non-null `encompass_gate` marker (B2 §7.2's template flag, which then *requires* a confirmed binding before sign-off — the marker makes "unmapped" itself a block: "This condition requires an Encompass match — link it first"). Phase 1 seeds the template flag on a handful of underwriting-critical templates only; everything else gates only when a human maps it. This keeps the default experience identical to today.

## 5. CTC gate specifics

### 5.1 The rule (`ctc_encompass_agreement`)

Predicate (all clauses AND; each independently reported in the blocker list):

```jsonc
{ "all": [
  { "kind": "loan_linked" },                                        // clause 0 (linkage)
  { "kind": "ctc_milestone_done", "semantics": "completed" },       // clause 1 (configurable milestone)
  { "kind": "no_open_prior_to_closing_conditions",
    "priorto_set": "ctc" }                                          // clause 2 (all PTC cleared)
]}
```

- **Clause 0 — linked loan required.** A file at CTC with no Encompass loan bound cannot be verified ⇒ block (E2 I0). The override ("this deal genuinely isn't in Encompass") is the documented exception path.
- **Clause 1 — configurable milestone.** CTC is not a core Encompass milestone (C6 §1.4); detection is config: bootstrap settings sync pulls `GET /encompass/v3/settings/milestones` → admin maps rows in `encompass_milestone_map` (`encompass_milestone_setting_id, name_at_map, is_ctc_signal, is_funded_signal, semantics reached|completed`) — suggestions prefilled by keyword (`/ctc|clear.?to.?close/i`), **humans approve** (the `src/clickup/status.js:21-98` two-layer pattern, per C6 §10). Fail-closed: no mapping, or a mapped setting id missing from the latest settings sync ⇒ `UNKNOWN` ⇒ block with "The Encompass clear-to-close milestone isn't configured/was changed — an admin must update the milestone mapping."
- **Clause 2 — all prior-to-closing conditions cleared.** Over the mirrored condition set for the loan (correct API family per loan): every `is_removed=false` condition whose `prior_to` maps into the `ctc` bucket set must be terminal (V1 `status ∈ {Cleared, Waived}`; V3 `status_open=false` + configured terminal status). The `prior_to → bucket` map (`encompass_priorto_map`) is config seeded at discovery — default `ctc` set = {Approval, Docs, Closing}, `funded` set adds {Funding} (mirrors the portal's own `CTC_SEVERITIES`/`FUND_SEVERITIES` split at `staff.js:3894-3895`; C5 §13.3 argued for including Funding at CTC — admin's call, one config row). This clause is what makes **Encompass-only conditions** (never materialized portal-side) still gate CTC.
- **Optional corroboration (config-off by default):** fieldReader `PCC.NOTCLEARED` empty — enable only after sandbox verification of its rendering (C5 §11); corroborating, never primary.

### 5.2 Enforcement points (all doors)

1. `advancementBlockers(appId, 'clear_to_close')` gains the `encompass` group → PATCH `/applications/:id` 409 body and the `/gating` readiness widget both show it automatically (`staff.js:4261-4266`, `:3926-3937`).
2. `POST /applications/:id/internal-status` (`staff.js:4298`) — currently **skips blockers entirely** — calls the same evaluation when the derived external status would enter `clear_to_close` (closes the pre-existing gap flagged by B2 §5.4; inbound ClickUp *sync* remains flag-not-block: a mismatch row, since the fact already exists upstream).
3. `borrower_notify:ctc` — even when status arrives via sync, the celebratory `major` borrower email (`staff.js:4283-4288`, `MAJOR_STATUSES` `:3888`) holds until the rule passes; a `review` row explains the held email.
4. Phase 2 (optional belt-and-suspenders, `db/069` precedent): DB trigger refusing `applications.status='clear_to_close'` unless a fresh `encompass_ctc_verified_at` stamp exists — pure SQL against a stamp column, no network in triggers.

### 5.3 Override

Admin `force:true` continues to work, but when the `encompass` blocker group is non-empty force additionally requires `override_encompass_gate` + a reason, and the history/audit rows record `forced_over_encompass: true` (extends `staff.js:4275`). Value-aware: re-arms if the Encompass evidence changes after the forced CTC.

### 5.4 Encompass-only conditions

Phase 1 does **not** materialize them as portal conditions (B2 open question 3 resolved conservatively): they appear read-only in the per-file "Encompass conditions" panel and they gate CTC via clause 2. Materialization (a new `origin_kind='encompass'`, CHECK extension at `db/037:69-73`) is a Phase-2 decision once mapping coverage is understood.

### 5.5 Regression after CTC (gate, not actor — applied consistently)

If a refresh shows a decided-CTC file no longer satisfies the rule (milestone un-finished — C6 §1.2 says `doneIndicator` can go backwards; new PTC condition added; condition reopened): **no auto-demotion, no auto-reopen.** A `review` row (kind `ctc_revoked`) + one targeted LO notification + one-click offer to move the file back through the normal, audited status PATCH (E2 G2). This deliberately diverges from B2 §7.3.3's 071-style auto-reopen recommendation: the 071/096 trigger family reacts to *portal-internal* economics edits (first-party facts); Encompass drift is *third-party evidence*, and per the assignment's "gate, not actor" rule it informs humans rather than acting. If shadow-period data shows staff reliably accept these suggestions, an auto-reopen promotion is a Phase-2 registry decision.

## 6. UI touchpoints

### 6.1 Per-file Encompass chip (staff file header)
One chip, three states: green **"Encompass: in agreement (as of 10:42 AM)"** · amber **"Encompass: 2 things to look at"** (expands to rows) · gray **"Not linked to Encompass"**. Never more than one banner per screen (E2 §6.2). The green state matters: staff *see* verification working when all is well.

### 6.2 Condition-row badges & the block dialog
- Mapped condition rows get a small status badge: `Encompass: Requested` (amber) / `Encompass: Cleared ✓` (green) / `Encompass: removed` (red), tooltip = title, status, statusDate, as-of.
- **Block dialog (422/409) always shows the exact evidence** — both values side by side, verbatim from the mirror: *"Encompass still shows **'Appraisal review'** as **Requested** (status date Jul 14, data as of 10:05 AM). ### 6.3 Mismatch dashboard (`/internal/encompass-drift`)
Separate table from `sync_review_queue` (different lifecycle — drift rows **auto-close on agreement**; review rows resolve by decision):

```
encompass_drift(id, application_id, rule_key, severity, portal_value, encompass_value,
                first_seen, last_seen, evidence_snapshot_at,
                state open|acknowledged|overridden|auto_closed,
                owner_staff_id, dismissed_value_pair jsonb)
```
Dedupe: partial unique index on `(application_id, rule_key, value_pair)` WHERE state='open' (the `uq_sync_review_open` pattern, `db/108:42-45`). Value-aware sticky dismissals (guard #67); snooze 7 days for `review` rows only; hard_block rows are never snoozeable. Filters: rule / severity / LO / age. Header KPIs: open by severity, oldest row, feed heartbeat (minutes since last successful sweep), overrides last 30d, unlinked files at underwriting+, shadow-mode would-have-fired counts. Notifications are state-transition-only, deduped, digest-capped (guards #66/#79): Phase 1 sends targeted LO notifications **only** for `ctc_revoked`.
