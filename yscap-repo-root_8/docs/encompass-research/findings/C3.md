Key architectural consequence: **a "GET-only" HTTP-method policy is impossible** — the three most important discovery reads (pipeline query, pipeline report, audit trail, fieldReader) are all POST. The read-only guard must be a **method+path allowlist**, e.g. allow POST only when the path matches `^/encompass/v3/loanPipeline(/report)?$`, `^/encompass/v1/loanPipeline$`, `^/encompass/v3/loans/[^/]+/auditTrail$`, `^/encompass/v3/loans/[^/]+/fieldReader$`; allow GET broadly under `/encompass/...`; deny PUT/PATCH/DELETE everywhere; deny `loanBatch` and `resourceLocks` for POST/DELETE explicitly.

### Why Batch Update is doubly dangerous
Batch Update applies the same field values to *many* loans at once, selected either by explicit `loanGuids` or by a **pipeline-style filter** (Postman "02 - Update Loans - with filter" uses the identical `filter` contract as the pipeline query). Per public docs, it requires Administrator access, is asynchronous, and **bypasses business rules, field triggers, Field/Loan Access rules, and does not run calculations** ([reference/batch-update](https://developer.icemortgagetechnology.com/developer-connect/reference/batch-update), [reference/get-status](https://developer.icemortgagetechnology.com/developer-connect/reference/get-status), [qualityexcellence.info write-up](https://qualityexcellence.info/blog/2023/03/17/using-encompass-api-to-batch-update-lender-name/)). A copy/paste error between a pipeline read body and a batch-update body could silently mass-overwrite the book. Mitigation beyond the allowlist: request an **API client (OAuth client) whose persona has no batch-update/admin rights** so the call fails 403 even if attempted.

### Why Resource Locks are banned even though they "protect" data
`POST /resourceLocks` with `{"resource":{"entityId":"{loanId}","entityType":"loan"},"lockType":"exclusive"}` **writes a lock record in Encompass**: an exclusive lock blocks loan officers in the app and other integrations from saving the loan until it is deleted ([docs/loan-locks-bp](https://developer.icemortgagetechnology.com/developer-connect/docs/loan-locks-bp), [reference/lock-resource-1](https://developer.icemortgagetechnology.com/developer-connect/reference/lock-resource-1)). An orphaned lock from a crashed worker would freeze production loan files. Reads never need a lock — per the Loan Locks best-practices page, locks exist to hold write intent across multiple calls; single-call updates get a transient "session-less" lock automatically (irrelevant to us since we never write). GET on `resourceLocks` is read-only and actually *useful* in Phase 1 (e.g., display "loan currently open/locked in Encompass" in staff UI).

## 2. Loan Pipeline API — the discovery workhorse

Docs: [docs/loan-pipeline](https://developer.icemortgagetechnology.com/developer-connect/docs/loan-pipeline), [reference/loan-pipeline](https://developer.icemortgagetechnology.com/developer-connect/reference/loan-pipeline), [reference/view-pipeline-with-pagination-1](https://developer.icemortgagetechnology.com/developer-connect/reference/view-pipeline-with-pagination-1) (V3), [reference/view-pipeline](https://developer.icemortgagetechnology.com/developer-connect/reference/view-pipeline) (V1), [reference/v3-create-cursor](https://developer.icemortgagetechnology.com/developer-connect/reference/v3-create-cursor) (V3 report cursor).

### 2.1 Read semantics despite POST
`POST /encompass/v3/loanPipeline` is a **query**: the JSON body is the search specification (fields to return, filter, sort). Nothing in Encompass changes. POST is used because the query contract is too rich for a query string. All pagination state lives either in `start`/`limit` query params or in a server-side cursor.

### 2.2 Data source and freshness caveat
The Pipeline API reads the **Encompass Reporting Database (RDB)**, which is updated **asynchronously** from the loan store — recent saves may not appear immediately ([docs/loan-pipeline](https://developer.icemortgagetechnology.com/developer-connect/docs/loan-pipeline)). Two consequences:
1. `Fields.####` canonical names only work for fields **provisioned into the RDB** by the Encompass admin (Reporting Database tool). Standard `Loan.*` canonicals always work. If YS Capital's hard-money custom fields (CX.*) must be discoverable/filterable, the Encompass admin must add them to the RDB first. Verify with `GET /encompass/v3/loanPipeline/canonicalFields`.
2. For gating decisions (e.g., "does Encompass show clear-to-close *right now*"), the pipeline result may lag; the authoritative check should re-read the live loan (`GET /v3/loans/{id}` or `POST .../fieldReader`) — pipeline is for *discovery*, not final verification.

### 2.3 Request body contract (V3, `LoanPipelineQueryBaseContract`)
Observed in Postman ("02 - Simple Query Pipeline" through "06b") and confirmed by [reference/view-pipeline-with-pagination-1](https://developer.icemortgagetechnology.com/developer-connect/reference/view-pipeline-with-pagination-1):

```jsonc
{
  "fields": ["Loan.Guid?", "Loan.LoanFolder", "Fields.4000", "Loan.LoanNumber",
             "Loan.LoanAmount", "Loan.LastModified", "Loan.BorrowerName"],  // columns to return
  "loanIds": ["<guid>", "..."],          // OR retrieve specific loans by GUID (no filter needed)
  "loanFolders": ["My Pipeline", "Prospects"],  // restrict search to named folders
  "filter": { /* QueryCriterionContract — see 2.4 */ },
  "sortOrder": [ {"canonicalName": "Loan.LastModified", "order": "Descending"} ],
  "orgType": "Internal",            // "Internal" (default) | "TPO"
  "tpoId": { ... },                 // only for TPO queries
  "loanOwnership": "AllLoans",      // "AllLoans" (default) | "MyLoans" (scoped to token's user)
  "includeArchivedLoans": true      // boolean; include loans in Archive folders
}
```

- Either `filter` or `loanIds` drives selection (`loanIds` mode returns those loans without pagination — Postman "04 - Retrieve using loanIds (no pagination)").
- `includeArchivedLoans` (Postman "05 - Include Archived Loans") is essential for a lender back-book: funded/old loans are commonly moved to Archive-type folders and are otherwise invisible to the query.
- `loanOwnership:"AllLoans"` + a service-account persona with pipeline access to all folders is required to see the entire book; admin persona also has documented performance benefits ([docs/loan-pipeline](https://developer.icemortgagetechnology.com/developer-connect/docs/loan-pipeline)).
- Sort: multi-key `sortOrder`; V3 uses `Ascending`/`Descending`, the V1 samples use `asc`/`desc`.

### 2.4 Filter syntax (`QueryCriterionContract`)
Two shapes, freely nestable (Postman "03 - Complex Query Pipeline" nests an AND inside an AND):

Leaf term:
```json
{ "canonicalName": "Loan.LastModified", "value": "2026-07-01", "matchType": "greaterThanOrEquals", "precision": "day", "include": true }
```
Composite:
```json
{ "operator": "and", "terms": [ <term|composite>, ... ] }
```

- **canonicalName** — `Source.Field` format; `Loan.*` = loan-store canonicals, `Fields.<fieldId>` = RDB fields ([reference/loan-pipeline](https://developer.icemortgagetechnology.com/developer-connect/reference/loan-pipeline)). Enumerate valid names via `GET /encompass/v3/loanPipeline/canonicalFields` (V3) or `GET /encompass/v1/loanPipeline/fieldDefinitions` (V1).
- **matchType** values (docs + Postman samples): `exact`, `equals`, `notEquals`, `greaterThan`, `greaterThanOrEquals`, `lessThan`, `lessThanOrEquals`, `contains`, `startsWith`, `isEmpty`, `isNotEmpty`, and `MultiValue` (value = array; Postman "05 - Simple Pipeline Query with Loan Folder" filters `Loan.LoanNumber` against a list). Source: [reference/view-pipeline](https://developer.icemortgagetechnology.com/developer-connect/reference/view-pipeline) and search-confirmed docs.
- **precision** — applies **only to date/dateTime values**: `exact` (default), `day`, `month`, `year`, etc. ([reference/view-pipeline-with-pagination-1](https://developer.icemortgagetechnology.com/developer-connect/reference/view-pipeline-with-pagination-1)).
- **operator** — `and` / `or`; **include** — boolean on a term (Postman "04 - Retrieve Only GUID" uses `"include": false` on the `(Trash)` folder term to *exclude* trash).
- Useful canonicals seen in samples: `Loan.LoanFolder`, `Loan.LoanNumber`, `Loan.LoanAmount`, `Loan.LoanRate`, `Loan.LastModified`, `Loan.BorrowerName`, `Loan.CurrentMilestoneName`, `Fields.4000/4002` (borrower first/last name), `Fields.CX.*` (custom, if in RDB).

### 2.5 Pagination — three modes

1. **Offset paging (`?start=&limit=`)** — simplest; V3 `start` is a zero-based index, `limit` the page size. **Max 1000 loans per page**, and **the server may override your `limit` downward** based on requested loans × fields ([reference/view-pipeline-with-pagination-1](https://developer.icemortgagetechnology.com/developer-connect/reference/view-pipeline-with-pagination-1) + docs search results). Each page re-runs the query — results can shift if loans change between pages, so sort on a stable key (e.g., `Loan.LastModified` ascending + `Loan.Guid`) during backfill.
2. **Random-access cursor (V1: `?cursortype=randomAccess`)** — first call creates a server-side **snapshot**; cursor ID is returned in the **`X-Cursor` response header** (Postman test scripts read `x-cursor` and store `pip_cursor_id`; total count comes back in **`X-Total-Count`**). Subsequent calls `POST /encompass/v1/loanPipeline?cursor={id}&start=3&limit=200` pass only the `fields` body — **the filter is cached server-side**. Snapshot = consistent paging even while loans change.
3. **V3 report cursor (`POST /v3/loanPipeline/report`, then `...?cursorId={id}`)** — same two-step snapshot pattern, explicitly "for report generation only; for all real-time pipeline results use `/v3/loanPipeline`" ([reference/v3-create-cursor](https://developer.icemortgagetechnology.com/developer-connect/reference/v3-create-cursor)).

**Cursor lifecycle limits** ([reference/v3-create-cursor](https://developer.icemortgagetechnology.com/developer-connect/reference/v3-create-cursor)): expire after **5 minutes of inactivity**, hard max lifetime **1 hour**, max **10 cursors per instance** (an 11th evicts the oldest-expiring). Design the backfill to drain a cursor promptly and treat cursor-expired errors as "recreate and resume by offset".

Other query params: `include=LockInfo` (V3 — piggyback current lock/alert info per loan; read-only), `ignoreInvalidFields` (default true — silently drops bad field names; consider setting `false` in CI-style validation so typos surface).

### 2.6 Response shape
An array of row objects, one per loan: loan identifier (loan GUID) plus a `fields` map of the requested canonicals → values. `X-Total-Count` gives the total matching count; `X-Cursor` the cursor ID when a cursor was created. (Shape per docs/Postman flows; capture an exact sample in the sandbox before coding the parser.)

### 2.7 V1 vs V3
Both remain published; V3 adds `loanFolders`, `includeArchivedLoans`, `orgType`/`loanOwnership`/`tpoId`, `include=LockInfo`, and the report-cursor variant, and is where ICE invests (26.x release notes). **Recommendation: build on V3 only**; keep V1 out of the allowlist to shrink surface area.

## 3. Loan Folders API

- `GET /encompass/v3/loanFolders` — lists **all folders the token's user can access, including Trash**; response includes folder name, folder type (Regular / Archive / Trash) and duplicate-check inclusion ([reference/get-list-of-loan-folders](https://developer.icemortgagetechnology.com/developer-connect/reference/get-list-of-loan-folders)).
- `GET /encompass/v3/loanFolders/{folderName}` — one folder's details.

Use at integration startup and daily thereafter to (a) build the `loanFolders` scope list for pipeline queries, (b) detect new folders an admin creates (a new LO pipeline folder must not silently escape sync), (c) identify Archive folders so `includeArchivedLoans` is used intentionally, and (d) exclude `(Trash)` (Postman samples filter it out with `Loan.LoanFolder` + `include:false`).

**Write cousins to forbid** (same area of the API, Loan Management folder, index line 337): `PATCH /encompass/v1/loanfolders/{folder}/loans?action=add` **moves loans between folders — WRITE, FORBID**.

`GET /encompass/v1/loans/{loanId}/metadata` (Postman "04 - Retrieve loan metadata", index line 334) returns loan-file metadata (create/modify information) without pulling the full loan payload — a cheap per-loan freshness probe when deciding whether a full re-fetch is needed. ## 5. Field Audit Trail API — the verification weapon

`POST /encompass/v3/loans/{loanId}/auditTrail` ([reference/pull-field-audit-data](https://developer.icemortgagetechnology.com/developer-connect/reference/pull-field-audit-data)). **Read-only despite the POST verb and the Postman name "04 - Create Audit Trail"** — it *pulls* field audit data from the Reporting Database.

- Body: `{"fieldIds": ["4000", "1109", "Log.MS.Comments.Approval"]}` — standard field IDs, and log-style IDs work (milestone comment log in the sample).
- Query params: `includeHistoricalData` (default false = **latest change only**; true = full change history per field), `ignoreInvalidFields` (default false), `start`, `limit` (**max 1000** records).
- Response: per change — the user (ID and name) who modified the field, the timestamp, and value(s) ([docs/audit-trail page](https://developer.icemortgagetechnology.com/developer-connect/docs/audit-trail); Encompass admin help confirms each audit record = date of change, ID and name of user, and the value the field was changed to — [help.icemortgagetechnology.com audit_trail](https://help.icemortgagetechnology.com/admintools/16.1.0/Content/audit_trail.htm)).
- **Prerequisite (critical open item):** audit trail is only captured for fields the Encompass **System Administrator has flagged for audit in the Reporting Database tool** ([admin help](https://help.icemortgagetechnology.com/admintools/16.1.0/Content/audit_trail.htm)). Before the portal's "condition cannot clear unless Encompass agrees" rules rely on audit data (who set CTC field, when), YS Capital's Encompass admin must confirm/enable audit trail on the specific fields (e.g., 2301/CTC-related, milestone dates, key underwriting fields). Fields never audited return nothing — the portal must degrade to value-comparison-only verification in that case.

Use cases for the portal rules engine: prove *who* cleared a condition-relevant field and *when*; detect retroactive edits after a portal approval; timestamp-order checks (Encompass change must precede portal clear).

## 6. Rate limits, 429s, payload sizes

- Encompass does **not** use classic `X-Rate-Limit` requests-per-window headers; it enforces **concurrency limits**: default **30 concurrent API calls per lender environment**, shared across ALL integrations (partner + lender initiated) ([docs/concurrency-limits](https://developer.icemortgagetechnology.com/developer-connect/docs/concurrency-limits)).
- Every response carries **`X-Concurrency-Limit-Limit`** and **`X-Concurrency-Limit-Remaining`**. Guidance: lender-initiated usage should stay under **80%** of the limit ((Limit−Remaining)/Limit < 0.8); ISV partners under 20%.
- Exceeding the limit returns **429 Too Many Requests**, and 429s continue until remaining > 0. Recommended handling: **exponential backoff (with jitter) + proactive metering** of the headers. Limits can be raised via ICE (commercial discussion).
- **Response payload cap: 6 MB** at the API Gateway; exceeding it returns **400 "Response exceeds maximum size (6 MB)"** — the fix is smaller `limit`/fewer `fields`, not retries ([docs/response-payload-size-limit](https://developer.icemortgagetechnology.com/developer-connect/docs/response-payload-size-limit)). This is why the server may override pipeline `limit`: it sizes pages to loans×fields. A separate "Loan Size Limit" doc governs full-loan payloads (out of C3 scope).

## 8. Recommended discovery strategy for YS Capital's existing book

### 8.1 One-time initial backfill
1. **Folder census** — `GET /v3/loanFolders`; classify Regular vs Archive vs Trash. Persist the folder list.
2. **Canonical-field census** — `GET /v3/loanPipeline/canonicalFields`; verify every field the portal needs (loan number, amount, rate, milestone, CTC indicator, CX.* hard-money fields) is queryable; file an Encompass admin request for any missing RDB/custom fields and for audit-trail enablement on verification-critical fields.
3. **GUID sweep** — `POST /v3/loanPipeline?start=0&limit=1000` with a minimal field list (`Loan.Guid`-equivalent id, `Loan.LoanNumber`, `Loan.LoanFolder`, `Loan.LastModified`, `Loan.CurrentMilestoneName`, `Loan.BorrowerName`), filter = exclude `(Trash)`, `includeArchivedLoans: true`, `loanOwnership: "AllLoans"`, sorted by `Loan.LastModified` ascending. Page by `start` until `X-Total-Count` is exhausted. For a hard-money book of ~500–5,000 loans this is **1–5 calls** (wider field sets: still ≤ ~25 calls; watch the 6 MB cap and let the server shrink `limit`).
4. **Match & link** — join to `applications.ys_loan_number`; queue ambiguous matches to the human review queue; store `encompass_loan_guid` on the application row (new column, another agent's scope).
5. **Deep fetch** — per-GUID full loan reads (C-other agents' scope: `GET /v3/loans/{id}`) fed from the GUID sweep, throttled (see 8.3). ~1–2 calls per loan → for 2,000 loans ≈ 2,000–4,000 calls, drainable in well under an hour at modest concurrency.

### 8.2 Incremental updates (steady state)
- **Watermark poll**: every 5–15 minutes, `POST /v3/loanPipeline` with filter `Loan.LastModified >= (watermark − 30 min overlap)` (RDB lag safety), `includeArchivedLoans: true`, minimal fields. Dedupe by GUID+LastModified. Typical result: 0–50 changed loans per poll for a shop this size → **1 call per poll**, ~100–300 calls/day at a 5-min cadence.
- **Targeted re-verification** (rules engine): when a portal user tries to clear a condition or issue CTC, do a **live** check — `POST /v3/loans/{id}/fieldReader` (or GET loan) for current values, and `POST /v3/loans/{id}/auditTrail` (`includeHistoricalData=true` on the few relevant fieldIds) for who/when. 2 calls per verification event; even 200 verifications/day ≈ 400 calls — negligible.
- **Nightly reconciliation**: full GUID sweep (as in 8.1 step 3) + folder census diff, comparing against the local mirror to catch moves to Archive/Trash, deletions, and any events the watermark missed. ~5–25 calls/night.
- **Do not poll per-loan**: never loop `GET /loans/{id}` across the book on a timer; the pipeline watermark query is the fan-in. (Encompass webhooks are the longer-term push alternative — out of C3 scope, flagged for the webhooks agent.)

**Total steady-state volume estimate: ~500–1,500 calls/day**, dominated by the watermark poll — comfortably inside a 30-concurrency environment *if* serialized properly.

### 8.3 Throttling design
- **Single global gate**: one process-wide semaphore for ALL Encompass calls (backfill, polls, verifications) with max **4–6 concurrent** requests — deliberately far below 30 because the limit is shared with every other integration the company runs (LO web sessions' integrations, disclosures vendors, etc.).
- **Header-driven adaptation**: after every response, read `X-Concurrency-Limit-Remaining`/`X-Concurrency-Limit-Limit`; if utilization across the environment exceeds ~50%, halve local concurrency; resume gradually.
- **429 handling**: exponential backoff with jitter (e.g., 1s, 2s, 4s… cap 60s), retry idempotent reads only (all our calls are reads, so all retryable), circuit-break after N consecutive 429s and alert.
- **6 MB / 400 handling**: on payload-size 400, halve `limit` (or split `fields`) and retry — never blind-retry.
- **Cursor hygiene**: prefer offset paging for the small book; if using report cursors, drain within 5-min inactivity / 1-hour lifetime and never hold >2 cursors (instance max 10, shared).
- **Priority lanes** in the worker: user-facing verification calls (condition clear / CTC gate) preempt background backfill/poll jobs — same interval-worker pattern as `src/sync/queue.js` with `run_after` backoff columns.
