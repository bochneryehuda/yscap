## 1. Executive summary

- **Base Developer Connect API access is an Encompass-client capability, self-provisioned** by a Super Admin from the developer portal (My Account > API Key); ICE does not publish a separate list price for base API access, and no public source states it is separately licensed. However, **several capabilities are premium/entitled**: Webhook **Custom Auth** is explicitly labeled "Premium" (the 26.2 Postman collection itself has a folder `Webhook Custom Auth - Premium`), the **Enhanced Field Change (EFC)** webhook requires an ICE support-ticket enablement per instance (R2T first, then production), concurrency-limit increases are a "pricing" discussion, ICE BI Dashboard flags need feature enablement, and bulk data replication is sold as a separate product (**Encompass Data Connect**).
- **Releases are now roughly quarterly majors** (25.1, 25.2, 25.4, 26.1, 26.2...) with monthly service packs and critical patches; each major ships to a **Release-to-Test (R2T)** environment ~4–8 weeks before production. Majors routinely carry **breaking changes** (attribute removals, null-vs-empty-string semantics, response truncation defaults) and a steady deprecation/sunset drumbeat (V1 eFolder Attachment APIs sunset in 26.3; SDK retirement extended to Dec 31, 2026 with paid "transitional access" after).
- **Biggest platform risks for YS Capital**: (1) discovering late that EFC/premium webhooks or a test instance are not entitled; (2) breaking changes in quarterly majors hitting a compliance-critical gate (portal CTC mirrors Encompass CTC); (3) the default **30 concurrent API call** limit per lender environment being consumed by naive polling; (4) Encompass outages making the portal's "Encompass-must-agree" rules block operations unless the portal degrades gracefully.

Repo context: the portal already anticipates this integration — `sync_queue` is described as the "ClickUp/Encompass, later" retry queue (`/home/user/yscap/yscap-repo-root_8/src/sync/queue.js:4`, `/home/user/yscap/yscap-repo-root_8/README.md:100`, `/home/user/yscap/yscap-repo-root_8/CLAUDE.md:45`, `/home/user/yscap/yscap-repo-root_8/package.json:5`). No Encompass code exists yet, so all licensing/enablement decisions are still open.

## 2. How API access is licensed for lenders

### 2.1 Base access
- The developer portal states Developer Connect "is intended for ICE Mortgage Technology Encompass® clients only" ([Welcome](https://developer.icemortgagetechnology.com/developer-connect/docs/welcome)).
- The **API key (Client ID + Client Secret)** is obtained by an Encompass user with the **super administrator persona** from the portal under My Account > API Key; "an API key is linked to an Encompass Client ID, which means that it cannot be used across all Encompass instances" — i.e., one key per instance ([Get an API Key](https://developer.icemortgagetechnology.com/developer-connect/docs/get-an-api-key)). Regenerating the secret "breaks any current Developer Connect applications using this API key."
- The API key grants no extra data rights: "The features and data users can access with the APIs is determined by their assigned Encompass persona" ([API Keys](https://developer.icemortgagetechnology.com/developer-connect/docs/get-an-api-key)).
- **No public source found** that says base Developer Connect API usage is metered or separately invoiced for a lender's own instance. Encompass itself is subscription/enterprise-priced with negotiated seats; ICE does not publish pricing ([mortgagetech.ice.com/products/encompass](https://mortgagetech.ice.com/products/encompass), [Software Advice profile](https://www.softwareadvice.com/product/133471-Encompass360/)). Treat "base APIs included" as **probable but unconfirmed — confirm in the YS Capital order form/master agreement**.
- Auth grant nuance: ICE docs state "grant type client_credentials is for ISV partners only" ([Authorization](https://developer.icemortgagetechnology.com/developer-connect/docs/authentication)); the Postman collection nevertheless includes both "Resource Owner Password Credentials" and "Client Credentials (API User)" under Authentication (endpoint index lines 7–8). YS Capital should confirm which grant its lender-side service account is expected to use (likely password grant with a dedicated API user) — this affects secret-management design.

### 2.3 API users (service accounts) and seats
- API users are created in Encompass admin (Company/User Setup) with the **"API User" checkbox**, must be assigned a **persona**, and inherit only that persona's rights ([Stitchflow Encompass API guide](https://www.stitchflow.com/user-management/encompass/api), [Jungo setup instructions](https://ijungo.com/encompass-api-admin-instructions/), [Proof setup article](https://support.proof.com/hc/en-us/articles/1500003014402-Set-up-API-users-in-Encompass)).
- Third-party guidance reports that users flagged as API User **"count against the Lender's enabled user license limits"** and that seat counts are negotiated at contract time ([Stitchflow](https://www.stitchflow.com/user-management/encompass/api)). I could not confirm this in an official public ICE page — **verify with the ICE account manager** whether a dedicated read-only API user consumes a billable seat on YS Capital's contract.
- Best practice for Phase 1: one dedicated API user with a **custom read-only persona** (no loan-edit rights) — this is also the strongest architectural "cannot write" control and doubles as a licensing question (does a restricted persona seat cost less?).

## 3. Environments: test/sandbox vs production

- **Partner Connect sandboxes are for ISV partners**, not lenders: partners get one sandbox + two production key pairs; "Transactions originating from sandbox environments will be considered mock, non-billable transactions" ([EPC Getting Started](https://docs.partnerconnect.elliemae.com/partnerconnect/docs/what-do-i-need-to-get-started)). That page also references "a Lender's sandbox/test instance," confirming lender test instances exist as a concept.
- **R2T ("Release to Test") instances**: ICE ships each major release to R2T weeks before production — e.g., "the scheduled Release to Test (R2T) of Encompass 25.1 on February 15, 2025" (production 25.1 landed later), and 25.3's R2T on July 12, 2025 ([25.1 release notes](https://developer.icemortgagetechnology.com/developer-connect/changelog/251-major-release), [25.3 release notes](https://developer.icemortgagetechnology.com/developer-connect/changelog/253-major-release)). EFC enablement explicitly assumes the lender has an R2T instance ([EFC How to Enable](https://developer.icemortgagetechnology.com/developer-connect/docs/efc-webhook-how-to-enable)).
- **No self-serve developer sandbox for lenders is publicly documented**; whether YS Capital already has (or must purchase) a test/R2T instance alongside production `BE11397907` is a contract question. Industry convention is that test instances carry a distinct instance-ID prefix (commonly reported as "TEBE" vs production "BE"); I could not verify the prefix in official public docs — confirm with ICE.
- **Implication**: if YS Capital has no test instance, the first integration build would run against production data (real borrower PII) — a significant GLBA/testing risk given the portal's PII posture (`/home/user/yscap/yscap-repo-root_8/CLAUDE.md:45` — SSN encryption, audit_log GLBA trail).

## 4. Release cadence and how APIs change

### 4.1 Cadence
From the official [Release Notes log](https://developer.icemortgagetechnology.com/developer-connect/changelog/release-notes-log):

| Release | Date |
|---|---|
| 26.2 August Service Pack Preview | Aug 8, 2026 (scheduled) |
| **26.2 Major Release** | **Jul 11, 2026** |
| **26.1 Major Release** | **Mar 20–21, 2026** |
| 25.4 Major Release | Nov 8, 2025 |
| 25.2 Major Release | May 17, 2025 |
| 24.3 Major Release | Dec 7, 2024 |
| 24.1 Major Release | Apr 6, 2024 |
| 23.3 Major Release | Oct 7, 2023 |
| 22.3 Major Release | Oct 15, 2022 |

Pattern: ICE moved from ~3 majors/year (through 2024) to ~**4 majors/year** (25.1–25.4), each preceded by an R2T drop, with service packs and critical patches in between. The Postman collection YS Capital holds is the **26.2** vintage (current as of July 2026).

### 4.2 Breaking changes are routine
The [Breaking Change Notices](https://developer.icemortgagetechnology.com/developer-connect/docs/breaking-change-notices) page shows the flavor of change a read-only consumer must absorb:
- 26.3 (upcoming): Get External Users returns minimal data and only 10 records by default without pagination params; obsolete rate-lock entities consolidated into `RateLocks`; entity type value renames ("Organization" → "ExternalOrg").
- 26.2: `fundingFees` attribute **removed** from loan contract responses; custom fields return `null` instead of empty string when no value.
- 26.1: `resolutionComment` capped at 1,000 chars (effective Apr 16, 2026); super-admin persona now required to retrieve Custom Data Objects.

These are exactly the kinds of silent semantic changes (null vs "", attribute removal, default truncation) that can corrupt a condition-matching/CTC-mirroring rules engine if the sync layer doesn't validate contracts per release.

### 4.3 Deprecation & sunset history
From [Deprecation and Sunset Notices](https://developer.icemortgagetechnology.com/developer-connect/docs/deprecation-notices) (definitions: *Deprecated* = still works, unsupported, will eventually sunset; *Sunset* = removed):

| Item | Deprecated | Sunset | Replacement |
|---|---|---|---|
| `createAuthCode` method | 23.3 (Oct 7, 2023) | 27.1 | `getAccessToken()` |
| **V1 eFolder Attachment APIs (6 endpoints)** | earlier | **26.3** | V3 attachment APIs (some without direct replacement) |
| TPO Connect app-object methods (getLoanSummary, getLoanData, createAuthCode) | — | 26.2 | Loan/Auth objects |
| V1 Loan Importer (Create Loan) | 26.1 | TBD | V3 Converter import |
| eFolder document `status` attribute | 26.1 | TBD | `documentStatus` |
| Partner Services (Appraisal/Verifications/Flood) | 25.4 | TBD | EPC Service Ordering |
| Task Group APIs (14 endpoints) | 24.3 | TBD | feature being redefined |
| EPPS V1 APIs | 21.2 | Jun 5, 2021 | EPPS V2 |

**V1 Loan APIs status**: per ICE's own [V1 vs V3 guide](https://developer.icemortgagetechnology.com/developer-connect/docs/v1-vs-v3-encompass-apis-whats-the-difference-1): "Although there have not been any formal announcements of sunsetting the Loan V1 APIs for cases where there is V3 parity, over time the V1 APIs will be deprecated." V1 loan APIs are REST wrappers over legacy SOAP/WCF; V3 are true REST with better performance. **Phase 1 should be V3-only wherever parity exists.**

### 4.4 SDK sunset (precedent for platform-policy risk)
- ICE announced (Sept 2024) retirement of the Encompass SDK and legacy service ordering with deadline **Oct 31, 2025** ([ICE transition page](https://mortgagetech.ice.com/explore/encompass-legacy-service-ordering-and-sdk-transition-deadline-october-31-2025)), then **extended to Dec 31, 2026**; no new SDK features after Nov 1, 2025; post-deadline "SDK Transitional Access" needs special approval and **monthly fees based on SDK activity starting January 2027** ([HousingWire](https://www.housingwire.com/articles/ice-delays-encompass-sdk-transition-until-end-of-2026/), [National Mortgage News](https://www.nationalmortgagenews.com/news/ice-mortgage-technology-pushes-sdk-sunset-to-end-of-2026), [Lender Toolkit](https://lendertoolkit.com/encompass-sdk-and-epc-transition-deadlines-extended-to-december-31-2026/)).
- Takeaway for YS Capital: building on **Developer Connect REST is the strategically safe side** of this transition (it is the designated successor), but the episode shows ICE will retire integration surfaces and attach fees to legacy access — plan for API-surface churn contractually and architecturally.

## 5. Support channels

- **ICE Resource Center**: 24×7 knowledge base + case submission; standard live support Mon–Fri roughly 8am–6/8pm ET; **24×7 for production-critical severity** (except Thanksgiving/Christmas) ([ICE support page](https://mortgagetech.ice.com/support), [technical support](https://icemortgagetechnology.com/support/technical-support)).
- **Premier Support / Premier Services**: paid tier ICE markets for faster/deeper help (e.g., Premier Support customers can request SDK usage analysis) ([ICE support](https://mortgagetech.ice.com/support), [HousingWire SDK coverage](https://www.housingwire.com/articles/ice-mortgage-technology-encompass-sdk-to-api-transition-2025/)).
- **Professional Services (PSO)** engagements via account manager for migration/integration builds.
- **Developer Forum is being sunset June 25, 2026**: the Community menu is removed from the Developer Connect site in release 26.2; blogs move under Guides ([Deprecation notices](https://developer.icemortgagetechnology.com/developer-connect/docs/deprecation-notices), [changelog](https://developer.icemortgagetechnology.com/developer-connect/changelog)). Net effect: community/self-help for API developers is shrinking; formal support tickets (and the enablement-ticket path used for EFC) become the primary channel.
- Feature enablement (EFC etc.) happens through support tickets with **pre-determined change dates** — lead time must be planned.

## 6. Outage / status posture

- Official status: **emstatuscenter.elliemae.com**, which now 301-redirects to the consolidated **[ICE Status Center](https://www.ice.com/status)** (Mortgage Technology section) covering "service disruptions, scheduled maintenance, or performance issues."
- Third-party trackers give outage history: e.g., IsDown recorded an Encompass (Web) incident Aug 10, 2025, and an "Encompass is experiencing issues" incident Mar 6, 2026; recent 90-day window showed 1 incident, median duration ~2h18m ([IsDown Encompass page](https://isdown.app/status/ice-mortgage-technology/encompass), [incident](https://isdown.app/status/ice-mortgage-technology/incidents/429112-encompass-web-version-is-experiencing-issues), [StatusGator](https://statusgator.com/services/ellie-mae)).
- Scheduled maintenance windows exist during which "access to Encompass, Encompass services… may be unavailable" ([status center](https://emstatuscenter.elliemae.com/)). Quarterly major-release weekends are effectively planned downtime/behavior-change windows.
- **Design consequence**: portal rules like "cannot clear a condition unless Encompass agrees" and "no CTC unless Encompass shows CTC" must define behavior when Encompass is unreachable or stale: fail-closed for CTC issuance (safe), but with explicit staleness indicators and cached last-known-good data so operations aren't blind during a 2-hour outage.

## 7. Terms-of-use / data-extraction constraints

- The public developer portal does **not** expose a standalone API terms-of-use document; API usage is governed by the Encompass master agreement plus the portal's login-gated terms. This must be obtained from ICE/legal — flagged as an open question.
- Enforced technical constraint: **default concurrency limit of 30 concurrent API calls per lender environment**; breach returns **429 Too Many Requests** until capacity frees; responses carry `X-Concurrency-Limit-Limit` / `X-Concurrency-Limit-Remaining`; lenders are told to keep utilization ≤80% (ISV partners ≤20% of the lender's budget); increases are a relationship-manager "pricing" conversation ([Concurrency Limits](https://developer.icemortgagetechnology.com/developer-connect/docs/concurrency-limits)). Per-minute rate limits are **not publicly documented**.
- API-based extraction is a **documented, sanctioned use case**: ICE's own "[Loan Data Extracts](https://developer.icemortgagetechnology.com/developer-connect/docs/ucm-loan-data-extracts)" guidance covers Pipeline API + Get Loan + field-change webhooks for "reporting, data warehousing, and batch operations," with payload-size/pagination limits. So enriching the borrower DB from Encompass is within intended use — but at scale ICE steers customers to **Encompass Data Connect** (separate product: full field schema, near-real-time, hosted Snowflake or direct-write, 3 years history, auto schema sync) ([Data Connect product page](https://mortgagetech.ice.com/products/encompass-data-connect)).
- For YS Capital's volume (private/hard-money lender; likely hundreds, not tens of thousands, of active loans) the API path should stay comfortably inside limits if webhook-driven rather than full-pipeline polling.

## 9. Questions for the ICE account manager (ask before building)

1. Is Developer Connect API access included in our current Encompass subscription for instance `BE11397907`, or is it a separate entitlement/SKU? Are there any per-call, per-loan, or monthly API charges?
2. Does a dedicated **API user** (service account) consume a billable Encompass seat? Is there a reduced-cost seat class for API-only users? How many API users are we entitled to?
3. Do we have (or can we get) a **test instance and/or R2T instance**? What does it cost, how is it refreshed from production, and can PII be masked in it?
4. What is required (and what does it cost) to enable: (a) the **Enhanced Field Change (EFC) webhook**, (b) **Webhook Custom Auth (Premium)**, on both R2T and production? What are typical enablement lead times and "change dates"?
5. What are our instance's current **concurrency limit** and any per-minute rate limits? Who else (ISV partners, POS, ICE services) is consuming that budget today, and what does an increase cost?
6. Can you provide the **API terms of use / relevant master-agreement sections** covering programmatic extraction of our loan data into our own borrower database, and confirm our read-only portal use case is permitted without Encompass Data Connect?
7. At what data volume/use case would ICE require or recommend **Encompass Data Connect** instead of Developer Connect polling/webhooks, and what is its pricing for a lender our size?
8. Which support tier do we have (standard vs **Premier**), what are the SLAs for API-related production incidents, and how do we get pre-release notification of **breaking changes** relevant to the endpoints we use?
9. Confirm the sanctioned **OAuth grant** for a lender-built server integration (password grant with API user vs client credentials), secret rotation policy, and whether IP allowlisting/mTLS options exist for our instance.
10. Are any endpoints we plan to use (V3 loans, pipeline, eFolder/attachments, milestones/conditions, webhooks) scheduled for **deprecation/sunset** in 26.3/27.x beyond what's published, and can we be added to advance-notice communications for instance `BE11397907`?
11. During **major-release weekends and maintenance windows**, what API behavior should we expect (downtime, read-only, degraded), and is there a calendar feed we can consume programmatically?
12. Is there any restriction on retaining Encompass-sourced data after a loan closes or after an Encompass contract ends (data portability/exit terms)?
