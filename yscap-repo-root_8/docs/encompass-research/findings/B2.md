## 3. How rules are declared and evaluated

- **Registry** (`src/lib/conditions/field-registry.js`): ~45 typed fields (money/number/percent/text/enum/boolean/date) in groups Loan & program / Property / Deal economics / Borrower & experience / Entity. Raw DB free-text is normalized to canonical enums (`normState`, `normStrategy`, `normLoanPurpose`, `normPropertyType`, `normRehabType`, `normCitizenship`, `normOccupancy`, lines 39-107). `writable:true` fields are legal `info_field` targets; the whitelist of write destinations is `WRITE_TARGETS` (lines 229-249) â€” the ONLY columns a borrower answer may touch. Admin-defined custom fields (`custom_fields` table, `db/038`) merge in at runtime (`fieldMap()`, lines 298-303; answers in `application_field_values` jsonb); inactive custom fields stay resolvable for old rules.
- **Rule trees** (`src/lib/conditions/rules.js`): stored on `checklist_templates.rule_logic` as `{combinator:'and'|'or', rules:[{field,operator,value}|group]}`, max one nesting level, â‰¤50 rows per group (lines 58-82). Typed operator whitelist per field type (lines 25-33). Evaluation is a pure whitelisted walk â€” nothing eval'd, nothing interpolated into SQL; invalid trees evaluate false; **missing data never fires a rule** except explicit `is_empty` (lines 113-195). `summarizeRule()` renders the plain-language summary snapshotted into `origin_detail`.
- **Engine** (`src/lib/conditions/engine.js`):
  - `loadRuleContext(appId)` (lines 49-137) joins applications + borrower + llcs + current `product_registrations`, computes `loan_to_arv`/`loan_to_cost`, sums verified experience across both borrowers, and merges custom-field answers.
  - `evaluateApplication(appId)` (lines 175-272): only for open statuses `new, in_review, processing, underwriting, approved, clear_to_close` (line 33) and non-deleted files. For every active template with `auto_apply IN ('always','rules')`: match & absent â†’ instantiate (`instantiateTemplate`, lines 144-160, with `origin_kind='auto'` + snapshot); no-longer-matching â†’ **retract only if engine-created AND untouched** (`origin_kind='auto'`, status `outstanding`, no sign-off/review/payload/docs/notes â€” lines 239-246). Duplicate suppression is per (application, template): ANY existing instance blocks re-issuance (lines 19-21). Audit action `conditions_auto_evaluated`; one borrower notification per pass; templates that are borrower-visible without borrower wording are demoted to staff-only (#78 guard, lines 220-230).
  - Re-evaluation hooks (grep results): application create/submit (`borrower.js:2881`), borrower profile save (`borrower.js:233`), product register (both portals: `borrower.js:698`, `staff.js:1624`), SOW save (`borrower.js:994`, `staff.js:1757,1833`), info-condition answer (`borrower.js:893`), details PATCH (`staff.js:4028`), **status change** (`staff.js:4278, 4322` â€” `status` is a rule field), track-record verify/unverify (`staff.js:3791`) and doc reject/delete (`staff.js:5297, 5402`), LLC link (`src/lib/vesting.js:106`), manual re-run per file (`staff.js:2077`) and global (`admin-conditions.js:304, 355, 438`).
- **Creation chokepoint** (`src/lib/conditions/ensure.js`): `ensureFileConditions(appId)` derives everything from the DB row, runs idempotent `generateChecklist`, and asserts the "never zero checklist items" invariant (loud audit `conditions_invariant_violation`); backstopped by boot backfill `db/095`.

## 4. Lifecycle & who can do what

### Item statuses in practice
`outstanding` â†’ (borrower uploads / answers / uses tool) â†’ `received` (borrower actions NEVER auto-satisfy) â†’ staff **sign-off** â†’ `satisfied` + `signed_off_at/by`. Staff push-back/reject â†’ `issue` + borrower-visible `issue_reason` (clears every stamp). Waive (optional items only) â†’ `satisfied` + `waived_at/by`. Auto-reopen triggers (below) flip cleared items back to `received` / `issue` / `outstanding` with `[auto]` notes.

### Capability model (`src/lib/permissions.js:34-60`, doc `docs/condition-center.md` Â§Roles)
| Capability | Default holders | Grants |
|---|---|---|
| `review_conditions` | loan_officer, processor, underwriter, loan_coordinator, admin | The lighter "reviewed" stamp â€” never completes |
| `sign_off_conditions` | processor, underwriter, loan_coordinator, admin (NOT loan_officer) | Complete/sign off checklist items; clear first-class conditions; accept documents |
| `waive_conditions` | underwriter, admin | Waive first-class conditions with reason; (checklist waives use `sign_off_conditions` + item must be optional) |
| `manage_conditions` | admin, software_setup | Author the Condition Studio library/rules |
| super_admin | â€” | everything, incl. the zero-document override |

### Enforcement points
- `PATCH /api/staff/checklist/:itemId` (`staff.js:2542-2689`): sign-off/`satisfied` requires `sign_off_conditions` (line 2563-2566); review stamp requires either capability (2570); **waive requires sign-off capability AND `is_required=false`** (2576-2581, 422 otherwise); push-back requires reviewer+reason (2587-2594) and notifies the borrower with the reason (2667-2687). Sign-off first passes `signOffGate` (2599-2602, 422 with plain-language reason).
- First-class conditions: `POST /loan-conditions/:cid/clear` requires `sign_off_conditions` (`staff.js:2250-2261`); `/review` writes `reviewed_by/at` WITHOUT changing status (`staff.js:2266-2282`, schema `db/083`); `/waive` requires `waive_conditions` + reason + per-file `canTouchApp` (`staff.js:2283-2299`).

### The review workflow (db/083)
`db/083_condition_review.sql` adds `reviewed_by/reviewed_at` to `conditions`: a **two-tier control** â€” an LO marks a condition REVIEWED ("I looked at it / I believe it's done") but cannot clear; clearing stays with `sign_off_conditions` holders. The stamp never changes status â€” the condition remains `open` (a blocker) until cleared/waived. The identical pattern exists on `checklist_items` (reviewed columns referenced throughout `staff.js` and reset by every reopen trigger). This mirrors exactly the two-phase model the Encompass gate needs: *review* (human eyeball) vs *clear* (authoritative completion).

### Data-integrity sign-off gates (`signOffGate`, `staff.js:2400-2540`)
The chokepoint pattern the Encompass gate should extend. Current branches:
- **Zero-document gate**: a required document condition can never be signed off with no current non-rejected document; only super_admin overrides (2428-2437).
- **Slot gates**: insurance needs `binder` + `invoice` slots; title needs any doc; fraud needs `background` (+ `criminal` when registered program matches /gold/) (2442-2466).
- **Product gate** (`rtl_p1_product` / tool_key `product_pricing`): a product must be registered (2486-2489).
- **Budget gate** (`rehab_budget`): first-page SOW target = line-item total = file budget = registered product budget **to the cent**, plus Gold â‰¥5% contingency (2490-2513); belt-and-suspenders DB trigger `db/069` refuses `status='satisfied'` at the SQL layer too.
- **Experience gate** (`track_record`): claimed experience must be VERIFIED against â‰¤3-year track records or the item can't be signed off (2514-2539).

### Automatic reopen triggers (the 071â†’096 family)
One DB trigger `trg_reopen_on_budget_change` on `applications` (function `reopen_conditions_on_budget_change`), successively broadened:
- `db/071`: rehab-budget change reopens `product_pricing` (â†’`received`) and `rehab_budget` SOW (â†’`issue`), clearing sign-off + review stamps, `[auto]` notes, only if previously cleared (`IS DISTINCT FROM` no-churn).

## 5. How clear-to-close is issued today

1. **Statuses**: `APP_STATUS` (`staff.js:3883`); decision statuses `approved, clear_to_close, funded, declined` are restricted to `seesAll` roles (admin/underwriter/loan_coordinator) â€” `DECISION_STATUSES` gate at `staff.js:4246-4257`.
2. **Blockers** (`advancementBlockers`, `staff.js:3895-3923`): moving to `clear_to_close` requires:
   - zero `conditions` rows `open/borrower_responded` with severity `standard|prior_to_docs` (funding adds `prior_to_funding`; `post_closing` never blocks);
   - zero required, non-gate checklist `document|condition` items not satisfied/signed off (tasks are workflow and don't block â€” audit finding S5-07);
   - zero `is_gate=true` items not signed off.
3. **Transition** (`PATCH /applications/:id`, `staff.js:4247-4292`): blocked with 409 + blocker list; an **admin may force** (`force:true`, recorded as `forced` in `application_status_history` and the audit log). Status change re-runs the rule engine and notifies borrower + staff.
4. **Second door**: `POST /applications/:id/internal-status` (`staff.js:4298-4325`) re-derives the borrower-facing status from the 38-value ClickUp status map; it enforces DECISION_STATUSES but does **NOT** call `advancementBlockers` â€” an internal-status change mapping to `clear_to_close` bypasses the conditions gate. Any Encompass CTC gate must cover this door too (repo's own "all sides" rule).
5. **Readiness widget**: `GET /applications/:id/gating` (`staff.js:3926-3937`).
6. Reaching CTC **locks** parts of the file (e.g. vesting entity, `staff.js:1371-1372`).

## 7. Conceptual design â€” the ENCOMPASS AGREEMENT GATE (not implemented; design only)

### 7.1 Read-only mirror, never a live call in the gate path
A pull-only sync worker (reusing `sync_queue` target `encompass`, direction `pull`) snapshots per linked loan into new mirror tables, e.g.:
- `encompass_loans` (application_id, encompass_loan_guid, loan_amount, rate, term, loan_type, status/current milestone, ctc_milestone_done_at, underwriting_status, snapshot_at)
- `encompass_conditions` (encompass condition id, loan guid, title, category/type â€” enhanced|underwriting|preliminary, status e.g. Added/Requested/Received/Reviewed/Cleared/Waived, cleared_by/at, snapshot_at)
- `encompass_condition_map` (checklist_template code or checklist_item_id â†” encompass condition id/title-pattern; plus field-level maps like portal `loan_amount` â†” Encompass loan amount)

Linkage key: a new `applications.encompass_loan_guid` (admin-set or resolved by pipeline query on the Encompass loan number â†” `ys_loan_number` / `investor_loan_number`, which already exist in `db/schema.sql:236-237`). Write-incapability is structural: the Encompass HTTP client module exposes **only** GET helpers (the ClickUp `client.js` chokepoint discipline, but with no write functions compiled at all), and the DB constrains encompass queue rows to `pull`.

### 7.2 The flag: "requires Encompass match"
Add to `checklist_templates` + mirrored onto `checklist_items` (new numbered idempotent migration, per repo rules):
- `encompass_gate` jsonb, e.g. `{ "kind": "condition", "match": {"titlePattern": "...", "conditionType": "underwriting"}, "requiredStatus": ["Cleared","Waived"] }` or `{ "kind": "field", "portalField": "loan_amount", "encField": "loanAmount", "tolerance": 0 }` or `{ "kind": "ctc" }`.
NULL = today's behavior (fully backward compatible; per the "previous AND future" rule, a backfill migration flags existing templates like `rtl_p2_enc`, the liquidity/credit-score items, and `rtl_cond_signedts` candidates). This is the same design shape as `field_key`/`rule_logic`: declarative metadata on the template, evaluated centrally.

### 7.3 Where each check sits in the current lifecycle
1. **Clear-time (primary): a new branch in `signOffGate` (`staff.js:2400`).** Exactly like the budget/experience branches: if `encompass_gate` is set, read the mirror; return a plain-language 422 reason unless (a) the mapped Encompass condition's status âˆˆ requiredStatus, or (b) the mapped field values agree within tolerance, AND (c) the snapshot is fresh (`snapshot_at` within a configured window â€” **fail closed** on stale/missing data, with the reason "Encompass data is stale/unlinked â€” refresh or link the loan"). The same branch guards `POST /loan-conditions/:cid/clear` for first-class conditions. Waives stay human (a waive is precisely the "Encompass disagrees but we accept it" escape hatch â€” require `waive_conditions` + reason, already the model).
2. **Belt-and-suspenders DB trigger** (the `db/069` SOW pattern): a `BEFORE UPDATE` trigger on `checklist_items` refuses flipping an `encompass_gate`-flagged item to `satisfied` unless a fresh agreeing row exists in the mirror tables (pure SQL against the mirror â€” no network in triggers). COALESCE-guard non-gated items exactly as 069 does.
3. **Post-clear auto-reopen** (the 071/096 pattern, but on the MIRROR): an `AFTER INSERT OR UPDATE` trigger on `encompass_conditions`/`encompass_loans` (or app-layer in the sync worker, which is simpler and can notify): when a pull changes a mirrored value such that a previously signed-off flagged item no longer agrees (Encompass condition reopened, loan amount drifted), reopen it â€” `status='received'` (or `'issue'`), clear `signed_off_*`/`reviewed_*`, `[auto]` note "Encompass no longer agrees: â€¦", `IS DISTINCT FROM` no-churn, audit row. This makes Encompass drift behave exactly like an economics change does today.
4. **Issue-time (optional, rules): expose read-only Encompass fields in the registry** â€” e.g. `enc_status`, `enc_ctc` (boolean), `enc_loan_amount`, `enc_amount_matches` (computed) added to `loadRuleContext()` from the mirror, `writable:false`. Admins can then author auto-apply rules like "attach 'Resolve Encompass discrepancy' while `enc_amount_matches` is no". Zero engine changes needed beyond context fields â€” the whitelisted rule walk already handles it.
5. **CTC issuance gate**: extend `advancementBlockers(appId, 'clear_to_close')` (`staff.js:3895`) with an Encompass clause: block unless the mirrored loan shows Encompass CTC (v3 milestones list shows the "Clear to Close"/"Approval" milestone finished, or the chosen underwriting-status field equals CTC). Surfacing through `advancementBlockers` automatically lights up the readiness widget (`/gating`) and the 409 blocker list. Two doors must be covered: `PATCH /applications/:id` AND `POST /applications/:id/internal-status` (which today skips blockers entirely â€” pre-existing gap worth closing in the same change). Policy decision needed: whether admin `force` may still override an Encompass-CTC block (recommended: keep force but make the audit record explicit, e.g. `forced_over_encompass`; or require super_admin).

### 7.5 Fit with existing invariants
- Snapshot semantics preserved: the gate reads live mirror state at decision time, wording stays snapshotted.
- The engine's conservative retraction rules are untouched; Encompass drift uses the reopen family, not deletion.
- Borrower-facing copy must never name Encompass internals beyond neutral wording (capital-partner rule); `[auto]` notes are staff-side (`notes` is internal-only per `db/075`).
- Freshness handling should follow the sync-review discipline: a persistent mismatch or unlinkable loan becomes a `sync_review_queue`-style row with options, never a silent block.
