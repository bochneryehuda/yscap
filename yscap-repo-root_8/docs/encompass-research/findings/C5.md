## 1. The two condition systems (and how to know which one a loan uses)

Encompass has **two mutually exclusive condition systems per loan**:

| | V1 "Standard" Loan Conditions | V3 "Enhanced" Conditions |
|---|---|---|
| API base | `/encompass/v1/loans/{loanId}/conditions/{underwriting\|preliminary\|postclosing}` | `/encompass/v3/loans/{loanId}/conditions` |
| Condition types | Fixed: Underwriting, Preliminary, Post-Closing (three separate endpoint families) | Lender-defined condition **types** and **templates** (e.g. "Underwriting", "X-3", …) configured in Encompass Settings; one endpoint, filter with `conditionType` |
| Status model | Fixed enum on the condition: `Added, Expected, Requested, Received, Rerequested, Fulfilled, Reviewed, Sent, Cleared, Waived, Expired, Rejected` + boolean/date pairs (`isRequested/requestedDate`, `isReceived/receivedDate`, `isFulfilled/fulfilledDate`, `isRerequested/rerequestedDate`) | Lender-configurable **status tracking entries** (`tracking[]` array of `{status, user, date}`) + read-only derived `status`, `statusDate`, `statusOpen` |
| Introduced | legacy (SDK parity) | Encompass 20.2+ |
| Discriminator | — | Loan field **`ENHANCEDCOND.X1`** / JSON path **`loan.useEnhancedConditionIndicator`**: `true` → the loan uses Enhanced Conditions; `false` → Standard Conditions ([Loan Enhanced Conditions reference](https://developer.icemortgagetechnology.com/developer-connect/reference/loan-enhanced-conditions)) |

**Integration implication:** the portal's condition-sync worker must read `ENHANCEDCOND.X1` (fieldReader) or `useEnhancedConditionIndicator` per loan first, then pull from the correct API family. Building only one path will silently miss conditions on the other kind of loan. YS Cap should confirm which mode BE11397907 actually runs (open question — likely standard/V1 for a hard-money shop unless admins enabled Enhanced Conditions).

Sources: [Loan Conditions overview](https://developer.icemortgagetechnology.com/developer-connect/reference/loan-conditions), [Loan Enhanced Conditions](https://developer.icemortgagetechnology.com/developer-connect/reference/loan-enhanced-conditions), [V3 Get All Enhanced Conditions](https://developer.icemortgagetechnology.com/developer-connect/reference/get-all-enhanced-conditions).

## 2. V1 Standard Loan Conditions — endpoints and object shape

### 2.1 Read endpoints (Postman folder "Encompass Loan / Loan Conditions", index lines 283–307)

| Purpose | Method | Path |
|---|---|---|
| List underwriting conditions | GET | `/encompass/v1/loans/{loanId}/conditions/underwriting` |
| Get one underwriting condition | GET | `/encompass/v1/loans/{loanId}/conditions/underwriting/{conditionId}?view=entity` |
| List preliminary conditions | GET | `/encompass/v1/loans/{loanId}/conditions/preliminary` |
| Get one preliminary condition | GET | `/encompass/v1/loans/{loanId}/conditions/preliminary/{conditionId}?view=entity` |
| List post-closing conditions (can include removed) | GET | `/encompass/v1/loans/{loanId}/conditions/postclosing?view=entity&isRemoved=true` |
| Get one post-closing condition | GET | `/encompass/v1/loans/{loanId}/conditions/postclosing/{conditionId}?view=entity` |

(Write operations exist — POST create, PATCH `action=add|update|remove`, comment and document sub-resources — the portal must never register them. All reads are plain GETs, so the V1 conditions surface fits a GET-only allowlist cleanly.)

### 2.2 Underwriting condition object (from the collection's canonical sample, "01 - Create an Underwriting Condition")

```jsonc
{
  "id": "GUID",                              // returned on reads
  "conditionType": "Underwriting",           // or "Preliminary" / "Postclosing"
  "title": "Create UWC Test",                // short name — the portal's primary match key
  "description": "…",                        // full text
  "priorTo": "Approval",                     // when it must be cleared: Approval | Docs | Funding | Closing | Purchase
  "category": "Credit",                      // Assets | Credit | Income | Legal | Misc | Property
  "source": "Manual",                        // Escrow | Investor | Recorder's Office | Borrowers | FHA | VA | MI Company | Other | Manual | Condition Set | Automated Conditions
  "owner": "Loan Officer",
  "status": "Received",                      // Added|Expected|Requested|Received|Rerequested|Fulfilled|Reviewed|Sent|Cleared|Waived|Expired|Rejected
  "statusDate": "2018-03-30T06:18:53Z",
  "isRemoved": false,                        // soft delete — removed conditions stay retrievable
  "isRequested": true,  "requestedDate": "…", "requestedBy": {"entityId":"…","entityType":"User"},
  "isRerequested": true, "rerequestedDate": "…", "rerequestedBy": { … },   // (prelim/post-closing samples)
  "isReceived": true,   "receivedDate": "…",  "receivedBy": { … },
  "isFulfilled": true,  "fulfilledDate": "…", "fulfilledBy": { … },
  "expectedDate": "…", "daysToReceive": 4, "requestedFrom": "devadmin",
  "printExternally": true, "printInternally": true,
  "forAllApplications": true,                // or "application": {"entityId":"All"|"<borrower-pair-id>","entityType":"Application"}
  "createdDate": "…", "createdBy": { … },
  "comments": [ { "comments": "…", "forRoleId": 1, "forRole": {"entityId":"1","entityType":"Role"},
                  "dateCreated": "…", "createdBy": "devadmin", "createdByName": "…" } ],
  "documents": [ { "entityId": "<eFolder documentId>", "entityType": "Document" } ]   // link to eFolder docs
}
```

Status enum, priorTo, category and source enumerations confirmed on the official [V1 Manage Preliminary Conditions](https://developer.icemortgagetechnology.com/developer-connect/reference/manage-preliminary-conditions) reference. Preliminary/post-closing objects share the same shape (post-closing sample adds the `isRerequested` trio; preliminary omits `owner`).

**Key read semantics**
- `status` is the current tracking state; `Cleared` and `Waived` are the terminal "done" states. `Fulfilled` exists as a distinct state (used in prelim/post-closing samples) — treat `Cleared`/`Waived` as the only clear-gate states and treat `Fulfilled` as "satisfied but not underwriter-cleared" unless YS ops decides otherwise.
- The boolean/date pairs are the **audit trail** of the tracking checkboxes (requested → received → fulfilled), not the gate.
- `isRemoved=true` conditions are soft-deleted but still returned when asked (`isRemoved=true` query on the list call); the portal must always filter `isRemoved=false`.
- `documents[]` carries entity references to eFolder documents — this is how a condition "knows" which document folder satisfies it.

## 3. V3 Enhanced Conditions — endpoints and full contract

### 3.1 Read endpoints (Postman "Encompass Loan / Enhanced Conditions", index lines 264–279)

| Purpose | Method | Path |
|---|---|---|
| List conditions | GET | `/encompass/v3/loans/{loanId}/conditions` (`conditionType`, `view=Summary|Detail|Full`, `includeRemoved`) |
| Get one condition | GET | `/encompass/v3/loans/{loanId}/conditions/{conditionId}?view=full` |
| Get comments | GET | `/encompass/v3/loans/{loanId}/conditions/{conditionId}/comments` |
| Get status tracking entries | GET | `/encompass/v3/loans/{loanId}/conditions/{conditionId}/tracking` |
| Get condition's documents | GET | `/encompass/v3/loans/{loanId}/conditions/{conditionId}/documents` |

`view`: Summary → basics; Detail → adds tracking + definitions; Full → adds comments ([V3 Get All Enhanced Conditions](https://developer.icemortgagetechnology.com/developer-connect/reference/get-all-enhanced-conditions)). The single-condition GET with `view=details` also returns an `x-elli-permissions` response header (e.g. `"CanEditCondition","CanAddComments","CanAssignDocuments"`) describing the calling user's rights (official PDF, p.13).

Writes (never to be registered by the portal): PATCH `?action=Add|Update|Remove|Duplicate` on the collection, comment/document/tracking PATCH sub-resources, and 26.2's `delegatedTrackingStatuses` update.

### 3.2 EnhancedConditionContract (official "Enhanced Conditions and Settings API Reference" PDF, pp. 7–12 — local copy `scratchpad/research/c5work/enhanced-conditions-ref.pdf`)

| Attribute | Type | Notes |
|---|---|---|
| `id` | string GUID | unique condition id |
| `conditionType` | string, read-only | e.g. Preliminary, Underwriting, Post-Closing — from the condition template |
| `title` | string | e.g. "Income - Personal Tax Returns" |
| `internalId` / `internalDescription` / `internalPrint` | string/string/bool | internal-facing id/text/print flag |
| `externalId` / `externalDescription` / `externalPrint` | string | TPO/borrower-facing id/text/print flag |
| `source` | string | source system, e.g. "Fannie Mae" |
| `application` | ApplicationReferenceContract | borrower-pair scope |
| `category` | string | lender-defined; examples Assets, Credit, Income, Legal, Misc, Property |
| `priorTo` | string | lender-defined; examples Approval, Docs, Funding, Closing, Purchase |
| `recipient` | string | lender-defined, e.g. "MERS" |
| `startDate` / `endDate` | ISO datetime | effective window |
| `requestedFrom` | string | e.g. "Borrower" |
| `daysToReceive` | int | |
| `status` | string, **RetrieveOnly** | name of current status, e.g. "Requested" |
| `statusDate` | ISO datetime, RetrieveOnly | when the current status was applied (GMT) |
| `statusOpen` | bool, RetrieveOnly | **whether the condition is open or satisfied per the definition** — the cleanest machine-readable "done" flag |
| `commentsCount` | int, RetrieveOnly | |
| `assignedTo` | Array[EntityReferenceContract] | **documents the condition is assigned to** |
| `createdBy` / `createdDate` | EntityReference / datetime | |
| `lastModifiedBy` / `lastModifiedDate` | RetrieveOnly | change detection for incremental sync |
| `isRemoved` | bool | soft delete ([FAQ](https://developer.icemortgagetechnology.com/developer-connect/docs/faqs-enhanced-conditions): all eFolder logs are soft-deleted; use `includeRemoved=true` to see them) |
| `tracking` | Array[TrackingEntryContract] | `{status, user (EntityReference), date}` — each status checkbox marked complete |
| `comments` | Array[LogCommentContract] | `{id, addedby, comments, addedDate, forRole, reviewedBy, reviewedDate, isExternal}` |
| `definitions` | EnhancedConditionDefinitionsContract | option lists for this condition: `categoryDefinitions`, `priorToDefinitions`, `recipientDefinitions`, `sourceDefinitions`, `trackingDefinitions` (incl. which roles may mark each status) |

`EntityReferenceContract.entityType` values: User, Role, Condition, Application, Attachment, Document, MilestoneSetting, Milestone (PDF p.12).

**Critical difference vs V1:** enhanced statuses, categories and priorTo values are **tenant-configurable**. The portal cannot hard-code "Cleared" — it must read the instance's tracking definitions (per condition via `definitions`, or globally via `/encompass/v3/settings/loan/conditions/types` / `/templates`) and map YS Cap's terminal statuses in configuration. Also note 26.1+ added a read-only `sourceOfCondition` attribute ([release notes](https://developer.icemortgagetechnology.com/developer-connect/changelog/261-major-release)).

**Webhook resource:** Enhanced Conditions is a first-class webhook resource category (create/update/delete events), including `EnhancedConditionTemplate` and `EnhancedConditionType` resources with extra-payload attributes ([webhook overview](https://developer.icemortgagetechnology.com/developer-connect/reference/webhook), [extra payload — EnhancedConditionTemplate](https://developer.icemortgagetechnology.com/developer-connect/reference/extra-payload-attributes-enhancedconditiontemplate)). Webhooks are reads (Encompass → portal) and are compatible with a strictly read-only posture; subscription management is the only write, and it writes to the webhook service, not the loan.

## 4. Conditions ↔ eFolder documents linkage

- **V1:** condition `documents[]` = entity references `{entityId: <documentId>, entityType: "Document"}`; sub-resource `GET /encompass/v1/loans/{loanId}/documents/{docId}` resolves them; document objects in turn expose their attachments (`GET …/documents/{docId}/attachments`).
- **V3 enhanced:** condition `assignedTo[]` (entity refs) + dedicated `GET /encompass/v3/loans/{loanId}/conditions/{conditionId}/documents`.
- Both directions are browsable: a document also relates back to conditions ([eFolder Manage Documents overview](https://developer.icemortgagetechnology.com/developer-connect/reference/efolder-document-1) lists Conditions as a related object).

So the evidence chain the portal can mirror is: **condition → document(s) → attachment(s) → bytes**, each hop a read.

## 6. eFolder Attachments — metadata vs bytes

Two storage generations matter: **legacy/media storage** and **cloud storage**. V3 APIs handle both; V1 attachment reads may miss cloud-stored files ([eFolder Documents and Attachments tips](https://developer.icemortgagetechnology.com/developer-connect/docs/document-tips-and-tricks)).

### Metadata (plain GET)
| Purpose | Method | Path |
|---|---|---|
| List attachments (V3) | GET | `/encompass/v3/loans/{loanId}/attachments?includeRemoved=true` |
| Get attachment (V3) | GET | `/encompass/v3/loans/{loanId}/attachments/{attachmentId}` |
| List/Get attachments (V1) | GET | `/encompass/v1/loans/{loanId}/attachments[/{id}]` |
| File-level metadata (page count, contentType, size, owner) | GET | `/efolder/v1/loans/{loanId}/files/{attachmentId}?includeMetaData=true` — note `includeMetaData` is **case-sensitive** (known issue) ([V1 Get Attachment Metadata](https://developer.icemortgagetechnology.com/developer-connect/reference/v1-get-attachment-metadata)) |

Attachment records carry `id` (format `Attachment-<guid>.<ext>`), `title`, `type` (`Native` vs image/cloud), active/removed state, and page structure for image attachments.

### Bytes (POST that generates a time-limited download URL — read-oriented despite the verb)
| Purpose | Method | Path |
|---|---|---|
| Download original attachment (V3, cloud + legacy) | **POST** | `/encompass/v3/loans/{loanId}/attachmentDownloadUrl` body `{"attachments":["<id>", …]}` → `{url, authorizationHeader, expiration}`; then plain GET on the returned URL (pass `elli-signature` auth header for cloud) ([V3 Create Download Attachment URL](https://developer.icemortgagetechnology.com/developer-connect/reference/get-download-attachment-url)) |
| Attachment URL (V1 legacy) | POST | `/encompass/v1/loans/{loanId}/attachments/{id}/url` |
| Single page image URL (V1) | POST | `…/attachments/{id}/pages/{pageId}/url` |
| Page thumbnail URL (V1) | POST | `…/attachments/{id}/pages/{pageId}/thumbnail/url` |

Caveat: if the attachment was imported as part of a larger file, `attachmentDownloadUrl` returns the **entire imported document**; use the Export Attachments API to isolate one attachment.

**Architectural note for the "incapable of writing" requirement:** a pure "GET-only" HTTP filter would break byte retrieval. The read-only client must allowlist these specific **read-semantics POSTs** by exact path: `attachmentDownloadUrl`, `attachments/*/url`, `pages/*/url`, `thumbnail/url`, `exportJobsCreator`, `exportjobs`, `fieldReader`, plus (if used) `loanPipeline`. Everything else — every PATCH/PUT/DELETE and every other POST under `/loans/` — must be structurally unreachable.

## 7. eFolder Export Attachments — async batch export (read-oriented POST)

Flow (Postman index lines 260–262 + official refs):

1. **POST** `/efolder/v1/loans/{loanId}/exportJobsCreator?skipPersonaChecks=false&includeNotActive=true` — body contains `requestId` and `exportEntity.exportMeta[]`, each entry `{fileName, entities:[{entityType:"urn:elli:encompass:document", entityId:"<docId>"}], requestId}`; documents, attachments **and conditions** are supported as export entities; max **10** requests per call; response **202** with job IDs / Location header ([Export Files Job Creator](https://developer.icemortgagetechnology.com/developer-connect/reference/export-files-job-creator), added 24.3 ([changelog](https://developer.icemortgagetechnology.com/developer-connect/changelog/243-major-release))).
2. **GET** `/efolder/v1/exportjobs/{jobId}` — poll status; when done the response carries the **`entityUri`** to download the merged file ([Get Export Status per 24.3 notes](https://developer.icemortgagetechnology.com/developer-connect/changelog/243-major-release)).
3. Plain GET on the returned URI.

The older single-job variant is **POST** `/efolder/v1/exportjobs` (`fileName`, `annotationSettings` — controls whether Private/Public annotations are burned into the export, `source`, `entities`; query `includeNotActive`, `errorOnBackgroundAttachment`) ([Export Attachments](https://developer.icemortgagetechnology.com/developer-connect/reference/export-attachments)). Exports merge multi-page/cloud/native files into a single downloadable PDF — the right tool when the portal wants "the whole credit condition package as one file".

Nothing in this flow mutates the loan: it creates a server-side export job only. Safe for the read-only allowlist; recommend pinning `annotationSettings.visibility` to exclude private annotations for borrower-visible copies.

## 8. eFolder History

`GET /encompass/v3/loans/{loanId}/histories/efolder` (`start` default 0, `limit` default 100 / max 1000) — retrieves the eFolder audit history for the loan (document/attachment/condition activity with user + timestamp). Added in 24.2; a UTC-conversion bug in its timestamps was fixed in 25.2 — treat pre-25.2 timestamps with suspicion. ([V3 Get eFolder History](https://developer.icemortgagetechnology.com/developer-connect/reference/v3-get-efolder-history), [24.2 changelog](https://developer.icemortgagetechnology.com/developer-connect/changelog/242-major-release)).

Use: cheap tamper-evident feed for "who cleared what, when" displays, and a reconciliation source when portal and Encompass disagree.

## 9. Disclosure Tracking logs — the compliance ledger

Endpoints (index lines 205–229):

| Purpose | Method | Path |
|---|---|---|
| List DT logs (V3) | GET | `/encompass/v3/loans/{loanId}/disclosureTracking2015Logs` |
| Get one DT log (V3, opt. snapshot) | GET | `…/disclosureTracking2015Logs/{id}?includeSnapshot=true` |
| Snapshot of one log | GET | `…/disclosureTracking2015Logs/{id}/snapshot` |
| Snapshots of all logs | GET | `…/disclosureTracking2015Logs/snapshots` |
| Email message for a log | GET | `…/disclosureTracking2015Logs/{id}/emailMessage` |
| V1 equivalents | GET | `/encompass/v1/loans/{loanId}/disclosureTracking2015[/{logId}][/snapshot]` |
| DT settings | GET | `/encompass/v3/settings/loan/disclosureTracking` |

Log object (from [V3 Update a Disclosure Tracking Log](https://developer.icemortgagetechnology.com/developer-connect/reference/update-a-disclosure-tracking-log) contract + collection samples):

- Identity/typing: `id`, `provider` (Encompass/Other), `status` (Active/Pending), **`disclosureType`** (Initial, Revised, Final, PostConsummation), `contents[]` (**LE, CD, ServiceProviderList, ServiceProviderListNoFee, SafeHarbor**), `includedInTimeline`.
- Delivery: `disclosedMethod` (ByMail, Fax, InPerson, Email, Phone, Signature, Other) + description, `disclosureRecipients[]` (per-recipient method + received dates), `disclosedDate` (user vs computed), `disclosedBy`, `fulfillments[]` (manual mail fulfillment records: method, orderedBy, processedDate, recipients w/ actualDate).
- Compliance state: `intentToProceed {intent, date, …}`, per-borrower **eConsent** (recorded via the DT log; the V3 API supports recording eConsent from external delivery platforms — [23.1 changelog](https://developer.icemortgagetechnology.com/developer-connect/changelog/231-major-release)), `changedCircumstanceIndicator` + `changeInCircumstance(+Comments)`, `disclosedApr`, `disclosedFinanceCharge`, `disclosedDailyInterest`, `feeLevelDisclosuresIndicator`, `feeChanges`, `useForUcdExport`, `linkedLogId`, `documents[]` (the eDisclosure package docs), read-only context (`loanProgram`, `loanAmount`, `applicationDate`, `propertyAddress`, `disclosureCreatedDate`).
- **`snapshot`**: read-only capture of loan data at disclosure time — the audit basis for "what APR/fees were disclosed".

What it reveals for the portal: whether an LE exists within 3 business days of application, whether a revised LE followed each changed circumstance, whether the **CD** was delivered and when receipt is presumed/actual (the CD 3-business-day waiting period before consummation is computed off these dates), eConsent state per borrower, and intent-to-proceed. For a CTC gate on consumer-purpose loans this is the compliance evidence; for YS Cap's business-purpose hard-money loans TRID generally does not apply, so treat DT as **optional evidence, present only when the instance actually discloses** (open question §13).

## 10. AUS Tracking logs — DU/LP findings state

Endpoints (index lines 183–187): `GET /encompass/v1/loans/{loanId}/ausTrackingLogs` (list), `GET …/{id}`, `GET …/{id}/snapshot`. (POST/PATCH exist; excluded.)

The log mirrors the most recent row of the **Underwriting Decision History** table in Encompass; rows are created automatically by DU/LP orders through the partner network or manually by staff ([AUS Tracking reference](https://developer.icemortgagetechnology.com/developer-connect/reference/aus-tracking)). Fields (collection samples): `application` (borrower-pair ref), `uwRiskAssessType` (e.g. DU/LP/LQA/Other — the engine used), **`recommendation`** (the findings result, e.g. Approve/Eligible), `submissionDate`, `firstSubmissionDate`, `submissionNumber`, `duCaseIdOrLpAusKey` (DU casefile ID / LPA key), `submittedBy`, `version`, `docClass`. `…/snapshot` gives the point-in-time view.

Portal use: display "latest AUS decision" and optionally require a satisfactory `recommendation` before CTC. For hard-money lending AUS is usually absent — treat as optional evidence.

## 11. fieldReader and the PCC.* virtual fields

- **V1:** `POST /encompass/v1/loans/{loanId}/fieldReader` with a JSON array of field IDs.
- **V3:** `POST /encompass/v3/loans/{loanId}/fieldReader?invalidFieldBehavior=Include` (also accepts custom fields, borrower-pair suffixes like `4000#2`, virtual fields).

Both are POSTs with pure read semantics (body = list of field IDs) — must be on the read allowlist.

The collection's own V1 fieldReader sample (folder "Loan Management / V1 Manage Loan / 03 - Loan Field Reader") reads, among others:

```
"CoreMilestone", "Log.MS.CurrentMilestone", "Log.MS.Stage", "Log.MS.LastCompleted",
"PCC.ALL", "PCC.NOTCLEARED", "LOCKRATE.RATESTATUS", …
```

**What PCC.ALL / PCC.NOTCLEARED return:** they are read-only **virtual loan fields** over the eFolder **Preliminary (closing) Conditions** log. `PCC.ALL` returns the full set of the loan's preliminary/closing conditions rendered as a single concatenated text blob (title/description lines); `PCC.NOTCLEARED` returns the same rendering restricted to conditions **not yet cleared** — i.e. it is empty exactly when every preliminary/closing condition is cleared. Public web documentation of these two IDs is thin (they appear in the official Postman sample and in Encompass virtual-field tooling; the reporting-database counterpart family for underwriting conditions is `UWC.AC` … `UWC.WAIVEDCOUNT`). **Treat the exact rendering as verify-in-sandbox**: they are string blobs, so use `PCC.NOTCLEARED === ""` only as a cheap corroborating signal, never as the primary matching key — the structured conditions APIs (§2–3) are the source of truth. Virtual-field definitions can be enumerated via [V3 Get Virtual Fields](https://developer.icemortgagetechnology.com/developer-connect/reference/get-virtual-fields) (`/encompass/v3/schemas/loan/virtualFields`).

`Log.MS.CurrentMilestone` / `Log.MS.LastCompleted` / `Log.MS.Stage` return the milestone log's current/last-completed milestone names — the cheapest CTC probe. Milestone detail comes from `GET /encompass/v3/loans/{loanId}/milestones` (+ `/{milestoneId}`) (index lines 158–159). For pipeline-wide polling, the Loan Pipeline API exposes `Loan.CurrentMilestoneName` as a canonical filter/report field (collection sample "V3 Loan Pipeline / 06a").

## 13. RECOMMENDED CONDITION-MATCHING DATA CONTRACT

### 13.1 Linking (one-time per condition)

Store on the portal condition (new columns on `conditions` / `checklist_items`):

| Portal column | Value |
|---|---|
| `encompass_loan_guid` | Encompass loan GUID (on the application row) |
| `encompass_condition_id` | condition GUID |
| `encompass_condition_api` | `v1_underwriting` \| `v1_preliminary` \| `v1_postclosing` \| `v3_enhanced` (from `useEnhancedConditionIndicator`) |
| `encompass_title_at_link` | `title` snapshot (drift detection) |
| `encompass_last_seen` | sync watermark (`lastModifiedDate` for v3; full-object hash for v1, which has no lastModified) |

Match by staff action in the UI (pick-list of Encompass conditions), with title/category auto-suggest. Never auto-match on title alone — titles are free text.

### 13.2 Clear-gate — the exact Encompass fields compared before a portal condition may be cleared

A portal `conditions.status → 'cleared'` transition is allowed **only if** a fresh (re-fetched at decision time, not cached) read of the linked Encompass condition satisfies ALL of:

**V1 standard condition** (`GET /encompass/v1/loans/{guid}/conditions/{type}/{conditionId}?view=entity`):
1. condition exists and `id` matches `encompass_condition_id`;
2. `isRemoved === false`;
3. `status ∈ {"Cleared","Waived"}` (config flag decides whether `Waived` maps to portal `waived` instead of `cleared`; `Fulfilled`/`Reviewed`/`Received` are NOT sufficient);
4. `statusDate` present and ≥ the portal condition's `created_at` (stale-clear guard);
5. persist as evidence: `{id, conditionType, title, priorTo, category, status, statusDate, isRemoved, receivedDate, fulfilledDate, documents[].entityId}` — store the raw JSON in an `encompass_evidence` jsonb column with fetch timestamp.
6. If the condition has `documents[]`, optionally verify each referenced eFolder document has `isReceived=true` (and `isReviewed=true` for underwriter-grade gates) via `GET /v3/loans/{guid}/documents/{docId}`.

**V3 enhanced condition** (`GET /encompass/v3/loans/{guid}/conditions/{conditionId}?view=full`):
1. exists, id matches; 2. `isRemoved === false`;
3. **`statusOpen === false`** AND `status ∈ configured_terminal_statuses` (a per-instance config table seeded from `definitions.trackingDefinitions` / `/v3/settings/loan/conditions/types` — default `{"Cleared","Waived"}` names if present);
4. `tracking[]` contains an entry `{status: <terminal>, user, date}` — capture `user`/`date` as who/when cleared in Encompass;
5. `statusDate`/`lastModifiedDate` sanity vs portal timestamps; evidence snapshot as above, including `tracking[]` and `assignedTo[]`.

Mismatch behavior: block the clear, surface the Encompass `status`/`statusDate` next to the portal condition, and enqueue a re-sync — never auto-clear the portal from a background sync alone (a human clears; Encompass agreement is the gate).

### 13.3 CTC evidence set — what "Encompass shows clear-to-close" means

The portal may set `applications.status = 'clear_to_close'` only when ALL of the following read-side checks pass in a single evaluation run (all values persisted as the CTC evidence bundle):

1. **All prior-to-closing conditions terminal.** Enumerate every condition (correct API family per `useEnhancedConditionIndicator`, `includeRemoved` left false):
   - V1: union of `/conditions/underwriting` + `/conditions/preliminary`; every condition with `priorTo ∈ {Approval, Docs, Funding, Closing}` (i.e. everything except `Purchase`-timed and post-closing types) must have `status ∈ {Cleared, Waived}`. Post-closing conditions are ignored for CTC and instead mirrored into `post_closing_items`.
   - V3: every non-removed condition whose `priorTo` maps (via the config table built from `priorToDefinitions`) to at-or-before closing must have `statusOpen === false` with terminal status.
   - Cross-check (cheap, corroborating only): fieldReader `["PCC.NOTCLEARED"]` returns empty/whitespace.
2. **Milestone state.** fieldReader `["Log.MS.CurrentMilestone","Log.MS.LastCompleted","Log.MS.Stage","CoreMilestone"]`: `Log.MS.LastCompleted` (or current) must equal the instance's CTC-equivalent milestone name — configurable, since BE11397907's milestone template names are lender-defined (default Encompass templates include "Clear to Close"). Corroborate with `GET /encompass/v3/loans/{guid}/milestones`: the CTC milestone log entry must show finished/done with its completion date captured.
3. **Loan not in a dead state:** pipeline/loan read confirms the loan folder isn't an archive/adverse folder and the loan is active.
4. **Optional evidence, when present on the instance:** latest Disclosure Tracking log with `contents` ⊇ `CD` and `disclosureType ∈ {Revised, Final}` exists with receipt date consistent with the closing date (§9 — only if YS ever runs disclosures through Encompass); latest AUS tracking `recommendation` in an approved set (§10 — usually N/A for hard money).
5. **Freshness:** every read in the bundle fetched within a configured TTL (suggest ≤ 5 minutes) at the moment CTC is issued; bundle stored immutably (jsonb + timestamps) for audit.

### 13.4 Sync cadence

Poll per open application: conditions list + documents list every 10–15 min (or on webhook events for Enhanced Conditions loans); milestone/fieldReader probe on the same cycle; full re-verify synchronously at the moment of any gate decision (clear or CTC). eFolder History (§8) nightly for reconciliation.

## 14. Read-only architecture notes specific to this domain

1. The condition/document/DT/AUS read surface is almost entirely GET; the unavoidable read-semantics POSTs are: `fieldReader` (v1+v3), `attachmentDownloadUrl` (v3), `attachments/{id}/url|pages/*/url|thumbnail/url` (v1), `efolder/v1/loans/{id}/exportJobsCreator`, `efolder/v1/exportjobs` (create) + its GET status, and (if used) `loanPipeline`. Enforce with an **exact-path allowlist** in the HTTP client, not a verb rule.
2. Everything in the eFolder is **soft-deleted** (`isRemoved`) — sync must not treat disappearance as deletion; query with `includeRemoved=true` during reconciliation and keep tombstones.
3. Enhanced Conditions vocabularies (statuses, priorTo, categories) are tenant-defined → ship a settings-sync job (`/v3/settings/loan/conditions/types|templates`, DT settings, document settings) and a mapping table rather than constants.
4. Timestamps: Encompass returns GMT/UTC ISO strings; eFolder History had a UTC bug pre-25.2; V1 conditions lack lastModified → hash-diff for change detection.
