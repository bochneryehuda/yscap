## 2. Method and classification rules

- **AUTH** — `POST /oauth2/v1/token` (all grant types incl. impersonation actor/subject tokens).
- **READ_VIA_POST** — POST/PATCH verb but read/compute semantics, confirmed by request body inspection where needed: `fieldReader`, `loanPipeline` query/cursor/report, `tradePipeline`, contact `*Selector`, schema `pathGenerator`/`contractGenerator`, `auditTrail` field-history query (body = `{"fieldIds":[...]}` + `start/limit`), token introspection, eFolder `exportJobsCreator` (read-export job), attachment/page/thumbnail *download*-URL generators, stateless calculators, MISMO→loan-object converter, EPPS pricing queries, `encwsearch` search, trade-documents pipeline query (body = `sortOrder` only), opportunity/loan contract *generators*.
- **WRITE_LOAN** — any create/update/delete of loans, loan sub-collections, conditions, milestones, rate locks, eFolder docs/attachments/uploads, disclosure orders and delivery, contacts, opportunities, trades, tasks, custom data objects, resource locks, service orders, POS sessions, borrower invitations.
- **WRITE_CONFIG** — `/webhook/v1/subscriptions*`, `/webhook/v1/functions/auth*`, `/encompass/v{1,3}/settings/*` writes, `/loanOpportunity/v1/settings/*`, `/workflow/v1/settings|templates/*`, internal/external user management, SCIM, EPPS user mappings, ECS settings.
- **AMBIGUOUS** — evidence insufficient to prove read-only (see §4 for per-request reasoning).
- GET defaults to READ; §3 lists every GET I suspect could have side effects.

## 6. PROPOSED PHASE-1 ALLOWLIST

Design rule: the Encompass client must enforce **default-deny** — an explicit `(method, path-regex)` table is the only thing that can pass; everything else throws before any HTTP request is built. `{id}`/`{loanId}` denote path segments (GUIDs). Query strings must be validated separately (notably: **strip/deny any `action` query parameter on allowlisted routes** — in this API `action=` is the mutation switch on PATCH and appears erroneously on two GETs, index lines 448-449).

## 7. Notable classification judgments (defensible edge cases)

1. **`exportJobsCreator` = READ_VIA_POST** — creates a job *resource* but is the sanctioned bulk-download mechanism; no loan/business data changes. Pair with `GET /efolder/v1/exportjobs/{id}` (index line 261).
2. **Pipeline cursors** (v1 `?cursortype=randomAccess`) create server-side ephemeral cursors — read semantics retained.
3. **Resource locks = WRITE_LOAN** (index lines 385-392): no data written, but locking a loan blocks/alters other users' editing — a state change the portal must never cause.
4. **Calculators = READ_VIA_POST** (timerCompletion, loan, amortization, standardPrintForms): compute-only over posted loan objects or loanId; excluded from the Phase-1 allowlist anyway (not needed). `standardPrintForms` should be re-verified for eFolder side effects before ever enabling. `automatedConditions` calculator is the exception → AMBIGUOUS (may apply conditions).
5. **Custom Data Objects = WRITE_LOAN** (company/user/loan `customObjects` writes): arbitrary business-data storage, not configuration.
6. **Token introspection = READ_VIA_POST**, token grants = AUTH; only these two `/oauth2` paths exist in the collection (no revocation endpoint present).
7. **MISMO converter (`/encompass/v3/converter/loans`) = READ_VIA_POST** — stateless transform; but its sibling `01b Create Loan from MISMO` and `.../importer` are WRITE_LOAN.
8. **Contract generators = READ_VIA_POST** (opportunityContractGenerator POST, loanContractGenerator GET): return generated objects; the subsequent create calls are the writes.

## 8. Open questions

1. **`GET /encompass/v3/loans/{loanId}/recipients`** — does it mint disclosure auth codes on read, or only return existing ones? Needs doc confirmation before any use (currently: deny).
2. **`documentAudits` / `planCodes/{id}/evaluator`** — the `"import":"all"` body flag on the evaluator suggests it can write plan-code data into the loan; confirm with ICE docs (currently: deny; not needed Phase 1).
3. **EPPS `rateSelector`** — `requestAction`/`requestLockStatus` body fields imply pricing/lock staging; confirm read-vs-write (currently: deny).
4. **Does the API-user's Encompass persona allow removing write rights entirely?** The strongest "architecturally incapable of writing" control is persona-level (instance BE11397907): a read-only persona + this client allowlist = defense in depth. The collection cannot answer what persona granularity is available.
5. **`standardPrintForms`** — verify generated PDFs are returned/linked only and not auto-filed into the eFolder.
6. **Postman collection is 26.2-era** — the live API may expose endpoints not in this collection; deny-by-default (not deny-known-writes) is therefore the only safe posture.
