## 1. Where files physically live

### 1.1 Storage abstraction — `src/lib/storage.js`

- Pluggable provider selected by `STORAGE_PROVIDER` (`src/config.js:160`): `local` (default), with **stub** `s3` and `sharepoint` providers that throw "not configured" (`storage.js:134-145`). The interface is uniform: `save(buf,{filename}) → {ref, provider, bytes}`, `read(ref)`, `stream(ref)`, `stat(ref)`, `remove(ref)`, `probe()` (`storage.js:14-20`).
- `local` persists under `STORAGE_DIR` (default `uploads`, prod: Render persistent disk `/var/data/uploads`, `storage.js:2-5`). If the configured dir is not writable, it **falls back to `os.tmpdir()/yscap-uploads`** with a loud warning — uploads degrade to non-persistent rather than fail (`storage.js:55-75`); `/api/health` surfaces `persistent` via `probe()` (`storage.js:123-131`).
- Refs are **server-generated, opaque, sharded**: 16 random hex bytes + sanitized short extension, laid out `ab/ab…hex.pdf` (`storage.js:91-104`). The human filename lives only in the DB.
- Safety properties: path-traversal defense on every ref resolution (`safePath`, `storage.js:77-86`), and **atomic write** (temp file + rename) so a crash can't leave a half-written document (`storage.js:99-103`).
- Note: `db/schema.sql:273` declares `storage_provider text NOT NULL DEFAULT 'sharepoint'` on `documents`, but every insert path passes the actual provider returned by `storage.save()` (e.g. `src/routes/borrower.js:2004-2009`), so rows record `'local'` in practice.

## 2. Upload validation chokepoint — `src/lib/upload-bytes.js`

The single decoding contract (`{filename, contentType, dataBase64}`, raw base64, never a `data:` URL):

- **`decodeUploadBase64()`** (`upload-bytes.js:76-86`) is mandatory on every path (owner-directed root fix 2026-07-15). It strips a `data:` prefix, removes whitespace, normalizes URL-safe base64, then **strictly rejects (400)** anything else — because Node's lenient decoder silently skips invalid chars and garbles the whole file (`upload-bytes.js:8-26`). Empty decode rejected; optional `maxBytes` → 413. Returns `{buf, sha256}`.
- **Size cap**: `MAX_UPLOAD_MB` (default 20 MB) enforced per route (`src/config.js:164`; e.g. `borrower.js:1985-1986`, `staff.js:5106-5107`).
- **`sniffKind()` / `expectedKind()`** (`upload-bytes.js:94-127`): magic-byte sniffing (PDF anywhere in first 1KB, PNG/JPG/ZIP-Office/GIF/HEIC/TIFF/HTML) used by the SharePoint integrity audit to flag files that were **already garbage at upload** ("source-suspect", e.g. an HTML e-sign error page saved as `.pdf`).
- **`safeFilename()`** (`upload-bytes.js:138-155`): strips control/path/Windows-reserved chars, caps at 160 chars preserving extension — filename flows into DB and email templates.

### Virus / malware scanning — what exists and what doesn't

- **The portal itself performs NO antivirus scanning** at upload time. `grep -rni 'virus|clamav|malware'` matches only the SharePoint layer.
- **Indirect scanning via Microsoft Defender on the SharePoint mirror**: `src/lib/sharepoint.js:208` selects the `malware` facet on driveItem metadata; the verify audit stamps `malware-flagged` and opens a `sync_review_queue` row telling staff to check the source file (`src/lib/sharepoint-backup.js:991-1009`); admin health rollup counts `malware_flagged` (`sharepoint-backup.js:1304`).
- Validation is therefore: strict base64 decode → size cap → (async) magic-byte type-mismatch sniff → (async) Defender verdict on the mirrored copy. **Download-side XSS defense** compensates for the lack of inline scanning (see §6).

## 3. Review lifecycle — `db/013_document_review.sql`

- Columns (`db/013:11-18`): `review_status IN ('pending','accepted','rejected','superseded')` default `'pending'`; `rejection_reason`; `reviewed_by` (FK `staff_users`); `reviewed_at`; `replaces_document_id`; `is_current` boolean default true. Indexes on `(application_id, review_status)` and `checklist_item_id` (`db/013:20-21`).
- **Core invariant** (header, `db/013:3-7`): *only `accepted` + `is_current` documents ever count as part of the file* — the clean-file/TPR export builds on this (`getApprovedDocuments`, `staff.js:5420-5430`: `review_status='accepted' AND is_current=true`, including the vesting LLC's docs).
- **Review endpoint** `POST /documents/:id/review` (`staff.js:5182-5255`):
  - `accept` requires the `sign_off_conditions` capability (processor/admin); **anyone on the file may `reject`**, and a rejection reason is mandatory (`staff.js:5188-5192`).
  - Accept flips the linked checklist item to **`received` — NOT `satisfied`** (owner-directed 2026-07-12): the condition only closes via the separate `signOffGate`, so a multi-document condition can't "fly away" when one doc is accepted (`staff.js:5244-5249`). Reject → item `issue`.
  - **Accept + request-more**: doc stays accepted, condition reopens `outstanding` with a required borrower-facing "Still needed: …" note written into `borrower_hint` (`staff.js:5194-5237`).
- **Supersede mechanics**: every re-upload marks prior versions `is_current=false` and flips `pending|rejected → superseded` (accepted stays accepted, just non-current) — same SQL pattern at all upload sites (`borrower.js:2016-2046`, `staff.js:5128-5155`, `tpr-export.js:529-533`). Scoping rules: with a `slot_label` (or explicit `replaceDocumentId`) only that slot's chain supersedes; `doc_kind='track_record_doc'` evidence is never superseded by a condition-row upload; `term_sheet` supersedes app-wide by kind.
- **Hard delete** `DELETE /documents/:id` (`staff.js:5327-5417`): gated on `sign_off_conditions`; unlinks local bytes best-effort, hard-deletes the row, reopens the condition if no accepted+current doc remains, and **revokes LLC / track-record verification** if the deleted doc was current evidence. Never issues a SharePoint delete (mirrored copy stays).
- Downstream consumers of review state: LLC slot gating (`src/lib/llc.js:198-201, 249-250`), activity feed wording (`src/lib/activity.js:150-151`), TPR export excludes rejected (`src/lib/tpr-export.js:241,272,285`), staff condition views (`staff.js:2433,2445`).

## 4. Visibility model — `db/014_document_visibility.sql`

Born as a **privacy fix**: internal chat attachments were leaking into the borrower's document library (`db/014:23-31`).

- `source_type IN ('borrower_upload','staff_upload','chat_attachment','document_request','condition','tpr','post_closing','system')` default `'borrower_upload'`; `visibility IN ('borrower','staff_only','internal')` default `'borrower'`; `message_id` FK to messages (`db/014:34-40`). Backfill classifies existing chat attachments by channel (`db/014:48-54`).
- Semantics in practice:
  - **`borrower`** — visible/downloadable by the borrower. Borrower library list filters `visibility='borrower' AND source_type <> 'chat_attachment'` (`borrower.js:2150-2162`); borrower download hard-requires `visibility='borrower'` plus ownership scope plus a **co-borrower privacy clause** (a co-borrower on the shared file cannot pull the other person's personal docs — only own/file-level/LLC/staff/system docs, `borrower.js:2169-2188`).
  - **`staff_only`** — staff-audience condition uploads (Insurance/Title etc., `staff.js:5100-5102`), internal chat attachments.
  - **`internal`** — system artifacts that must never reach a borrower **or a buyer package**: TPR export zips are `visibility='internal'` explicitly so they are "structurally excluded from every future buyer package" (`tpr-export.js:516-526`).
- Staff-side access is *not* visibility-based but assignment-based: `canSeeDocument()` (`staff.js:5436-5460`) — `seesAll` roles see everything; otherwise an app-scoped doc requires assignment to **that** application (never borrower-wide fallthrough), borrower/LLC-scoped docs allow any assigned file of that borrower. Downloads stream via `serveDocument` after this check and are audit-logged (`staff.js:5462-5471`, `borrower.js:2186`).

## 5. Dedup — two complementary layers

### 5.1 Upload idempotency — `src/lib/doc-dedup.js`
`recentDuplicateDocId()` (`doc-dedup.js:21-43`): collapses a byte-identical re-POST (React double-invoke / double drop / client retry) of the **same filename+size+uploader+context** (`application_id`, `checklist_item_id`, `llc_id`, `track_record_id`, `slot_label`, `doc_kind`) within a **120s window** onto the already-saved `is_current` row — one document row, one notification. A real replacement (different bytes/size, or no longer current) inserts normally (`doc-dedup.js:15-18`). Callers: `borrower.js:264,1870,1998`; `staff.js:3420,3576,5109`. Returns 201 `{deduped:true}` with the existing id.

### 5.2 Mirror byte-dedup — `src/lib/sharepoint-backup.js`
Before uploading to SharePoint, exact **sha256+filename+scope** match against an already-healthy mirror ⇒ the row is settled pointing at the existing copy with `sharepoint_skipped_reason='duplicate bytes …'` — no second copy in SharePoint (`sharepoint-backup.js:629-676`). Never dedups a re-mirror of a diagnosed-corrupt copy. Additionally, autosaved regen-kind snapshots (`track_record_html`, `tpr_export`, `*_export`) **coalesce onto one documents row** within a 10-min window while un-mirrored (`track-record-snapshot.js:40-75`), and superseded-before-mirror snapshots settle without uploading.

## 6. Serving documents — `src/lib/serve-document.js`

- Caller must authorize first; then `storage.stream()` pipes with `Content-Length`. Bytes gone (pre-persistent-disk uploads) → **410 "file no longer available"** (`serve-document.js:17-25`).
- `content_type` is treated as attacker-controlled: non-printables stripped, fallback `application/octet-stream` (`:30`).
- **Inline rendering allowlist** (stored-XSS defense — an inline SVG/HTML would run on the portal origin and could read the auth token): only `application/pdf`, `image/(png|jpe?g|gif|webp|bmp)`, `text/plain` may render inline; everything else forced to `attachment` (`:31-36`). Plus `X-Content-Type-Options: nosniff` and a sandboxing CSP `default-src 'none' … sandbox` (`:39-42`).
- `safeName()` sanitizes the Content-Disposition filename (`:11-15`).

## 7. Packaging — `src/lib/zip.js` + TPR export

- `zip.js` is a dependency-free ZIP writer (STORE only, hand-assembled headers + CRC32 — repo policy allows only `express`+`pg`) (`zip.js:1-6`).
- Used by the TPR ("third-party review" / buyer package) export (`tpr-export.js:147,508`): pulls **accepted+current, never-rejected** docs into `Subject/<condition>/`, `Term Sheet/`, `Track Record/` folders with a `00_INDEX.txt` manifest naming who accepted each doc and listing open conditions with no document. The zip itself is persisted back as a `documents` row (`doc_kind='tpr_export'`, `source_type='system'`, `visibility='internal'`), superseding the previous export (`tpr-export.js:512-538`).

## 9. Inbound email attachments — `src/routes/inbound-file-email.js` + `src/lib/file-inbox.js` + `src/lib/file-address.js`

A **parallel, non-persisting** file path worth knowing about:

- Every file notification email carries `Reply-To: file+<applicationUUID>@CHAT_REPLY_DOMAIN` (`file-address.js:26-31`); replies hit the Resend `email.received` webhook at `/api/inbound/file-email`, signature-verified over raw bytes, fail-closed when unconfigured (`inbound-file-email.js:35-50`).
- The webhook has **metadata only**; the body is fetched from the Resend Receiving API and **forwarded (branded) to every active assignee** on the file — sender's raw HTML is never inlined (phishing/tracking vector), only extracted text + hyperlink targets (`file-inbox.js:269-306`).
- **Attachments are forwarded transiently, capped (8 MB each / 15 MB total, 2.5 MB under Graph, max 10) and best-effort — they are NEVER saved into `documents` or local storage** (`file-inbox.js:49-58,184-217`). If attachment forwarding fails, text retries alone.
- Ledger table `inbound_file_emails` (`db/116:70-86`): unique `resend_email_id` idempotency claim + per-app `app_results`, retryable-vs-terminal statuses (`received, forwarded, no_recipients, unknown_app, archived_app, auto_reply, rate_limited, chat_posted, retrieval_failed, forward_failed, lookup_failed, failed_permanent`), lease-based reclaim of crashed runs (10-min stuck window, 8 attempts), auto-reply/bounce suppression (RFC 3834 heuristics), 20 forwards/file/hour breaker, **no email bodies ever persisted** (`db/116:67-68`, `file-inbox.js:19-40, 328-362, 370-613`).
- Gap worth noting: a borrower emailing a document to the file address gets it to staff inboxes, but it never enters the portal's document/review/mirror pipeline — staff must manually re-upload it.

## 10. SharePoint backup linkage — `db/091` + `src/lib/sharepoint-backup.js`

- `db/091` (originally a research spike, now live) adds the mirror-tracking columns (§1.2) and a partial index for the pending scan (`db/091:19-21`). The mirror is **one-way, append-only, no-delete** (policy in `docs/SHAREPOINT-POLICY.md`).
- Architecture: a **reconciler chokepoint**, not per-endpoint code — it scans un-mirrored `documents` rows and uploads bytes from local storage to `Pipeline Drive/<Officer>/<Borrower>/<Address>/Synced by Pilot/<Condition>/`; upload endpoints call `kick()` so new docs mirror within seconds (`sharepoint-backup.js:1-29`; kick calls at `borrower.js:2054`, `staff.js` upload paths, `tpr-export.js:537`, `track-record-snapshot.js`).
- Routing: `scopeKeyFor()` (`sharepoint-backup.js:188-199`) — photo IDs and track-record docs always file at the **borrower profile** level; otherwise app scope, then borrower. Version streams per checklist item or per kind (`stateKeyFor`, `:207-212`); human uploads get the owner's Version-1/2/N shuffle on supersede; regen kinds never version-shuffle (`:44-58`).
- Integrity: local size check before mirroring, sha256, QuickXorHash verification, Office-format promotion exception, `malware` facet check, corrupt-mirror re-upload as "(fixed copy)" with the single seven-guarded sanctioned delete (`:604-700, 852-1060`).
- Failure surfacing ties into **`src/lib/sync-file-review.js`**: reasons `sharepoint_match_uncertain` → action `sp_rematch` (invalidate folder cache, re-match after a human fixes folders) and `sharepoint_mirror_failed` → `sp_retry_doc` (reset `sharepoint_backup_attempts/error/backed_up_at/skipped_reason` and `kick()`) / `sp_rematch` (`sync-file-review.js:41-47, 198-238`). All stuck-state resolutions run the sync's own guarded machinery; a reviewer's dismiss sticks. This is the portal's established "stuck integration state → review row WITH OPTIONS" pattern.

## 11. Lifecycle summary (end-to-end)

1. **Ingest**: client sends `{filename, contentType, dataBase64}` → `decodeUploadBase64` (strict, 400 on junk) → size cap → `recentDuplicateDocId` idempotency (120s) → `storage.save` (atomic, sharded, opaque ref) → `INSERT documents` (`review_status='pending'`, `is_current=true`, visibility per audience) → supersede prior versions in the same slot/kind stream → checklist item `received` → notifications + audit row → `sharepoint-backup.kick()`.
2. **Review**: processor accepts (item `received`, awaiting sign-off) or anyone rejects with reason (item `issue`); accept+request-more keeps the condition open. `signOffGate` alone flips `satisfied`.
3. **Consumption**: borrower sees only `visibility='borrower'` non-chat docs; staff access by assignment; all downloads stream through the hardened `serveDocument`; the "clean file" (exports, buyer package) = accepted+current only.
4. **Mirror**: async one-way SharePoint copy with byte-dedup, versioning, integrity audit, Defender malware verdicts, and review-queue escalation.
5. **Removal**: sign-off-capable staff can hard-delete (bytes + row); verification built on deleted current evidence is revoked; the SharePoint copy is never deleted by the system.

### 12.1 Recommended representation: metadata first, bytes later

**Phase 1a — metadata-only "pulled artifacts."** Do NOT insert Encompass docs into `documents` while byteless. The whole pipeline assumes `storage_ref` points at real local bytes: `serveDocument` would 410, the SharePoint reconciler selects any row with bytes-expectations and would fail rows into the retry/review machinery, `getApprovedDocuments`/TPR would try to read them, and `DELETE` semantics differ. Instead create a sibling table (new idempotent migration, repo pattern), e.g. `encompass_documents`:

- Identity: `application_id` (portal FK), `encompass_loan_guid`, `efolder_document_id`, `attachment_id`, `title/type`, `received/created dates`, `file_size`, `page_count`, `is_removed` (from `includeRemoved=true`), `condition_ids` (from the condition-documents endpoint), `last_seen_at`, `source_hash` of the metadata for change detection.
- This mirrors how the portal already treats ClickUp: an external system's state is indexed locally (`clickup_task_index` snapshot pattern) and compared, never trusted as portal-native rows.
- Upsert on `(encompass_loan_guid, efolder_document_id, attachment_id)`; never delete — mark `is_removed`/`superseded_in_encompass`, matching the portal's own "supersede, never destroy" document philosophy (`db/013`).

**Phase 1b — bytes on demand / lazy hydration.** When staff actually opens an Encompass artifact (or a rule needs page-level evidence), call `attachmentDownloadUrl`, fetch bytes server-side, then land them through the **existing chokepoints**: `decodeUploadBase64`-equivalent is unnecessary (bytes arrive binary), but reuse `sha256hex`, `sniffKind` mismatch flagging, `safeFilename`, and `storage.save()`; only then insert a `documents` row so serving, dedup, integrity audit, and the SharePoint mirror all work unchanged. Suggested stamps: `source_type='system'` (or a new enum value `'encompass'` added via CHECK-widening migration, pattern `db/123:95-98`), `doc_kind='encompass_efolder'` (or per-type), `uploaded_by_kind='staff'` with a dedicated service actor, plus `encompass_documents.document_id` linking metadata → hydrated bytes. Byte-dedup falls out for free: `documents.sha256` + the mirror's sha-dedup prevent double-storage when Encompass holds a copy of something the borrower already uploaded (and a sha256 match is itself a useful "portal doc == Encompass doc" agreement signal for condition rules).
- Consider making `doc_kind LIKE 'encompass%'` a **non-version-shuffling stream** in the mirror (extend `isRegenKind`/`REGEN_KIND_SQL`, `sharepoint-backup.js:57-58`) or excluding it from mirroring entirely — Encompass is itself a durable system of record; re-mirroring its bytes to SharePoint may be pure churn. Decide deliberately; default OFF is safer.

### 12.2 Review/visibility rules for Encompass-sourced documents

- **`review_status`: introduce a distinct terminal state rather than reusing the human ladder.** Encompass docs are not "pending borrower-review"; they are already-adjudicated LOS artifacts. Two workable options: (a) insert hydrated rows with `review_status='accepted'`, `reviewed_by=NULL`, and a marker (`source_type='encompass'`) so audit/activity wording can say "pulled from Encompass" instead of "accepted a document" (`activity.js:150` would need a branch); or (b) widen the CHECK with `'external'`. Option (a) is less invasive but blurs "a human accepted this"; option (b) is cleaner for compliance because `reviewed_by` stays honest. **Recommendation: (b)** — a new `review_status='external'` (or `source_verified`) that `getApprovedDocuments`/TPR treat as accepted-equivalent only if the business decides Encompass docs belong in buyer packages at all.
- **Portal staff must NOT be able to accept/reject/delete an Encompass-sourced row in any way that implies changing Encompass**: the read-only doctrine means portal-side review actions on these rows are local annotations at most. Reject/delete endpoints should refuse (`403`/`409`) on `source_type='encompass'` rows, or clearly re-badge as "hide in portal" — never wording that suggests the eFolder changed. The existing hard-delete revocation logic (`staff.js:5380-5407`) must never fire off an Encompass row deletion.
- **Supersede/versioning**: driven ONLY by Encompass state (a new attachment version observed on poll ⇒ old hydrated row `is_current=false`), never by a portal upload. A borrower re-upload must not supersede an Encompass artifact and vice versa — keep them in separate `doc_kind` streams so the existing slot/kind supersede SQL can't cross them (the `COALESCE(doc_kind,'') <> 'track_record_doc'` exclusion at `borrower.js:2036-2044` is the exact precedent to copy).
- **`visibility`: default `staff_only`.** eFolder contents routinely include appraisals-in-review, title work, internal underwriting worksheets, credit and background material that the portal's borrower-facing rules (capital-partner-name scrubbing, Gold Standard aliasing, co-borrower privacy) have never sanitized. Borrower exposure should be a deliberate per-document (or per-eFolder-document-type allowlist) staff action that flips `visibility` to `borrower` — matching the existing model where staff-audience condition uploads are born `staff_only` (`staff.js:5100-5102`). `internal` for anything machine-generated (export jobs, audit bundles), which also keeps it out of buyer packages structurally (`tpr-export.js:516-521`).
- **Condition-agreement rules** (the Phase-1 headline feature): the `conditions/{id}/documents` linkage + `encompass_documents` metadata gives the evidence needed for "a portal condition cannot be cleared unless the matching Encompass data agrees" — wire it as an additional branch in `signOffGate` (the same place the SOW budget and experience gates live), and surface disagreements through the established `sync_review_queue` pattern with explicit reviewer OPTIONS (`sync-file-review.js` `REASON_ACTIONS`), e.g. `encompass_doc_disagrees: ['refresh_from_encompass','acknowledge_mismatch']`. Nothing auto-clears; nothing writes back.

### 12.3 Open questions

1. Should Encompass-hydrated bytes mirror to SharePoint at all (churn + duplication vs. one-tree completeness)? Default proposal: no.
2. Do Encompass docs ever belong in the TPR/buyer package (`getApprovedDocuments`)? If yes, `'external'` review status must be added to its WHERE clause deliberately.
