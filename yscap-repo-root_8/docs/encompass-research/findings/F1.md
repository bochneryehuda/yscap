# F1 — Read-Only Enforcement Design: making a write to Encompass architecturally impossible

**Agent:** F1 ("Read-only enforcement design") | **Date:** 2026-07-19 | **Design only — no implementation.**

## 0. Design thesis

The portal already has a proven doctrine for integration safety, stated in `src/clickup/client.js:15-21` and `docs/CLICKUP-DATA-SAFETY.md`: **structural incapability** — dangerous operations are not "not called," they *throw at a single chokepoint* so that no present or future code path, refactor slip, or copy-paste can perform them. Phase 1 Encompass inverts the ClickUp problem: instead of "writes are allowed but destructive writes are impossible," the invariant is simpler and absolute — **no write, ever**. That simplicity lets us go further than ClickUp did:

1. **There are zero write functions** (the strongest guarantee — there is nothing to call; the `src/clickup/enqueue.js:6-9` "no delete op exists" precedent, and the `sharepoint.remove()` permanent throwing no-op precedent cited in CLAUDE.md).
2. **The transport itself refuses anything not on an explicit method+path allowlist** (deny-by-default), because in this API neither the HTTP method nor the path alone is a safe signal (§2).
3. **Five independent layers** each independently sufficient to stop a write: Encompass-side persona → client allowlist → no-write-function surface → kill switches → CI guard suite; with a READ journal making every outbound call queryable and attributable.

A bug would have to defeat all five at once. That is what "architecturally incapable of writing" means here.

## 1. Why method-only filtering fails (the case for an explicit method+path allowlist)

A naive client that "only allows GET" or "blocks PUT/PATCH/DELETE" fails **in both directions** against this API surface (all counts/lines from C7 and `postman-endpoint-index.md`):

### 1.1 Reads that are POSTs — a GET-only client cannot do Phase 1's job

| Read-via-POST endpoint | Why Phase 1 needs it |
|---|---|
| `POST /oauth2/v1/token` (+ `/token/introspection`) | The OAuth2 grant itself. No POST → no token → no integration. |
| `POST /encompass/v{1,3}/loans/{id}/fieldReader` | The bulk field-value workhorse — the primary primitive for the "portal condition cannot clear unless Encompass agrees" rule (C7 §6.1 B3/B4). |
| `POST /encompass/v{1,3}/loanPipeline` (+ `/report`) | Loan discovery/sync — how the portal finds which loans exist and changed (C7 §6.1 C1–C3). |
| `POST /encompass/v3/loans/{id}/auditTrail` | Field-change history query; body is `{"fieldIds":[...]}` + paging — verified pure read (C7 index line 182 / findings line 295). |
| `POST /encompass/v1/schema/loan/pathGenerator`, `contractGenerator` | Field-ID → JSON-path/contract translation (schema tooling, C7 §6.1 D3/D4). |
| `POST .../attachmentDownloadUrl`, v1 attachment `/url` generators, `exportJobsCreator` | The only sanctioned document *download* mechanisms if Phase 1 pulls any eFolder content (C7 §6.1 G5–G7). |

Of the collection's 200 POSTs, only 49 are read-semantics and 4 are auth (C7 §1). **"Allow POST" admits ~147 write requests; "deny POST" kills the integration.** Method-only filtering is therefore unusable in either polarity.

### 1.2 GETs that are not safely readable — an allow-all-GET rule is also wrong

- **Two GETs carry `action=Add`** (`GET .../urlaAlternateNames?action=Add&view=entity`, index lines 448-449) — query strings copied from PATCH siblings. In this API `?action=` is *the mutation switch* on PATCH (`action=add|update|delete|clear|confirm|extend|relock|move`). Almost certainly ignored on GET, but a strict client must **strip/deny any `action` query parameter** rather than trust that (C7 §3, §6 design rule).
- **`GET /encompass/v3/loans/{id}/recipients`** may *mint/rotate disclosure auth codes* (credential issuance — a side effect) and returns borrower access secrets. Deny (C7 §3, open question 1).
- **`GET .../loanContractGenerator`** is literally named "Create Loan Contract from Scenario" (index line 41) — generator semantics, but "verify before ever allowing."
- **11 opaque `{{locationHeader}}`/presigned GETs** (index lines 112-143, 531) have URLs unknown at allowlist time — an allowlist keyed on our own path templates can't see them; they need resolved-host+path-prefix validation (§3.6).
- **PATCH is the write verb of this API** (214 of 800 requests) and several writes hide behind innocuous-looking POSTs (`planCodes/{id}/evaluator` with `"import":"all"` in the body; EPPS `rateSelector` staging lock state; `calculators/automatedConditions` possibly applying conditions — all 8 AMBIGUOUS are denied, C7 §4).
- **The collection is 26.2-era.** The live API may expose endpoints the collection doesn't list; only **deny-unknown** (never "deny-known-writes") survives an API upgrade (C7 §8.6).

**Conclusion:** the guard must be the `guardTaskUpdatePayload` shape (ALLOWLIST, `src/clickup/client.js:80-99`), not the `guardNoTaskDeletion` shape (targeted blocklist, `client.js:29-39`): an explicit table of `(method, anchored-path-pattern, query rules, body rules)`, deny-by-default, enforced inside the one function that owns the network socket.

## 2. Layer 1 — the single-chokepoint client (`src/encompass/client.js`)

### 2.1 Module layout

```
src/encompass/
  allowlist.js   ← FROZEN data module: the (method,path,query,body) table + tripwires. No logic beyond compile.
  client.js      ← FROZEN transport: token handling, guardReadOnly(), request(), pacing, retry, journal hook.
  reads.js       ← typed read helpers (getLoan, readFields, queryPipeline, getConditions, getMilestones,
                    getDocuments, …). The ONLY module the rest of the codebase imports.
```

House-style anchors: one chokepoint like `call()` (`src/clickup/client.js:173-214`); guards exported *only* for the no-network test suite (`client.js:310-312` precedent); PII-free, value-free error strings (`httpError`, `client.js:139-145`); token bucket + Retry-After + timeout ported from `client.js:101-213`.

Key differences from the ClickUp client, all deliberate:

1. **`request()` is module-private.** ClickUp exports `call` (`client.js:306`); Encompass must not — every caller goes through a named, typed read helper in `reads.js`. A generic "call any path" function is exactly the hole the allowlist would otherwise have to be perfect against. (The guard still runs inside `request()` — belt and suspenders — but the export surface never offers raw path construction to callers.)
2. **Zero write functions exist.** Not "blocked writes" — *absent* writes. No function in any `src/encompass/*` module ever sets `method` to PUT/PATCH/DELETE or builds a non-allowlisted POST. ClickUp keeps `removeTaskFromList` for interface symmetry and lets the guard block it (`client.js:280-283`); Encompass keeps nothing. The literal strings `'PUT'`, `'PATCH'`, `'DELETE'` should not appear in `src/encompass/` at all (CI-enforced, §4.3) — deny-by-default means the guard needs no verb enumeration to refuse them.
3. **IDs, never paths, cross the module boundary.** `reads.js` helpers accept a loan GUID etc., validate its shape (`/^[0-9a-fA-F-]{32,36}$/` — reject anything containing `/ ? # % .`), and interpolate into a compiled template. A caller cannot smuggle `"{id}/fieldWriter?x="` through an ID parameter.

### 2.2 The allowlist (`src/encompass/allowlist.js`)

Data shape (illustrative sketch — design, not implementation):

```js
// FROZEN FILE — see docs/ENCOMPASS-READ-ONLY-POLICY.md before touching.
// Every entry is a proven-read endpoint from the C7 endpoint atlas.
// Adding an entry = policy change: hash pin + owner sign-off + audit gate.
const ALLOW = [
  // A. Auth plumbing
  { id: 'A1', method: 'POST', re: /^\/oauth2\/v1\/token$/,                       query: [] },
  { id: 'A2', method: 'POST', re: /^\/oauth2\/v1\/token\/introspection$/,        query: [] },
  // B. Loan reads
  { id: 'B1', method: 'GET',  re: /^\/encompass\/v3\/loans\/[0-9a-f-]{32,36}$/i, query: ['view','entities'] },
  { id: 'B3', method: 'POST', re: /^\/encompass\/v3\/loans\/[0-9a-f-]{32,36}\/fieldReader$/i,
              query: [], body: bodyIsFieldIdArray },
  { id: 'B5', method: 'POST', re: /^\/encompass\/v3\/loans\/[0-9a-f-]{32,36}\/auditTrail$/i,
              query: ['includeHistoricalData','ignoreInvalidFields','start','limit'], body: bodyIsAuditTrailQuery },
  // C. Pipeline … D. Schema … E. Conditions … F. Milestones … G. eFolder metadata … H. Webhook reads
  //   (full membership = C7 §6.1 tables A–H, verbatim; nothing else)
];
```

Rules baked into the matcher:

- **Anchored regexes** (`^…$`) against the *decoded, normalized* path: percent-decode once, reject a path that decodes to contain `..`, `\`, `//`, or another `%` (double-encoding); strip and separately validate the query string. Path normalization happens **before** matching so `/encompass/v3/loans/x%2FfieldWriter` can't slip a template.
- **Per-entry query allowlist.** Unknown query params are **denied, not stripped-and-forwarded** (fail closed, log which param tripped). `action` is globally denied on every entry — even where a copied `action=Add` would probably be ignored (index 448-449); "probably ignored" is not a guarantee we accept.
- **Per-entry body validator** for the read-via-POST endpoints. `fieldReader` bodies must be an array (or `{fieldIds:[…]}`) of field-ID strings; `loanPipeline` bodies must contain only `filter/fields/sortOrder` keys; `auditTrail` only `fieldIds` + paging. This prevents an allowlisted POST from being *repurposed* — and specifically blocks the `"import"` key family that makes `planCodes/{id}/evaluator` ambiguous (C7 §4). Unknown body keys → throw. (Same allowlist-of-keys philosophy as `TASK_UPDATE_ALLOWED_KEYS`, `src/clickup/client.js:84`.)
- **Host pinning.** `request()` builds URLs from a single `BASE` constant (the Encompass API host from `src/config.js`); it never accepts a caller-supplied absolute URL. The optional Location-header/presigned follow (§2.4) is the one structured exception.
- **Two distinct refusal codes**, both value-free:
  - `ENCOMPASS_NOT_ALLOWLISTED` — anything that simply isn't on the table (deny-by-default).
  - `ENCOMPASS_WRITE_FORBIDDEN` — the request *additionally* matched a **deny tripwire**: the high-risk look-alikes from C7 §6.3 (`fieldWriter`, `.../importer`, `loanBatch/updateRequests`, `attachmentUploadUrl`/`attachmentUrl`, `documentOrders/**` + `/delivery`, `RatelockRequests`, `resourceLocks`, `?action=` mutations, folder moves, `/consumers/v1/invitations|reminders`, partner `transactions`, `ComplianceReports`, `/scim2/`, users/settings writes, `customObjects` writes). Tripwires never *allow* anything (they are a subset of the default deny); their sole purpose is a louder failure: a distinct code, a `console.error`, an `audit_log` row, and a `blocked=true` journal row (§4.4) — because a tripwire hit means **portal code attempted a write**, which is a build-breaking bug, not a runtime condition to retry.
- The matrix (allowlist + tripwires + the §1 counterexamples) is locked by the guard test suite (§4.3), run with **no DB / no network** like `scripts/test-clickup-delete-guard.js:5`.

### 2.3 `request()` internal order of operations

```
request(entryId, params) →
  1. kill-switch check (cfg.encompassEnabled; off → throw ENCOMPASS_DISABLED)      [§4.2]
  2. resolve allowlist entry by id; build path from template + validated IDs
  3. guardReadOnly(method, path, query, body)  ← throws before ANY socket work
  4. read-budget token check (per-window call budget)                              [§4.6]
  5. token bucket pacing (RPM under the vendor limit)                              [client.js:147-161 port]
  6. fetch with timeout; Retry-After/backoff retry contract                        [client.js:173-213 port]
  7. journal the call — success, failure, or blocked — to encompass_request_log    [§4.4]
```

The guard runs *inside* the same function that owns `fetch` — there is no code path from "guard passed" to "different URL sent." Error objects carry `e.status/e.retryable/e.retryAfter` for the durable pull queue, message format `"Encompass GET /encompass/v3/loans/{id} -> 429"` — **path template, never the raw path** in messages that might carry a loan GUID into borrower-visible errors, and never values (the `httpError` discipline, `client.js:138-145`).

### 2.4 The Location-header / presigned-URL exception (deferred by default)

If Phase 1 ever polls docs-service status or downloads report files, those URLs arrive opaque (index lines 112-143, 531). Design: a dedicated `followServiceUrl(url, { expectHostSuffix, expectPathPrefix })` that parses the URL, requires an exact allowlisted host, an allowlisted path prefix (e.g. `/encompassdocs/v1/`), method GET, no `action` param — and journals under its own route id. **Not needed for the C7 §6.1 core allowlist; ship Phase 1 without it** and add it only with the same policy ceremony as any allowlist change.

### 2.5 Token handling

Client-credentials grant against a **read-only API user persona** (§3), tokens held in memory only — never DB, never logs, never `audit_log.detail` (A5 checklist A4). Config follows the house pattern: an `encompass:{}` block in `src/config.js` (env-only via `resolveSecret`-style handling, integration dormant until configured, surfaced in `src/lib/integrations/index.js configured()` — A5 §7).

## 3. Layer 0 — Encompass-side least-privilege persona (vendor-enforced read-only)

The strongest statement of "architecturally incapable" is one *our code cannot influence at all*: the credential itself cannot write.

- Provision a **dedicated API user** on instance **BE11397907** for the portal (client-credentials grant), with an Encompass **persona stripped of every write right** the admin console allows: no loan edit, no eFolder upload, no disclosure ordering, no rate-lock, no settings, no user management. Then even a bug that somehow defeated every code layer receives a 403 from ICE. (A1 open question 1; A5 checklist A3; C7 §8.4 — persona granularity must be confirmed with ICE, it is the top open question.)
- **Dedicated identity also buys attribution** — the ClickUp forensics lesson (`docs/CLICKUP-DATA-SAFETY.md` §1: a second automation writing under the same token identity muddied an incident; SYNC-GUARD-MATRIX #80). Every Encompass audit-trail entry from this integration must be unambiguously "the portal read this."
- **Two credentials, not one** (C7 §6.2): the runtime read-only credential, and a separate **admin credential used only for webhook subscription management** (§5) that the portal server process never loads. If ICE personas can't fully remove write rights, this layer degrades gracefully — the four code layers still hold — but push hard for it; it converts "we promise" into "ICE enforces."

## 4. Layer 2 — defense in depth

### 4.1 Summary stack

| # | Layer | Stops a write because… | Precedent |
|---|---|---|---|
| 0 | Read-only Encompass persona (§3) | ICE returns 403 regardless of our code | matrix #80 lesson; A5 A3 |
| 1a | No write function exists (§2.1) | there is nothing to call | `enqueue.js` has no delete op; `sharepoint.remove()` no-op |
| 1b | Allowlist guard in `request()` (§2.2) | non-allowlisted method+path throws pre-socket | `guardTaskUpdatePayload` allowlist shape |
| 2 | Kill switches, default off (§4.2) | integration is dark until deliberately lit; no outbound flag exists to turn on | `config.js:214-232` |
| 3 | CI guard suite + source scan (§4.3) | a new call site / write shape fails the build | `test-clickup-delete-guard.js`; drift-check style `test-sync-file-review.js:179-195` |
| 4 | READ journal of every outbound call (§4.4) | every call (incl. blocked) is queryable; a write-shaped call is an alarm | `clickup_write_log`, `db/107` |
| 5 | DB constraint: no ('encompass','push') queue row (§4.5) | a push job is unrepresentable in storage | `sync_queue` schema `db/schema.sql:302-314`; `db/069` trigger style |
| 6 | Read-budget circuit breaker (§4.6) | a runaway loop stops hard | orchestrator breaker `:479-507`, journal-backed this time |

### 4.2 Environment kill switches (`src/config.js` additions — all default OFF)

| Flag | Default | Effect |
|---|---|---|
| `ENCOMPASS_ENABLED` | off | Master switch. Client (`request()` step 1), scheduler, and ingest all no-op/refuse when off — gate at every layer like `CLICKUP_SYNC_ENABLED` (orchestrator.js:214 / enqueue.js:28 / clickup-sync.js:1191). |
| `ENCOMPASS_DRYRUN` | off | Boot dry-run: authenticate, pull sample loans/pipeline, run the mapper, log what WOULD land in the portal, **start no loops, write no rows** (the `dryRunBackfill` pattern, clickup-sync.js:1102-1138). The go-live validation tool for BE11397907. |
| `ENCOMPASS_MAX_CALLS_10MIN` | e.g. 600 | Read-budget breaker cap (§4.6). |
| `ENCOMPASS_MAX_RPM` / `ENCOMPASS_MAX_TRIES` / `ENCOMPASS_TIMEOUT_MS` | ~conservative | Pacing/retry/timeout knobs (client.js:115-117 pattern). |

**Deliberately absent, permanently:** `ENCOMPASS_OUTBOUND_ENABLED`. On the ClickUp side outbound is a *gate to open* (`config.js:217`); on the Encompass side there is nothing behind the gate. The boot log states the mode the way `clickup-sync.js` states its mode: `"[encompass] READ-ONLY integration — zero write paths exist in this build"`. `/api/health` surfaces `encompassConfigured/encompassEnabled` like `storageWritable` (A5 F20). The CI scan (§4.3) greps that the string `ENCOMPASS_OUTBOUND` appears nowhere in `src/` — the flag can't even be *added* without failing the suite.

### 4.3 CI guard suite — `scripts/test-encompass-readonly-guard.js` (no DB, no network)

House style: standalone assertion script, `pass/fail` counters, exit 1 on failure (`test-clickup-delete-guard.js`), plus the `readFileSync` source-scanning drift-check style already proven in `scripts/test-sync-file-review.js:179-195`. Sections:

1. **Guard matrix — deny side.** Every C7 §6.3 tripwire, every §4 AMBIGUOUS endpoint, every PATCH/PUT/DELETE shape, `POST .../fieldWriter`, `GET .../recipients`, `GET …?action=Add`, an unknown made-up path, and a known-good path with an unknown query param → all must throw with the right code. Assert the §1 proof pairs explicitly: `POST fieldReader` allowed **and** `POST fieldWriter` (same prefix, one word different) blocked — the regression that proves method-only filtering was never reintroduced.
2. **Guard matrix — allow side.** Every allowlist entry id passes with representative params; body validators accept the documented shapes and reject `{"import":"all"}`-style extras.
3. **Export-surface audit.** `require('../src/encompass/client')` and assert: no exported symbol issues a write; no raw `request`/`call` export; only `reads.js` helpers + the guards-for-test exports exist.
4. **Grep-based source scan (the "no non-allowlisted call site" gate):**
   - The Encompass API host literal appears in exactly two files: `src/config.js` and `src/encompass/client.js`. Any other occurrence in `src/` (a second fetch path) fails.
   - `fetch(` does not appear in `src/encompass/` outside `client.js`.
   - The strings `'PUT'`, `'PATCH'`, `'DELETE'`, `fieldWriter`, `attachmentUploadUrl`, `documentOrders`, `resourceLocks` appear nowhere in `src/encompass/` (outside the tripwire table in `allowlist.js`, which is asserted by hash instead).
   - `ENCOMPASS_OUTBOUND` appears nowhere in `src/`.
   - Every `require(...'encompass/` import outside `src/encompass/` resolves to `reads.js` only (no direct `client.js`/`allowlist.js` consumers).
5. **Frozen-file pin.** sha256 of `src/encompass/allowlist.js` (and of the guard block of `client.js`, delimited by markers) equals a constant pinned in the test. Any edit — even a "harmless" one — is a two-file change that a reviewer must consciously approve (§6).
6. **Queue guard.** The pure enqueue helper refuses `direction:'push'` for `target:'encompass'` (unit-level mirror of §4.5).

Wire it into the "suites that must stay green" invariant list the way `docs/CLICKUP-DATA-SAFETY.md` §6.6 pins its suites; given SYNC-GUARD-MATRIX gap #19/#55 (no CI runs the green suites today), recommend this integration be the one that finally adds a minimal CI step (or at least a pre-merge hook) running `node scripts/test-encompass-readonly-guard.js` — the whole suite is dependency-free and sub-second.

### 4.4 The READ journal — `db/1NN_encompass_read_journal.sql` (mirror of `db/107`, pull side)

Append-only `encompass_request_log`, one row per **outbound call attempt** — success, HTTP failure, or guard-blocked:

| Column | Meaning |
|---|---|
| `route_id` text | allowlist entry id (`'B3'`) or `'(denied)'` |
| `method`, `path_template` text | template, **never** the raw path (GUIDs go in their own column; nothing free-form) |
| `loan_guid` text NULL | the loan touched, when applicable |
| `query_params` jsonb | sanitized param names+values (no PII lives in allowlisted params) |
| `status` int, `ok` bool, `duration_ms` int, `response_bytes` int | outcome; **response bodies are never stored here** |
| `blocked` bool, `block_code` text | `true` + `ENCOMPASS_WRITE_FORBIDDEN` / `ENCOMPASS_NOT_ALLOWLISTED` / `ENCOMPASS_BUDGET_OPEN` — refusals are journaled exactly like `clickup_write_log.blocked` (`db/107` header) |
| `source` text | `'poll' \| 'staff_refresh' \| 'boot_dryrun' \| 'webhook_replay'` |
| `actor_kind`, `actor_id` | `'system'` for scheduled pulls; real staff id for manual refreshes (A5 D13) |
| `created_at` | — |

Indexes per `db/079` style (time-desc, loan_guid, partial on `blocked`). Best-effort insert — a journal failure never blocks a read (orchestrator.js:531 precedent). Purpose, stated in the migration header exactly like `db/107`'s: *attribution* — "if a call is not in this journal, the portal did not make it," and the converse compliance proof: **a query showing zero write-shaped rows is standing evidence the integration is read-only in practice**, not just by design. The journal also feeds the deploy-surviving read budget (§4.6).

Sibling (owned by the ingest workstream, noted for completeness): an `encompass_pull_field_change` adoption journal — before/after portal values, SSN masked via `maskSSN` before insert (orchestrator.js:509-527 discipline) — is layered on top for every portal field changed *because of* Encompass data. The request log records "we asked"; the adoption journal records "we changed ourselves accordingly."

### 4.5 DB layer — make an Encompass push row unrepresentable

`sync_queue` already constrains `target IN ('clickup','encompass','graph')` and `direction IN ('push','pull')` (`db/schema.sql:302-314`) but permits the pair `('encompass','push')`. Since `schema.sql` is never edited (CLAUDE.md migration rule), add a new numbered idempotent migration:

- `ALTER TABLE sync_queue ADD CONSTRAINT sync_queue_encompass_pull_only CHECK (target <> 'encompass' OR direction = 'pull')` inside an idempotent guard (catalog check / `DO $$` block), **or** a `db/069_sow_budget_guard.sql`-style trigger if a data backfill wrinkle makes the CHECK unsafe to validate. This is the belt-and-suspenders layer A1 noted ClickUp achieves only in code (A1 §11.8).
- The app-level enqueue helper for Encompass work exposes **pull only** — no `direction` parameter at all (the "no delete op exists" shape, enqueue.js:6-9).

### 4.6 Read-budget circuit breaker (the inverted volume breaker)

Port the rolling-window breaker (orchestrator.js:479-507) as an API-call budget: more than `ENCOMPASS_MAX_CALLS_10MIN` calls in 10 minutes → every further `request()` throws `ENCOMPASS_BUDGET_OPEN` (queue retries later), opening audited once/minute. **Fix the known deploy-reset weakness this time** (SYNC-GUARD-MATRIX #4/#54): seed the in-memory window from `encompass_request_log` on boot (one cheap `count(*) WHERE created_at > now()-'10 min'`), so a crash-loop deploy storm can't reset the budget — this also addresses the boot-storm gap (#18/#53) before launch, per A1 §11.6. The single-instance deployment plus `sync_locks` leases for the poller (A5 open question) keep this simple.

## 5. The one permitted config write: webhook subscription management — policy decision framework

Receiving webhooks is read-only from Encompass's perspective; the only write the integration ever *needs* is subscription CRUD (`POST/PUT/DELETE /webhook/v1/subscriptions*` — WRITE_CONFIG, C7 §5.12/§6.2). Policy framework, applied in order — each question answered "no" escalates to the next tier:

**Q1 — Can a human do it in the Encompass admin UI instead?** If ICE's console can create the subscription(s) Phase 1 needs, choose that: the portal then makes **zero** config writes ever, and the runtime allowlist stays purely read. (Open question for the B-team; the collection can't answer it.)

**Q2 — If the API is required, does the *runtime* need it?** No — subscriptions are created once at setup and touched rarely. Therefore: **out-of-band admin tooling, separate credential** (C7 §6.2). Concretely:
- A standalone script (`scripts/encompass-webhook-admin.js`), modeled on `scripts/clickup-date-restore.js`: **dry-run by default, `--apply` to act** (script header :24 precedent), every action verified by re-read (`GET /webhook/v1/subscriptions/{id}`) and journaled + audited.
- It authenticates with **dedicated admin env vars the server runtime never reads** — not referenced anywhere in `src/config.js`'s exported config object, so the running portal process cannot even resolve the credential. The runtime client keeps subscription CRUD **denied** (it is not in §6.1; H2 grants GET only).
- Guardrails inside the script: subscription callback URLs must be under the portal's own `APP_URL`; the script may update/delete **only subscriptions whose endpoint matches that host** (never another integration's subscription — the `moveOwnItem` expected-parent philosophy); creation payloads come from a checked-in manifest, not CLI free-text.

**Q3 — Drift detection is a runtime READ, never a runtime fix.** The portal polls `GET /webhook/v1/subscriptions` (allowlist H2) and compares against the expected manifest: missing subscription → health warning; **unknown subscription pointing at a non-portal host → loud security alert** (someone else is exfiltrating events). The runtime never "heals" drift by writing — it reports, a human runs the admin script.

**Q4 — Anything beyond subscriptions (custom-auth functions, settings) stays denied** until it passes this same framework with owner sign-off (C7 defers `/webhook/v1/functions/auth*`; its `/test` endpoint triggers Encompass-outbound calls — keep denied).

This preserves the invariant in its honest form: *the portal runtime is architecturally incapable of any Encompass write; the organization retains a separately-credentialed, human-driven, dry-run-first path for the one config write the integration needs.*

## 6. Code-review rules and the frozen-file convention

### 6.1 Frozen files

Repo precedent: the pricing engines are "frozen source of truth… never change their numbers or logic except by an explicit owner-directed change (then update this note and re-freeze)" (CLAUDE.md). Apply the same convention, mechanically enforced:

- **`src/encompass/allowlist.js` and the guard block of `src/encompass/client.js` are FROZEN.** Each carries a header comment naming the policy doc and the rule: *"FROZEN — read-only enforcement chokepoint. Any edit requires: owner sign-off in their own words, an entry in docs/ENCOMPASS-READ-ONLY-POLICY.md, an updated hash pin in scripts/test-encompass-readonly-guard.js, and the mandatory two-audit-agent gate."*
