# F5 — Consolidated Failure-Mode Analysis & Risk Register (Encompass Read-Only Integration)

**Agent:** F5 ("Failure modes and risk register") | **Date:** 2026-07-19 | **Instance:** BE11397907 (production)
**Primary inputs:** A6 (incident history / NEVER-AGAIN list), C1 (auth & token lifecycle), C4 (webhooks & event architecture), D4 (platform/commercial risk); corroborated against F1 (read-only enforcement), F2 (pull-sync architecture), F3 (governance/retention), B1 (portal data model), C3 (pipeline discovery). All repo citations are against `/home/user/yscap/yscap-repo-root_8` (read-only inspection; no files modified, no network calls to any ICE endpoint).

**Framing fact that dominates every rating below:** this is not a "sync goes stale, someone re-runs it" integration. The portal will **gate lending decisions** (condition clearing, clear-to-close) on the Encompass mirror. Therefore a wrong, stale, or mis-linked snapshot is not a data bug — it is a **wrong lending decision**. The ClickUp sync history (16 incidents in 9 days, first corruption 32 minutes after go-live — A6 §1) is the empirical base rate for what happens without the controls listed here. Phase 1's saving grace: it is read-only, so *Encompass* cannot be damaged; every failure mode lands on the **portal side** (wrong gate decisions, PII exposure, stale data, dead pipelines).

## 4. Deep dives on the four scariest

### 4.1 Wrong-loan mismatch detection (FM-20) — the design, spelled out

The gate must never be able to consume a snapshot whose linkage is in doubt. Four cooperating layers:

1. **Bind once, deterministically.** Matching ladder (F2 §6): existing GUID binding wins; else unique loan-number hit **with borrower-surname agreement**; anything else (`conflict`/`ambiguous`/`unmatched`) is a human card. Phase 1 starts conservative: auto-match proposes, staff confirms (`manual_confirmed`). One live link per application enforced by partial unique index — double-binding is structurally impossible, not policed.
2. **Re-verify on every ingest.** Each `ingestLoan(guid)` re-checks that the snapshot's loan number and borrower surname still agree with the linked portal file. Contradiction ⇒ demote to `conflict` (snapshot keeps flowing; enrichment and gating pause for that loan; card raised). Never silently re-bind — an inherited "unique" key is a possible stale artifact, never an identity claim (the copied-loan-number lesson).
3. **Gate-side hard precondition.** Gate evaluation requires `match_status ∈ {auto_matched, manual_confirmed}` AND freshness within the rule's staleness budget AND the identity re-check embedded in the gate read (the gate query re-joins link → snapshot → application and asserts surname/loan-number agreement *at decision time*, logging the compared values to `encompass_gate_log`). A mismatch at decision time is a **blocked decision with a named reason**, not a fallback.
4. **Independent recon.** Nightly, outside the ingest code path (A6 NA-15): population pass (every gated portal loan ↔ exactly one live Encompass GUID; every bound GUID present in the census) and field pass (loan number, surname, property address diffed through the same shared transforms the gate uses). Breaks persist and age as recon rows; **mismatch rate is a first-class metric** (§5.3). Assume the pull path lies until the independent reader agrees.

### 4.2 Silent sync death (FM-30) — the never-silent contract

Every loop (webhook receiver, inbox drainer, watermark poller, fetch drainer, nightly sweep, recon, pruner) must satisfy: (a) **heartbeat row** in Postgres per completed pass (`sync_runtime_state`: loop name, last_started_at, last_completed_at, items, errors); (b) every exhausted retry ends as a parked row + review card; (c) staleness feeds the gates. Detection is then triple-redundant: the loop self-reports (heartbeat age), the data self-reports (freshness SLO), and an **external pinger** watches `/api/health` — so a wedged process, a crashed dyno, and a dead alert channel are all distinguishable. The dead-man's switch is non-negotiable: an external service must alert when the health endpoint stops being reachable or when heartbeat age exceeds budget, because in-process alerting dies with the process (A6 §5.4 R4 — the one dead email key that muted everything).

### 4.3 Credential leakage — rotation & leak-response runbook (FM-10/FM-11)

**Routine rotation (quarterly, alternating credentials so failures are diagnosable — C1 §8.7):**
1. Announce a cutover window (off-hours). 2. Super admin: Developer Connect portal → My Account → API Key → **Regenerate Secret** (this immediately breaks the running app — planned). 3. Paste the new secret directly into Render env (never chat/email/ticket); redeploy. 4. Verify: introspection asserts user/instance; one known-loan GET succeeds; a write attempt is rejected; freshness gauges recover. 5. Next quarter, rotate the *service-user password* instead (Encompass admin → env → redeploy → same verification).

**Leak response (pasted-secret incident — "a pasted secret is a burned secret," C1 §9.1):**
1. **Flip the kill switch** (DB-backed pause flag — no deploy needed, A6 NA-16). 2. Regenerate the secret / change the password *immediately* — do not wait for a convenient window; the break-the-app cost is the incident cost. 3. Best-effort `POST /oauth2/v1/token/revocation` for any cached token. 4. Preserve and scope: query the read journal (`encompass_pull_log`) for what was pulled, when, from where, under the burned credential (F3 runbook 1). 5. Purge the pasted artifact (chat message, commit — rewrite history if committed) and record the incident. 6. If unencrypted NPI of ≥500 consumers was plausibly acquired: FTC §314 30-day notice analysis + state breach matrix + notify ICE per contract (F3 runbook 1). 7. Post-incident: §7.1-8 verification checklist, then unpause. — The runbook only works because of two day-one design facts: **dedicated credentials used by nothing else** (attribution) and **every read journaled** (scoping). Both were absent in the ClickUp era and cost two forensic days on false alarms (A6 §2 #4, #7).

### 4.4 Maintenance windows & outage mode (FM-16)

State machine: `NORMAL → DEGRADED (breaker open / status-page incident) → RECOVERY (paced catch-up sweep) → NORMAL`. In DEGRADED: polling paused or slowed, watermark frozen, nothing classified deleted, gates fail closed with "Encompass unreachable since HH:MM" surfaced to staff, borrower-facing UI shows last-known-good with timestamp. Quarterly major-release weekends are *scheduled* DEGRADED windows — put them on the ops calendar from ICE's release notes, and treat the first post-release week as elevated-risk for FM-40 (run the replay fixtures against live samples before re-enabling enforced gates if any contract check fails).

## 5. Day-one monitoring & alerting spec (the integration must ship with this)

Nothing in this section is Phase-2. The ClickUp record shows every detector below paying for itself within days.

### 5.1 Heartbeats (liveness)

| Heartbeat | Source | Budget (alert when exceeded) |
|---|---|---|
| Watermark poll pass completed | `sync_runtime_state` row per pass | 3× configured cadence (15 min cadence ⇒ alert at 45 min) |
| Inbox drainer pass | same | 5 min |
| Fetch drainer pass | same | 10 min |
| Webhook delivery observed | inbox insert timestamp | 3× expected event rate **cross-checked against event history** — silence + events occurring ⇒ endpoint/subscription problem, auto-degrade poll to 5 min (F2 §4.2) |
| Hourly event-history reconcile | same | 2 h |
| Nightly sweep + recon + prune | same | 26 h |
| **External dead-man's switch** | third-party pinger on `/api/health` | endpoint unreachable OR any exposed age gauge over budget. This detector must live **outside** the app and page a channel that is not the app's own email path |
| Alert-channel self-test | weekly synthetic alert that a human must ack | 1 missed ack |

`/api/health` exposes: `encompass: { watermarkAgeSec, inboxDepth, fetchQueueDepth, parkedCount, circuitOpen, lastPollAt, lastReconAt, minFreshnessSec, tokenAgeSec, subscriptionDriftCount }` (F2 §9 pattern).

### 5.2 Freshness SLOs (consumed by the gates, not just graphed)

| Object | SLO | On breach |
|---|---|---|
| Any gated loan's snapshot age (`last_fetched_at`) | ≤ 2× poll cadence (steady state ≤ 30 min) | staleness alarm; S1/S2 gates already failing closed by construction (F2 §7) |
| CTC gate evaluation | on-demand re-fetch immediately before decision; hard fail if the re-fetch fails | decision blocked with named reason; card raised |
| Condition-clearing gate | snapshot ≤ 1 h old | fail closed → on-demand re-fetch → card if still stale |
| Whole-book floor | `min(last_fetched_at)` across active book ≤ 2× cadence; watermark advanced within 3 passes | sync-stalled alert (the watermark-stall detector) |
| Display surfaces | every Encompass-derived value rendered with "as of <timestamp>" | n/a (permanent UI rule) |

### 5.3 Mismatch-rate & correctness metrics

| Metric | Definition | Alert |
|---|---|---|
| `link_conflict_rate` | links demoted to `conflict`/`ambiguous` per day ÷ active links | any spike >3× 7-day baseline; any conflict on a loan with an open CTC/condition decision pages immediately |
| `recon_population_breaks` | nightly recon: gated loans without exactly one live GUID, or bound GUIDs missing from census | target 0; any nonzero break older than 48 h escalates |
| `recon_field_breaks` | field-pass diffs (loan number, surname, address) | same |
| `gate_blocked_stale` / `gate_blocked_mismatch` | gate decisions blocked by staleness vs identity, per day | trend review weekly; sudden zero *and* sudden spike are both signals |
| `intake_violations` | excluded/unknown fields or failed contract validations per pull | any occurrence alerts (persona drift or release drift — FM-26/FM-40) |
| `unknown_enum_count`, `unmapped_field_growth` | registry validation + nightly snapshot audit | weekly digest; step change after an Encompass release ⇒ hold enforced gates |
| `http_429_rate`, `concurrency_remaining_min` | client telemetry from `X-Concurrency-Limit-*` | sustained 429s or remaining < 20% of limit |
| `auth_error_rate` | 401-after-self-heal events | ≥2 in 10 min ⇒ page (password expiry / regeneration signature) |
| `parked_rows` (inbox ≥6 attempts, fetch ≥8) | drainer park counters | any parked row past 4 h SLA; feeds the error-budget freeze (FM-31) |
| `snapshot_table_bytes`, `snapshot_rows_per_loan_day` | storage telemetry | trend alarm; dedup ratio < 90% suggests hash bug |
| `review_cards_open`, `cards_per_day`, `dismiss_rate_by_reason` | review-queue telemetry | >5 staff emails/day = queue-design bug (NA-14); dismiss-rate >80% on a reason ⇒ auto-demote it |

### 5.4 Alert routing & rituals

- **Page immediately (S1):** dead-man's switch; CTC/condition gate blocked by mismatch; intake violation involving excluded PII; credential anomaly (auth errors + volume spike); circuit open > 30 min.
- **Same-day (S2):** freshness SLO breach; recon breaks; subscription drift; parked rows past SLA; watermark stall.
- **Weekly digest (S3):** unmapped-field growth, storage trends, dismiss-rate telemetry, 429 trends, release-notes review.
- **Rituals:** weekly 10-minute metrics review (A6 §5.4); quarterly rotation (§4.3); per-Encompass-release R2T regression + fixture refresh (FM-40); alert-channel self-test weekly.
- **Error budget:** while any S1/S2 item is open past SLA, no new Encompass feature merges (NA-20).
