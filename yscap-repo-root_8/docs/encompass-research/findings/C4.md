## 2. Subscription API

### 2.1 Request body schema

From the Create-a-Subscription reference
(https://developer.icemortgagetechnology.com/developer-connect/reference/create-a-subscription) and Postman
bodies:

```json
{
  "endpoint":  "https://<your-public-host>/<path>",   // HTTPS required
  "resource":  "Loan",
  "events":    ["change"],
  "signingkey": "<shared secret — optional; auto-assigned if omitted>",
  "filters":   { "attributes": [ "/applications/*/borrower/lastName" ] },
  "enableSubscription": true,
  "description": "human-readable description"
}
```

- Response: `201 Created`; the subscription id is returned in the `Location` header, pattern
  `/webhook/v1/subscriptions/{subscription-id}` (same reference page).
- `filters.attributes` is supported **only** for Loan `change` and `fieldchange` events (same reference page).
- **No validation on `filters.attributes` — invalid attributes are silently ignored** (same reference page).
  This makes automated verification of filter correctness (test loans + event history checks) essential.

### 2.2 Loan change attribute filters (JSON pointers)

Postman "06 - Update a Webhook Subscription" body shows the pointer syntax including wildcards:

```json
"filters": { "attributes": [
  "/applications/1/borrower/maritalStatusType",
  "/applications/*/coborrower/equifaxScore",
  "/contacts/3/*",
  "/customFields/100/*"
]}
```

Official docs add examples (https://developer.icemortgagetechnology.com/developer-connect/docs/say-goodbye-to-polling-in-the-encompass-lending-platform-part-2):
`/baseLoanAmount`, `/principalAndInterestMonthlyPaymentAmount`, `/applications/*/coborrower/experianCreditScore`,
`/milestoneLogs/*/doneIndicator`, and even a JSONPath-style predicate
`/contacts[?(@/contactType == \"ESCROW_COMPANY\")]/email`. The create-subscription reference also shows
`/fields/1109` (classic field-ID addressing) and `/contacts/*/contactType/name`. Field IDs map to v3 JSON paths
via the Loan Schema API.

### 2.3 Constraints and limits

(https://developer.icemortgagetechnology.com/developer-connect/docs/webhooks and .../docs/webhooks-bp)

- "An application should never exceed **25 Webhook Subscriptions per Lender**."
- "You cannot include more than **50 attributes** as part of a single subscription, but you may add more than
  one 'change' subscription."
- "The **resource + event + endpoint combination must be unique**" across subscriptions.
- Endpoint must be **public HTTPS**; for the closely-related Partner Connect platform ICE documents
  **CA-signed certificates only (no self-signed)** and **TLS 1.2+**
  (https://docs.partnerconnect.elliemae.com/partnerconnect/docs/webhooks) — assume the same for Developer
  Connect.

## 3. Notification payload shape — references, not loan data

From the Webhook API overview (https://developer.icemortgagetechnology.com/developer-connect/reference/webhook):

```json
{
  "eventId":   "<unique id>",
  "eventTime": "2025-02-03T20:58:11.599Z",
  "eventType": "change | create | milestone | move | ...",
  "meta": {
    "userId":       "admin",
    "resourceType": "Loan",
    "resourceId":   "<loan GUID>",
    "instanceId":   "<instance id, e.g. BE11397907>",
    "resourceRef":  "/encompass/v3/loans/{resourceId}",
    "payload":      { "correlationId": "...", "event": { /* event-specific extras */ } }
  }
}
```

Key property for YS Cap's architecture: the **standard notification carries no loan data / PII — only a
reference** (`meta.resourceRef`) that the consumer must dereference with an authenticated GET. That keeps the
webhook endpoint low-sensitivity. Two exceptions that DO put data in the payload:

1. **`move`** — includes source and target loan-folder paths in the extra payload (docs: "the 'move' event ...
   enhanced to include extra payload for loan folder source and target paths").
2. **`enhancedfieldchange` (EFC)** — includes previous + new field **values** (see §6). If subscribed, the
   endpoint and inbox storage become PII-bearing and must be treated accordingly (encrypt/redact at rest).

## 4. Signature verification

From the Signing Keys reference (https://developer.icemortgagetechnology.com/developer-connect/reference/signing-keys):

- Every notification carries three headers:
  - **`Elli-Signature`** — "the base64 encoded, SHA256 hash of the event (request body)" — in practice an
    **HMAC-SHA256 over the exact raw request body, keyed with the subscription's signing key (UTF-8 bytes),
    base64-encoded**. (Some docs/pages refer to the header as `X-Elli-Signature`; match case-insensitively and
    accept both spellings.)
  - **`Elli-Environment`** — always `prod`.
  - **`Elli-SubscriptionId`** — identifies which subscription (and therefore which signing key) produced the
    notification. Use it to select the key when running multiple subscriptions.
- The signing key "serves as a password that either you can assign or can be automatically assigned by the
  Encompass platform"; a lost key is recoverable via **Get a Subscription**.

Node equivalent of the documented C# sample:

```js
const expected = crypto.createHmac('sha256', Buffer.from(signingKey, 'utf8'))
                       .update(rawBodyBuffer)            // EXACT raw bytes
                       .digest('base64');
crypto.timingSafeEqual(Buffer.from(expected), Buffer.from(headerValue));
```

The repo already implements this exact pattern for ClickUp and Resend/Svix — raw-body route mounted before the
global JSON parser, HMAC over exact bytes, constant-time compare:
- `src/routes/clickup-webhook.js:1-49` (raw body + HMAC + fail-closed when secret unset in production)
- `src/server.js:26-28` (webhook route mounted BEFORE `express.json()`)
- `src/lib/resend-webhook.js:1-50` (Svix scheme, `timingSafeEqual`, 5-minute replay tolerance)
- `src/clickup/client.js:296` (`verifyWebhookSignature`)

Note: Encompass sends no timestamp header equivalent to Svix's, so replay defense = eventId dedupe (inbox
unique key) rather than a timestamp window.

## 5. Delivery semantics — NOT guaranteed

From Webhook Best Practices: Event Reconciliation
(https://developer.icemortgagetechnology.com/developer-connect/docs/webhook-best-practices-event-reconciliation-and-more):

- A delivery is considered failed on: **5XX response, invalid response, connection error, or no response within
  30 seconds**.
- On failure, redelivery is attempted; **after four total attempts the notification is discarded**.
- Consequence, verbatim from ICE: delivery "is not guaranteed" — "implement a reconciliation process to
  periodically fetch data from the Event History API and look for misconfigured subscriptions and undelivered
  notifications."
- No documented ordering guarantee; treat notifications as unordered at-least-zero-times hints.

## 6. Webhook Event History API (reconciliation backbone)

`GET https://api.elliemae.com/webhook/v1/events`
(https://developer.icemortgagetechnology.com/developer-connect/reference/202-get-all-events; Postman request
"Webhook / 08 - Get Event history for loan transactions").

Query parameters: `resourceType`, `resourceId` (loan GUID), `eventType` (loan examples: Create, Update, Submit,
Change, Move, Document, Milestone), `subscriptionId`, `startTime` / `endTime` (ISO8601; default previous day →
now), `start` (default 0), `limit` (default 100, **max 100 / 6MB page**), `status`.

Statuses (reference page + best-practices blog):

| Status | Meaning |
|---|---|
| `EventReceived` | Event received from the origin (Encompass) |
| `SubscriptionMatch` / `SubscriptionMatched` | Event matched ≥1 subscription |
| `NotificationDelivered` | Delivered AND acknowledged by your endpoint |
| `DeliveryAttempted` | Logged **only on endpoint failure** (check statusDetails for the error) |
| `NotificationFailed` | Delivery failed |
| `DeliveryFailedExhaustedRetries` | All 4 attempts exhausted; notification discarded |

Reconciliation recipe from the blog: periodically diff `EventReceived`/`SubscriptionMatch` events against your
inbox; anything without a matching received notification (or showing `DeliveryAttempted`/failed statuses) gets
re-fetched via the ordinary GET APIs. Also use it to detect **misconfigured subscriptions** (event received but
no `SubscriptionMatch` ⇒ filter/subscription bug). Retention window of the history store is **not documented**
(open question for ICE support); the default 1-day query window suggests running reconciliation at least daily,
preferably hourly.

## 7. Enhanced Field Change (EFC) events

Sources: EFC User & Setup Guide (…/docs/efc-webhook-user-setup-guide), How to Enable
(…/docs/efc-webhook-how-to-enable), Features & Usage Notes (…/docs/efc-webhook-features-and-usage-notes),
Best Practices (…/docs/efc-webhook-best-practices).

- Available from **Encompass 24.2 GA onward**; delivered for changes made via SDK/Desktop **and** V3 APIs.
- Subscribe with `resource: "Loan"`, `events: ["enhancedfieldchange"]` (Postman "05k - Create
  enhancedfieldchange Webhook Subscription" uses lowercase `"loan"`), but **subscription alone is NOT enough**:
  "A successful subscription ... does not indicate that the setup is completed." You must **open an ICE support
  ticket via the Resource Center including the Instance ID** (BE11397907), it is enabled **first in your R2T
  (test) instance, production only after successful R2T testing**, and notifications begin on the predetermined
  change date communicated in the ticket.
- Payload: `payload.event.fieldChangeEvents[]`, each entry
  `{ modifiedField, parentFieldId, encompass: { previousValue, newValue }, v3LoanModel: { previousValue, newValue } }`
  — both classic-field and v3 representations (formats can differ, e.g. `Y/N` vs `true/false`). Standard AND
  custom fields included.
- **No field-level filters — all field change events are emitted** (Features & Usage Notes). High volume; large
  payloads are **chunked** into multiple webhooks with `multipartIndicator` ("1/4"…) and a correlating
  `chunkId`.
- PII consequence: unlike baseline notifications, EFC payloads contain actual field values (names, scores,
  amounts). If YS Cap adopts EFC, the inbox must redact/encrypt sensitive values, mirroring the existing
  `redactClickupPayload` precedent (`src/routes/clickup-webhook.js:19-36`).

Phase-1 verdict: EFC is attractive (per-field before/after without a full re-pull) but not necessary; the
`change` event + attribute filters + re-GET pattern is simpler, PII-free at the webhook layer, and needs no
support ticket. Defer EFC to Phase 2 unless field-level audit trails become a requirement.

## 8. Webhook Custom Auth (premium, optional)

Postman folder "Webhook Custom Auth - Premium" (endpoint index lines 747–755):
`POST/GET /webhook/v1/functions/auth`, `PUT/PATCH/DELETE /webhook/v1/functions/auth/{functionId}`,
link/unlink to a subscription via `PATCH/DELETE /webhook/v1/subscriptions/{id}/functions/auth/{functionId}`,
list via `GET /webhook/v1/subscriptions/{id}/functions/auth`, and
`POST /webhook/v1/functions/auth/{functionId}/test`.

Body template (`templateType: "OAUTH2_CLIENT_CREDENTIAL"`, `functionType: "auth"`): ICE stores YOUR endpoint's
`tokenURL` + client credentials and fetches an OAuth2 bearer token from you before delivering notifications, so
your webhook endpoint can demand `Authorization: Bearer ...` in addition to the HMAC. The test endpoint returns
`tokenTestOKResponse` / structured `tokenTestFailureResponse` (changelog notes). This is a **premium** feature;
the **signing key remains the baseline** and is sufficient for Phase 1. Consider custom auth later if the
security team wants defense-in-depth at the network edge (e.g., reject unauthenticated POSTs before touching the
body).

## 10. Classification: is subscription management a "write"?

**Analysis.** `POST/PUT/DELETE /webhook/v1/subscriptions` writes *platform integration configuration* — it
creates delivery plumbing owned by the client application. It cannot alter any loan, borrower, condition,
document, milestone, or setting inside Encompass; it is invisible to loan files and to underwriters. The
"architecturally incapable of writing to Encompass" goal is about **loan/business data**. A strictly-GET-only
policy would force manual subscription setup via Postman (with long-lived credentials in a human's hands) —
arguably worse for security — or forgo webhooks entirely.

**Recommendation: allow `/webhook/v1/subscriptions` (and only that) as the sole permitted non-GET namespace**,
with these controls:

1. **HTTP-client-level allowlist (the real guarantee):** the single shared Encompass client module rejects any
   non-GET request unless `path.startsWith('/webhook/v1/subscriptions')` — deny-by-default, enforced in code at
   one choke point, unit-tested (mirror the write-guard test style of
   `scripts/test-clickup-write-guards.js` / `scripts/test-clickup-delete-guard.js`).
2. **Persona/permission-level enforcement:** ask the Encompass admin to grant the API user a read-only persona
   for loan resources; Developer Connect honors Encompass persona rights, so even a bug can't write loan data.
   (Verify with ICE that webhook subscription APIs work under that persona — open question.)
3. **Config-as-code:** desired subscriptions declared in a versioned JSON file; a `reconcileSubscriptions()`
   startup/daily job diffs desired vs `GET /webhook/v1/subscriptions` and creates/updates/disables only its own
   (match by endpoint host + description prefix, never touch subscriptions it didn't create — same "never touch
   what we don't own" discipline as the ClickUp sync guards).
4. **Audit log:** every subscription mutation recorded in Postgres (who/when/what body).
5. **Cap and lint:** enforce ≤50 attributes/subscription and stay far below the 25-subscription platform limit.

## 11. Recommended design: webhook + polling-reconciliation hybrid

**Principle: webhooks are an accelerator (low latency); polling + event-history reconciliation is the source of
truth (correctness).** ICE itself mandates this ("delivery not guaranteed ... implement a reconciliation
process").

```
Encompass ──webhook POST──▶ /api/encompass/webhook  (raw body, verify Elli-Signature,
                                dedupe by eventId, INSERT ... ON CONFLICT DO NOTHING
                                into encompass_webhook_inbox, ack 200 in <1s)
                                        │
                          inbox drainer (interval worker)
                                        │  GET meta.resourceRef (v3 loan) — read-only client
                                        ▼
                              portal mirror tables (loan snapshot, conditions,
                              milestones, CTC status) + rules engine
                                        ▲
        hourly reconciliation:  GET /webhook/v1/events (statuses vs inbox)  →  re-fetch missed loans
        scheduled full poll:    Loan Pipeline / v3 GETs over the active-loan universe
                                (lastModified cursor) — catches events webhooks can never see
```

Concrete elements, reusing proven repo patterns:

1. **Receiver** — clone `src/routes/clickup-webhook.js` (mounted before the JSON parser, `src/server.js:26-28`):
   `express.raw()`, verify HMAC (base64, UTF-8 key, `timingSafeEqual` as in `src/lib/resend-webhook.js`),
   select key by `Elli-SubscriptionId`, dedupe on `eventId` (Encompass provides one natively — better than
   ClickUp's body-hash workaround at `src/routes/clickup-webhook.js:56`), insert into
   `encompass_webhook_inbox`, return 200 immediately. Never process inline.
2. **Drainer** — interval worker copying `processInboxOnce` (`src/sync/clickup-sync.js:644-668`,
   `FOR UPDATE SKIP LOCKED`, crash-reclaim, attempts + last_error) that dereferences `meta.resourceRef` with the
   read-only Encompass client and upserts the portal mirror. Webhooks therefore only ever *trigger reads* —
   they carry no trusted data themselves (except `move` folder paths).
3. **Reconciliation (hourly)** — `GET /webhook/v1/events` for the window since the last run; every
   `EventReceived` for Loan without a corresponding inbox row, and every
   `DeliveryAttempted`/`NotificationFailed`/`DeliveryFailedExhaustedRetries`, enqueues a re-fetch of that
   `resourceId`. Also alert when events show no `SubscriptionMatch` (subscription drift/misconfig).
4. **Polling source of truth (daily minimum, ideally every 15–60 min)** — Loan Pipeline query over the active
   loan universe with a `lastModifiedDateTime` cursor, upserting mirrors regardless of webhook activity. This is
   the layer that catches: discarded notifications, endpoint downtime > 4 retries, Smart-Client-originated
   events that never fire ("API only" rows in §9), event-history retention gaps, and subscription mistakes.
5. **Rules gating** — portal rules (condition clearing, clear-to-close) read ONLY the mirror tables and their
   `synced_at` staleness stamp; a webhook can accelerate freshness but a stale mirror (> threshold) forces an
   on-demand re-fetch before allowing a gated action. Encompass remains authoritative; the portal can be more
   restrictive but never less.

**Subscriptions to create (Phase 1):** one Loan `change` subscription with ≤50 filter attributes covering
milestone/CTC/condition-adjacent fields (e.g. `/milestoneLogs/*/doneIndicator`, milestone current name/date,
underwriting status fields), plus Loan `update`, `move`, `delete` (and optionally `milestone`, `condition` —
API-only caveat). All to the same endpoint; remember resource+event+endpoint uniqueness.

## 12. Endpoint hosting requirements

- **Public HTTPS**, CA-signed cert (no self-signed), TLS 1.2+ (Partner Connect doc, assumed to apply). The
  portal already deploys on Render (`render.yaml`), which provides managed TLS on a public URL — suitable.
  No documented ICE egress IP ranges to allowlist (open question); auth must come from signature (+ optional
  premium custom auth), not network origin.
- **Fast ack:** respond 2XX well inside the 30s timeout — target <1s by doing only verify → insert → 200. Any
  5XX/timeout burns one of only 4 delivery attempts.
- **Queue-then-process:** inbox table + async drainer (§11); never call Encompass or run rules inline in the
  request handler.
- **Raw-body mounting:** before the global JSON parser, exactly like `src/server.js:26-28`.
- **Idempotency:** unique index on `eventId`; redeliveries and reconciliation re-fetches must collapse.
- **Fail-closed:** if the signing key env var is unset in production, return 503 and deliver nothing
  (precedent: `src/routes/clickup-webhook.js:46-49`).
- **Low sensitivity by design:** with baseline events the inbox stores references only. If EFC is ever enabled,
  add value redaction before persisting (precedent: `redactClickupPayload`,
  `src/routes/clickup-webhook.js:19-36`).
