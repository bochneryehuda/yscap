# E1 — Borrower Intelligence Ideas (Read-Only Encompass Feed)

_Research agent E1. Scope: what a strictly read-only Encompass Developer Connect feed (instance BE11397907) enables for borrower intelligence in the PILOT portal. No implementation; ideas + risks + review checkpoints only._

## 1. Grounding: what the portal already has

### 1.1 Borrower profile & track-record model

### 1.2 Experience engine (the thing Encompass data must respect)

- **Frozen 3-year window** (`src/lib/experience.js:5-18`): a deal counts only if its **exit** (sale date for flips; lease/refi date for holds) is within the last 36 months and not future-dated. `RECENT_EXIT_SQL` is the single reusable predicate.
- Buckets: `flips` / `holds` / `ground` via `bucketOf()` (`src/lib/experience.js:25-30`).
- File experience = primary + co-borrower summed (`src/lib/experience.js:54-78`).
- **Loans size on claimed experience; funding is gated on verification** (`CLAUDE.md` session rules; enforcement in `syncExperienceChecklistForApplication`, `src/lib/experience.js:86-221`). A negative verified-experience change makes the registration stale/FATAL (`src/lib/experience.js:119-163`).
- Implication: any Encompass-derived experience must feed the **verification side** (evidence for staff) or the counts themselves via new `track_records` rows — never bypass `RECENT_EXIT_SQL`.

### 1.3 Identity matching machinery (directly reusable)

- `resolveBorrower` (`src/clickup/ingest.js:310-405`) match ladder: (1) exact **SSN-hash**, (2) **email + corroboration** — corroboration = phone last-10 OR DOB OR full name, **never last name alone** (family-shared-email incident), see `emailMatchCorroborated` (`src/clickup/identity.js:113-142`); SSN conflict = definitely two people; uncorroborated email match → distinct profile + `borrower_dedup_candidates` row. No-identity guard: never mint a profile without name/SSN/DOB.
- `nameConflict` and `bestMatch` (threshold ≥2 identity fields) also exported (`src/clickup/identity.js:145-176`).
- **This exact ladder should be the Encompass↔portal borrower matcher.** Do not invent a second one.

### 1.4 Pre-wired integration hooks

- `sync_queue.target` already allows `'encompass'` (`db/schema.sql:306`) — the queue plumbing anticipates this integration (deferred).
- `sync_review_queue` (db/108, per CLAUDE.md) + the `borrower_dedup_candidates` pattern are the established "suspicious → human review, never silent" mechanisms to copy.

### 1.5 Portal-audit constraints that shape every idea below

From `docs/portal-audit/02-borrower-file.md`:

- **S2-04** (`02-borrower-file.md:125-149`): the borrower file endpoint is send-everything-then-hide (`src/routes/borrower.js:272-315`); *"the next internal field added leaks automatically."* **Every Encompass-derived column added to `applications`/`borrowers` will leak to borrowers by default** until the screen flips to a show-list.
- **S2-06** (`02-borrower-file.md:177-192`): track-record screen does `SELECT t.*` (`src/routes/borrower.js:1134-1141`) — Encompass provenance fields/notes added to `track_records` would ride to the borrower.
- Capital-partner-name wall (`02-borrower-file.md:29-48` + CLAUDE.md hard rule): Encompass data is saturated with investor/lender/trade names (loan folders, investor fields, trade data). **All Encompass-derived text must be classified staff-only by default.**
- S2-05 (`02-borrower-file.md:152-173`): borrowers can silently rewrite deal economics — an Encompass cross-check (portal value vs Encompass value) is also a *detector* for this class of tampering.

### 1.6 Encompass read surface available (from the Postman index)

From `scratchpad/research/postman-endpoint-index.md` (800 requests; the read-relevant subset):

| Capability | Endpoint(s) | Note |
|---|---|---|
| Bulk loan queries with field selection + filters | `POST /encompass/v3/loanPipeline` (+ `/report` cursor for big pulls); canonical field discovery `GET /encompass/v3/loanPipeline/canonicalFields` | POST-as-read; filter by `Loan.LoanFolder`, `Loan.CurrentMilestoneName`, `Loan.LastModified`, borrower name, loan number (Multivalue) |
| Full loan / selected entities | `GET /encompass/v3/loans/{id}` (`?entities=...`), `POST /encompass/v3/loans/{id}/fieldReader` (arbitrary field IDs incl. custom `CUST*`/`BE*` fields) | fieldReader is POST-as-read |
| Milestones & staff associates | `GET /v3/loans/{id}/milestones`, `/v1/loans/{id}/associates` | dates per milestone → stall/fallout forensics |
| Conditions (incl. post-closing) | `GET /v3/loans/{id}/conditions`, `GET /v1/loans/{id}/conditions/postclosing` | for the condition-mirroring rules (other agents' scope) but also EPO/post-close flags |
| Borrower pairs | `GET /v1/loans/{id}/applications` | co-borrower structure per loan |
| Vesting entities | `GET /v3/loans/{id}/closingDocument/vestingEntities` | LLC vesting per closed loan → entity graph |
| Borrower contacts (CRM) | `POST /encompass/v1/borrowerContactSelector` (query by `Contact.*` canonical fields), `GET /v1/borrowerContacts/{id}` | second identity oracle |
| Loan folders | `GET /encompass/v3/loanFolders`; pipeline `includeArchived` | folder moves (e.g. Adverse/Archive) ≈ fallout signal |
| Audit trail | `POST /v3/loans/{id}/auditTrail` | field-level change history |
| Loan schema / custom fields | `GET /v3/schemas/loan`, `/standardFields`, `/virtualFields` | discover YS-specific custom fields before mapping |
| Funding | `GET /v3/loans/{id}/fundingFees`, `/fundingBalances` | funded economics |
| Webhooks | `POST /webhook/v1/subscriptions` | push-based change detection (a *subscription* is a write to the webhook service, not to loan data — flag for the read-only architecture decision) |

**Read-only nuance:** several essential reads are HTTP POST (`loanPipeline`, `fieldReader`, selectors, `auditTrail`). "Architecturally incapable of writing" must therefore be an **endpoint-path allowlist**, not a GET-only verb filter.

## 2. Prioritized idea catalog

Legend: **Difficulty** S/M/L (integration + portal work, assuming a base read-only Encompass client and loan↔file linking exist). **QW** = quick win.

### Tier 1 — quick wins (build first)

---

#### I1. Repeat-borrower detection at intake **[QW]**

**What:** when a new lead/application/borrower is created (public intake, portal registration, ClickUp ingest), match the person against a locally cached index of Encompass borrowers (nightly `loanPipeline` + `borrowerContactSelector` pull). Surface a staff-only banner: "Likely repeat borrower — N prior YS loans in Encompass (X funded, Y fell out), last closed <date>," with links.

- **Value:** instant "known customer" context for LOs; repeat borrowers are the highest-converting, best-priced leads; stops re-underwriting people YS already knows; feeds every other idea (it *is* the identity join).
- **Encompass data:** pipeline query for borrower name/SSN-last4/email/phone/DOB per loan (`Fields.4000`-family canonical fields), loan number, folder, milestone, funded/closed dates; `borrowerContactSelector` for contact records. Cache locally; never query live per keystroke.
- **Difficulty:** **S–M.** Match ladder already exists (`src/clickup/identity.js`); the new work is the Encompass identity cache table + a hook at the 3–4 borrower-creation chokepoints.
- **Risks / false matches:** same-name relatives (father/son Jr/Sr with shared email) — exactly the class that caused the 2026-07-15 merge incident; SSN availability in the pull (avoid pulling full SSN — prefer hashing on ingest if available at all, or match on DOB+phone+email+name). A false "repeat borrower" banner could leak another person's loan history to the LO and mis-price the deal.
- **Manual-review checkpoint:** the banner is a **suggestion only** — it never auto-links. Staff click "confirm same person" / "not the same person"; verdicts recorded like `borrower_dedup_candidates` (`db/081`) with reason + reviewer. Only a confirmed link creates a durable `borrower_encompass_links` row.

---

#### I2. Auto-built borrower track records from closed Encompass loans **[QW with a caveat]**

**What:** for each borrower with a confirmed Encompass link, generate **unverified, provenance-stamped** `track_records` rows from YS's own closed/funded Encompass loans: property address, purchase price, loan amount, rehab budget, deal type, closing date, vesting LLC. Staff verify with one click (evidence is YS's own LOS — the strongest possible verification source).

- **Value:** kills the biggest borrower friction point (re-typing past deals YS itself funded); makes tier/experience counts more complete and more trustworthy; converts "trust me" into "we funded it."
- **Encompass data:** pipeline query over funded loans (folder/milestone = funded/closed), per-loan `GET /v3/loans/{id}` or `fieldReader` for property address, amounts, program/purpose, closing date, borrower pair, vesting entity.
- **Difficulty:** **M.** Mapping is mostly 1:1 with `track_records` columns (`db/schema.sql:123-147`); dedup against existing rows needed (address-canon exists: `src/lib/address-canon.js`).
- **THE caveat — exit dates:** the experience window keys on the **exit** (sale/lease/refi date, `src/lib/experience.js:13-18`), and Encompass origination data proves the *entry*, not the exit. Options: (a) treat a **payoff of the YS loan** as the exit proxy (needs payoff/servicing data availability — open question); (b) detect exits when the *same property* appears as the seller side / refi of a later loan; (c) create the row with entry data and leave exit blank → row exists but **counts toward nothing** until staff/borrower add the exit — still a big data-entry win, and it can never inflate the tier (safe-by-default).
- **Risks / false matches:** wrong borrower link propagates someone else's deal into a profile (pricing impact!); duplicate rows vs borrower-typed ones inflating counts (the autosave-duplicate incident class in CLAUDE.md — dedup by canonical address + date proximity); `SELECT t.*` leak (S2-06) exposing Encompass provenance fields to borrowers — fix the show-list first or keep provenance in a side table.
- **Manual-review checkpoint:** rows land `is_verified=false` with `source='encompass'`; a staff "verify against Encompass" screen shows the row next to the source loan; **verification stays a human sign-off** (matches the existing gate — sizing on claims, funding on verification). Never auto-verify, even from YS's own LOS, until the owner explicitly blesses that policy.

---

#### I3. Stated-vs-actual experience cross-check **[QW]**

**What:** on every file, compare claimed experience (`applications.requested_exp_*`, `db/schema.sql:185-187`) against (a) portal-verified counts and (b) the borrower's Encompass history (count of funded YS loans by type). Staff-only panel on the file + on the experience condition: "Claims 10 flips; portal verifies 4; Encompass shows 6 YS-funded flips, 1 fallout." Flag big divergences.

- **Value:** directly hardens the funding gate the system already enforces; catches inflated claims early (before underwriting time is spent); also catches *understated* experience (borrower deserves a better tier — a sales win).
- **Encompass data:** same funded-loan pull as I2; loan purpose/program to bucket into flips/holds/ground (`bucketOf`, `src/lib/experience.js:25-30`).
- **Difficulty:** **S** once I1+I2's cache exists — it's a comparison + display.
- **Risks:** Encompass only sees deals YS funded — a genuine 10-flip borrower who did 6 elsewhere is *not* lying; the panel must present Encompass as a **floor**, never a ceiling. Divergence copy must be staff-only (a borrower seeing "we think you're lying" is a relationship fire). False-match risk inherits from I1.
- **Manual-review checkpoint:** divergence creates an `[auto]` staff note / review row on the experience condition — never auto-reopens or auto-fails the condition, and never changes counts. The human verifier decides.

---

#### I4. Pre-filled applications for repeat borrowers **[QW]**

**What:** when a confirmed repeat borrower starts a new application, pre-fill identity (address, employment, citizenship, entity) and offer their known entities/track record from the Encompass-enriched profile: "Welcome back — we've pre-filled what we know."

- **Value:** conversion + speed; the classic repeat-customer moat; reduces typo-borne identity fragmentation (fewer new dedup candidates).
- **Encompass data:** borrower contact fields + latest funded loan's borrower section; vesting entities (I5).
- **Difficulty:** **M.** Pre-fill sources exist (the portal already pre-fills SOW from the application); the work is choosing precedence: portal profile → ClickUp → Encompass (freshest-wins with provenance), and only ever pre-filling **blank** fields (the S2-05 lesson: check the *stored* value, and audit-log every write).
- **Risks / false matches:** pre-filling from a mismatched person = a direct PII leak into another borrower's screen — the worst failure mode in this catalog. Stale data (borrower moved) silently propagating. Compliance: pre-filled ≠ attested; the borrower must still confirm.
- **Manual-review checkpoint:** pre-fill only after a **staff-confirmed** I1 link (never from a fuzzy match); borrower sees and confirms each pre-filled field (attestation step); all pre-fills audit-logged with `source='encompass_prefill'`.

### Tier 2 — high value, medium/large builds

---

#### I5. Entity/LLC relationship graph

**What:** build a graph: borrowers ↔ LLCs ↔ properties ↔ loans, merging portal data (`llcs`, `llc_borrowers`, `track_records.llc_id`) with Encompass vesting entities (`GET /v3/loans/{id}/closingDocument/vestingEntities`) and borrower pairs (`GET /v1/loans/{id}/applications`). Staff-only explorer: "This LLC vested 3 YS loans across 2 borrowers; this borrower operates through 4 entities; these two 'unrelated' applicants share an entity."

- **Value:** exposure/relationship truth (guarantor webs, straw-borrower patterns, partner splits); dedup assistance (shared entity is powerful corroboration); credit decisions on the *sponsor group*, not the individual; feeds concentration analytics (I7).
- **Encompass data:** vesting entities per loan, borrower pairs, business contacts; EIN if present (join key to `llcs.ein`, `db/schema.sql:106`).
- **Difficulty:** **L.** Entity name matching is messy ("123 Main St LLC" vs "123 Main Street, LLC"); needs an entity-canonicalization layer + graph storage/queries + UI. Phase it: table-based "shared entity" flags (M) before a visual graph (L).
- **Risks / false matches:** generic LLC names (property-address-named LLCs collide when two people flip the same address over time); EIN absent → name-only matching is weak; graph UIs invite over-reading (two borrowers sharing an attorney-registered-agent address are not partners). Staff-only forever — an entity graph is a partner/PII minefield for borrower eyes.
- **Manual-review checkpoint:** entity merges (this Encompass vesting entity = this portal `llcs` row) go through a review queue exactly like `borrower_dedup_candidates`; unreviewed matches render as dashed "possible" edges, never as fact.

---

#### I6. Borrower risk flags: fallout, EPO, adverse history

**What:** per confirmed borrower, staff-only flags derived from Encompass history: **fallout** (loans that died — `declined`/`withdrawn` analogues: adverse loan folder moves, terminal milestones, archived loans), **stalls** (milestone durations way over norm), **EPO / early payoff** (paid off inside the 3-month minimum-interest window — a YS economics rule, CLAUDE.md standing rule), repeated appraisal blow-ups, condition-heavy files. Rendered as neutral facts: "2 of 5 prior YS applications did not fund."

- **Value:** institutional memory at intake; protects capital-partner relationships (EPO penalties from note buyers are real money for a hard-money shop); prices relationship risk, not just deal risk.
- **Encompass data:** pipeline over **all** folders incl. archived (`includeArchived`), milestones + dates per loan, loan folder history, audit trail for status transitions; payoff data **if present** (biggest open question — payoff/servicing may live outside Encompass once loans are sold; post-closing conditions and funding balances are partial proxies).
- **Difficulty:** **L** (M for fallout-count-only). Fallout semantics must be reverse-engineered from how YS actually uses folders/milestones/status fields in instance BE11397907 — discover via `canonicalFields` + `schemas/loan` custom fields first.
- **Risks:** *mislabeling* is the big one — a loan "fell out" because YS repriced, the appraisal came low, or the borrower chose a competitor; a raw fallout count reads as a character judgment. Fair-lending/ECOA optics if flags ever influence decisions without a documented basis. Borrower-facing leakage (S2-04 class) would be catastrophic here. False matches inherit from I1 with higher stakes.
- **Manual-review checkpoint:** flags are **advisory badges with drill-down to evidence**, never auto-gates; an LO/underwriter can annotate or suppress a flag with a reason (recorded); any flag that would influence a decline must be human-confirmed and documented on the file.

---

#### I7. Portfolio-wide borrower concentration analytics

**What:** staff/admin dashboard: top borrower groups (borrower + entities, via I5) by open exposure (funded, not paid off), by pipeline exposure, repeat rate, geography overlap, per-officer dependence on a handful of sponsors; trend of new-vs-repeat volume.

- **Value:** risk management the owner can act on ("one sponsor is 14% of the book"); capital-partner reporting ammunition; marketing targeting (dormant repeat borrowers).
- **Encompass data:** pipeline reports cursor (`POST /v3/loanPipeline/report`) over funded loans: amounts, dates, borrower, property state/zip; funding balances; payoff status if available (else "open exposure" is approximate).
- **Difficulty:** **M** for tables/tiles over the I1/I2 cache; **L** with entity-group rollups (needs I5).
- **Risks:** garbage-in — concentration numbers are only as good as the identity/entity matching underneath (an unmatched duplicate borrower *understates* concentration — the dangerous direction); payoff blindness overstates open exposure. Contains partner/economic data → admin/staff-only.
- **Manual-review checkpoint:** dashboard displays a data-quality banner (N unresolved dedup candidates / unlinked loans that could shift these numbers) linking to the review queues; no automated action hangs off these analytics.

### Tier 3 — worth designing, not first

#### I8. Borrower 360 timeline (unified history panel)

**What:** on the staff borrower screen, one chronological feed: portal applications, ClickUp events, and Encompass loans (milestone dates, funded date, amounts) for the confirmed-linked borrower. Cheap to render once I1's cache exists (**S–M**). Value: context without tab-switching. Risk: internal names/fields in Encompass milestone comments — display a field allowlist, never raw payloads. Checkpoint: only confirmed links appear; "possible" matches show as a review prompt, not as history.

#### I9. Encompass as a dedup oracle for the existing queue

**What:** enrich `borrower_dedup_candidates` resolution UI with each side's Encompass matches ("profile A links to Encompass borrower with SSN-last4 1234; profile B to 9876 → distinct"). **S** once I1 exists. Value: turns undecidable email-share cases into decidable ones. Risk: an Encompass contact record can itself be a duplicate — treat as evidence, not verdict. Checkpoint: unchanged — the human records the verdict; merge stays manual (`db/081_borrower_dedup_candidates.sql:9-11`).

#### I10. Dormant repeat-borrower reactivation list

**What:** borrowers whose last Encompass funded loan closed 6–24 months ago with no open portal file — exported for LO outreach. **S** on the cache. Risk: contacting someone about "their history" they don't associate with YS (broker-sourced loans); marketing-consent hygiene. Checkpoint: LO reviews the list before any outreach; no automated email to borrowers from this feature.

## 3. Cross-cutting build order & guardrails

1. **Foundations first (everything above depends on these):** read-only Encompass client with an **endpoint-path allowlist** (POST-as-read endpoints exist — verb filtering is insufficient); a local `encompass_loans` / `encompass_borrowers` cache (pipeline + selector pulls, incremental on `Loan.LastModified`); `borrower_encompass_links` with `status IN ('suggested','confirmed','rejected')` + reviewer + reason — modeled on `db/081`.
2. **Reuse the identity ladder** (`src/clickup/identity.js`) verbatim: SSN-hash strong; email needs corroboration (phone/DOB/full-name — never surname); <2 corroborating fields = suggestion only. All Encompass matching flows through the same module so a fix (like the 2026-07-15 surname lesson) lands everywhere.
3. **Fix the leak surfaces before shipping any Encompass-derived columns:** S2-04 show-list conversion (`src/routes/borrower.js:272-315`) and S2-06 track-record show-list (`src/routes/borrower.js:1134-1141`). Alternative that avoids the dependency: keep **all** Encompass-derived data in side tables (`encompass_*`) that borrower routes never select from — recommended regardless.
4. **Partner-name discipline:** every Encompass text field (folder names, investor fields, milestone comments, condition titles) is staff-only by default; display through explicit allowlists; never write Encompass wording into `borrower_label`/`borrower_hint` or any borrower-visible field (S2-01/02/03 class).
5. **Provenance + freshness on everything:** each derived row/flag carries source loan GUID, pulled-at timestamp, and staleness display ("as of Encompass sync 2h ago") so staff never mistake a cache for live truth — mirrors the `clickup_shadow` pattern but staff-only.
6. **Human-in-the-loop symmetry with existing patterns:** suggestions → review queues (`borrower_dedup_candidates`, `sync_review_queue` style); auto-actions limited to *provably safe* ones (creating unverified rows, staff-only badges); anything that changes pricing inputs, experience counts, condition status, or borrower-visible data requires a recorded human verdict.

## 4. Open questions (for the owner / other agents)

1. **Payoff/EPO data availability:** does instance BE11397907 carry payoff dates / servicing outcomes after loans are sold to note buyers, or does that live only in a servicing system? Decides how much of I2's exit-date proxy and I6's EPO flag is buildable. Check custom fields via `GET /v3/schemas/loan` first.
2. **How does YS mark fallout in Encompass** — a folder ("Adverse"?), a milestone, a status custom field? I6's semantics depend entirely on the instance's actual conventions.
3. **SSN handling in the pull:** can the pipeline pull be scoped to SSN-last4 / avoid full SSNs entirely (match on DOB+phone+email+name+ssn_last4)? Pulling full SSNs expands the GLBA blast radius the split `borrowers`/`borrower_auth` design exists to contain.
4. **Historical depth:** how many years of closed loans exist in Encompass (pre-Encompass history in ClickUp only)? Determines how complete auto-built track records can be, given the 36-month exit window.
5. **Webhook subscriptions** are POSTs to the webhook service — does "architecturally incapable of writing" permit them, or is Phase 1 polling-only (pipeline on `Loan.LastModified`)?
6. **Does the owner want Encompass-verified track records to auto-verify** (YS's own funded loan as self-evident proof), or keep the human sign-off? Recommendation: keep the human sign-off in Phase 1.
