## 1. The two integration rails: Developer Connect vs. Partner Connect (EPC)

ICE Mortgage Technology exposes two distinct integration platforms. Understanding which rail a vendor rides explains almost everything about how their integration behaves.

| Dimension | **Encompass Developer Connect (EDC)** — *this project* | **Encompass Partner Connect (EPC)** |
|---|---|---|
| Who it's for | The **lender itself** (or an ISV acting with the lender's API key) building custom integrations against its own instance | **Service providers/ISVs** (credit, appraisal, title, MI, verifications, doc prep, PPE) who "build once, use everywhere" across all ICE lender-facing apps |
| Credential model | Lender's own OAuth2 client (client-credentials "API user", resource-owner password, or client assertion w/ private key + cert); scoped to one instance (e.g., BE11397907) | Partner-level credentials issued by ICE; lender "orders a service"; entitlements govern which loan data the partner may request/return |
| Data model | Direct REST access to loans, pipeline, eFolder, conditions, milestones, users, settings | Transaction request/response: partner receives an order (loan data + attachments per **request data entitlements**), services it, posts a response that EPC applies to the loan per **response data entitlements** |
| UI surface | None required — headless APIs | Runs in a **sandboxed iFrame inside Encompass**; interacts with the LOS through the EPC framework |
| Event delivery | **Webhook API** (`/webhook/v1/subscriptions`) subscribed by the lender | EPC webhooks notify the partner of transaction lifecycle events; payloads signed with a shared secret |
| Distribution | Private to the lender | Listed in the ICE Marketplace; any Encompass lender can enable it |

Key citations:
- EDC overview — "These are premium APIs for Encompass customers only (for Lender use). Access requires special licensing entitlements." (https://developer.icemortgagetechnology.com/developer-connect/docs/overview)
- "Grant type client_credentials is for ISV partners only" and API-User-for-ISV pattern (https://developer.icemortgagetechnology.com/developer-connect/docs/api-user-isv-partner)
- EPC definition: "cloud-native, 3rd-party-services integration platform… real-time, bi-directional data flow between lenders and partners" (https://docs.partnerconnect.elliemae.com/partnerconnect/docs/what-is-encompass-partner-connect; https://awesometechinc.com/what-is-encompass-partner-connect; https://www.valuelinksoftware.com/what-is-encompass-partner-connect-epc/)
- EPC transaction flow: partner pulls the transaction from `…/partner/v2/transactions`, services it, posts response; EPC processes asynchronously and notifies via signed webhook (https://docs.partnerconnect.elliemae.com/partnerconnect/docs/how-does-epc-work; …/transaction-fulfillment; …/webhooks)

**Implication for YS Capital:** we are squarely on the **Developer Connect / lender rail**. EPC is not applicable (we are not distributing a product to other lenders), but EPC's *entitlement* concept — a hard, declarative list of what data may flow in each direction — is the pattern worth imitating: our "response entitlement" is simply the empty set.

A third, adjacent rail matters for context: **Encompass Data Connect** — ICE's own managed replication of *all* loan fields to a lender-owned warehouse ("Direct Write") or an ICE-hosted **Snowflake** instance, in near real time, with ~3 years of history and schema auto-updates (https://mortgagetech.ice.com/products/encompass-data-connect; https://www.prnewswire.com/news-releases/ellie-mae-announces-new-data-delivery-method-encompass-data-connect-now-includes-a-hosted-delivery-choice-301069872.html). It is the industry's canonical **read-only, one-way** Encompass integration — architecturally the closest cousin to what YS Capital is building, just implemented by ICE instead of by the lender.

## 2. The ICE Marketplace ecosystem

- The Marketplace (https://mortgagetech.ice.com/products/marketplace) lists thousands of integrations across service-ordering categories: **Appraisal, Closing Fees, Documents, Flood, Fraud, Homeowners Insurance, Mortgage Insurance, Pricing, Tax Service, Title, Verifications** (https://mortgagetech.ice.com/partners/industry-service-provider-partners).
- ICE splits partners into **Industry Service Providers** (order/fulfill services on a loan — EPC rail) and **Solution Providers / ISVs** (software on top of the platform — mostly EDC rail with lender-granted API users) (https://mortgagetech.ice.com/partners/solution-provider-partners).
- ICE increasingly packages whole categories natively (e.g., a "Mortgage Insurance Center" with all major MI providers via EPC) (https://www.fintechfutures.com/press-releases/ice-launches-mortgage-insurance-center-for-encompass-digital-lending-platform-with-integrations-to-all-major-mi-providers).
- ICE also sells its own first-party products on the same APIs: **Encompass TPO Connect** (broker/correspondent portal), **Encompass Consumer Connect** (POS), **Encompass Data Connect** (replication), **ICE PPE** (pricing). TPO Connect is instructive: a web portal for third-party originators offering "bi-directional communication," configurable pipeline views, doc exchange — i.e., ICE's own version of "portal in front of Encompass" (https://mortgagetech.ice.com/products/encompass-tpo-connect; https://www.housingwire.com/articles/37918-ellie-maes-tpo-connect-allows-seamless-collaboration-with-lenders/). YS Capital's borrower portal is functionally a private, borrower-facing analog of TPO Connect, minus the write path.

## 3. Sync mechanics available on the Developer Connect rail (verified locally)

From the local Postman collection / endpoint index (`postman-endpoint-index.md`):

- **Webhook API** — `GET /webhook/v1/resources`, `GET /webhook/v1/resources/{resource}/events`, `POST /webhook/v1/subscriptions` (index lines listing folder "Webhook", e.g. requests `01 - GET all Resources` … `05a - Create a Loan Webhook Subscription`). Example subscription bodies in the collection show resources `Loan`, `Transaction` (EPC), `Trade`, `ExternalUser(s)`, `ExternalOrganization`, `InternalUsers`, `Timer`, `userGroup`, `DocumentOrder`.
- **Loan resource webhook events** (official reference, https://developer.icemortgagetechnology.com/developer-connect/reference/wbhks-re-cat-loan): `create`, `update`, `submit`, `move`, `delete`, `document`, `attachment`, **`condition`** (Enhanced Conditions: create/update/assign/comment/status-change subevents), **`milestone`** (updateMilestones, finishMilestones), `change` (specified attributes), **`fieldchange`** (specific field IDs), **`enhancedfieldchange`** (includes previous + new values), `lock/unlock`, `alertchange`, `disclosureTracking` (beta). Payloads carry `eventId`, `eventTime`, `eventType`, and `meta` (`userId`, `resourceType`, `resourceId`, `instanceId`, `resourceRef`) — i.e., a **thin event**: you get "loan X changed," then call back to fetch the data. The docs warn: "Webhook notifications are not guaranteed to be in real-time."
- **Polling / bulk read** — `POST /encompass/v3/loanPipeline` (filterable, cursor + `start/limit` paging, up to `limit=2000`, canonical field names via `GET /encompass/v3/loanPipeline/canonicalFields`), plus `POST /encompass/v3/loanPipeline/report` two-step cursor flow for large extracts (index: "Encompass Loan / Loan Pipeline / V3 Loan Pipeline" rows). Per-loan reads: `GET /encompass/v3/loans/{id}`, `POST /encompass/v3/loans/{loanId}/fieldReader` (read arbitrary field IDs), `GET /encompass/v3/loans/{loanId}/conditions`, `…/conditions/{id}?view=full`, `…/conditions/{id}/tracking`, `…/conditions/{id}/documents`, `GET /encompass/v3/loans/{loanId}/milestones` — every read need for the portal's condition/CTC gating exists as a GET.
- **Concurrency/rate limits** — default ~30 concurrent API calls per lender environment, shared across *all* consumers of the instance (partner + lender initiated); response headers `X-Concurrency-Limit-Limit` / `X-Concurrency-Limit-Remaining`; 429s when exceeded; increases available on request (https://developer.icemortgagetechnology.com/developer-connect/docs/concurrency-limits). ICE's stated best practice: **"Use webhook events to listen for changes instead of polling via GET calls."** (echoed by https://mortgageworkspace.com/blog/exploring-api-integrations-for-encompass-whats-possible-in-2025).
- **Webhook security** — signing-key validation plus optional "Webhook Custom Auth" premium functions (`/webhook/v1/functions/auth`, OAUTH2_CLIENT_CREDENTIAL policy) to make Encompass authenticate *to our* receiving endpoint (Postman folder "Webhook Custom Auth - Premium").

## 5. Deep dives: publicly documented sync approaches (4–6 vendors)

### 5.1 Blend (POS) — webhook-driven, asymmetric, deliberately *not* full two-way
- **Encompass → Blend is webhook-consuming**: "This solution consumes webhooks from Encompass to let the service decide what needs to change based on the Blend loan and the current state of the borrowers in Encompass." Triggered "any time borrower information is added, edited, or removed in Encompass, and when a Partner Document Delivery package is received." (https://help.blend.com/support/solutions/articles/156000371700-loan-updates)
- **Explicitly scoped**: "This is not a full two-way sync of application data" — only a defined field list (borrower identity/contact, NBO, loan amount, property, TRID date, assignees, **milestone**, archive status, **conditions**) syncs down.
- **Blend → Encompass is event-gated**, not continuous: exports fire on borrower app submit, LO manual export, TRID trigger, credit/AUS/fees events (https://help.blend.com/support/solutions/articles/156000371699-application-sync).
- Some deployments use a KensieMae plugin for in-Encompass sync-status visibility (https://www.kensiemaellc.com/product/blend).
- **Lesson**: even the biggest POS treats Encompass as the system of record, listens via webhooks, re-reads current state before acting, and keeps the synced field set small and explicit.

### 5.2 Floify (POS) — lender-supplied Developer Connect credentials, dedicated API user
- Setup requires **Encompass Client ID + Client Secret (from the Developer Connect portal, My Account → API Key), an Encompass username/password, and the Instance ID**; Floify explicitly recommends "a dedicated Floify account with a non-expiring password" rather than personal credentials, "needed to periodically refresh your company's connection" (https://help.floify.com/hc/en-us/articles/360042244091-Encompass-Developer-Connect).
- Feature articles show milestone sync ("Encompass Company Milestone Sync"), **Encompass conditions → Floify document requests** ("Enable Encompass Conditions to Floify Document Requests" — the exact conditions-mirroring pattern YS Capital wants, in the read direction), and a separate "Bi-directional Data Sync Encompass Plugin" for deeper two-way field sync (https://help.floify.com/hc/en-us/articles/360056854832; …/360041825392; …/7745936627853).
- **Lesson**: dedicated non-expiring service account, lender-owned API key, conditions surfaced to borrowers as document requests derived from Encompass state.

### 5.3 Total Expert Fusion (CRM) — mapped, filtered, one-way loan feed
- Fusion is a configuration layer: lender maps Encompass fields → TE journey fields per connection/loan folder; **"Loan Data Sync Control"** lets the lender designate a gate field so "the system only attempts to send loan files with the field of your choice checked… when they are created or updated" (https://totalexpert.freshdesk.com/support/solutions/articles/22000286617-encompass-best-practices; https://totalexpert-trainingteam.s3.us-east-2.amazonaws.com/TE_Encompass/Fusion-+Encompass+Feature+and+Configuration+Guide.pdf).
- Provisioning: TE "will request access to your instance via API from ICE" (~5 business days) and returns a company-specific API Client ID — the standard ISV-with-lender-API-user flow.
- **Lesson**: sync on create/update events, but let an explicit field/flag gate *which* loans flow; map only named fields, not whole loans.

### 5.4 Truv / The Work Number (verifications) — EPC transactions + Automated Service Ordering
- Truv verifies income/employment "within the Encompass Partner Connect platform"; **ASO** auto-fires orders "when a loan fulfills certain conditions"; completed orders deposit all documents (reports, paystubs, W-2s) into a **Verifications eFolder**, and connected accounts are re-monitored to regenerate reports through origination (https://truv.com/mortgage-lending/los-integrations/encompass; https://docs.truv.com/docs/encompass-integration-guide).
- **Lesson**: rule-triggered automation keyed off loan state, results land in the eFolder — when YS Capital reads conditions/documents, expect vendor-deposited artifacts there.

### 5.5 Optimal Blue / Polly (PPE) — the opposite archetype (bidirectional EPC)
- Both are canonical EPC citizens: pricing/lock inside Encompass, loan-level results written back to the loan; OB automates lock desk actions end-to-end, Polly receives "for-sale pipeline from Encompass and [returns] loan-level pricing… upon loan commitment" (URLs in §4.3). Included as the contrast case: this is what YS Capital is deliberately **not** building.

### 5.6 Mortgage Coach (borrower-facing analytics) — read-mostly Developer Connect
- Pulls live loan data via EDC to auto-generate borrower TCA presentations with SSO from Encompass; no meaningful write-back (URLs in §4.6). Closest well-known product to the YS Capital shape: *lender-authorized read of loan data to power a borrower-facing experience*.

## 6. Integration archetypes — and where YS Capital fits

1. **Service-transaction partner (EPC)** — order/response, entitlement-governed, writes results into the loan/eFolder. (OB, Polly, DocMagic, Truv, TWN.) *Not us.*
2. **Bidirectional POS/CRM companion (EDC + webhooks)** — Encompass is system of record; companion listens to webhooks, re-reads state, pushes a scoped field set back on defined events. (Blend, Floify, BeSmartee, SimpleNexus, Jungo.) *Half of us — we take the read half only.*
3. **One-way mapped feed (EDC)** — create/update-triggered export of mapped fields, gated by a control flag. (Total Expert Fusion, Surefire.) *Close.*
4. **Read-only replication / analytics** — full or wide loan-data copy into an external store; portal/BI reads the copy, never the LOS, and never writes back. (Encompass Data Connect, Richey May, RETR-style products, Mortgage Coach for per-loan reads.) *This is us.*

**YS Capital's Phase 1 = archetype 4 implemented with archetype 2's plumbing**: webhook-triggered (`Loan` resource: `milestone`, `condition`, `fieldchange`/`enhancedfieldchange`, `update`, `move`, `delete`) thin events → authenticated GET-only re-read (`loanPipeline`, `loans/{id}`, `conditions`, `milestones`, `fieldReader`) → local Postgres mirror that the portal's rule engine (condition-clearing gate, CTC gate) evaluates. A scheduled `loanPipeline` reconciliation sweep backstops missed webhooks — ICE itself warns webhooks "are not guaranteed to be in real-time."

Notably, the repo already runs this exact shape against ClickUp: a pull/ingest processor fed by "the live webhook processor and the historical backfill," idempotent and keyed on an immutable external ID (`/home/user/yscap/yscap-repo-root_8/src/clickup/ingest.js:1-12`), with a queue worker whose header says "Encompass/Graph targets slot in here behind the same interface later" (`/home/user/yscap/yscap-repo-root_8/src/sync/queue.js:1-6`); `sync_queue` is already earmarked "ClickUp/Encompass, deferred" (`/home/user/yscap/yscap-repo-root_8/CLAUDE.md:45`, `README.md:100`).

## 7. What the vendors do that we should copy

1. **Dedicated, non-expiring service account + lender-owned API key** (Floify) — a named "portal-encompass-reader" API user so audit trails and entitlements are clean.
2. **Webhooks-first, poll-as-reconciliation** (Blend + ICE's own guidance) — subscribe to thin events, re-read current state before acting; never trust the event payload as the data.
3. **Explicit, small, versioned field map with a sync-gate flag** (Total Expert Fusion, Blend's enumerated field list) — enumerate exactly which canonical fields the portal consumes; use `fieldReader`/pipeline field lists, not whole-loan dumps, to stay under the ~30-concurrent-call instance budget shared with every other integration.
4. **Entitlement thinking from EPC** — encode "response entitlement = none" structurally: an HTTP client that only exposes GET (plus the POST-that-are-reads: `loanPipeline`, `fieldReader`, selectors), no PATCH/PUT/DELETE code paths at all.
5. **Local mirror as the serving layer** (Data Connect / Richey May) — the portal and its rules read Postgres, not Encompass, per request; keeps borrower page loads off the concurrency budget and makes the integration failure-isolated.
6. **Idempotent, external-ID-keyed ingest with backfill** — already proven in-repo for ClickUp (`src/clickup/ingest.js`); reuse the pattern with the Encompass loan GUID as the immutable key, plus `move`/`delete` handling for folder moves and soft deletes (Blend handles `move` as archive signal).
7. **Signed/authenticated webhook endpoint** — use the Webhook API signing key and consider the Custom Auth premium function so our receiver can verify origin (Postman folder "Webhook Custom Auth - Premium").
8. **Condition→borrower-task mirroring** (Floify's "Encompass Conditions to Floify Document Requests") — precedent for the portal's core rule that a portal condition reflects, and can only clear in agreement with, the Encompass Enhanced Condition (`GET /encompass/v3/loans/{loanId}/conditions/{id}?view=full` + `/tracking`).
