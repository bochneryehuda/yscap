# B1 — Portal Data Model Map (YS Capital Borrower Portal)

Repo: `/home/user/yscap/yscap-repo-root_8` (Node/Express + Postgres). Base schema: `db/schema.sql` (335 lines, non-idempotent, applied once on an empty DB); evolution: idempotent numbered migrations `db/002_*.sql` … `db/125_*.sql`, applied in order on every boot by `src/migrate-boot.js` (CLAUDE.md:33–35).

Design rules stated in the schema header (`db/schema.sql:6-12`):
- **Borrower is the BASE entity**; each mortgage = a new `applications` row.
- **LLCs + track records live on the BORROWER**, not the application; an application links to ONE LLC (vesting entity).
- Checklist items are scoped `application | borrower_profile | llc`.
- SSN encrypted at rest (pgcrypto app-side AES-256-GCM); PII access audit-logged (GLBA).
- `db/schema.sql:4`: "Syncs bidirectionally to ClickUp (now) and **Encompass (later)** via sync_*" — Encompass was anticipated from day one.

## 4. Domain: Applications (loan files)

`applications` (schema.sql:155-213) — one borrower : many applications. Key fields grouped:

- **Natural keys**: **`ys_loan_number`** (schema.sql:164; see §11), **`investor_loan_number`** (schema.sql:165).
- **Product**: `program` (Fix & Flip w/ Construction / Bridge / DSCR…), `loan_type` (Purchase / Refi R&T / Refi Cash-Out / Ground up), `lender`, `channel` (Wholesale / Delegated Corr / …), `occupancy`.
- **Property**: `property_address jsonb`, `property_type`, `units`, `sqft_pre/post`.
- **Economics (snapshot)**: `purchase_price`, `as_is_value`, `arv`, `rehab_budget`, `rehab_type`, `loan_amount`, `ltv`, `dscr_ratio`, `rate_pct`, `term`, `ppp`; 016:6-8 assignment purchase: `is_assignment`, `underlying_contract_price`, `assignment_fee`; 030 `requested_ir_months`, 073 `requested_ir_amount`; 032:19-22 `requested_exp_reo`, `payoff_amount`, `original_purchase_price`, `acquisition_date`; claimed experience `requested_exp_flips/holds/ground` (schema.sql:184-186).
- **Appraisal/DSCR extras from ClickUp** (041:23-24, 047:8-26): `actual_appraised_value`, `approx_appraised_value`, `appraised_rental_value`, `approx_appraised_rental_value`, `cda_value`, `appraiser_name`, `actual_rate`, `desired_rate`, `property_taxes`, `property_insurance`, `property_hoa`, `rental_income`, `prepayment_penalty`, `title_company(_contact)`, `insurance_company(_contact)`, `first_lien`, `second_lien`, **`encompass_status` (047:25)**, `application_submitted`.

## 12. Candidate JOIN KEYS: portal application ↔ Encompass loan, ranked

The repo already solved this exact problem for ClickUp (`src/clickup/identity.js:14-17`): 8 identity fields — `address, loanNumber, borrowerName, dob, email, ssn, phone, purchasePrice` — **≥2 must agree** to declare "same loan"; one field is never enough. Reuse this module wholesale.

1. **`applications.ys_loan_number` = Encompass Loan Number** — the strongest existing natural key. Globally unique among live files (db/048 partial unique index excludes soft-deleted rows). Compare `lower(btrim())`; skip placeholder values (`isPlaceholderLoanNumber`). Caveats: nullable (early-stage files have none), sourced today from ClickUp text, an archived (soft-deleted) file can share the number.
2. **A stored Encompass loan GUID — does not exist yet; add it.** The durable binding analogous to `clickup_pipeline_task_id`/`clickup_task_index`. Recommend `applications.encompass_loan_guid` (+ partial unique index mirroring db/048) plus an `encompass_loan_index` table cloned from `clickup_task_index` (044:39-51) carrying match_status/snapshot like 045/046. Once set after a first successful match, it is rank-0 reliable and every other key becomes verification only.
3. **`borrowers.ssn_hash`** (044:10-12, HMAC-SHA256 keyed by `SSN_MATCH_KEY`) — deterministic borrower-level join; hash the SSN read from Encompass with the same key and match, never store plaintext. Strong for borrower identity, but one borrower has many loans — must be combined with a per-loan key.
4. **Normalized property address + borrower identity** — `identity.js normAddress()` (strips unit/apt, ", USA", punctuation) against Encompass subject-property address, combined with borrower last name/DOB. Address alone uniquely identifies a loan per borrower in practice (ingest.js:559 comment) but street-suffix/abbreviation variance makes it a fuzzy key; `address_canon_cache` (124) exists to help.
5. **Multi-field composite (≥2-of-8)** — the general fallback: DOB + email, name + purchase price, phone + address, etc., exactly as `isMatch()` implements. Route sub-threshold matches to a human queue (`borrower_link_candidates` / `sync_review_queue` patterns).
6. **`investor_loan_number`** — secondary; only if YS's Encompass instance carries it consistently.
7. **Weak/rejected keys**: borrower email alone (shared spouse emails — see 114 borrower_profile_links), phone alone, borrower name alone, `encompass_status` (label only), ClickUp task id (no Encompass meaning).

### Read-only guardrail observation
`sync_queue.target` already allows `'encompass'` with `direction IN ('push','pull')` (schema.sql:306-307). For the "architecturally incapable of writing" requirement, Phase 1 should add a DB CHECK/trigger (new numbered migration, per repo convention) rejecting `target='encompass' AND direction='push'` rows, in addition to shipping no push code path — matching this repo's belt-and-suspenders trigger pattern (069/071/072/103).
