# F3 — Data Governance Plan for Encompass-Sourced Data (Design Only)

_Research agent F3 · 2026-07-19 · YS Capital borrower portal ← Encompass Developer Connect, instance BE11397907, Phase 1 strictly read-only._

Repo: `/home/user/yscap/yscap-repo-root_8` (paths below relative to that root). Builds on A5 (security posture), B1 (data model / join keys), D3 (compliance norms) — full findings in the sibling files. This is a **design**; no repo files were changed.

**Governing principles** (derived from D3 §6-8 and the repo's own doctrine):

1. **Bounded operational cache, not a mirror.** ICE's public API terms restrict "data replication (e.g., data warehouses or analytics repositories)" without written approval (EPC T&Cs §3.1(k), per D3 §8). Every pulled field must be on a signed-off allowlist that a defined portal workflow reads. This same boundedness is the GLBA data-inventory artifact examiners request (D3 §4).
2. **Minimize at the source, filter at intake, redact at storage, scrub at output** — the portal's existing four-layer doctrine (`src/lib/pii-guard.js`, `src/lib/redact.js`, `src/lib/crypto.js`, `src/lib/borrower-safe.js`; A5 §1).
3. **Encompass stays the system of record.** The portal durably retains only *evidence of its own decisions* ("condition X cleared at T because Encompass snapshot S showed Y"), not the loan file itself (D3 §7).
4. **No new access paths.** Encompass-sourced data inherits `VISIBLE_OFFICERS_SQL` / `canSeeBorrowerId` scoping (`src/routes/staff.js:136-141, 2812-2826`) and the capability model (`src/lib/permissions.js:28-61`). Borrowers see derived outcomes only.

## 1. Field-level intake policy

Four tiers. The allowlist is enforced in **three places**: (a) server-side in Encompass via a read-only API-user persona with field-level denies ("Persona Access to Field" removes denied field IDs from API responses — D3 §6); (b) request-side by preferring `POST /encompass/v3/loans/{loanId}/fieldReader` with an explicit field-ID list over `view=full` loan pulls (endpoint present in the local Postman index, line 15); (c) intake-side by a strict allowlist filter + extended `redactPII()` before anything persists.

### Tier 1 — PULL (structured, plaintext OK)

Non-sensitive operational fields the gating rules and staff display actually read:

| Group | Fields | Consumed by |
|---|---|---|
| Loan identity | loan GUID, Loan Number (field 364), investor loan number | Matching to `applications.ys_loan_number` / `investor_loan_number` (`db/schema.sql:164-165`; B1 §12) |
| Lifecycle | current milestone, milestone log (dates, done-by), loan folder, **clear-to-close indicator + date**, key dates (est./actual closing, lock) | CTC gate on `applications.status` via `signOffGate` (B1 §5); replaces the manual ClickUp-fed `applications.encompass_status` (`db/047:25`) |
| Conditions | UW condition list: id, title, status, cleared date/by, category | Condition-parity gate for `checklist_items` (incl. `rtl_p2_enc`, `rtl_p2_loprep` — `db/005:74-84`) |
| Economics | loan amount, note rate, term, amortization, loan purpose/type, lien position | Structure-match check `rtl_p2_enc` ("loan amount & structure match the file"); compare vs `product_registrations.note_rate/total_loan` |
| Property | subject property address, property type, units, appraised value | Loan matching (address key) + condition checks |
| Parties (non-sensitive) | borrower/co-borrower **name, email, phone**; entity/vesting name; loan team (LO/processor/UW names, NMLS) | Identity matching (≥2-of-8, `src/clickup/identity.js`); team display |
| Credit summary | representative credit score(s) (number only) | `rtl_p3_credit2` "scores entered in Encompass & YS portal, verified" |

### Tier 2 — PULL MINIMIZED (aggregate only; income/asset minimization)

The liquidity condition (`rtl_p3_liq`, "required liquidity verified against the Encompass F&F sheet") needs a **single aggregate number**, not account detail:

- **Pull:** total verified liquid assets (one figure), and/or a computed `liquidity_ok` boolean + the threshold used.
- **Exclude:** account numbers, per-account balances, institution+account pairs, asset statements, income line items, employer income figures, tax-return data. If the F&F sheet only exists as per-account rows, sum server-side in memory during the pull and persist only the total.
- Same pattern for any future income check: store the derived verdict (and at most the aggregate), never the source detail.

### Tier 3 — HASH / DERIVE (transient in memory; only the derivative persists)

| Datum | Handling |
|---|---|
| **SSN** | **Never persist a full SSN from Encompass — no second ciphertext.** Default: don't request it at all (persona-denied); match on loan GUID → loan number → name/email/address composite. If borrower-identity matching measurably needs it: pull transiently, compute the existing HMAC via `SSN_MATCH_KEY` (`src/config.js:87-90`), compare/store against `borrowers.ssn_hash` (`db/044:10-12`), optionally retain `last4` for staff cross-check display, and discard plaintext in the same request scope. Any full-SSN need routes through the single chokepoint `ssnForStorage()` (`src/lib/crypto.js:136-140`) — which Phase 1 should never invoke for Encompass data. |
| **DOB** | **Do not persist a full Encompass DOB in Phase 1.** For identity verification: pull transiently, compare in memory to `borrowers.date_of_birth`, persist only `dob_match: true/false` (+ birth **year** if a display aid is wanted). A mismatch never overwrites — it opens a `sync_review_queue` row (two-sided, `db/110`), consistent with the `decideDob()` doctrine that a DOB change is always a human decision (CLAUDE.md §3b). If a future phase must store an Encompass DOB, it goes in the AES-256-GCM envelope (`crypto.js:109-127`), date-only `'YYYY-MM-DD'` strings end-to-end, validated by `sanitizeDob`. |

### Tier 4 — EXCLUDE (never requested; persona-denied; intake-filter rejects on arrival)

- Full SSN / ITIN (field 65 etc.), full DOB (beyond Tier 3 transient), driver's-license / passport numbers.
- Bank/asset **account numbers**, card numbers, per-account statements, VOD/VOA detail.
- Credit **report payloads** (tradelines, inquiries) — the portal pulls its own via Xactus under FCRA; only the score number crosses (Tier 1).
- Income documents / figures beyond the Tier 2 aggregate; tax transcripts; HMDA GMI (race/ethnicity/sex) — no portal workflow reads them.
- **eFolder attachments/documents** (Phase 1): heaviest PII (1003s, bank statements). If a later phase pulls them, they must flow through `src/lib/storage.js` + `src/lib/serve-document.js` with `canSeeDocument` gating and `download_document` auditing (A5 open question).
- Encompass conversation/notes free text (uncontrolled PII), except specific condition comments if a workflow requires them (then pii-guard-scanned first).

**Intake enforcement (belt and suspenders):**
1. Allowlist filter: the pull adopts **only** keys on the versioned allowlist; any unexpected key in a response is dropped **and** raises an `encompass_intake_violation` review/alert row (the persona should have prevented it — its appearance means server-side config drifted).
2. Extend `SSN_KEY` in `src/lib/redact.js:7` (or add a sibling `ENCOMPASS_EXCLUDE_KEY`) to Encompass spellings — `taxIdentificationNumber`, `ssn`, `socialSecurityNumber`, `birthDate`, `dateOfBirth`, `accountNumber`, `bankAccount` — and run every stored jsonb through `redactPII()` exactly as `raw_intake` is scrubbed today (`src/routes/borrower.js:398`, `intake.js:118`, `leads.js:171`).
3. Journal masking: Encompass values entering any journal reuse the `maskSSN`/`maskCard` pattern (`src/clickup/orchestrator.js:509-527`); cleartext PII never lands in `clickup_write_log`-style tables (`db/107` header) and the same holds for the new pull journal.

**Change control:** the allowlist lives as a code constant + a versioned `docs/ENCOMPASS-FIELD-INVENTORY.md` (MISMO/ULAD names as neutral vocabulary, mirroring `docs/FIELD-MAP.md` for ClickUp). Adding a field = code change + doc update + sign-off by the compliance owner/Qualified Individual (open question 5, D3 §11). This doc is the exam-binder data-inventory artifact.

## 2. Storage design

### 2.1 Proposed tables (new numbered idempotent migrations, per repo convention)

| Table | Role | Retention class |
|---|---|---|
| `encompass_loan_index` | Durable binding portal file ↔ Encompass loan (clone of `clickup_task_index`, `db/044:39-51`): `loan_guid` UNIQUE, `application_id`, `loan_number`, `match_status`, `matched_fields jsonb`, `first_matched_at`, `last_pull_at`. Plus `applications.encompass_loan_guid` with a partial-unique index mirroring `db/048` (B1 §12). | Life of the file |
| `encompass_loan_state` | **Current-state operational cache**, one row per bound loan: structured columns for Tier-1 fields (milestone, ctc_flag, ctc_date, loan_amount, note_rate, term…), `fields jsonb` for the remaining allowlist (redacted per §1), `pulled_at`, `encompass_modified_at`. Overwritten each pull — current state only. | Overwritten; deleted with the file |
| `encompass_snapshots` | Append-only per-pull snapshot of the **allowlisted field set only** (never `view=full` output), redacted; `loan_guid`, `application_id`, `snapshot jsonb`, `sha256`, `pulled_at`. Exists for drift diagnosis and review-queue display. | **Pruned** (§5) |
| `encompass_pull_log` | Append-only fetch journal modeled on `clickup_write_log` (`db/107`): endpoint class, loan GUID, HTTP status, record/field counts, duration, `actor_kind/actor_id` (system vs staff-triggered), `adopted jsonb` (field → old/new **masked** values actually written into portal columns), `blocked` flag for guardrail refusals. | ≥ 5 years |
| `encompass_gate_log` | **Decision evidence** (the NYDFS 500.06 "reconstruct the transaction" bar, D3 §10): one row per rule evaluation that consumed Encompass data — gate name (`condition_parity`, `ctc`, `credit_scores`, `liquidity`), `checklist_item_id`/application, verdict pass/fail/indeterminate, and the **decision-time copy of the exact field values relied on** (field id, value, Encompass last-modified, pull ts). Because relied-on values are copied here, `encompass_snapshots` stays freely prunable. | ≥ 5 years, legal-hold aware |

Enrichment of portal-native columns (e.g. filling a blank `applications.investor_loan_number` or `borrowers.fico`) is allowed only fill-blank/COALESCE-style (never overwrite a differing value — that goes to `sync_review_queue`, `db/108/110`), stamped with the `origin` pattern (`borrowers.origin`, `db/044:11`) extended with an `'encompass'` value, and journaled in `encompass_pull_log.adopted`.

### 2.2 Raw snapshot JSONB rules

- Snapshot = the fieldReader response for the allowlisted IDs, post-filter, post-`redactPII()`. Storing a full loan object is prohibited (ICE replication clause + minimization).
- No Tier-3/4 datum may appear in any jsonb, log line, error message, or `audit_log.detail` — same rule that keeps SSNs out of `raw_intake` today (`redact.js:1-6`).

### 2.3 Encryption posture

- **In transit:** TLS to Encompass (default); OAuth2 client-credentials token held in memory only, never persisted (A5 checklist §A4).
- **At rest:** DB-level encryption (Render Postgres). Because Tier-3/4 exclusions keep sensitive values out of the store entirely, **no new field-level ciphertext is needed in Phase 1** — that is the point of the intake policy. Any later-approved sensitive field (e.g. DOB) uses the existing AES-256-GCM envelope (`crypto.js:109-127`) with keys env-only via `resolveSecret` (`src/config.js:34-50`); never a second encrypted-SSN copy (two ciphertexts under two lifecycles = double breach surface, D3 §6).
- Encryption posture documented in writing → preserves the FTC §314 and state breach-law **encryption safe harbor** (D3 §2.2, §9).
- Credentials: `encompass: {...}` block in `src/config.js`, master switch `ENCOMPASS_ENABLED=0` default-off, `configured()` status in `src/lib/integrations/index.js` (A5 §7); rotation documented; a credential pasted in chat is compromised and rotated (CLAUDE.md standing rule).

## 3. Access control

### 3.1 Who sees what

| Audience | Sees | Gate |
|---|---|---|
| Staff assigned to the file (LO/processor/assistant/delegate) | Encompass panel on the file: milestone, CTC, condition parity results, economics comparison — Tier-1/2 values | The `/applications/:id` scoped middleware + `VISIBLE_OFFICERS_SQL` (`staff.js:136-141`); **no new parallel query path** |
| `see_all_files` roles (admin, underwriter, loan_coordinator per `ROLE_DEFAULTS`, `permissions.js:49-61`) | Same, all files | Existing capability |
| Borrower-level Encompass data (name/email/phone match results, credit-score presence, dob_match) | Staff who pass `canSeeBorrowerId` (`staff.js:2812-2826`) — the documented GLBA horizontal gate | Existing gate |
| Raw snapshot / pull journal / gate log viewer | Staff with `view_audit_log` | Existing capability |
| Connection setup, allowlist status, kill switch | `platform_setup` | Existing capability |
| **Borrowers** | Derived outcomes only ("condition cleared", "clear to close") through the existing audience model (`borrower_label`/`borrower_hint`, B1 §5); every borrower-facing string through `borrower-safe.scrubText` (investor names occur inside Encompass loan data — lender/investor fields) | Existing borrower routes; `activity.js` rows `borrowerSafe:false` by default |

No new capability is required for Phase 1; if a later phase pulls anything sensitive, introduce `view_encompass_sensitive` via `sanitizeOverrides`-whitelisted capability keys (`permissions.js:96-101`) rather than role checks.

Last-4/derived SSN artifacts (if Tier 3 is enabled) follow the existing reveal rules: masked display; there is nothing to "reveal" because no full value is stored — the audited `view_ssn` endpoint (`staff.js:3440-3448`) remains the only SSN reveal path and serves the portal-native ciphertext only.

### 3.2 Provenance labeling in UI

Every Encompass-sourced value rendered to staff carries provenance so staff always know a value came from Encompass and how fresh it is:

- API responses wrap Encompass values in a standard shape: `{ value, source: 'encompass', asOf: <pulled_at>, encompassModifiedAt, loanGuid }` — never silently merged into portal-native fields.
- UI badge: a small "Encompass · as of Jul 19, 2:14 PM" chip next to the value; stale data (> configurable threshold since `pulled_at`, e.g. 24h) renders amber with "may be out of date — refresh"; a failed last pull renders the last-known value with an explicit warning, never blank (mirrors the notify.js "degrade, don't break" doctrine).
- Gate results always show both sides: "Portal: cleared · Encompass: condition still open (as of …)" — the two-sided display pattern already established by `sync_review_queue` (`db/110` header).
- Enriched portal columns show an origin marker (from the `origin` stamp) in edit surfaces, so staff know an edit will diverge from Encompass and trigger review on the next pull.
- The per-file activity feed gets staff-only rows for adopted changes ("Encompass milestone moved to Resubmittal") — `borrowerSafe:false` unless deliberately whitelisted in `AUDIT_RENDER` (A5 §4.3).

## 4. Audit requirements

**Every fetch and every rule evaluation that consumed Encompass data is logged.** Two layers, matching the repo's split between `audit_log` (who/what/when) and journals (full values):

1. **`audit_log` actions** (`db/schema.sql:322-335`; labels registered in `src/lib/audit-actions.js` under a new `encompass` category sibling to `sync`; unknown codes already render via `describeAction()` fallback):
   - Fetch: `encompass_pull_loan`, `encompass_pull_batch`, `encompass_refresh_requested` (staff-triggered — `actor_kind='staff'` + real id; scheduled pulls `actor_kind='system'`), `encompass_pull_failed`.
   - Rule evaluations: `encompass_gate_eval` (every evaluation, pass or fail), `encompass_condition_parity_block` (a sign-off refused because Encompass disagreed), `encompass_ctc_gate_pass` / `encompass_ctc_gate_fail`.
   - Matching: `encompass_loan_linked`, `encompass_link_review_queued`, `encompass_link_unlinked`.
   - Admin/ops: `encompass_config_change`, `encompass_credential_rotated`, `encompass_circuit_open`, `encompass_intake_violation`, `encompass_snapshot_prune`.
   - `detail` jsonb carries loan GUID, endpoint class, counts, verdicts — **never PII** (the pii category precedent, `audit-actions.js:17-22`).
2. **Journals:** `encompass_pull_log` (every API batch — including failures and guardrail blocks, so "the cross-system history never goes dark," the `db/107` lesson) and `encompass_gate_log` (decision-time field snapshots, §2.1). Append-only: the app DB role gets INSERT/SELECT only on both (no UPDATE/DELETE), the immutability posture D3 §10.5 recommends.
3. **Telemetry/anomaly:** pull-rate and volume counters surfaced on `/api/health`; alert on volume spikes or off-hours pulls (possible credential misuse — ICE monitors its side too, EPC §3.4); outbound circuit breaker opening is itself audited (mirror `outbound_circuit_open`).
4. **Sampling into governance:** pull-journal and gate-log samples go into the QI annual report and the CSBS exam binder (D3 §4: examiners request the design doc, field inventory, vendor file, audit samples).

## 5. Retention & snapshot pruning

Per D3 §7 there is no single "GLBA retention period" — floors (state licensing 3-5 yrs typical; Reg B 12-month business-credit; NYDFS 5-year if applicable) plus the FTC §314.4(c)(6) 2-year-disposal ceiling absent documented need. Do **not** copy the inaccurate "GLBA ≥ 6 years" line from `docs/SHAREPOINT-SECURITY-COMPLIANCE.md:38`.

| Data class | Policy |
|---|---|
| `encompass_loan_state` (current cache) | Overwritten every pull; deleted when the application is purged (extend the audited `purge_application` cascade, `db/018`, `audit-actions.js`) |
| `encompass_snapshots` | **Prune aggressively**: rolling window `ENCOMPASS_SNAPSHOT_RETENTION_DAYS` (default **90**) AND keep-last-N per loan (default **10**); loans that reach funded/withdrawn/declined drop to keep-last-1 after 30 days. Safe because every relied-on value is copied into `encompass_gate_log`. Pruning job is idempotent, lease-guarded via `sync_locks`, and audited (`encompass_snapshot_prune` with counts). Thin snapshots are also the ICE §3.1(k) "not a warehouse" evidence. |
| `encompass_pull_log`, `encompass_gate_log`, related `audit_log` rows | Retain **≥ 5 years** (NYDFS bar; covers typical state floors) — final number set by the written retention schedule once the licensed-state list is confirmed (D3 open question 1) |
| `encompass_loan_index` binding | Life of the file + file retention period |
| Anything with no documented need | Dispose within 2 years of last use (FTC) |
| **Legal hold** | A hold flag (per application/borrower) suspends pruning and purge cascades; hold set/cleared is audited |

The written retention schedule (a one-page doc naming each table, its period, and its legal basis) is itself the compliance artifact.

## 6. Incident playbook basics

Pre-staged scenarios, first moves, and evidence sources. All start with the same two levers: the **kill switch** (`ENCOMPASS_ENABLED=0` — integration dormant, portal keeps working) and **log preservation** (audit_log + encompass_pull_log are append-only; snapshot pruning paused during any incident).

| # | Scenario | Playbook |
|---|---|---|
| 1 | **Credential compromise** (secret leaked, anomalous pull volume, ICE notice) | Kill switch → rotate the API-user secret in the Encompass admin console → revoke tokens → preserve logs → scope query on `encompass_pull_log` (what was pulled, when, from where) → QI + counsel; if unencrypted NPI of ≥500 consumers acquired: FTC notice ≤30 days (D3 §2.2) + state matrix (all-50-state laws keyed to the individual, D3 §9); notify ICE per contract; document in exam binder |
| 2 | **Excluded field appears in a response** (persona drift — an SSN/DOB/account number arrives) | Intake filter already dropped it and raised `encompass_intake_violation`; verify nothing persisted (grep snapshots/state by pull id); fix the Encompass persona field-security config; if anything persisted: targeted purge + audit row + assess as a potential internal exposure (encrypted-at-rest DB → likely safe-harbor, but log the analysis) |
| 3 | **Wrong-loan match** (Encompass data adopted onto the wrong borrower's file = cross-borrower NPI exposure) | Unlink (`encompass_link_unlinked`), pull-log `adopted` entries identify every enriched field → revert; check who viewed the file meanwhile (audit_log by entity); open a `sync_review_queue` row for re-match; tighten the ≥2-of-8 threshold if systemic |
| 4 | **Portal DB breach involving Encompass-sourced rows** | Standard IR plan (Safeguards §314.4(h)) — scope is small by design: Tier-1/2 fields are largely non-trigger data under state breach statutes (no SSN/DOB/account numbers stored); the field-inventory doc makes the "what was in there" answer immediate |
| 5 | **ICE-side block/throttle** (EPC §3.6 remedies) | Circuit breaker opens; portal degrades to last-known values with stale badges (§3.2); gates that require fresh Encompass data fail **closed** (a condition cannot clear on stale/absent data — never fail open); contact ICE; review call-volume budget |
| 6 | **Rule regression** (gate cleared something it shouldn't have) | `encompass_gate_log` reconstructs every decision with the exact field values relied on; re-run the corrected rule against the log; conditions wrongly satisfied are reopened via the established reopen pattern (`db/071/072`) with `[auto]` notes |

Runbook doc lives in `docs/` (e.g. `docs/ENCOMPASS-INCIDENT-RUNBOOK.md`) naming the QI, counsel/insurer contacts, the Encompass admin who rotates credentials, and the notification decision tree — the §314.4(h) written-IR-plan element and cyber-insurance notice timelines (often shorter than statutory; D3 open question 6).
