## 1. Where the Encompass developer community actually lives

- A StackExchange API search for `elliemae` returned **1 result**; `"encompass api"` returned **1 result**; `encompass loan` returned only **7 questions**, most from the 2017-2018 SDK era (e.g., ["Get field data outside of reporting database using Encompass360 SDK"](https://stackoverflow.com/questions/43456273/get-field-data-outside-of-reporting-database-using-encompass360-sdk), ["How to query loans with Custom field in encompass?"](https://stackoverflow.com/questions/53669491/how-to-query-loans-with-custom-field-in-encompass)). ## 4. Practitioner pain points (with evidence)

### 4.1 Token expiry handling
- Official token semantics ([Authorization docs](https://developer.icemortgagetechnology.com/developer-connect/docs/authentication)): access tokens live **30 minutes, max 24 hours**, and **must be used at least once within 15 minutes** or they expire early; scope is always `lp`. Grant types: Authorization Code, **Resource Owner Password (recommended for lenders)**, Client Credentials (**ISV partners only** — a lender like YS Capital authenticates with an API *user* + password grant), and User Impersonation.
- Community pain: syncs that "die after hours of success" from expired tokens ([MortgageWorkspace](https://mortgageworkspace.com/blog/exploring-api-integrations-for-encompass-whats-possible-in-2025)); token sharing/recycling across serverless workers ([EncompassRest #302](https://github.com/EncompassRest/EncompassRest/issues/302)); whether/when to revoke on shutdown ([#77](https://github.com/EncompassRest/EncompassRest/issues/77), [#275](https://github.com/EncompassRest/EncompassRest/issues/275)); intermittent auth failures ([#360](https://github.com/EncompassRest/EncompassRest/issues/360)).
- Regenerating the client secret **immediately invalidates** old credentials across all integrations (Authorization docs) — coordinate rotation.

### 4.2 Rate limiting / concurrency
- Encompass enforces a **concurrency** limit, not a classic request-per-second limit: default **30 concurrent in-flight calls per lender environment**, shared by *all* integrations (lender + partners). Exceeding it returns **429** until in-flight count drops. Monitor response headers **`X-Concurrency-Limit-Limit`** and **`X-Concurrency-Limit-Remaining`**; ICE recommends exponential backoff and can raise limits on request ([Concurrency Limits](https://developer.icemortgagetechnology.com/developer-connect/docs/concurrency-limits)).
- Additional volume-based rate limits exist but are **not publicly documented** — confirm your tier with ICE support ([MortgageWorkspace](https://mortgageworkspace.com/blog/exploring-api-integrations-for-encompass-whats-possible-in-2025)).
- Community evidence: concurrency-limit exceptions during `GetLoan` bursts ([EncompassRest #338](https://github.com/EncompassRest/EncompassRest/issues/338)); gateway timeouts frequent enough that the library added opt-in **auto-retry on timeouts** after a 17-comment thread ([#117](https://github.com/EncompassRest/EncompassRest/issues/117), [#120](https://github.com/EncompassRest/EncompassRest/issues/120), [#303](https://github.com/EncompassRest/EncompassRest/issues/303)); a lone SO answer about RestSharp gateway timeouts where Postman worked (client timeout config).

### 4.3 V1/V3 inconsistencies
- Per ICE's own blog ([V1 vs V3](https://developer.icemortgagetechnology.com/developer-connect/docs/v1-vs-v3-encompass-apis-whats-the-difference-1)): V1 = "RESTful wrappers on legacy SOAP/WCF"; V3 = true REST on a new codebase. **Different data contracts** — V1 mixes fixed-size and variable-size collections in one contract and uses inconsistent (sometimes position-based) IDs; V3 separates them and guarantees stable system-generated IDs. **No full parity**: some capabilities are still V1-only; all new APIs ship V3-only. Both "stable and supported," but V1 "will eventually depreciate" (no date).
- V1 pieces are already being sunset one by one: V1 eFolder Attachment APIs (26.3), V1 Loan Importer (26.1), `createAuthCode` (27.1), Task Group APIs (24.3) — [Deprecation Notices](https://developer.icemortgagetechnology.com/developer-connect/docs/deprecation-notices). There is a standing [Breaking Change Notices](https://developer.icemortgagetechnology.com/developer-connect/docs/breaking-change-notices) page — subscribe to release notes; "minor" releases have broken integrations (e.g., eFolder `status` → `documentStatus` attribute rename, 26.1).
- Practical effect: an integration almost always ends up **mixing V1 and V3** endpoints (e.g., pipeline exists in both, conditions are V1 `loan conditions` vs V3 `enhanced conditions`), with different pagination, casing, and error shapes.

### 4.4 Field ID confusion
Encompass has **three overlapping addressing schemes**: classic field IDs (`4002`, `CX.MYFIELD`, `FR0108#1`), pipeline **canonical names** (`Fields.4002`, `Loan.LastModified`), and the V3 **loan JSON schema paths**. Community evidence of confusion:
- Field exists on the URLA form but can't be found by ID ([EncompassRest #376](https://github.com/EncompassRest/EncompassRest/issues/376) — FDE0151).
- **Borrower-pair indexing**: FieldReader always reads the *first* application; per-pair scoping "is not possible through the field reader API" ([#390](https://github.com/EncompassRest/EncompassRest/issues/390); [EncompassRest wiki](https://github.com/EncompassRest/EncompassRest/wiki/Retrieving-Loan-Data)); `#1`/`#2` suffix writes appended duplicate records instead of updating ([#286](https://github.com/EncompassRest/EncompassRest/issues/286)).
- Locked-by-default fields (e.g., `ULDD.X1`) and virtual fields (`LOCKRATE.2286`) behave differently than in the SDK ([#427](https://github.com/EncompassRest/EncompassRest/issues/427), [#372](https://github.com/EncompassRest/EncompassRest/issues/372)).
- The **V3 fieldReader** (`POST /encompass/v3/loans/{loanId}/fieldReader`, with `invalidFieldBehavior=Include`) is the community-preferred way to read a specific list of classic field IDs without pulling the whole loan — it appears three times in the local Postman collection (index: `/tmp/.../scratchpad/research/postman-endpoint-index.md`, "V3 Manage Loan / 05 - Field Reader"). This is the natural primitive for yscap's "portal condition cannot clear unless Encompass field agrees" rule.
- Schema discovery: [V3 Get Field Schema](https://developer.icemortgagetechnology.com/developer-connect/reference/get-field-schema-1) + [Loan Data Dictionary Guide](https://developer.icemortgagetechnology.com/developer-connect/docs/encompass-loan-data-dictionary-guide); pipeline canonical names via `GET /encompass/v1/loanPipeline/fieldDefinitions` or `GET /encompass/v3/loanPipeline/canonicalFields` (both in the Postman index).

### 4.5 Pipeline query quirks
- Default page **1,000 loans**, expandable to **25,000**; results "may not include all loans if dataset is too large" — a real silent-truncation report exists ([EncompassRest #189](https://github.com/EncompassRest/EncompassRest/issues/189); [wiki](https://github.com/EncompassRest/EncompassRest/wiki/Retrieving-Loan-Data); [Loan Pipeline reference](https://developer.icemortgagetechnology.com/developer-connect/reference/loan-pipeline)).
- **Cursors**: max **10 active cursors**; cursor expires after **5 minutes idle**, absolute max **1 hour** ([wiki](https://github.com/EncompassRest/EncompassRest/wiki/Retrieving-Loan-Data); [Loan Pipeline docs](https://developer.icemortgagetechnology.com/developer-connect/reference/loan-pipeline)). Design nightly full-sweeps to page briskly, never lazily.
- Filter oddities: `Empty`/`NonEmpty` match types are effectively date-filter constructs with a required-but-ignored value ([EncompassRest #83](https://github.com/EncompassRest/EncompassRest/issues/83), a 42-comment thread); querying a field not in the reporting database errors unless `ignoreInvalidFields` is passed ([#191](https://github.com/EncompassRest/EncompassRest/issues/191)); multiple simultaneous filters on saved pipeline views are limited ([#408](https://github.com/EncompassRest/EncompassRest/issues/408)).
- **Only fields in the Encompass reporting database are queryable/returnable via pipeline** — custom fields must be added to the reporting DB by the Encompass admin first (SO: [query loans by custom field](https://stackoverflow.com/questions/53669491/how-to-query-loans-with-custom-field-in-encompass); SDK-era: [reporting database gap](https://stackoverflow.com/questions/43456273/get-field-data-outside-of-reporting-database-using-encompass360-sdk)).
- `GetCanonicalNames` responses can blow the payload cap ([#217](https://github.com/EncompassRest/EncompassRest/issues/217)).

### 4.6 Webhook reliability
- Payloads are **thin references** (event ID, timestamp, event type, resource GUID + API URL), not full loan data — you must call back into the API ([Webhook overview](https://developer.icemortgagetechnology.com/developer-connect/reference/webhook)).
- Delivery: retries **up to 3 times, 20 seconds apart**; a **30-second callback timeout** counts as failure ([EPC webhook docs](https://docs.partnerconnect.elliemae.com/partnerconnect/docs/webhooks)). After that the event is only recoverable via the **Get All Events API** — the documented reconciliation path.
- ICE **auto-deletes "bad" subscriptions** (>30 days old with undeliverable endpoints / >1,000 events/week failing / 5XXs & timeouts) — your integration can silently go deaf ([Webhook Subscriptions docs](https://developer.icemortgagetechnology.com/developer-connect/docs/webhook-subscriptions)).
- Signature: HMAC-SHA256 of the raw body with your subscription **signing key**, Base64, compared to the **`Elli-Signature`** header (with `Elli-Environment` alongside); key complexity regex is documented; a lost key is retrievable via Get Subscription ([Signing Keys](https://developer.icemortgagetechnology.com/developer-connect/reference/signing-keys)). Custom key registration had bugs historically ([EncompassRest #363](https://github.com/EncompassRest/EncompassRest/issues/363)).
- No ordering guarantee is documented; consumers treat events as "something changed, go fetch."

### 4.7 Big-loan JSON payload sizes
- Hard platform numbers: **6 MB API-gateway response cap** ([Response Payload Size Limit](https://developer.icemortgagetechnology.com/developer-connect/docs/response-payload-size-limit)) and a **40 MB loan file size limit** ([Loan Size Limit](https://developer.icemortgagetechnology.com/developer-connect/docs/loan-size-limit); [25.1 R2T error-change notice](https://developer.icemortgagetechnology.com/developer-connect/docs/ka-large-loan-sizes-251-r2t)). A full `GET /loans/{id}?view=entity` on a heavy loan can exceed 6 MB and fail with `400 "Response exceeds maximum size (6 MB)"`.
- Mitigations per docs/community: request **`entities=`** subsets (Postman collection uses e.g. `view=full&entities=disclosureTracking2015Logs,documents`), use fieldReader for scalar values, paginate collection endpoints, always send **`Accept-Encoding: gzip`**. ICE itself had to stop returning two log collections in `view=entity` because of payload bloat (22.3 release note).

### 4.8 Custom field handling
- Custom fields (`CX.*`) are the lifeblood of hard-money shops (yscap's own ClickUp maps a free-label "Encompass" dropdown today — `src/clickup/mapper.js:122`, `docs/CLICKUP-DATA-MAPPING.md:183`). Community-reported quirks: string custom fields holding dates get a server-side `dateValue` added and are **nulled out on subsequent writes**; date/integer/decimal customs nulled; timestamps mutated ([EncompassRest #401](https://github.com/EncompassRest/EncompassRest/issues/401)) — read-only Phase 1 dodges the write bugs but must expect `dateValue`/`stringValue` duality when *reading*.
- Listing all custom field definitions required V1 settings endpoints and caching ([#212](https://github.com/EncompassRest/EncompassRest/issues/212), [#164](https://github.com/EncompassRest/EncompassRest/issues/164)); Business Contacts custom-field API parity arrived only ~25.1 (SDK migration guide).
- To make a custom field pipeline-queryable it must be in the reporting database (§4.5).

### 4.9 Test environment availability
- There is **no generic public Encompass sandbox**. Test capacity is a per-lender provisioned instance: **`TEBE…` prefix = test, `BE…` = production** ([Jungo setup guide](https://ijungo.com/encompass-api-admin-instructions/)). **YS Capital's stated instance `BE11397907` is a production instance** — Phase 1 development would hit live loan data unless ICE provisions a TEBE test instance (often an extra cost/contract item).
- ICE "recommends developing and testing your new APIs in your non-production environment prior to utilizing them in production" (ibid.), and the Postman collection assumes environment variables per instance.
- Repo grounding: the portal already anticipates this integration — a deferred `sync_queue` worker for "ClickUp/Encompass" (`src/sync/queue.js:4`, `README.md:100`, `CLAUDE.md:45`, `package.json:5`) and a planned integrations audit chapter (`docs/portal-audit/00-MASTER-PLAN.md:108`).

## 5. SDK sunset → what it means for API stability

Timeline (well covered by trade press):
- **Original plan:** SDK and legacy integrations sunset **Oct 31, 2025** ([National Mortgage News](https://www.nationalmortgagenews.com/news/ice-mortgage-technology-to-sunset-encompass-sdk)).
- **Revised after customer pushback:** deprecated (feature-frozen) **Nov 1, 2025**, still maintained, with the hard transition deadline moved to **Dec 31, 2026**; post-deadline SDK access requires special approval **and monthly fees** ([HousingWire](https://www.housingwire.com/articles/ice-delays-encompass-sdk-transition-until-end-of-2026/), [Scotsman Guide](https://www.scotsmanguide.com/news/ice-mortgage-technology-adds-grace-period-after-encompass-sdk-sunset-date/), [ICE transition page](https://mortgagetech.ice.com/explore/encompass-sdk-and-legacy-integrations-transition)).
- Replacements: **Encompass Partner Connect (EPC)** for vendors, **Developer Connect (EDC) REST APIs** for lenders, or native Encompass features ([migration guide](https://developer.icemortgagetechnology.com/developer-connect/docs/sdk-to-api-migration-getting-started-guide)).

Implications for yscap:
1. **The REST APIs are the strategic, invested platform** — the entire SDK install base is being forced onto them by end-2026, so expect continued investment, but also **crowding**: every vendor's migrated integration shares your instance's 30-call concurrency budget.
2. **V3 is the future, V1 is next in line for retirement** — ICE has already begun sunsetting V1 endpoints piecemeal (§4.3). Prefer V3 (`/encompass/v3/loans`, v3 fieldReader, v3 loanPipeline) wherever parity exists; isolate unavoidable V1 calls behind an adapter.
3. **Feature-parity gaps are real** (ICE concedes "no V3 parity for every V1 API"), and the migration wave keeps surfacing them; the deadline slipped once precisely because customers weren't ready.
4. Even ICE's official client bindings were sunset (Jan 2022) — **don't couple to any client library**; couple to the HTTP contract and the Postman collection (800 requests, local copy indexed at `/tmp/.../scratchpad/research/postman-endpoint-index.md`).

**Adjacent read-only enrichment option worth evaluating:** [Encompass Data Connect](https://mortgagetech.ice.com/products/encompass-data-connect) — ICE's managed near-real-time replication of *all* loan fields to a hosted Snowflake instance or direct-write to your warehouse, "without impacting Encompass performance" and without field limits. For heavy borrower-DB enrichment it sidesteps concurrency limits, 6 MB caps, and pipeline quirks entirely (licensed separately); the API integration then handles only the low-latency per-loan checks (conditions/CTC gating). See also [Loan Data Extracts](https://developer.icemortgagetechnology.com/developer-connect/docs/ucm-loan-data-extracts).

## 6. Top 10 practitioner lessons (each with a one-line mitigation)

1. **Tokens die fast and quietly** (30-min life, 15-min idle kill; expired tokens are the #1 cause of syncs that die mid-day). → Centralize one token-manager that caches, refreshes ~5 min early, and retries exactly once on 401.
2. **The limit is concurrency (default 30 in-flight per environment, shared with every other vendor), and 429s hit everyone at once.** → Run all Encompass calls through one queue with a configurable concurrency ceiling (start ≤5), watch `X-Concurrency-Limit-Remaining`, back off exponentially with jitter on 429.
3. **V1 and V3 are different products with different contracts, and V1 is being retired piecemeal.** → Default to V3; wrap any V1 call in an adapter module and subscribe to the Deprecation/Breaking-Change notice pages in your release checklist.
4. **Field addressing is a trap: classic field IDs, canonical pipeline names, and V3 JSON paths don't line up, and borrower-pair (`#2`) semantics are buggy.** → Build one field-mapping table (portal concept → V3 fieldReader ID → canonical name) as config, validated at startup against the field-schema endpoint.
5. **Pipeline queries silently truncate and cursors expire (1,000 default/25,000 max rows; 10 cursors; 5-min idle expiry).** → Page with cursors promptly, order by immutable keys, cross-check counts, and pass `ignoreInvalidFields` deliberately never habitually.
6. **Webhooks are hints, not truth: thin payloads, 3×20s retries, and ICE auto-deletes failing subscriptions.** → Ack in <30s before processing, verify `Elli-Signature` HMAC, and run a scheduled Events-API + pipeline sweep reconciler that assumes events were missed.
7. **Big loans break responses: 6 MB gateway cap on a loan file that can be 40 MB.** → Never `GET ?view=entity` in production paths — request `entities=` subsets or fieldReader values, and always send `Accept-Encoding: gzip`.
8. **Custom fields lie subtly** (`dateValue`/`stringValue` duality, values not in the reporting DB are invisible to pipeline queries). → Enumerate custom-field definitions once per day, get needed `CX.*` fields added to the reporting database by the Encompass admin, and validate types on ingest.
9. **There is no public sandbox — `BE11397907` is production** (test = `TEBE…`). → Get a TEBE test instance provisioned before writing code; in the meantime enforce read-only *architecturally*: an HTTP client that only permits GET plus an allowlisted set of read-style POSTs (fieldReader, loanPipeline), a token for an API user whose persona has no write rights, and no PATCH/PUT/DELETE code paths at all.
10. **The community is thin and shrinking (SO ~7 questions; ICE forum deleted; flagship libraries unmaintained since 2021).** → Budget for ICE support tickets and in-house docs; mine EncompassRest's issue tracker as the de-facto FAQ before filing.
