# A3 — Manual Review Queues: How the Portal Routes Risky Sync Changes to Humans

Research agent A3, 2026-07-17. Repo root: `/home/user/yscap/yscap-repo-root_8` (all paths below relative to it).

## 1. Executive summary

The portal (PILOT) already contains a mature, battle-tested **human review gate** for its bidirectional ClickUp sync, built after a 2026-07-15 DOB-corruption incident. Its governing philosophy (stated in code comments and `docs/SYNC-ACTION-PLAN-2026-07.md`):

- **Auto-resolve only what is PROVABLE; genuine ambiguity waits for a person** (`src/lib/sync-review.js:2-21`, `src/lib/sync-autoresolve.js:4-29`).
- **Nothing suspicious is ever silently applied or silently dropped** — it becomes a queue row (`db/108_sync_review_queue.sql:4-15`).
- **Every stuck state is a review row WITH OPTIONS** — whole files, not just fields (`src/lib/sync-file-review.js:3-21`).
- **"Nothing is silent" is a standing guarantee** — LO notification on row creation, 3-day reminder, 7-day admin escalation, weekly digest (`src/lib/sync-review.js:184-235`).
- **The review queue should contain only decisions** (north-star invariant 4, `docs/SYNC-ACTION-PLAN-2026-07.md:14`).

This is directly reusable as the template for an Encompass read-only manual review gate — with one crucial inversion: the ClickUp resolver writes the winner to **both** systems, while an Encompass gate must be architecturally incapable of writing back (see §9).

## 3. What enters review vs. what auto-applies

### 3.1 Auto-resolution engine (`src/lib/sync-autoresolve.js`)

`decideDob()` (`sync-autoresolve.js:52-96`) is the ONE decision function consulted by every DOB-conflict site (inbound heal, outbound push gate, restore script), so behavior is identical regardless of which side surfaces a disagreement. Outcomes:

| Outcome | Rule | Example (each maps to a real incident) |
|---|---|---|
| `agree` | Same day in different storage forms (2-digit-year artifact `0095-10-19` pivots to `1995-10-19`) → canonicalize silently | test `scripts/test-sync-autoresolve.js:25-26` |
| `adopt` | One side blank → the other fills (`portal_blank`/`clickup_blank`) | `:57-62` |
| `adopt` | One side impossible (toddler/future/over-120), other a plausible adult DOB → plausible wins (`*_implausible`) | the "12/11/2022 toddler vs 12/11/2002" class |
| `adopt` | ClickUp typed-2-digit-year artifact + portal value has NO human provenance (`borrowers.origin='clickup_backfill'`) → human typing beats sync-derived copy | `:68-70`, "Shaindel Schwimmer" class |
| `adopt` | Backdating rule: portal profile sync-created AND **proven** never-human-edited (`portalHumanEdited === false`; unknown/null never qualifies) → ClickUp current wins | `:81-83` |
| `review` | Two plausible adult DOBs that differ with human provenance → human decision (`kind:'differs'`) | `:84` |
| `review` | Neither side plausible — with vetted pivot proposal when one exists; `kind:'same_impossible'` when both carry the SAME impossible value ("this DOB can't be right", not "they differ") | `:91-95` |

`adoptDobEverywhere()` (`:104-158`) applies a provable winner to the portal profile **and every linked ClickUp task**: idempotent, per-task best-effort, journaled (`clickup_write_log`), audited (`sync_dob_auto_resolve` with before-image), counted into the ONE outbound circuit breaker (`:132`), gated by the outbound switch, and it **closes stale open reviews** for the borrower's DOB (`:152-156`) — a fix at the source clears the queue with no clicks.

### 3.3 Queue hygiene inside `queueReview()` (`src/lib/sync-review.js:42-88`)

- `suppressIfRejected` (`:49-55`): file-level rows are re-produced every sync pass while a file stays stuck — a reviewer's DISMISS must stick, or the row respawns 5 minutes later. Field-value rows don't use it (a re-blocked write is a fresh event).
- **DOB rows dedupe per BORROWER**, not per task (`:60-67`) — a borrower with three linked tasks used to queue three identical rows.
- Best-effort: a queue insert failure never breaks a sync pass (`:87`).
- `closeStaleReviews()` (`:167-182`): whenever the sync observes the two systems agreeing (or auto-adopts a canonical value), open rows for that field auto-close as `status='resolved', auto_resolved=true` with an explanatory note — kept as history, never deleted. Called from ~15 sites (ingest DOB agreement, cleared-DOB vacate, identity audit convergence, file materialization, loan-number self-heal, SharePoint recovery, `adoptDobEverywhere`).

## 4. The two-sided review model

Design (owner-directed 2026-07-15 evening, `db/110` header + `src/lib/sync-autoresolve.js:160-164`):

1. Every row records **what ClickUp holds AND what PILOT holds** (`clickup_value`/`portal_value`), derived from direction if the caller didn't pass them explicitly (`sync-review.js:68-74`). Stored values are **display-only**.
2. Resolution picks a **WINNER** — `clickup` | `portal` | `custom` (a typed value, db/112).
3. The winning value is **re-read LIVE at resolve time** from the winning system (never trusted from the row) and applied to **BOTH systems** through audited, journaled writes. SSNs are never stored in the queue — masked display (`✱✱✱-✱✱-1234`) only.

`applyReviewWinner(row, winner, customValue)` (`sync-autoresolve.js:192-398`) is the single applier, per field kind:

| Field kind | Applier behavior |
|---|---|
| `date_of_birth` (`:211-230`) | Borrower-level: re-read winner live, `sanitizeDob` (adult plausibility 18–120), then `adoptDobEverywhere` → portal + every linked task |
| `expected_closing`/`actual_closing`/`acquisition_date` (`:233-266`) | App-level date: sanitize, write `applications` column, push epoch to ClickUp (breaker-counted, journaled) |
| `ssn` (`:269-309`) | Live re-read/decrypt; custom requires full 9 digits; portal write = encrypt + last4 + match-hash; ClickUp write journaled masked |
| `status` (`:312-321`) | winner=clickup → `ingestOne` (normal inbound path); winner=portal → scoped `pushApplication({only:['status'], approvedReview:true, force:true})` |
| identity fields `email`/`cell_phone`/`first_name`/`current_address` (`:329-395`) | winner=portal → scoped re-push with review bypass; winner=clickup/custom → validated portal write via one shared `applyIdentityValue` (email format + uniqueness 409, phone ≥10 digits, placeholder-name rejection, address→jsonb), then custom pushes out so both systems carry it |
| anything else | 422 "resolving '<field>' two-sided is not supported yet — use approve/reject" (`:397`) |

Error discipline: `httpError()` sets `expose=true` on the module's own validation errors (`:176`); ClickUp client errors carry `.status` but are NEVER relayed (an upstream 401 would read as session-expiry and log staff out) — the route maps them to 502 (`src/routes/staff.js:6070-6074`).

There is also a legacy **one-sided approve** (`staff.js:6007-6042`): inbound → write the proposed date into the portal column with before-image audit; outbound → re-push the blocked field with `approvedReview:true` (the deliberate human action the DOB guard demands).

## 6. Notifications, aging, escalation, digest

- **On row creation** (`sync-review.js:86, 97-156`): the file's **loan officer** is notified in-app + branded email with a deep link to `/internal/sync-reviews` — "reviews are theirs to resolve, not an admin-only backwater." HARD SCOPE GUARD (`:107-117`, after an owner-reported leak): a FILE-scoped row notifies ONLY that file's assigned LO — never the borrower-wide fan-out. Borrower-level rows (DOB) notify every LO across the borrower's active files, deduped (`:118-127`). Unassigned file → nobody emailed, but the admin queue view, sidebar badge, and 7-day escalation still surface it (`:128`). `notified_at` prevents double-send.
- **File-level rows get action-oriented copy** — "the review screen offers ACTIONS, not sides" (`:134-142`); two-sided rows get compare-and-choose copy.
- **Aging** (`remindStaleReviewsOnce`, `:191-235`; scheduled on boot +60s and daily in `src/sync/clickup-sync.js:1261-1270`): open >3 days → one LO re-notify (`reminded_at`, ≤50/run); open >7 days → one escalation to every active admin with a bulleted list (`escalated_at`, ≤25/run). Bounded, best-effort, never breaks the sync.
- **Weekly digest** (`sendReviewDigestOnce`, `:243-282`): 7-day stats (opened / auto-closed / human-resolved / dismissed / open-now / open>14d) + top reasons, emailed to admins; self-gated to ≤1 per 6 days via an `audit_log` stamp — proof the system is worked + early flood warning.

## 9. Recommendation: the manual review gate for Encompass-sourced data

Phase 1 Encompass is STRICTLY READ-ONLY (pull-only from Developer Connect, instance BE11397907). The existing machinery maps almost 1:1, with one deliberate inversion and several reuse decisions:

### 9.1 The architectural inversion: one-sided application
The ClickUp resolver's defining feature — "the winner is applied to BOTH systems" — is exactly what an Encompass gate must NOT do. Recommended contract:

- **`winner:'encompass'`** → re-read the loan LIVE from the Encompass snapshot/staging table (never trust the row's stored display value — same live-re-read discipline as `applyReviewWinner`), sanitize through the same field validators (`sanitizeDob`, `normalizeTypedDate`, email/phone/SSN chokepoints), write the **portal** record, audit with before-image, journal.
- **`winner:'portal'`** → **no write anywhere.** The resolution note records "portal value kept; Encompass disagrees — fix it in Encompass if Encompass is wrong." Pair with `suppressIfRejected`-style stickiness so the same (loan, field, Encompass-value) tuple never respawns; a CHANGED Encompass value is a fresh event (exactly the `db/108` unique-index semantics: proposal is part of the dedup key).
- **`winner:'custom'`** → typed value applies to the portal only, flagged in the note that Encompass still differs (the row's auto-close will fire when a later pull shows Encompass matching — the `closeStaleReviews` pattern).
- There is **no Encompass applier module at all** — the write-incapability should be structural (no client method that can PATCH/POST to Encompass; mirror the `sharepoint.remove()` throwing-no-op precedent and the `guardTaskUpdatePayload` allowlist philosophy from `src/clickup/client.js`).

### 9.2 What auto-applies vs. what stops for review (Encompass edition)
Follow the "provable vs. ambiguous" split:

**Auto-apply (enrichment, no card):**
- Filling a blank portal field from Encompass (the COALESCE fill-only pattern — a blank can never clear the portal, and a fill is not an overwrite).
- Format canonicalization that provably represents the same fact (date storage forms, phone formatting, address canonicalization via the existing `address-canon` machinery).
- Pure Encompass-namespace data that has no portal counterpart (milestones, lock data landed in dedicated `encompass_*` columns/tables) — new facts, not conflicts.
- Additive contacts (extra emails/phones) → `borrower_contacts`, exactly as today (`clickup-sync.js:555-566`).

**Review (a card, decided by a human before it touches portal records):**
- **Conflicting borrower identity**: Encompass borrower name/DOB/SSN/email/phone/address differs from a non-blank portal value → `encompass_identity_differs` rows, two-sided display (`encompass_value`/`portal_value`), SSNs masked-never-stored. Run DOB through `decideDob`-style logic first (implausible values, artifacts) so only genuine ambiguity cards.
- **Mismatched loan fields that gate decisions**: loan amount, rate, closing dates, loan number, program, property address differing from portal → review before overwriting. Note the loan-number lesson (`copied_loan_number_needs_assignment`): any globally-unique matcher key can arrive as a stale copy — never treat an inherited Encompass loan number as an identity claim.
- **Entity-resolution ambiguity**: an Encompass loan (GUID) that matches >1 portal application, or matches a different borrower's file → `encompass_loan_not_matched_ambiguous` with candidate enrichment in `raw_value` and file-level actions `link_existing` / `create_file` / dismiss — a direct port of `file_not_materialized_ambiguous`.
- **"One profile, two people"** signatures (name+SSN disagreement on a profile with another relationship) → reuse `borrower_identity_conflict`/`split_borrower` as-is; Encompass is just a second evidence source.
- **Encompass loan disappeared/archived** while the portal file is active → the `task_deleted_needs_decision` analog.
- **Pull-pipeline dead letters**: a loan whose pull fails terminally → a review row (never a silent skip) — invariant 1.

**Gates, not reviews** (the layered rules in the project brief): "a portal condition cannot clear unless Encompass agrees" and "no clear-to-close unless Encompass shows CTC" are **blocking gates** in the sign-off path (like `signOffGate`) — a 422 at clear-time, not a queue row. But a *standing disagreement* (portal condition marked satisfied while Encompass reopened it) should ADDITIONALLY queue a Tier-A review card so it isn't discovered only at CTC time.

### 9.3 Reuse decisions
- **Same queue or sibling?** Recommend the **same `sync_review_queue`** with a `source` column (`'clickup'`|`'encompass'`, default clickup) added by a new idempotent migration — the notify/aging/digest/badge/bulk/scoping machinery, the LO-ownership model, and the UI shell all come for free; only per-reason copy and appliers are new. The established extension checklist already exists (CLAUDE.md rule 5d / 3c): new reason = producer + `REASON_ACTIONS` entry + applier + `REASON_COPY`/buttons in `SyncReviews.jsx` + drift-check test coverage.
- **Provenance from day one** (the WO-11 lesson): stamp `origin='encompass_backfill'` on records the integration creates and track per-field last-writer, so a future `decideField()` can auto-adopt Encompass values over sync-derived portal copies while NEVER overriding a human-entered portal value without a card — the exact rule that made `decideDob` safe.
- **Stage, then diff**: land every pull in an immutable `encompass_loan_snapshots` staging table first; the diff engine (not the fetcher) decides auto-apply vs. review. This gives the queue its live-re-read source at resolve time and a forensic `raw_value`.
- **Tier from day one** (WO-7): identity conflicts, loan-field mismatches, and CTC disagreements are Tier A (email the LO immediately); informational drift is Tier B (badge + digest) — otherwise the initial Encompass backfill floods LOs exactly the way the boot heal passes threatened to (the bulk endpoint and per-boot LIMIT/sweep bounds exist for that reason; copy the ≤100-rows-per-boot + starvation-aware NOT EXISTS pattern from `flagUnsyncableFilesOnce`, `clickup-sync.js:169-200`).
- **Auto-close on convergence**: every producer must pair with a `closeStaleReviews` call on the next agreeing pull — "once it's fixed in Encompass, the review goes away with no clicks."

### 9.4 Open questions
1. Should Encompass rows share `sync_review_queue` (recommended) or get a parallel table? Sharing risks slug collisions and mixed-queue volume; the `source` column + tiering mitigates.
2. Expected initial-backfill conflict volume against ~hundreds of loans — needs a dry-run diff report (the `audit-evidence-report.js` pattern) before the queue goes live, or the first deploy floods LOs.
3. Which loan fields does Encompass own outright (authority `encompass` in a WO-10-style registry → portal value auto-yields with journal but no card) vs. two-sided-review fields? Needs an owner decision per field.
4. The CTC/condition gate needs a defined staleness policy: how old may the last successful Encompass pull be before the gate refuses to answer (fail closed) rather than trusting a stale snapshot?
