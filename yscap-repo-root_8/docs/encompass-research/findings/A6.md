## 1. Context: what happened, in one paragraph

The portal↔ClickUp bidirectional sync was **built in ~8 hours overnight (Jul 6→7, 2026)** and outbound writes were enabled with the design's own safeguards explicitly waived — the go-live commit message read *"The ONLY protection is no deletion"* (`docs/AUDIT-2026-07-15-SYNC-ANNUAL.md:28`). **The first data-corruption incident hit 32 minutes after go-live** (`AUDIT:29`). Over the following 9 days: **16 forensically reconstructed incidents**, ~45 reactively added guard layers (~35 in one 40-hour window), 19 review-card types, 13 deploys in a single day, and 491 commits (`AUDIT:14, 22, 63`). 5 incidents needed 2–3 fix attempts. Two "incidents" were false alarms that each cost a forensic day, both traceable to a **shared API token** making the portal's writes indistinguishable from anyone else's (`AUDIT:63`). The final program conclusion (2026-07-17): keep the monolith, harden at four chokepoints, move policy into data, move volatile safety state into Postgres, and adopt the invariant *"every failure ends as a retry or a review row — never silence"* (`docs/SYNC-FINAL-CONCLUSIONS-2026-07.md:15-63, 129-130`).

### 2.1 The date incident in detail (the flagship, `docs/CLICKUP-DATE-INCIDENT.md`)

Three legs, one class — *a pure calendar date represented as an instant-in-time (JS `Date` / epoch ms)* (`:12-15`):

1. **Outbound convention (`:17-35`):** ClickUp stores dates as epoch ms and renders in each viewer's timezone; its UI pins no-time dates at 4 AM in the setter's timezone. The portal pushed **UTC midnight**, which is the previous evening in New York — every date the portal ever pushed displayed one day early. Worse (owner's activity-history audit, `:122-131`): **ClickUp normalized the UTC-midnight epoch to the previous NY calendar day and stored it at 4 AM like a native entry** — the damage was physical, not display-only, and the normalization **erased the fingerprint**, so a fingerprint scan undercounts; ClickUp's own activity history (not exposed via API) was the authoritative damage source.
2. **pg parse leg (`:37-46`):** node-postgres parsed `date` columns into JS `Date`s at server-local midnight; `toISOString()` shifted the day on any non-UTC host. Fixed at one chokepoint: `types.setTypeParser(1082, v => v)` — dates are `'YYYY-MM-DD'` strings everywhere. Browser leg (`new Date('YYYY-MM-DD')` → `toLocaleDateString()` shows previous day in EST) fixed via `app-v2/src/lib/dates.js`.
3. **Mid-typing saves (`:48-57`):** save-on-every-`onChange` persisted year-0026 epochs (found live: `-61329625438000`). Fixed at **three layers**: draft-commit UI, server-side strict `YYYY-MM-DD` + 1900–2100 validation, and the push chokepoint returning null for out-of-range years.

Explicitly ruled out (`:59-67`): the portal DB was NOT mass-corrupted (inbound DOB was fill-only via `COALESCE`), which made the portal DB the trusted restore source.

## 3. Systemic defects the annual audit confirmed (beyond individual incidents)

- **F-C1 (CRITICAL): failed field writes swallowed, job marked done** — a ClickUp 429/500 on the write = the user's edit silently dropped, no retry, no review row; the next inbound pull can revert the portal to the stale value (`AUDIT:93`, `src/clickup/orchestrator.js:346-349`). Reads failed closed; **writes failed silent**.
- **F-H1: zero 429/Retry-After handling** anywhere in the ClickUp client — while every deploy fired unbounded boot sweeps straight into ClickUp's 100 req/min/token limit, manufacturing the failures F-C1 then swallowed (`AUDIT:97, 183`). The repo's own `sharepoint.js:105-125` already implemented the correct retry pattern — it was simply never ported.
- **F-H4: every deploy re-ingested the entire linked portfolio** — in-memory watermark reset to a 24h lookback on boot, no LIMIT, no pacing; ×13 deploys/day = a daily read/write storm and the direct multiplier of the review-queue email flood (`AUDIT:103, 138`).
- **Review-queue divergence:** 0→19 card types in ~40 hours; dismissals didn't stick for the noisiest types; no notification rate cap; two-thirds of card volume informational yet emailed identically to work-blocking cards; some card types could **never auto-close** (`AUDIT:134-143`). One card type (`identity_mismatch_audit`) was ~90-95% noise (`SYNC-FIELD-FORENSICS:190`).
- **Three foundational blueprint items never built:** dedicated bot identity, rate-limit handling, per-field authority as data (`clickup_field_mappings` table existed in the DB, read by nothing) (`AUDIT:14, 163`).
- **SharePoint mirror never ran in production** — shipped Jul 13, env vars never set; the team may have believed borrower documents were backed up when the Render disk was the only copy (`AUDIT:71-82`). Separately its fuzzy matcher would auto-file on a single name-only match — two "David Cohen"s under one officer would cross-file SSN documents (F-H2, `AUDIT:99`) — dodged only because it never executed.
- **Partial-value comparisons everywhere:** SSN mismatch checked last-4 only; phones last-4; names first-word; addresses house-number + two street tokens — real conflicts structurally invisible (`SYNC-FIELD-FORENSICS:68-72, 92, 156`).
- **Latent-value coercions:** money parser turned "N/A"/"TBD" into a real $0; the wipe-guard guarded *empty* but not *zero* (`SYNC-FIELD-FORENSICS:175-176`).
- **Blueprint drift:** the blueprint still says "PROPOSAL — nothing implemented" while describing deleted machinery — "materially misleading" (`AUDIT:165`; `docs/CLICKUP-BIDIRECTIONAL-SYNC-BLUEPRINT.md:4`). The locked owner decision to run on the owner's personal token with **no bot seat** (`BLUEPRINT:24, 90-95`) made echo-suppression "mandatory, not optional" — and that mandatory layer was then never wired.
- **Test coverage vs incidents:** 4 of 16 incidents covered, 6 partial, 6 uncovered; `ingest.js` (1,545 lines carrying fixes for 9 of 16 incidents) had **zero** direct test coverage; no `npm test`, no CI (`AUDIT:169-173`).
- **Firefighting metric:** 88% of all commits touched a file changed in the previous 24 hours; **100% of ClickUp error-fixes did** (`SYNC-FIELD-FORENSICS:230`). Migration-number collisions ×3 (033, 088, 113) with no `schema_migrations` ledger (`AUDIT:37, 127`).

## 4. What recovery actually required

1. **A purpose-built forensic/restore tool** (`scripts/clickup-date-restore.js`, 302 lines) with a strict discipline: dry-run by default; a **classification taxonomy** for every live epoch (`native-4am` / `portal-utc-midnight` / `garbage-year` / `other-offset` — `:72-82`, exploiting the fact that a human cannot produce a 00:00Z epoch through ClickUp's UI); cross-checks against the portal DB + `audit_log` + `sync_queue` history; CSV evidence output; with `--apply`, **every write verified by re-reading the task** and journaled to `clickup_write_log` + `audit_log` (`:232-247`); 700ms pacing between tasks (`:257`); never clears a field, never touches what it can't classify; undecidable values **flagged for human review, not guessed** (`CLICKUP-DATE-INCIDENT.md:70-79`).
2. **The repair tool itself had to be repaired twice.** (a) Its garbage-year branch rewrote a DOB from a wrong portal value (Shaindel Schwimmer) → all DOB decisions were centralized into ONE decision function, `decideDob()` in `src/lib/sync-autoresolve.js`, that auto-settles only provable verdicts and queues everything else (`:144-168`). (b) Its wipe-don't-guess branch wiped off a **stale up-front snapshot**, destroying a DOB healed earlier in the same run → wipes became a deferred post-pass that re-reads the LIVE value and dedupes per borrower (`:109-128, 260-293`).
3. **Human forensics outside the API:** the owner had to pull ClickUp's own task activity history (not API-exposed) to establish the true damage inventory, because ClickUp's normalization had erased the write fingerprints (`CLICKUP-DATE-INCIDENT.md:122-131`).
4. **The actual restore:** 10 DOBs rewritten +1 day, 1 wiped closing date restored, each verified by re-read and logged in a restore table (`CLICKUP-DATE-INCIDENT.md:144-163`); 2 were already human-fixed and left untouched; open items (Yuda Elbaum's year-0026 closing, the Yaniv Erez 8-field wipe) required a portal-value review or a post-deploy admin repush — **"deploying the code alone does not rewrite history"** (`:165-169`). `SYNC-FIELD-FORENSICS:42`: *"Fixing the code is not the same as fixing the data."*
5. **A human review-queue system** (`sync_review_queue`, db/108; two-sided rows with both systems' values, db/110; LO-owned resolution applying the winner to BOTH systems through the guarded path) — because several classes of conflict are decidable only by a human.
6. **Reconstruction of missing history:** the write journal (`clickup_write_log`, db/107) was created *after* the incidents precisely because there was no API history — its provable negative ("not in the journal ⇒ the portal didn't do it") is what ended false-alarm forensics (`AUDIT:196`).
7. **Backfill migrations for past files** — the repo's recovery pattern for "this fix must reach existing data": e.g. `db/036_conditions_backfill_audit.sql` (templates added in 032 never reached pre-existing files; the backfill also starts items as `received` where the underlying data already existed, avoiding re-asking borrowers) and `db/060_fix_phantom_borrower_conditions.sql` (a NULL `borrower_label` rendered as the placeholder "An item your loan team needs"; a misconfigured auto-apply template + re-firing rule-engine sweeps **minted phantom conditions nobody knowingly added**; recovery = idempotent label backfill + retiring orphans to staff-only + fixing the engine default).

## 5. Structural rules adopted afterward (the codified regime)

### 5.1 Structurally-dead failure classes (provably impossible now, `AUDIT:65`)
Task deletion; field wipes in both directions (push skips empties, pull writes through COALESCE, no delete path exists); task rename/description writes (status-only allowlist in `client.js`); the echo/overwrite loop; the date day-walk; garbage years reaching ClickUp; scoped-push task creation; silent stuck files; un-reviewed automated DOB rewrites; email+surname merges; identity-less profile creation.

### 5.2 The chokepoint doctrine
The audit's central empirical finding (`AUDIT:16, 204`): **"chokepoint fixes stayed dead; call-site/instance fixes recurred within a day."** Everything that was fixed once at a single chokepoint (the date convention, `guardNoFieldClearing`, the client hard-stops, `decideDob`) never regressed; everything fixed per-instance (stamp → loan-number → race → defer; role re-seat ×3) recurred. Hence `SYNC-FINAL-CONCLUSIONS:207-209`, rule 12: *"No new guards as scattered code — anything guard-shaped must land at one of the four chokepoints or in the registry."*

### 5.3 The binding invariants (now in `CLAUDE.md`, CI-enforced per SYNC-FINAL-CONCLUSIONS Stage 0)
- Date-only values are `'YYYY-MM-DD'` strings end-to-end; ClickUp date writes only via `dateOnlyToClickUpEpoch()` with a **throw-on-violation round-trip assertion** (refuse to emit an epoch our own pull would read as a different day).
- Every outbound write journaled with before-image; no-op suppressed (read first, skip equivalent).
- Suspicious changes stop at the review queue; nothing suspicious is silently applied or silently dropped.
- A DOB change is ALWAYS a human decision unless provable — one decision function, consulted by every conflict site.
- Wipes structurally impossible both directions; every mid-typing artifact blocked at three layers; 2-digit years resolve via one helper (`normalizeTypedDate`), never a bare regex per field.
- Every stuck sync state = a review row WITH resolution actions; dismissals stick.
- **The program's new invariant:** *every sync operation that exhausts retries MUST end as a queued retry or a review row — a silent "done" is a bug by definition* (`AUDIT:206`; `SYNC-FINAL-CONCLUSIONS:129-130`).

### 5.4 The decided redesign (Contract-Gated Ledgered Sync, `SYNC-FINAL-CONCLUSIONS`)
- Transport contract at `client.js` (retry/429/token-bucket/deadline); truth contract at the orchestrator ("done" derived from the write journal, not set as a flag); **per-field policy as data** (typed field registry: direction, authority, PII class, validator — parity-tested cutover, byte-identical before legacy path deletion); **provenance as a required function argument**; volatile safety state (breaker, watermark, boot-sweep ledger) in Postgres so deploys can't reset guards.
- Standing rules: **error-budget deploy freeze** (no feature work while dead-letters > 0 — the only control aimed at the demonstrated root cause: velocity over safety, 491 commits/9 days); weekly 10-minute metrics ritual; never-regress list CI-enforced.
- Explicit "will NOT do" list (`:176-209`): no event-sourcing rewrite, no fuzzy identity expansion ever ("deterministic-only auto-merge; F-M5 is how different-named people got merged"), no new bidirectional fields by default, no webhook autocreate, no SharePoint enablement before the misfile-recall guard.
- Definition of done is **measured, not felt** (`:213-231`): recon breaks = 0, dead-letters from transient errors = 0, 3 independent document copies + a tested restore, <5 LO emails/day, two consecutive green weeks.
- Detection independent of the write path: nightly read-only recon with expected-state from the write journal (`EXPECTED_WRITE_NOT_LANDED` detector class), dead-man's switch on alerting (R4: one dead email key currently mutes ALL alerting — "the program's single biggest live residual"), `/api/health` exposing queue age for an external pinger.

### 5.5 Process lessons codified in CLAUDE.md
Fix the root, then every place the root can surface; every fix is all-surfaces; **previous AND future** (deterministic idempotent backfill migrations — the db/036 lesson); verify every write endpoint by re-fetch ("returned 200 but didn't save" is the #1 recurring class); a green build ≠ a rendering page; merge conflicts must preserve BOTH sides; never let a fix depend on a comment being read (the `fromEpochMs` comment *lied* about a guard it lacked — `SYNC-FIELD-FORENSICS:57`).

### 5.6 Handoff-doc lessons (`docs/HANDOFF-REMAINING-WORK.md`)
- **"Guessing the field mapping would corrupt data"** (`:48`) — the ClickUp CRM sync was deliberately blocked until the owner supplied the real board structure. Precedent: never build an integration mapping from assumption.
- Code-done ≠ live: multiple features fully built but inert until one owner config action (`:100-104`); the SharePoint mirror is the cautionary tale of assuming shipped = running.
- Docs go stale mid-flight and must be superseded in place (`:39-41` corrects its own earlier webhook instruction).

## 6. NEVER-AGAIN LIST — failure classes to design out of the Encompass read-only integration from day one

Phase 1 is strictly read-only pulls from Encompass Developer Connect (instance BE11397907), portal-side rule layering (condition clearing gated on Encompass agreement; CTC gated on Encompass CTC), and borrower-DB enrichment. Every item below is a demonstrated failure class from this repo's history, restated as a day-one structural requirement.

**NA-1. "Read-only" must be structural at the HTTP chokepoint, not behavioral.** The Jul 7 lesson: safeguards described as mandatory were waived at go-live ("The ONLY protection is no deletion") and corruption followed in 32 minutes. The Encompass client must be a single module that is *incapable* of emitting POST/PUT/PATCH/DELETE (method allowlist enforced in the client; ideally also OAuth scope/API-user permissions restricted server-side at Encompass). No caller, present or future, can bypass it — the exact pattern of `guardNoFieldClearing`/status-only-allowlist in `src/clickup/client.js`, which is the one class of guard that never regressed. Add a CI check that greps the Encompass client for non-GET verbs.

**NA-2. Dedicated integration identity + full read journal from day one.** The shared-token ambiguity cost two forensic days on false alarms and left a foreign automation permanently unattributable (incidents 4, 7). Use dedicated Encompass API credentials used by *nothing else*; journal every API call (endpoint, loan GUID, fields, timestamp, result) so "not in the journal ⇒ the integration didn't do it" holds from the first request. The "API history went dark" problem must never exist here.

**NA-3. Rate limits and retries at the chokepoint before the first sweep.** F-H1: zero 429/Retry-After handling while boot sweeps fired unbounded portfolio reads. The Encompass client ships with Retry-After honor, capped backoff + jitter, a token-bucket pre-throttle, request timeouts, and a per-job wall-clock deadline — on day one, not after the first throttling incident. Surface rate-limit headroom in health.

**NA-4. Deploy-neutral sync state.** F-H4/F-M16: in-memory watermarks and breakers reset by 13 deploys/day = daily read storms and guard resets. All cursors/watermarks/breaker windows/boot-sweep ledgers live in Postgres (`sync_runtime_state` pattern); full-portfolio sweeps run at most once per time window, bounded and paced, never per deploy; watermarks advance only past cleanly-consumed pages.

**NA-5. Silent failure is banned by invariant.** F-C1: a failed pull/refresh must end as a queued retry or a review row — never a silent "done", and never a stale value presented as current. Because portal rules *gate real decisions* on Encompass data (condition clearing, CTC), a swallowed fetch error is not a sync bug, it's a wrong lending decision.

**NA-6. Fail closed + staleness as a number for every Encompass-gated rule.** The portal must refuse to clear a condition or issue CTC when its Encompass copy is unavailable, stale beyond a threshold, or unverifiable — and data freshness (last successful pull per loan) must be a visible metric with an alert, not a feeling (the WO-18 heartbeat lesson; all 16 incidents were caught by humans noticing wrong data). Pair with a dead-man's switch on the alert channel itself (R4).

**NA-7. Dates are calendar strings end-to-end; every representation conversion gets a round-trip assertion.** The DOB disaster's class was "a date-only value represented as an instant." Encompass returns a mix of date and datetime formats across V1/V3 APIs. Parse at one boundary into `'YYYY-MM-DD'` strings (or explicit instants for true timestamps), keep the pg 1082 string parser, render via `dates.js`, and assert that parse(format(x)) is identity for every date/enum/currency transform. Never `new Date('YYYY-MM-DD')` in the browser.

**NA-8. A typed field registry as data before the second field is mapped.** F-M13: the garbage-year guard was a hardcoded 2-field list, so the next date field shipped unguarded; four divergent placeholder-name definitions and three fake-email definitions split behavior. Every Encompass field pulled gets one registry row — Encompass field ID, portal column, type, direction (all `inbound` in Phase 1), authority, PII class, validator, staleness tolerance — and the puller, validators, enrichment writer, rule-gates, and review producers all iterate the registry. Adding a field = adding a row; every guard applies automatically.

**NA-9. Never trust values — classify them.** Placeholders ("TBD", "N/A", "Unknown"), zeros, mid-typed years, and empties all corrupted real data. One shared classifier decides real/placeholder/empty/garbage per type; a money parser must never coerce "N/A" to $0; zero and empty are distinct; unknown enum/option values go to review, never silently substituted with a default or dropped (guard #13, `SYNC-FINAL-CONCLUSIONS:153`). Never assume Encompass option/index encodings symmetric between read paths (the ClickUp dropdown writes-UUID-reads-index bug, `BLUEPRINT:405-416` — verify Encompass field option encodings empirically per endpoint).

**NA-10. Enrichment is fill-only with provenance; conflicts go to review.** COALESCE fill-only inbound saved the portal DB in the DOB incident and made it the restore source — keep it: Encompass data fills empty portal fields, never overwrites a human-entered value without a review card. But close residual R2: a wrong-but-plausible *first fill* becomes the defended truth — stamp every enriched value with provenance (`encompass_unverified` until a human checkpoint) so "who set this" is a lookup, not archaeology. Provenance is a required argument, not a convention.

**NA-11. Identity joins on the immutable Encompass loan GUID only — never the loan number, never fuzzy.** The Salamon/Gruber class: duplicated records copy "unique" business keys; loan numbers get reused/copied. Incident 14: weak-key corroboration (shared family email/phone) merged different people and leaked file access. Deterministic-only matching; a name conflict is a hard veto; ambiguity is a human card, never an auto-merge; every auto-link journaled and reversible. Handle the copied-key form of ANY new matcher key on the day it's added.

**NA-12. No partial-value comparisons in any "does Encompass agree?" rule.** Last-4 SSN, first-word names, and 2-token addresses made real conflicts permanently invisible. The condition-clearing gates must compare canonical, fully-normalized values through the same transform on both sides — and a comparison that can't be made canonically must fail closed to review, not pass.

**NA-13. PII minimization decided before the first pull, not after.** The plaintext-SSN-in-ClickUp exposure (and full card+CVV) is the standing largest exposure. Decide per-field what Encompass PII the portal actually needs (prefer last-4 + verification flags over full values), encrypt what is stored, mask journals/logs/review cards, and record the decision in a memo (the WO-15 pattern — including the R1 lesson: a minimization migration must not silently kill an identity-matching key).

**NA-14. Review queue designed as a system, not accreted per incident.** 0→19 card types in 40 hours; noisy, undismissable, uncapped emails. Day one: tiered reasons (blocking vs informational), value-keyed sticky dismissals, per-user notification coalescing/rate cap, auto-close on convergence, borrower/loan grouping, per-reason dismiss-rate telemetry with auto-demotion. Every stuck state gets a card with concrete resolution actions.

**NA-15. Detection independent of the pipeline.** A nightly read-only recon (population pass: every gated portal loan ↔ exactly one Encompass loan; field pass: diff through shared transforms; joined on the GUID only), with persisted breaks that age rather than re-card, plus `EXPECTED_STATE_NOT_OBSERVED`-style checks against the journal. Assume the pull path lies until an independent reader agrees.

**NA-16. Runtime kill switches + circuit breaker, DB-backed.** Pause flags (per-direction, per-loop) flippable without a deploy — stopping the runaway ClickUp sync required redeploys. A volume breaker on reads (an Encompass sweep gone wrong is a rate-limit/lockout incident even when read-only), persisted so restarts don't reset it.

**NA-17. Tests, CI, migration ledger, and replay fixtures from day one.** The 8-hour-overnight build with no CI, no `npm test` wiring, no `schema_migrations` ledger, migration-number collisions ×3, and a zero-coverage 1,545-line ingest module is the anti-pattern. Record real Encompass API responses as replay fixtures and assert identical ingest decisions in CI (the WO-19 idea — the only credible test strategy for an ingest module); parity-test any mapping refactor byte-identically before deleting the old path.

**NA-18. Code fix ≠ data fix; build the repair discipline before it's needed.** Any bad data that reaches the portal requires an explicit repair operation: dry-run default, classification taxonomy, cross-checks against journals, verify-every-write by re-read, never-clear/never-guess, deferred destructive actions re-checking LIVE values (the restore script's stale-snapshot near-miss, `clickup-date-restore.js:109-128`), pacing, and human review for undecidables. Also: auto-generated portal conditions derived from Encompass data must carry complete display metadata and an origin marker so re-firing sweeps can't mint phantom "An item your loan team needs" conditions at scale (db/060).

**NA-19. The blueprint must match reality, and its safeguards are not optional.** The sync blueprint's mandatory echo-suppression was written, never wired, deleted; the doc still says "nothing implemented" while machinery runs; the deferred bot seat and deferred duplicate hold each became an incident (`AUDIT:155-165`). For Encompass: no staged safeguard is waivable at go-live; when reality diverges from the design doc, the doc is updated in the same change; "shipped" is verified as "running in production" (the SharePoint mirror ran zero times while everyone assumed backups existed — verify the first successful pull against the live instance, and monitor that the integration keeps executing).

**NA-20. Velocity governance.** The demonstrated root cause of the whole mess was speed over safety (built overnight, 13 deploys/day, 88–100% next-day-rework churn). Adopt the error-budget rule for the Encompass program from day one: while any Encompass dead-letter or blocking review card is open past SLA, no new integration feature merges — only fixes.
