The SharePoint integration is the portal's **one-way, append-only document mirror** into the team site (`yscapgroup.sharepoint.com/sites/SharedData`, `Documents` library, `Pipeline Drive` tree). It is the company's most heavily-guarded integration and the closest existing analog to the planned read-only Encompass integration ‚Äî except its directionality is inverted (write-only TO the external system, vs. read-only FROM Encompass). Its guard architecture is directly transferable.

## 2. One-way backup design

- **Outbox reconciler, not dual-write** (`sharepoint-backup.js:1-29`): every upload path INSERTs a `documents` row; a background pass (`pendingBatch`, `sharepoint-backup.js:219-281`) selects rows where `sharepoint_backed_up_at IS NULL`, reads bytes from local storage, and uploads. Upload endpoints only `kick()` (debounced 4s, `sharepoint-backup.js:1216-1225`); a SharePoint failure can never break a borrower/staff upload. Interval sweep every `SHAREPOINT_BACKUP_POLL_SEC` (default 300s) retries and performs the first-run full-history backfill oldest-first.
- **Self-gated**: inert unless `SHAREPOINT_BACKUP_ENABLED=1` AND `MS_*` creds present (`enabled()`, `sharepoint-backup.js:134-136`; `src/config.js:176-206`). Every risky behavior has an env kill switch.
- **Directionality enforced structurally** (`sharepoint.js:16-17`): the module never downloads document bytes from SharePoint. The only reads are folder LISTINGS (names/ids, `listChildren` `sharepoint.js:191-200`) for matching, and item METADATA (`itemMeta`, `sharepoint.js:228-233` ‚Äî size + Graph's own hashes) for the integrity audit. The verify pass explicitly documents "bytes are never downloaded back; one-way holds" (`sharepoint-backup.js:848-865`).
- **Target layout**: everything the mirror writes lives ONLY inside a portal-owned `Synced by Pilot` / legacy `YS portal syncing` leaf folder; category subfolders per condition; special layouts for LLC docs, Photo ID, `REO/<address>` track-record docs, `Term Sheet/Unsigned` (`categoryPathFor`, `sharepoint-backup.js:162-177`).
- **Version history**: a supersede event bumps a per-condition version counter ‚Äî first replacement creates `Version 1`, moves ONLY the portal's own prior copies in, and the new doc lands in `Version 2`; later supersedes add `Version 3+` (`isSupersedeEvent` `sharepoint-backup.js:333-400`, `shuffleRootIntoVersion1` `:410-471`, versioning in `mirrorRowInner` `:749-788`). Regen kinds (autosaved snapshots/exports, `isRegenKind` `:58`) NEVER version-shuffle and superseded-before-mirror snapshots settle without uploading (`settleSupersededSnapshots` `:288-311`) ‚Äî the "Version 47" incident fix.

## 3. The hard-rule guard stack (policy ‚Üí code)

Binding policy is `docs/SHAREPOINT-POLICY.md`; the enforcement:

1. **No delete** ‚Äî `remove()` is a permanent throwing no-op (`sharepoint.js:613-615`). No general Graph DELETE may exist.
2. **ONE sanctioned delete exception** (owner amendment 2026-07-16) ‚Äî `deleteReplacedCorruptMirror()` (`sharepoint.js:404-483`) deletes a diagnosed-corrupt mirror copy only after its verified "(fixed copy)" replacement exists. **Seven layered guards, ALL mandatory**: G1 kill switch (`SHAREPOINT_DELETE_REPLACED_CORRUPT=0`); G2 DB-ownership (itemId must come from the row's own `sharepoint_backup_ref`); G3 replacement-first (replacement re-read live, size must equal local bytes); G4 same-bytes-as-diagnosed (current size == size at diagnosis); G5 expected-parent; G6 Pilot-tree ancestry walk (‚â§8 hops must hit a `Synced by Pilot`/`YS portal syncing` ancestor ‚Äî the delete can never reach outside the portal's own tree); G7 If-Match eTag pinning (concurrent human change ‚Üí 412 ‚Üí nothing happens). Refusals throw; caller treats it as best-effort; every delete/refusal is audited. Deletes are soft (93-day recycle bin) so even the exception is recoverable.
3. **No overwrite** ‚Äî all uploads use `conflictBehavior:'fail'`; conflicts uniquify or adopt (`uploadNew` `sharepoint.js:286-377`; candidate names + `adoptIfIdentical` `sharepoint-backup.js:504-540`).
4. **No move/rename except one** ‚Äî `moveOwnItem()` (`sharepoint.js:388-402`) requires `expectedParentId` (a human-moved file becomes permanently unmovable by the sync) plus If-Match eTag; callers additionally verify DB ownership. Used only for the Version-N shuffle inside portal-created folders. Matched human folders are never renamed (hard no-rename; legacy folder names are matched forever, `sharepoint-map.js:52-72`).
5. **Never guess** ‚Äî fuzzy matching is conservative: exact-normalized or first/last-token borrower match (never substring: "Moshe Katz" ‚âÝ "Moshe Katzman"); address match requires identical house number + exactly-equal suffix-normalized street tokens; typo fallback is Damerau-Levenshtein ‚â§1 per token, single-candidate only, always review-flagged (`sharepoint-map.js:113-201`). Ambiguity (2+ fuzzy hits, none exact) ‚Üí create a fresh marked folder, never guess (`pickMatch` `sharepoint-map.js:217-223`); every uncertain resolution queues a sync-review row (`sharepoint-map.js:388-412`).
6. **Policy over permission**: the Azure app holds `Sites.ReadWrite.All` (owner declined `Sites.Selected` downgrade, 2026-07-13) which technically permits deletes ‚Äî so the no-delete guarantee is enforced by code + binding policy + a CLAUDE.md audit gate on every change touching a Graph call path (`docs/SHAREPOINT-POLICY.md:67-77`). `docs/SHAREPOINT-SECURITY-COMPLIANCE.md` still recommends the `Sites.Selected` downgrade as "the real fix."

## 4. Integrity verification (without ever reading bytes back)

- **At upload**: `verifySize` in `uploadNew` fails any commit whose driveItem is missing id/size or whose size differs from bytes sent (`sharepoint.js:291-314`); a failed check = a failed row = retry; an unverifiable item is never recorded as a mirror.
- **QuickXorHash** ‚Äî Microsoft's per-item content hash is computed LOCALLY (`sharepoint.js:245-259`, validated against an independent spec implementation in `scripts/test-sharepoint-integrity.js:58-74`), so transfers verify against Graph-reported hashes with zero byte downloads. It **self-calibrates**: one hash match on a size-verified fresh upload ‚Üí trusted; one disagreement ‚Üí size-only forever (never mass-flags) (`calibrateQuickXor`, `sharepoint-backup.js:105-117`).
- **Recurring audit** (`verifyRow`/`verifyOnce`/`drainVerify`, `sharepoint-backup.js:919-1147`): boot + every 6h (`SHAREPOINT_VERIFY_POLL_SEC`), each mirrored doc re-checked ‚â§ every 30 days. Compares local bytes (size/sha256/QuickXor) vs the item's METADATA. Verdicts stamped on the row: `ok`, `mismatch` (re-queued ‚Üí good copy re-uploads as "(fixed copy)" into the same folder; sanctioned delete may then remove the corrupt original), `source-suspect` (magic-byte sniff says bytes were garbage BEFORE upload ‚Äî human must request fresh copy; re-mirroring can't fix), `item-missing` (human deleted ‚Äî human decides via review), `local-missing` (mirror may be the only surviving copy ‚Äî never touched), `malware-flagged` (Defender blocked ‚Äî never blind re-upload), `modified-in-sharepoint` (lastModified > upload time ‚Üí never auto-replaced; avoids fighting humans/SharePoint).
- **Office-format exception** (2026-07-16 root fix): SharePoint "property promotion" REWRITES xlsx/docx seconds after upload (size AND hash drift). Post-upload byte comparison is meaningless for those; integrity is proven once from the PUT response, and identity thereafter is **provenance** ‚Äî `createdByThisApp` (createdBy.application.id == our client id) + exact name + our folder (`sharepoint.js:210-223`, `sharepoint-backup.js:1012-1023`).
- **Anti-duplicate**: sha256 byte-dedup within scope (only against healthy mirrors, `sharepoint-backup.js:640-674`); adopt-on-conflict absorbs upload-succeeded-but-unrecorded retries; cross-process DB leases (`sync_locks`, `db/115_sharepoint_integrity.sql:45-49`; `acquireLease` `sharepoint-backup.js:72-103`) prevent deploy-overlap double-drains; per-document lease renewal mid-batch.
- **Local integrity gate**: bytes on disk must match `size_bytes` recorded at upload, or the row fails ‚Äî corruption is never pushed INTO SharePoint (`sharepoint-backup.js:620-627`). Upstream, all uploads decode through the strict `decodeUploadBase64` chokepoint (the `data:`-prefix garbling class is structurally impossible; canary test at `scripts/test-sharepoint-integrity.js:41-42`).

## 5. Metadata stamping (roadmap R1, built 2026-07-16)

After each upload the mirror best-effort stamps four text columns on the driveItem's listItem: `PilotDocumentId`, `PilotFileId`, `PilotBorrower`, `PilotSyncedAt` (`PILOT_COLUMNS`, `sharepoint.js:523-572`). Columns are created once per drive (`ensurePilotColumns`; `PilotFileId` indexed), visible to staff, and make identity **id-based and rename/move-proof**. Gated by `SHAREPOINT_STAMP_METADATA` (default on); a stamp failure never affects the mirror (bytes already recorded). Coverage tracked via `documents.sharepoint_stamped_at` (`db/121_sharepoint_metadata_stamp.sql`) and surfaced as an ID-stamp-coverage % in the reconciliation controls block.

## 6. Sync-state tracking (DB)

- `db/091_document_sharepoint_backup.sql` ‚Äî per-document outbox columns: `sharepoint_backup_ref` (`sp:<driveId>:<itemId>`), `sharepoint_web_url`, `sharepoint_backed_up_at`, `sharepoint_backup_error`, `sharepoint_backup_attempts` (budget 8, `MAX_ATTEMPTS`), `sharepoint_backup_attempted_at`; partial index on un-mirrored rows.
- `db/092_sharepoint_sync_state.sql` ‚Äî `sharepoint_version`/`sharepoint_parent_id` on documents; `sharepoint_folder_cache` (scope_key ‚Üí resolved sync folder + `details` jsonb recording every match/ambiguity flag for review); `sharepoint_condition_state` (per-condition Version-N counter).
- `db/115_sharepoint_integrity.sql` ‚Äî `sha256`, `sharepoint_item_size`, `sharepoint_verified_at`, `sharepoint_integrity` (verdict), `sharepoint_skipped_reason` (distinguishes "deliberately not uploaded" ‚Äî superseded snapshot / duplicate bytes ‚Äî from "mirrored"); verify index; `sync_locks` lease table.
- `db/121` ‚Äî `sharepoint_stamped_at` + pending-stamp index.
- Retry lifecycle: exhausted rows get a fresh chance daily and on every boot (30-min time-guard against deploy bursts, `sharepoint-backup.js:1233-1244`); exhaustion on a **human-actionable** error queues a sync-review card with the real error, while transient infra noise (429/503/timeouts) never pages ("no fake cards", `recordFailure` `sharepoint-backup.js:808-846`).

## 7. Auth handling (Microsoft Graph)

- **App-only client credentials** against `login.microsoftonline.com/<tenant>/oauth2/v2.0/token`, scope `https://graph.microsoft.com/.default` (`sharepoint.js:84-112`).
- **Dual credential with fallback**: certificate preferred (`MS_CLIENT_CERT_PEM`, self-built RS256 client-assertion JWT with x5t thumbprint, `certAssertion` `sharepoint.js:58-73`), client secret (`MS_CLIENT_SECRET`) fallback ‚Äî one credential failing doesn't stop the sync.
- **Token cache + single-flight** (`sharepoint.js:77-82`): 60s early-expiry refresh; concurrent callers share one token fetch; one automatic refresh on 401 inside `graph()`.
- **Throttling discipline** (`graph()`, `sharepoint.js:118-154`): honors `Retry-After` on 429/503/504 exactly (capped 120s), exponential backoff + jitter otherwise, `MAX_TRIES=6`; upload-session chunk loop parses Retry-After defensively (HTTP-date can't become `sleep(NaN)`) and resumes 416s via `nextExpectedRanges`; 300ms pacing between uploads. Every Graph error carries Microsoft's `request-id` for support correlation.
- **Credential-lifecycle watchdog** (`credentialHealth`, `sharepoint.js:493-510`): parses the cert's notAfter, reports expiry date/days-left on the admin health probe, warns ‚â§30 days ‚Äî expiry becomes a planned rotation, never a silent outage. `reconciliation().healthy` goes false on a credential warning.
- Secrets live only in env (`src/config.js`); the original research doc records that a pasted secret was treated as compromised and required rotation (`docs/SHAREPOINT-INTEGRATION-RESEARCH.md:28-42`).

## 8. Admin/observability surface

`/api/admin/sharepoint/*` (requireAuth + `platform_setup`; every action audit-logged): `GET /health` (probe, backlog, integrity verdict histogram), `GET /reconciliation` ‚Äî the chain-of-custody report classifying EVERY document into exactly one bucket (mirrored/pending/exhausted/skipped/unverified/ok/mismatch/source-suspect/malware/item-missing/local-missing), oldest-pending age, SLO verdict, and a `controls` block (sanctioned-delete gate state, stamp gate + coverage %, total attributed sanctioned deletes, credential health) ‚Äî the screen "an examiner reads to confirm the guarantees are actually enforced, not just claimed" (`sharepoint-backup.js:1289-1346`, `docs/SHAREPOINT-SECURITY-COMPLIANCE.md:74-81`). `POST /verify`, `/mirror`, `/retry-exhausted`, `/doc/:id/remirror` for operations. An SLO watchdog notifies admins once per breach episode when the oldest un-mirrored doc exceeds `SHAREPOINT_BACKLOG_SLO_HOURS` (default 6) or anything is exhausted (`checkBacklogSlo`, `sharepoint-backup.js:1348-1380`). Compliance posture: SEC 17a-4 "audit-trail alternative to WORM" ‚Äî every mirror action/verdict/delete attributed in `audit_log`; Purview retention label recommended so even the sanctioned delete would fail safe.

## 9. Policy rules imposed (summary of docs/SHAREPOINT-POLICY.md)

1. Never delete/recycle (one seven-guard exception). 2. Never overwrite. 3. Never move/rename (one guarded own-item exception). 4. One-way flow ‚Äî folder names read for matching only; document content never read out of SharePoint. 5. Uncertainty ‚Üí new marked folder + recorded decision, never a guess. Enforcement is defense-in-depth: code guards, binding policy doc, CLAUDE.md hard rules, mandatory audit of every Graph-touching change, and unit tests asserting the guards refuse (`scripts/test-sharepoint-integrity.js:118-135` ‚Äî kill switch, missing-replacement, missing-parent, missing-id refusals, and `remove()` still throws).

## 10. Lessons for the Encompass integration (how one-directional stays one-directional)

The SharePoint mirror is write-only outbound; Encompass Phase 1 is read-only inbound. The direction is inverted but the discipline is identical ‚Äî "the forbidden verb class must be structurally impossible, not merely unused":

1. **One chokepoint client, forbidden verbs as throwing no-ops.** All Graph traffic flows through `sharepoint.js`; deletion isn't "not called," it's `remove(){throw}` with a comment pointing at the policy. Encompass analog: ONE `src/lib/encompass.js` client whose exported surface contains only read operations; export explicit `create/update/delete/patch = () => { throw }` stubs so future code that tries to write fails loudly and greppably. No caller may hold a raw token or call `fetch` at api.elliemae.com directly.
2. **Guard at every layer, because permission scope alone is not the guarantee.** SharePoint's scope (`Sites.ReadWrite.All`) permits deletes, so code+policy+audit carry the guarantee ‚Äî and the compliance doc still calls scope reduction "the real fix." For Encompass, do BOTH from day one: request least-privilege (a read-only Encompass user/persona + minimal API scopes) AND enforce read-only in code. Unlike SharePoint, don't let the least-privilege downgrade be declined.
3. **Every exception is: owner-sanctioned, one named function, N mandatory layered guards, an env kill switch, fully audited, and best-effort (a refusal never breaks the pipeline).** Phase 1 Encompass should have ZERO write exceptions ‚Äî but this is the template if one is ever demanded.
4. **Policy-as-code + binding doc + change-audit gate.** `docs/SHAREPOINT-POLICY.md` binds "any AI assistant ever given SharePoint access"; CLAUDE.md mandates auditing every change that touches a Graph call path for delete/move/rename. Write an `ENCOMPASS-POLICY.md` with the same force: every change touching the Encompass client is audited for non-read verbs before it ships.
5. **Tests that assert the guards refuse.** `test-sharepoint-integrity.js` proves `remove()` throws and the sanctioned delete refuses without its preconditions ‚Äî no DB/network needed. Encompass needs the same: a suite asserting the client exposes no write path and that write stubs throw (plus a grep-style CI check that no code outside the client references the Encompass base URLs).
6. **Verification without violating direction.** The mirror proves integrity via metadata (size + QuickXorHash) without ever downloading bytes ‚Äî read-side verification that respects the write-only rule. Encompass analog: layer portal rules (condition-clear gate, CTC gate) purely on data READ from Encompass; record per-field provenance and verdicts on portal rows (like `sharepoint_integrity`), and never "fix" Encompass ‚Äî a disagreement is a review row, not a write.
7. **Outbox/reconciler shape + operational hygiene transfer wholesale**: gated-off-by-default master switch; DB `sync_locks` leases against deploy-overlap double-runs; retry budget with daily/boot re-arm; transient-vs-actionable error split (no fake review cards); Retry-After honored exactly with jittered backoff and pacing; request-id captured on every error; dual-credential fallback + credential-expiry watchdog on the health probe; an SLO watchdog; and a single `reconciliation()` chain-of-custody report an examiner can read. The Encompass poller should replicate each of these.
8. **Conservative entity matching, review-not-guess.** Folder matching never guesses (exact-first, anchored fuzzy, single-candidate, review-flagged, ambiguity ‚Üí safe default). Portal-file ‚Üî Encompass-loan matching should be exact on loan GUID/number; anything ambiguous goes to the existing `sync_review_queue` machinery with concrete resolution actions, never a silent link. The duplicated-task/copied-loan-number lessons from ClickUp apply directly.
9. **Never-clear semantics on the inbound side.** The ClickUp pull writes through COALESCE so a remote blank can't wipe portal data; Encompass enrichment should adopt the same rule ‚Äî Encompass data enriches, provably-fresher wins by policy, blanks never clear.

## Open questions

- Whether the Encompass Azure-equivalent (API credential persona) can be provisioned truly read-only in Encompass admin ‚Äî the strongest structural guarantee, and the one SharePoint lacks.
- Whether the Encompass pull should reuse `sync_locks` and `sync_review_queue` (recommended ‚Äî one operational surface) or grow parallel tables.
- R2 (delta-query drift healer) and R5 (webhooks) are designed but unbuilt on the SharePoint side; the Encompass design should decide poll-vs-webhook with the same "delta feed is the safety net" stance.
