# E4 — Long-Term Vision After the Read-Only Phase (with a Simplicity Filter)

Research agent E4. Scope: what YS Capital should (and should not) build on top of the Encompass integration AFTER the strictly read-only Phase 1 proves out. Every idea below is evaluated against the owner's own bar — **"not too complicated, user friendly, helping out"** — and against hard evidence from this repo of what integration complexity actually costs this team.

Sources: repo at `/home/user/yscap/yscap-repo-root_8` (read-only), the Encompass Developer Connect 26.2 Postman collection index at `scratchpad/research/postman-endpoint-index.md` (800 requests: 329 GET, 200 POST, 214 PATCH, 27 PUT, 30 DELETE), and `CLAUDE.md` / `docs/` for repo history.

## 1. The Simplicity Filter, made concrete

The owner is a non-developer (`CLAUDE.md:7` — every message must be plain language). The filter is not a vibe; this repo contains a measurable record of what integrations cost:

| Integration | Direction | Documentation/incident tail in `docs/` |
|---|---|---|
| ClickUp sync | **Bidirectional** | 13 sync/ClickUp docs, including `CLICKUP-DATE-INCIDENT.md`, `SYNC-FIELD-FORENSICS-2026-07.md`, `SYNC-GUARD-MATRIX.md`, `AUDIT-2026-07-15-SYNC-ANNUAL.md`, `HANDOFF-SYNC-DATA-SAFETY.md` |
| SharePoint mirror | **One-way, additive-only** | 6 docs (`SHAREPOINT-POLICY.md`, `SHAREPOINT-SYNC-HARDENING-RESEARCH.md`, etc.), plus ~15 lines of hard rules in `CLAUDE.md:105-117` (no-delete policy, integrity verification, Version-N shuffle, corrupted-mirror audit) |

Lesson: **each direction of data flow roughly doubles the guard machinery and the incident classes.** The bidirectional ClickUp sync required a guard matrix, forensics, an annual audit, and produced at least one data-corruption incident (`CLICKUP-DATE-INCIDENT.md`). Even the *one-way* SharePoint mirror needed a no-delete policy document, hash verification, and a seven-guard exception process (`CLAUDE.md:106`). Encompass is the system of record for a regulated lending business — a write-back mistake there is strictly worse than either precedent.

**The filter as five questions** (an idea must pass all five, or be redesigned until it does):

1. **One-sentence test** — can the owner explain the feature to a borrower or an LO in one plain sentence?
2. **Direction test** — does data still flow one way, with Encompass remaining the sole source of truth for loan facts?
3. **Failure-mode test** — when it breaks, does it degrade to *"information temporarily missing"* (acceptable) or to *"a wrong action was taken / a wrong fact was shown"* (unacceptable)?
4. **Ceremony-budget test** — is the guard machinery smaller than the feature? (ClickUp failed this test in hindsight.)
5. **Ongoing-care test** — does it add a new queue/reconciler/incident class somebody must babysit forever?

A useful repo-specific corollary: **never render a raw Encompass string to a borrower.** `CLAUDE.md:78` forbids exposing capital-partner names (BlueLake, Temple View, RCN, Churchill, Fidelis…) on any borrower-facing surface; Encompass milestone names, condition titles, and comments can embed exactly those names. Every borrower-facing feature below therefore requires a **portal-owned label map**, not passthrough.

## 2. Idea-by-idea evaluation

### 2.1 Controlled write-back (the big one)

The API surface makes writing *easy* — that is the danger, not the opportunity: `POST /encompass/v3/loans/{id}/fieldWriter`, `PATCH /encompass/v3/loans/{id}` (index lines ~321, 337, and the whole V3 Manage Loan folder), condition PATCHes (lines 266-273), milestone "Finish" PATCHes (lines 162, 173), rate-lock lifecycle (lines 366-381). Phase 1's "architecturally incapable of writing" guarantee must survive any Phase 3 expansion — which means **write-back, if it ever happens, lives in a separate service with a separate credential**, never as a relaxation of the read-only client. Note that method-based blocking ("GET only") is *not* sufficient even for reads: pipeline queries (`POST /encompass/v3/loanPipeline`, line 344), `fieldReader` (line 321), and attachment download-URL minting (line 256) are POST-shaped reads — so the read-only guarantee must be an **endpoint allowlist**, and the write service must be a physically separate allowlist/credential.

Ranking of writes by safety, best-first:

| Write | What it is | Safety character | Verdict |
|---|---|---|---|
| **eFolder document push** | Portal-collected docs uploaded as *new* attachments: create document container (`PATCH …/documents?action=add`, line 230/147), `POST …/attachmentUploadUrl` (line 251), PUT bytes (line 253) | **Additive-only.** Never modifies or deletes anything; identical in spirit to the SharePoint mirror (no-delete, no-overwrite, uniquify-on-conflict, `conflictBehavior:'fail'`, `CLAUDE.md:107`). Failure mode = a missing/duplicate document, visible and fixable by a human | **The one write worth doing.** First candidate for Phase 3 |
| **Custom-field breadcrumb** | Write one `CUST.X` field per loan: portal deep-link + last-sync timestamp (via `fieldWriter` restricted to that one field ID; custom-field *definitions* readable at `/encompass/v3/settings/loan/customFields`, lines 576-578) | Touches the loan object → can collide with an open-in-editor loan, fire business rules, and appears in the audit trail. But it's one field, portal-owned, idempotent, and no human decision depends on it | **Acceptable second write**, after the doc push proves the ceremony works |
| **Condition status write-back** (mark received/cleared in Encompass) | Condition PATCHes (lines 266-273 and the v1 underwriting/preliminary/postclosing sets) | Inverts the whole Phase-2 premise ("portal cannot clear unless Encompass agrees" becomes "portal changes what Encompass says"). Circular-truth hazard; failure mode = wrong underwriting state | **No.** Fails direction test permanently — do not roadmap it |
| **Milestone finish, rate locks, loan data edits, disclosure sends** | Everything else write-shaped in the collection | Regulated artifacts, compliance timing, money | **Never.** Out of scope for a borrower portal |

**The write ceremony** (prerequisite for ANY write, modeled on precedents already proven here):

1. Owner sign-off *in the owner's own words* per write type (the frozen-baseline convention, e.g. `CLAUDE.md:82`).
2. A binding policy doc, sibling to `docs/SHAREPOINT-POLICY.md`: additive-only; no update, no delete, no overwrite; enumerated endpoints.
3. Separate OAuth client/credential for the write service; the read client keeps zero write endpoints in its allowlist.
4. Kill switch env var, default **off** (precedent: `SHAREPOINT_DELETE_REPLACED_CORRUPT=0`, `CLAUDE.md:106`).
5. Dry-run mode first: journal what *would* be written for 2-4 weeks; a human reviews the journal.
6. Read-after-write verification of every write (already house law: `CLAUDE.md:93` "verify every write endpoint"), journaled in an `encompass_write_journal` table.
7. Idempotency: every write carries a portal-side natural key so a retry can never double-apply (the lesson of the autosave-duplicate incident, `CLAUDE.md:71`).

Prerequisite trust milestones: Phase 2 fully passed (below) + 6 months of clean read-only operation + doc-mirror parity (portal already knows exactly which docs exist in the eFolder from Phase 1 reads, so the push can be reconciled read-side).

Complexity cost: **high** (new service, new credential, journal, reconciliation). Filter verdict: **passes only for the two additive writes above**, precisely because their failure modes are visible and non-destructive. Everything else fails question 3.

### 2.2 Investor reporting / capital-markets analytics

Data source is pure read: `POST /encompass/v3/loanPipeline` + `/loanPipeline/report` cursor endpoints (index lines 343-350) with `canonicalFields` for field discovery, plus (if ever needed) read-side trade pipeline queries (`/secondary/v1/tradePipeline`, lines 490-493). The portal DB is *enriched by Phase 1 anyway*, so v1 of this feature is **SQL over the portal's own mirror tables + a scheduled xlsx/CSV export** — no new Encompass surface at all.

- Trust milestones: field-mapping parity proven (staff confirm the portal's numbers match Encompass side-by-side for ~4 weeks); loan-matching rate ≥99%.
- Complexity cost: **low-to-moderate** if it stays "generated spreadsheet on a schedule"; **high** if it becomes a live BI platform. Do the former.
- Filter verdict: **passes** ("every Monday you get the investor tape as a spreadsheet, built from what Encompass says"). Stays read-only forever.

### 2.3 Servicing handoff packages

Encompass has a purpose-built read path: **eFolder export jobs** (`POST /efolder/v1/loans/{id}/exportJobsCreator` + job-status polling, index lines 260-262) bundle attachments; combine with a fieldReader extract for the servicing data sheet. Output: one ZIP/PDF package per funded loan. Relevant house context: draw management is deliberately sandboxed to a separate portal (`CLAUDE.md:104`) — servicing lives *outside* this portal by design, so a handoff **package** (a deliverable file) is the right shape, not a servicing *module*.

- Trust milestones: document-inventory reads proven accurate (Phase 1); one manual pilot handoff assembled by staff using portal data before automating.
- Complexity cost: **moderate** (async job polling, large-file handling — but the storage/streaming plumbing already exists in `src/lib/storage.js`).
- Filter verdict: **passes** ("when a loan funds, the portal assembles the complete file to hand to the servicer"). Read-only.

### 2.4 AI-assisted condition matching

This is not really a Phase-3 luxury — it is the enabling problem of **Phase 2**: the portal's `checklist_items`/`checklist_templates` (`db/schema.sql:220-256`) must be paired with Encompass conditions (enhanced conditions `GET /encompass/v3/loans/{id}/conditions`, lines 264-265; v1 underwriting/preliminary/postclosing sets) before "cannot clear unless Encompass agrees" can be enforced. The right architecture, in order:

1. **Deterministic first**: a maintained mapping table (portal template `code` ↔ Encompass condition title/type patterns). The portal already keys conditions by stable codes (`rtl_p1_product`, `rtl_p1_budget`, `rtl_p3_reo`… — see `signOffGate`, `src/routes/staff.js:2400-2414`), so most of the map is a static config, not AI.
2. **AI as a suggester only**: semantic matching proposes pairings for the unmapped residue; a staff member confirms each pairing once; the confirmation is stored and becomes deterministic thereafter.
3. **AI never acts**: a fuzzy match may *block* nothing and *clear* nothing on its own. An unmatched portal condition simply shows "not yet linked to Encompass" and falls back to today's human sign-off path.

- Trust milestones: deterministic map covers ≥90% of active-file conditions; suggestion acceptance rate measured before widening.
- Complexity cost: **low** for the deterministic core; **moderate** for the suggester (an LLM call + a review screen). The trap to refuse: auto-acting AI, which converts every model error into a wrong underwriting state (fails filter question 3).
- Filter verdict: **passes as suggest-and-confirm; fails as autopilot.**

### 2.5 Document OCR cross-checks

Idea: OCR portal/eFolder documents and cross-check extracted values against Encompass fields (insurance coverage vs. field value, signed term-sheet amount vs. registered loan amount). Honest assessment: **highest complexity-to-benefit ratio on this list.** It adds a model/vendor dependency, a false-positive triage queue, and per-document-type extraction logic — a whole new incident class (filter question 5), for checks a human already performs at sign-off. The repo's own history warns about byte-level document fragility (the corrupted-upload incident chain, `CLAUDE.md:113-115`).

- If ever built: **advisory-only** (a yellow "the numbers may not match — please double-check" note on the condition, never a block), scoped to *one* high-value document type first (insurance evidence is the classic hard-money pain point).
- Trust milestones: Phase 2 stable; a measured false-positive rate <10% on a shadow run before any staff-visible surfacing.
- Filter verdict: **defer — fails the ceremony-budget test today.** Revisit late Phase 3 only if staff explicitly ask for it.

### 2.6 Warehouse-lender reporting

Same machinery as §2.2: a pipeline query filtered to funded/warehouse-line loans, emitted as a scheduled export in the warehouse lender's template. It is §2.2 with a different column set and audience.

- Complexity cost: **low** (incremental over §2.2 — build them on one shared "report definition → scheduled export" primitive rather than as two features).
- Filter verdict: **passes**, same conditions as §2.2.

### 2.7 Borrower-facing status timeline from Encompass milestones

Data source: `GET /encompass/v3/loans/{id}/milestones` (index line 158-159). This is the most *user-friendly* item on the list — borrowers ask "where is my loan?" constantly — and it is pure read. Three non-negotiable design rules:

1. **Label map, not passthrough**: Encompass milestone names → portal-owned borrower-friendly stage names (also satisfies the `CLAUDE.md:78` capital-partner-name rule).
2. **Staff-first rollout**: show the milestone strip on the staff file view for ~a month; only after staff trust it does it appear to borrowers. Borrower-visible wrongness is the most expensive kind (the "two people see two different dates of birth" incident, `CLAUDE.md:90`, was exactly this class).
3. **Freshness gating**: the timeline shows "as of \<sync time\>" and hides itself rather than show stale data past a threshold — degrade to *missing*, never to *wrong* (filter question 3). It must also never contradict the portal's own gates — the portal's clear-to-close structure lock (`src/lib/file-lock.js:3-16`) and CTC status remain governed by the Phase-2 rule "portal shows CTC only if Encompass shows CTC."

- Complexity cost: **low.** Filter verdict: **strong pass — recommend it as the first Phase-2 borrower-visible deliverable**, because it makes the read-only integration *felt* by customers without any enforcement risk.

### 2.8 Ideas considered and rejected outright

- **Two-way condition/status sync** — recreates the ClickUp bidirectional tail against the system of record. Permanent no.
- **Portal-initiated loan creation in Encompass** (the collection is full of Create Loan requests) — origination stays in Encompass. No.
- **Rate-lock or disclosure automation** — compliance-bearing writes. No.
- **Webhook subscriptions in Phase 1** (`POST /webhook/v1/subscriptions`, line 152): technically a platform-config write, not a loan write — but Phase 1's "incapable of writing" promise is cleanest if even this waits; poll first, adopt webhooks in Phase 2/3 as a freshness optimization with explicit owner sign-off that it does not breach the read-only pledge on *loan data*.

## 3. Staged roadmap with go/no-go gates

The repo is already shaped for this: the sync worker was explicitly designed so "Encompass/Graph targets slot in here behind the same interface later" (`src/sync/queue.js:4-5`), and `sync_queue` exists in the base schema (`db/schema.sql:302`).

### Phase 1 — Read-only mirror (prove the plumbing)
Pull loans, milestones, conditions, document inventory, and key fields into portal tables on a poll; enrich borrower/application records; staff-only "what Encompass says" panel on the file view. Zero enforcement, zero borrower visibility, zero writes (endpoint-allowlisted client — allowlist by *path*, not by HTTP method, because core reads are POSTs).

**Gate 1 → 2 (go/no-go):**
- **Match rate**: ≥99% of active portal files auto-matched to exactly one Encompass loan; every unmatched/ambiguous file triaged with a documented cause.
- **Parity**: staff side-by-side review for ≥4 consecutive weeks with zero "the portal shows a different number than Encompass" reports outstanding.
- **Freshness SLO**: data age ≤ agreed threshold (e.g. 15 min) ≥99% of polling windows; staleness is visibly stamped in the UI.
- **Write-incapability audit**: egress/audit log review confirms zero non-allowlisted calls; Encompass-side audit trail shows zero writes by the API user.
- **Operational calm**: ≥2 weeks with no sync incident requiring engineering intervention.
- **Condition map coverage**: deterministic mapping (§2.4 step 1) covers ≥90% of open conditions on active files — *built during Phase 1, required before Phase 2 can mean anything*.
- **No-go signals**: match rate stuck below target; staff distrust the panel; token/ops churn.

### Phase 2 — Rules enforcement + borrower-visible reads
Turn on the agreement gates inside the existing `signOffGate` chokepoint (`src/routes/staff.js:2400`): a mapped portal condition cannot be cleared unless the paired Encompass condition agrees; portal CTC requires Encompass CTC. Fail-closed on stale data, with an audited staff override (reason required) so a sync outage never silently blocks the business without a trace. Ship the borrower milestone timeline (§2.7) staff-first, then borrower-facing. Still zero writes.

**Gate 2 → 3 (go/no-go):**
- **False-block rate**: staff overrides of the Encompass-agreement gate <5% of clears, every override reasoned; trend flat or falling over ≥8 weeks.
- **Value shown**: at least one documented case where the gate caught a premature clear/CTC — or, minimally, months of zero wrongful blocks.
- **Adoption**: staff are not working around the portal (override log + usage tell the truth).
- **Borrower surface clean**: milestone timeline live to borrowers ≥4 weeks with no confusion incidents and no leaked internal/partner strings.
- **Per-item owner sign-off in the owner's own words** for each Phase 3 candidate (the house convention for anything consequential).
- **No-go signals**: gates being routinely overridden (the mapping is wrong — fix Phase 2, don't advance); any incident where stale data blocked or misled.

### Phase 3 — Selective expansion (in this order, each independently gated)
1. **Report exports** (§2.2 + §2.6 on one shared primitive) — read-only, immediate business value.
2. **Servicing handoff packages** (§2.3) — read-only.
3. **eFolder document push** (§2.1) — the *first and only initial* write, under the full seven-step ceremony, dry-run first, kill switch default-off, in a separate write service with its own credential. Bonus: it can retire double-uploading for staff (borrower uploads once to the portal; Encompass receives it) — squarely "helping out."
4. **Custom-field breadcrumb** (§2.1) — only after ≥3 months of clean document pushes.
5. **OCR cross-checks** (§2.5) — advisory-only, only if staff ask.
- **Permanent non-goals** restated in the policy doc: condition/status write-back, loan creation/edit, milestones, locks, disclosures, deletes/overwrites of any kind.

## 4. Open questions for the owner / next agents

1. **Does the API user's Encompass persona support a read-scoped configuration** strong enough to make Phase 1's guarantee two-layered (Encompass-side persona restrictions + portal-side allowlist), and later a separate write persona limited to eFolder upload? (Cannot be verified locally — no credentials, no network per hard rules.)
2. **Loan matching key**: which field pairs portal `applications` to Encompass loans (loan number in a custom field? address+borrower fuzzy match, like the SharePoint resolver in `src/lib/sharepoint-map.js`?). Match-rate gates depend on choosing this deliberately.
3. **Polling vs. webhooks** for Phase 2 freshness: is the webhook-subscription config write acceptable under the owner's read-only pledge, or does polling remain the rule until Phase 3?
4. **Which report consumers first** — the warehouse lender template or the investor tape — and can both truly share one export primitive?
5. **Fail-closed tolerance**: how long an Encompass outage may block clears before the audited override becomes the expected path (this number should be written into the Phase 2 gate).
