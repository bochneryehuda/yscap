# E3 — Operational Automation Ideas from a Read-Only Encompass Feed

_Research agent E3. Scope: ideate ops automation the PILOT portal can build on a STRICTLY read-only Encompass Developer Connect feed (instance BE11397907). Ideas + sources + difficulty + noise risk; ends with a phase-1 shortlist. No implementation._

## 1. Grounding: the automation machinery the portal already has

Everything below can be reused as-is; a read-only Encompass poller only needs to *feed* these chokepoints.

### 1.1 Notification service (`src/lib/notify.js`)

| Capability | Where | Why it matters for Encompass automation |
|---|---|---|
| One chokepoint writes an in-app row + best-effort branded email | `src/lib/notify.js:2-8`, `_emailRow` L40-60 | Encompass events become notifications with zero new plumbing |
| Staff per-user email kill switch (`notifications_enabled`) | `notify.js:68-86` | Managers can mute a noisy rollout per person |
| Borrower category preferences (`messages / status_updates / documents / conditions / pricing / reminders / draws / other`) | `CATEGORY_OF` + `NOTIFY_CATEGORIES`, `notify.js:89-105` | Encompass-driven borrower pings slot into existing categories — no new pref UI |
| “Major moments only” borrower email gate (#88) | `BORROWER_MAJOR_EMAIL`, `notify.js:107-124`; `opts.major` per-event override used by status route (`MAJOR_STATUSES`, `src/routes/staff.js:3888`, applied at L4285) | The anti-noise pattern is already policy: only decision-grade events email the borrower |
| Must-see events bypass in-app muting | `ALWAYS_IN_APP`, `notify.js:103` | `condition_added` / `doc_rejected` always reach the borrower |
| **Capital-partner name scrub at the borrower chokepoint** | `notify.js:143-160` (`scrubTextExcept`) | Critical: raw Encompass condition/milestone text may contain note-buyer names (Churchill, RCN…). Anything borrower-facing must flow through `notifyBorrower`, and ideally only mapped `borrower_label` text — never raw Encompass wording |
| File-wide fan-out helpers | `notifyAppBorrowers` L174-181, `notifyAppStaff` L191-202 (assignees incl. assistants), `notifyAdmins` L205-217 | One call notifies the whole team of a file |
| `fileContext(appId)` meta block (loan #, address, borrower, program, amounts) | `notify.js:229-257` | Every Encompass alert can carry full file context for free |

### 1.2 Scheduler / dispatcher patterns

- **Minute-cadence reminder dispatcher**: `src/lib/reminders.js:260-311` (`dispatchDue` + `startDispatcher`, `setInterval` 60s, idempotent via `fired_at` stamps). Started from `src/server.js:410`.
- **Files that are `funded / on_hold / declined / withdrawn / file_intake` are MUTED** for reminders (`NOT_MUTED`, `reminders.js:270`) and excluded from active pipeline scopes (`INACTIVE_FILE_STATUSES`, `src/routes/staff.js:57`). ⚠️ Post-closing condition tracking (idea 10) collides with this — funded files are exactly the ones it must watch. Needs its own channel or an explicit exemption.
- **Daily upkeep + weekly self-gated digest already exist**: the ClickUp sync runs a `reviewUpkeep` pass on boot + every 24h (`src/sync/clickup-sync.js:1262-1271`), calling `sendReviewDigestOnce` (`src/lib/sync-review.js:243`) which self-gates to weekly. This is the exact template for an Encompass condition digest / reconciliation report job.
- **SSE event bus** for live UI updates (`src/lib/events.js:1-30`) — a war-room or side-by-side dashboard can update live without polling from the browser.

### 1.3 Review-queue pattern (the “never silently act” gate)

`sync_review_queue` (docs/OBSERVABILITY.md:82-90; `src/lib/sync-review.js`, `src/lib/sync-file-review.js`) parks every suspicious cross-system disagreement with BOTH sides’ values and concrete resolution actions, and notifies the file’s LO. A read-only Encompass feed should reuse this shape for mismatches: **report + queue for a human, never auto-fix either system** (portal writes stay possible, Encompass writes are architecturally off the table).

### 1.4 Observability standing rule

docs/OBSERVABILITY.md:3-8 (owner-directed 2026-07-17): **every integration — including any future one — must journal what was sent/received/refused so troubleshooting works from the database alone.** Every Encompass poll cycle, snapshot diff, and suppressed/sent notification must land in a queryable stream (its own `encompass_*` journal + the per-file timeline at `GET /api/staff/applications/:id/observability`, OBSERVABILITY.md:23-42).

### 1.5 Portal condition & status model the feed must map onto

- Checklist/condition statuses: `outstanding / requested / received / satisfied / issue` (`db/schema.sql:243`); conditions carry `borrower_label`/`borrower_hint` vs internal wording (CLAUDE.md frozen rule).
- Condition categories already include `post_closing` (`src/lib/conditions/types.js:27-35`).
- Application statuses: `file_intake, new, in_review, processing, underwriting, approved, clear_to_close, funded, declined, withdrawn` (`db/schema.sql:195`; labels `src/routes/staff.js:3884`).
- Sign-off gating funnels through `signOffGate` (`src/routes/staff.js:2400`) — the natural single place to later add “Encompass must agree” checks (other agents’ scope).
- Status transitions are journaled in `status_history` (OBSERVABILITY.md:102-104) — the Encompass milestone log is its cross-system twin.

## 2. Encompass read-only sources (from the Postman collection, 26.2)

All endpoints below are GETs or read-style POST queries (query body, no state change), verified in `/tmp/…/scratchpad/research/postman-endpoint-index.md` and the collection JSON.

|---|---|---|
| Pipeline batch query | `POST /encompass/v3/loanPipeline?limit=1000` (+ `include=LockInfo`), canonical fields via `GET /encompass/v3/loanPipeline/canonicalFields` | Filter on `Loan.LastModified` `greaterThanOrEquals` = cheap delta polling (example bodies in collection: Simple/Complex Query Pipeline). Also `Loan.CurrentMilestoneName`, `Loan.LoanFolder`, `loanIds[]` batch fetch |
| Milestones | `GET /encompass/v3/loans/{id}/milestones` (+ v1 variants, `/associates`) | Per-milestone status/dates/associates |
| Milestone vocabulary | `GET /encompass/v3/settings/milestones` | Build the milestone→portal-status map once |
| Conditions (enhanced) | `GET /encompass/v3/loans/{id}/conditions`, `/{cid}?view=full`, `/{cid}/tracking`, `/{cid}/comments`, `/{cid}/documents` | Status tracking endpoint gives the state-change trail |
| Conditions (legacy) | `GET /encompass/v1/loans/{id}/conditions/underwriting`, `/preliminary`, `/postclosing` | Which system BE11397907 uses must be confirmed |
| Rate locks | `GET /encompass/v1/loans/{id}/ratelockRequests` (+ `?view=detailed`, `/snapshot`) | Plus pipeline `include=LockInfo` |
| Disclosure tracking | `GET /encompass/v3/loans/{id}/disclosureTracking2015Logs` (or loan entity `?entities=disclosureTracking2015Logs`) | LE/CD sent/received/eConsent dates |
| AUS logs | `GET /encompass/v1/loans/{id}/ausTrackingLogs` (+ `/snapshot`) | Likely n/a for hard money (no DU/LPA) |
| eFolder docs | `GET /encompass/v3/loans/{id}/documents?view=Full`, `/attachments` | Detect appraisal/title doc arrival by document title |
| Targeted field reads | `POST /encompass/v3/loans/{id}/fieldReader` | Collection examples confirm virtual fields: `Log.MS.CurrentMilestone`, `Log.MS.Stage`, `Log.MS.LastCompleted`, `PCC.ALL`, `PCC.NOTCLEARED` (post-closing conditions), `LOCKRATE.CURRENTSTATUS`, `LOCKRATE.CURRENTDATETIME`, `LOCKRATE.RATESTATUS` |

**Webhooks caveat:** real-time `enhancedfieldchange` / milestone webhooks exist (`POST /webhook/v1/subscriptions`) but creating a subscription is a *platform-config write* (not a loan write). Under the “architecturally incapable of writing” posture, Phase 1 should be **poll-only** (`Loan.LastModified` delta queries); webhooks are a Phase-2 decision for the owner.

**Prerequisite for everything below:** an `encompass_loan_snapshot` (+ per-collection child snapshots: milestones, conditions, locks) table diffed each poll. Every idea is “diff → route through existing notify/digest chokepoints”, and every diff/act/suppress is journaled (§1.4).

## 3. Idea catalog

Scoring: **Value** to ops, **Difficulty** to build honestly (incl. mapping), **Noise risk** = chance of alert fatigue / wrong-audience pings.

### 3.1 Milestone-change notifications (staff first, borrowers later)

- **What:** when a linked loan’s Encompass milestone advances (e.g. Processing → Submittal → Approval → Docs Out → Funding), notify the file team via `notifyAppStaff` (`type:'status_change'`); optionally the borrower via `notifyAppBorrowers` with a portal-safe label.
- **Source:** pipeline delta on `Loan.CurrentMilestoneName` (cheapest, one query for all loans) confirmed against `GET /v3/loans/{id}/milestones`; `Log.MS.*` via fieldReader for spot checks.
- **Value: HIGH.** LOs stop tab-switching into Encompass; the borrower stepper gains an authoritative twin.
- **Difficulty: LOW** (needs the snapshot diff + a milestone→label map from `settings/milestones`).
- **Noise risk: LOW–MED.** Milestones move a handful of times per loan. Borrower copy must (a) route through the scrub chokepoint (automatic, `notify.js:143-160`), (b) respect `status_updates` prefs + `major` gating — recommend borrower email only on decision-grade milestones, mirroring `MAJOR_STATUSES` (`staff.js:3888`). Staff copies: in-app always, email default on.

### 3.2 Rate-lock expiration countdown alerts

- **What:** for loans with an active lock, alert LO at T-7/T-3/T-1 days before `LOCKRATE` expiration; escalate if expiration < estimated closing date.
- **Source:** pipeline `include=LockInfo`; detail via `GET /v1/loans/{id}/ratelockRequests?view=detailed`; `LOCKRATE.CURRENTSTATUS/CURRENTDATETIME` fieldReader.
- **Value: UNKNOWN→MED.** Hard-money/business-purpose lending often doesn’t run Encompass lock desk at all. **Confirm YS actually uses ratelock requests in BE11397907 before building** — otherwise this is dead weight.
- **Difficulty: LOW** (deterministic countdown; reuse reminder-style idempotent stamps so each threshold fires once — `fired_at` pattern, `reminders.js:258-301`).
- **Noise risk: LOW** if thresholds are fixed and per-lock deduped.

### 3.3 Condition status digests (daily, per officer)

- **What:** one daily email/in-app digest per staffer: across *their* files, Encompass conditions added / status-moved / cleared / waived since yesterday, grouped by file. No per-condition drip.
- **Source:** `GET /v3/loans/{id}/conditions` (+ `/tracking` for precise transitions), or v1 lists if legacy conditions are in use; counts cross-checkable via `PCC.*`-style virtual fields.
- **Value: HIGH.** Processors live in condition churn; a digest converts it into one scannable unit.
- **Difficulty: MED** (per-condition snapshot + diff; grouping per assignee via `application_assignees`).
- **Noise risk: LOW** — the digest *is* the noise-control (mirrors the weekly `sendReviewDigestOnce` self-gating pattern, `src/sync/clickup-sync.js:1262-1271`). Keep it staff-only; raw Encompass condition titles must never reach borrowers (partner-name rule).

### 3.4 Document-needed nudges driven by Encompass condition state

- **What:** when an Encompass condition that maps to a portal checklist item is (still) open in Encompass and the portal item is `outstanding/requested`, nudge the borrower (`type:'doc_requested'` — already in `BORROWER_MAJOR_EMAIL` and category `documents`).
- **Source:** v3 conditions + the condition↔checklist mapping (built by the matching workstream, other agents).
- **Value: MED–HIGH** (chases docs without human effort).
- **Difficulty: HIGH.** Depends entirely on trustworthy condition matching; must compose with the portal’s own request flow (which already notifies) — otherwise double-nudging.
- **Noise risk: HIGH** — the worst idea to ship early. A wrong or duplicate nudge to a borrower burns trust. Mitigations if/when built: only nudge when BOTH systems agree the item is open, cooldown (e.g. ≥5 days since last portal notification on that item), weekly cap per borrower, borrower text = portal `borrower_label` only. **Recommend Phase 2.**

### 3.5 Pipeline dashboard — portal vs Encompass side-by-side

- **What:** staff screen listing every linked file: portal status vs Encompass milestone, loan amount / rate / closing date from both sides, open-condition counts both sides, lock + CTC flags; delta cells highlighted. Filterable per officer.
- **Source:** one `POST /v3/loanPipeline` with `loanIds[]` of all linked loans (batch, ~1 call per 1000 loans) + stored snapshots; no per-loan fan-out needed for the list view.
- **Value: HIGH — and it’s the trust-builder.** Before any alert fires, staff can *see* the mapping is right. Also the natural home for the “Encompass agrees?” badge the gating rules will need.
- **Difficulty: LOW–MED** (read model + one React screen; SSE for live refresh is optional).
- **Noise risk: NONE** (pull UI, zero notifications).

### 3.6 Daily reconciliation report of mismatches

- **What:** nightly job diffs portal vs Encompass on the agreed field set (status↔milestone, loan amount, rate, closing date, borrower name, property address, CTC flag, open-condition counts) → writes rows to an `encompass_review`/mismatch queue (reusing the `sync_review_queue` two-sided-value shape, OBSERVABILITY.md:82-90) → one summary email to admins via `notifyAdmins` (`notify.js:205-217`) with counts + top offenders.
- **Value: HIGH.** This is the operational backbone of Phase 1’s core promise (“portal can’t clear unless Encompass agrees”): gates are only trustworthy if drift is surfaced daily. Also the safety net for poller bugs.
- **Difficulty: MED** (field-normalization rules: money rounding, address canon — reuse `src/lib/address-canon.js`; name matching from ClickUp identity work).
- **Noise risk: LOW–MED.** One email/day, capped; mismatch rows need acknowledge/suppress so a known-acceptable difference doesn’t re-report forever (`suppressIfRejected` precedent in `sync-file-review.js`).

### 3.7 Closing-week war-room view

- **What:** dashboard of files whose estimated closing date is within N days (default 7): countdown, CTC status both sides, open conditions both sides, disclosure/doc-order state, lock expiry vs closing date, missing-doc list. Optional 8am daily war-room digest to processors/admins.
- **Source:** pipeline query filtered on the closing-date canonical field + the per-loan condition/lock/disclosure pulls already made for 3.3/3.2/3.8 (composite view — near-zero extra API cost).
- **Value: HIGH & very visible.** Closing week is where hard-money ops melts down; this is the screen the owner will show people.
- **Difficulty: MED** (composition + one screen; data all comes from other items).
- **Noise risk: NONE** as a view; LOW with a single scheduled daily digest.

### 3.8 Disclosure tracking visibility

- **What:** per-file panel of LE/CD sent / received / eConsent events from `disclosureTracking2015Logs`; alert if closing is near and required disclosure events are absent.
- **Value: LOW–MED for YS.** Business-purpose loans are generally TRID-exempt, so the 2015 DT log may be empty in this instance. **Verify against a real loan before investing.** If YS does any consumer-purpose lending, value jumps to HIGH (compliance).
- **Difficulty: MED**; **Noise risk: LOW** (staff-only, threshold-based).

### 3.9 Appraisal / AUS status surfacing

- **What:** (a) appraisal: detect appraisal-titled documents/attachments appearing in the eFolder (`GET /v3/loans/{id}/documents`) and notify the file team “appraisal received in Encompass”; surface ordered/received dates if tracked as fields. (b) AUS: `ausTrackingLogs`.
- **Value:** appraisal MED–HIGH (real hard-money bottleneck; the portal already handles appraisal *payment* via `src/lib/appraisal-card.js` — status surfacing completes the story). AUS: **near-zero for hard money** — skip.
- **Difficulty: MED** (doc-title matching heuristics per their eFolder naming).
- **Noise risk: LOW–MED** (title matching can misfire; keep staff-only).

### 3.10 Post-closing condition tracking

- **What:** after funding, track Encompass post-closing/trailing conditions (recorded deed, final title policy, insurance follow-ups): weekly staff digest of aging items; surfaced on a “funded – trailing docs” screen.
- **Source:** `GET /v1/loans/{id}/conditions/postclosing` and/or `PCC.ALL` / `PCC.NOTCLEARED` fieldReader virtuals (both confirmed in the collection).
- **Value: MED–HIGH** — trailing docs are chronically dropped precisely because funded files leave everyone’s active views.
- **Difficulty: MED, plus a product decision:** funded files are deliberately MUTED for reminders (`reminders.js:270`) and excluded from active-pipeline scopes (`staff.js:57`); the portal’s `post_closing` condition category exists (`types.js:27-35`) but needs its own visibility channel that doesn’t un-mute everything else.
- **Noise risk: LOW** (weekly digest cadence; small population).

## 4. Cross-cutting noise-control design (applies to every idea)

1. **Digest-over-drip default.** Per-event pings only for decision-grade moments (milestone advance, CTC, lock T-1); everything else aggregates daily/weekly — the codebase already votes this way (#88 major gating, weekly review digest).
2. **Diff-once, notify-once.** All alerts derive from snapshot diffs with idempotent “fired” stamps (the `reminders.js` `fired_at` pattern) so poll retries can never double-send.
3. **Borrower-facing = mapped text only.** Never raw Encompass condition/milestone wording to borrowers (partner-name freeze); the `notifyBorrower` scrub is the backstop, not the plan.
4. **Respect existing mutes**: borrower category prefs, staff `notifications_enabled`, inactive-file muting (except the deliberate post-closing exemption).
5. **Journal everything** (OBSERVABILITY standing rule): poll cycles, diffs, sent AND suppressed notifications, mismatch rows — queryable without ICE credentials.
6. **Read-only discipline:** every automation output lands in the portal (notification, dashboard, review row). Nothing ever calls a non-GET Encompass endpoint; mismatch resolution changes the *portal* or tells a human to fix Encompass in Encompass.

## 5. Recommended Phase-1 shortlist

| # | Item | Why first |
|---|---|---|
| 1 | **Side-by-side pipeline dashboard (3.5)** | Zero noise risk, immediate visible value, and it *validates the loan↔file linkage* every other feature depends on. One batch pipeline query powers it |
| 2 | **Daily reconciliation report (3.6)** | The backbone of the “Encompass must agree” gates; catches mapping/poller bugs before they bite; reuses the review-queue + notifyAdmins machinery |
| 3 | **Milestone-change notifications to staff (3.1)** | First push automation; low volume, low mapping risk, high daily utility. Borrower copies deferred until the map is proven |
| 4 | **Closing-week war-room view (3.7)** | Highest “wow” per unit of work — it composes data items 1–3 already pull, with no new alerting risk |

Optional 5th if confirmed in use: **rate-lock countdown (3.2)** — trivially cheap once lock data is in the snapshot, but only if YS actually runs locks in Encompass.

Explicitly deferred to Phase 2: borrower-facing document nudges (3.4 — highest noise risk, depends on condition matching), disclosure tracking (3.8 — likely TRID-exempt), AUS (3.9b — n/a), post-closing tracking (3.10 — wants a product decision on the funded-file mute first).

## 6. Open questions

1. Does BE11397907 actually use Encompass **rate-lock requests**, **AUS**, or **2015 disclosure tracking** (business-purpose lender → possibly none)? One read of a few real funded loans answers all three.
2. **Enhanced (v3) vs legacy (v1) conditions** — which does the instance use? Determines the condition endpoints and whether `/tracking` transition history is available.
3. Milestone vocabulary of the instance (`GET /v3/settings/milestones`) — needed for the milestone→portal-status map before 3.1 ships.
4. Loan-folder layout + active-loan volume — sets poll cadence and whether one `Loan.LastModified` delta query per few minutes suffices (rate-limit budget; other agents own limits research).
5. Is creating a **webhook subscription** (a platform-config POST, not a loan write) acceptable under the read-only mandate, or is polling the hard requirement? Phase 1 assumes polling.
6. Post-closing: owner decision on how funded-file tracking coexists with the deliberate funded-file mute (`reminders.js:270`).
7. Where should the loan↔file link live and how is it established (ys_loan_number / investor_loan_number ↔ Loan.LoanNumber)? Owned by the matching workstream (see E1/A-series findings) — every idea here presumes it exists.
