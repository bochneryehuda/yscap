## 1. Big picture

The ClickUp integration is a **DB-queue-mediated, loop-driven sync** running inside the single Express process (started from `src/server.js:404` when `RUN_SYNC=1`/dev). Nothing is processed inline in an HTTP request; every inbound event and outbound write passes through a durable Postgres table and is drained by interval loops in `src/sync/clickup-sync.js`.

```
ClickUp ──webhook (HMAC)──▶ /api/clickup/webhook ──▶ clickup_webhook_inbox (dedup by body hash)
ClickUp ◀──scoped field writes── sync_queue (outbox) ◀── enqueue-on-write from portal edits
        ──filtered poll (dateUpdatedGt watermark)──▶ reconcile loop ─┐
                                                                     ▼
                       ingest.ingestTask (idempotent, keyed on task_id)
                          ├─ resolveBorrower (identity graph, fill-only heal)
                          ├─ linkOrCreateApplication (stamp/loan#/identity match)
                          ├─ applyChecklistStatuses (pull-only, no-downgrade)
                          └─ clickup_task_index snapshot (masked, audit)
```

Four loops, all gated by `cfg.clickupSyncEnabled` (`src/sync/clickup-sync.js:1-9`):

| Loop | Function | Cadence | Direction |
|---|---|---|---|
| Outbox push | `pushOutboxOnce` (`clickup-sync.js:49-126`) | every 3 s (`:1294`), only if `CLICKUP_OUTBOUND_ENABLED=1` | portal → ClickUp |
| Inbox drain | `processInboxOnce` (`clickup-sync.js:644-672`) | every 4 s (`:1280`) | ClickUp → portal |
| Reconcile poll | `reconcileOnce` (`clickup-sync.js:739-761`) | every `CLICKUP_POLL_SEC` (default 300 s, `src/config.js:215`) | ClickUp → portal |
| Boot one-shots | backfill, `reconcileLinkedProgramsOnce`, `recoverUnlinkedFilesOnce`, `retryStuckTasksOnce`, audits (`clickup-sync.js:1206-1259`) | on boot | mostly ClickUp → portal |

Config switches (`src/config.js:209-232`): master `CLICKUP_SYNC_ENABLED`, separate `CLICKUP_OUTBOUND_ENABLED` (staged rollout — inbound can run alone), `CLICKUP_INBOUND_CREATE_FILES` (gates materializing NEW portal files from tasks), `CLICKUP_DRYRUN` (read-only validation boot that runs the mapper against live data and logs what *would* happen, `clickup-sync.js:1182-1189`), `CLICKUP_RUN_BACKFILL` (`'data'` = identity graph only, `'full'` = also create files), `CLICKUP_RUN_AUDIT`.

`src/sync/queue.js` is the **legacy** generic worker: its `op='create'` handler is permanently retired (throws — `queue.js:18-26`) and its claim query is defensively scoped to `op='create'` only so it can never steal the modern scoped-push jobs (`queue.js:44-57`). It is not started by `server.js`; the live drain is `pushOutboxOnce`.

## 2. Webhooks (real-time path)

`src/routes/clickup-webhook.js` — mounted **outside** the JSON body parser at `/api/clickup/webhook` (`server.js:28`) because HMAC verification needs the raw bytes:

- `express.raw()` for this route only (`clickup-webhook.js:39`); ClickUp signs the raw body with HMAC-SHA256, sent as `X-Signature`; verified constant-time by `client.verifyWebhookSignature` (`client.js:296-303`).
- **Fail closed in production** when the secret is not configured: 503 (`clickup-webhook.js:47-48`).
- **Idempotency key = `sha256(rawBody)`** (`clickup-webhook.js:56`) inserted into `clickup_webhook_inbox` with `ON CONFLICT (event_id) DO NOTHING` (`:61-64`) — ClickUp's at-least-once redeliveries collapse to one row.
- **PII redaction before persistence**: SSN/card custom-field values in `history_items` are replaced with `[redacted]` before the payload lands in the jsonb inbox (`clickup-webhook.js:19-36`).
- **Ack fast, process later**: the handler only inserts + returns 200 (even on insert failure — ClickUp will retry; `:65-69`). Processing is entirely asynchronous.

Inbox table `clickup_webhook_inbox` (`db/042_clickup_control.sql:35-49`): `event_id UNIQUE`, `status ∈ received|processing|done|error|ignored`, `attempts`, `last_error`, `received_at/processed_at`, plus `processing_started_at` added by `db/080` for crash-reclaim correctness.

Notably, the inbox drain **does not trust the webhook payload's field values at all** — it only takes the `task_id` and re-fetches the full task from the API (`processInboxOnce` → `ingestOne`, `clickup-sync.js:664, 677-685`). The webhook is a *trigger*, not a data source. This makes every ingest path (webhook, poll, backfill, boot retry) converge on one code path: `ingest.ingestTask(fullTask, options, {createFile})`.

## 3. Polling: reconcile + backfill (webhooks are treated as lossy)

The design assumption is explicit: webhooks catch most changes, a periodic **filtered team-task poll** catches the rest (`clickup-sync.js:5 "catch missed webhooks + hot duplicates"`).

- `reconcileOnce` queries `GET /team/{teamId}/task` filtered to all pipeline folders with `date_updated_gt = watermark`, `include_closed=true`, `subtasks=true` (`clickup-sync.js:745-747`, client at `client.js:261-275`). Every returned task is fully re-ingested (fetching full custom fields if the list payload lacks them).
- `runBackfill` (`clickup-sync.js:928-966`) is the one-shot **historical import**: per-folder pagination (`page` loop, `<100 tasks = last page`), each task through the same `ingestTask`, with a logged verification summary (linked/assigned counts, match_status spread). Can be scoped to one folder for a self-serve officer re-sync.
- `reconcileLinkedProgramsOnce` (`clickup-sync.js:863-923`) is a boot-time **full sweep of all linked files** (not windowed): re-ingests each, detects program descope, detects deleted tasks (hard 404) → orphan resolution.
- The folder set polled is derived from the routing registry (`PIPELINE_FOLDERS()`, `clickup-sync.js:21-27`): every officer/processor pipeline folder + Lead Capture.

## 4. Watermark / reconciliation strategy (the durable bookmark)

This is the piece most directly reusable for Encompass. History: the watermark lived in process memory; 13 deploys in one day each reset it and re-scanned 24 h of tasks — "a portfolio-wide re-ingest storm on every restart" (`db/125_sync_runtime_state.sql:4-9`).

Current design (`clickup-sync.js:687-761`, WO-4):

- **Persisted** in `sync_runtime_state` (`db/125`): a generic durable key/value store (`key text PK, value jsonb, updated_at`), key `clickup_reconcile_watermark`, value `{since_ms}`. `loadState`/`saveState` upsert (`clickup-sync.js:704-717`).
- **Captured BEFORE the query** (`preQueryMs`, `:741`) so a task updated *during* the pass falls after the new bookmark and is caught next time.
- **Advances only on a fully-clean pass**: any per-task ingest failure sets `hadFailure` and the bookmark keeps its old value, so the choked task is re-covered (`nextWatermark`, `:734-737`).
- **Overlap**: the new bookmark is `preQueryMs − 2 min` (`RECON_OVERLAP_MS`, `:700`) — boundary tasks are deliberately re-ingested (safe because ingest is idempotent).
- **Clamps** (`reconcileSince`, `:722-729`): no bookmark → default 24 h lookback; ancient bookmark (long outage) → max 72 h catch-up (never hammer the API with a month scan); future-skewed bookmark → clamped to now (self-corrects).
- The two helpers are **pure functions, exported and unit-tested** with no DB/network (`scripts/test-reconcile-watermark.js` — asserts resume-exact, 72 h clamp, never-future, advance-only-on-success, pre-query ceiling).

## 5. Snapshot auditing

Every ingest — RTL loan or not — upserts a **masked snapshot** of the task's fully-mapped read into `clickup_task_index` (`ingest.js:1159-1174`; columns from `db/044:39-51` + `db/046`):

- `buildMaskedSnapshot` (`ingest.js:49-78`): full mapped `app`/`borrower`/`llc` blocks with SSN reduced to `***-**-1234`, card reduced to `hasCard` boolean, co-borrower email partially masked/phone last-4, plus **`unmapped`** = every ClickUp field the mapper did not map (a standing "what are we missing" audit feed) and `rawProgram` (so non-RTL/long-term tasks are preserved as `kind='data_only'` rather than dropped).
- `snapshot_at` / `last_seen` timestamps; `task_name`, `folder_id`, `internal_status` also stored for non-materialized tasks (`db/046:12-16`).
- The snapshot is not just forensic — it is **used as evidence** by later logic:
  - the boot-time identity-mismatch audit compares borrower profile vs. last-ingest snapshot with **zero extra API calls** (`clickup-sync.js:200-515`);
  - the "human edited DOB at the source" detection compares the incoming DOB against the previous snapshot's DOB (`ingest.js:149-158`);
  - the "cleared DOB" evidence gate requires the previous snapshot to have carried a DOB before treating a blank as an intentional clear (`ingest.js:226-235`);
  - ops dashboards/audits aggregate `snapshot->'unmapped'`, funded-status gaps, etc. (`auditData`, `clickup-sync.js:973-1060`).
- Complementary audit surfaces: `clickup_pull_field_change` audit rows for every inbound value change on a linked file (before→after diff, `ingest.js:1413-1438`), `clickup_write_log` journal for every outbound write (orchestrator), and `audit_log` system rows for descope/orphan/roles-swap actions.

## 6. Retry / backoff (three layers)

**Layer 1 — in-call HTTP retry** (`client.js:101-214`, WO-2, tested by `scripts/test-clickup-retry.js`):
- Per-process **token bucket** pacing at `CLICKUP_MAX_RPM` (default 70, under ClickUp's 100/min) so 429s rarely happen at all (`:151-161`).
- `MAX_TRIES=3` quick attempts; retry only on 429/5xx/network/timeout; 4xx fails fast (`isRetryableStatus`, `:125-127`).
- Backoff honors `Retry-After`, else capped exponential 500 ms → 8 s + jitter (`backoffMs`, `:132-135`); 20 s per-request timeout via AbortController.
- Errors are **tagged** `e.status / e.retryable / e.retryAfter` and messages are value-free (never PII) — `httpError`, `:139-145`. Explicit division of labor: the short in-call budget smooths blips; "the durable Postgres queue owns the long game" (`:110-114`).

**Layer 2 — durable queue retry (outbox)** (`pushOutboxOnce`, `clickup-sync.js:89-124`):
- Normal failure: exponential `2^attempts` seconds capped at 3600; **dead at 8 attempts**.
- **Outage-class** failure (`CLICKUP_CIRCUIT_OPEN`, `CLICKUP_PREREAD_FAILED`, or `e.retryable`): fixed 10-min spacing, dead only after **40 attempts (~7 h)** — because the default budget was shorter than the outbound circuit breaker's own window, a legitimate edit could dead-letter during a breaker opening (`:91-103`).
- **Dead-letter is never silent**: a `sync_review_queue` row (`push_dead_lettered`) with a "Retry push" action is queued; any later successful push for the file auto-closes it (`:78-88, 107-123`).
- **Crash reclaim**: jobs stranded in `processing` for >5 min are reclaimed (re-push is idempotent) (`:50-61`).

**Layer 3 — inbox retry**: failed ingest → `attempts+1`, back to `received`; `error` (parked) after 6 attempts (`processInboxOnce`, `:666-670`). Rows stranded in `processing` >15 min **from claim time** (`processing_started_at`, db/080) are reclaimed — measured from claim, not receipt, so a slow backlog can't double-ingest a row still being worked (`:644-660`).

Push-side completeness (context, tested by `scripts/test-orchestrator-push-failures.js`): a partially-failed multi-field push must **throw** (`assertPushComplete`) rather than let the job be marked done; failure records carry only fieldId/status/code/retryable/message — never the value (GLBA). An all-transient failure set is retryable; one permanent failure dead-letters to a visible card.

## 8. Identity crosswalk — how external IDs map to portal entities

**Field/ID registry** (`src/clickup/fields.js`): the single source of truth for every ClickUp custom-field UUID (SHARED cross-space PII fields, PIPELINE loan fields, CHECKLIST status dropdowns with per-option UUIDs, SYNC integration fields like `portalFileId`/`portalFileLink`/`sendToPortal`, EXTRA staff-workflow fields). Runtime-editable overrides exist in `clickup_field_mappings` (`db/042:15-33`) with the constants as fallback.

**Enum crosswalk** (`src/clickup/crosswalk.js`): portal value ↔ ClickUp option **label** (stable), with the option UUID resolved at runtime against the live option list (`resolveWriteId`/`resolveReadValue`, `:170-183`) so option-ID churn never breaks mapping. Reads come back as `orderindex` integers, writes take UUIDs — `transforms.dropdownIndexToLabel/dropdownLabelToId` are the single translation point (`transforms.js:245-269`). Descope of non-RTL programs fires **only on a positively-recognized non-RTL label** (allowlist + conservative keywords, `crosswalk.js:144-164`) — never merely because mapping failed (mass-soft-delete protection).

**Status crosswalk** (`src/clickup/status.js`): two-layer model — `internal_status` = exact ClickUp status string (38-status workflow, mirrored verbatim), borrower-facing `status` = derived via `EXTERNAL_FOR` map + keyword fallback for unknown new statuses (`:79-99`); `isTerminal` uses **known statuses only** (fallback is display-only, never a materialization gate, `:115-124`).

**People routing** (`src/clickup/routing.js`): officers/processors keyed by **pipeline folder id → staff email** (the folder a task lives in is the primary ownership signal; the Loan Officer Email field is fallback only — a stale email field can't reassign a file out of its folder, `:117-132`); `clickupEmail` aliases handle portal-vs-ClickUp email drift; case-insensitive name index for outbound routing (`:144-162`); unknown/blank officer → Lead Capture folder. `staff_users.clickup_user_id` is backfilled by email from live workspace members on boot (`backfillMemberLinksOnce`, `clickup-sync.js:1149-1175`).

**Person identity** (`src/clickup/identity.js`): 8 identity fields (address, loanNumber, borrowerName, dob, email, ssn, phone, purchasePrice) with normalizers; **≥2 agreeing fields = same loan** (`isMatch`, `:79`), ≥2 populated fields = enough to materialize a file (`canMaterialize`, `:82`; plus not in scratch statuses — `clickup-sync.js:30-37`). SSN matching uses a **keyed HMAC-SHA256 hash** (`ssnHash`, `:26-30`) — plaintext never compared/stored in the graph. Email matches require **corroboration** (phone, DOB, or full first+last name; surname alone is banned after the family-shared-email wrong-merge incident, `:94-141`). `nameConflict` prevents ON-CONFLICT(email) adoption of a different person's row (`:143-161`).

**Borrower resolution** (`ingest.resolveBorrower`, `ingest.js:310-444`), in order: (1) exact SSN-hash → merge; (2) email match, merged only if SSN agrees or corroborated, SSN conflict = definitely two people; (3) otherwise create a **distinct** profile ("safe over-split") — with an `emailUnsafe` guard so the insert uses a synthetic `noemail+<taskId>@clickup.local` shadow email rather than ON-CONFLICT-merging into the person just refused; uncorroborated pairs are logged to `borrower_dedup_candidates` + a `shared_email` review card. A no-identity guard refuses to mint "Unknown Unknown" husk profiles (no name, no SSN, no DOB → `borrowerId=null`, materialization holds with a visible review card, `:363-374, 1497-1500`). Every resolution path runs `healBorrowerFields` — fill-only enrichment (COALESCE / placeholder-replacement only), with the DOB decision engine (`decideDob`), human-edit-wins, and wipe-don't-guess branches (`:107-308`).

**Task ↔ application binding** (`ingest.findExistingApp` `:564-748` + `linkOrCreateApplication` `:1185-1540`), in priority order:
1. `applications.clickup_pipeline_task_id = task.id` (`linked_task`) — the authoritative, immutable binding.
2. Portal-File-ID **stamp** written on the task (`linked_stamp`) — but a stamp resolving to an app bound to a *different live task* is a **stale copy** from the duplicate-a-task workflow and is ignored (`:569-593`).
3. `ys_loan_number` as a **global natural key** (`linked_loannum`) — same-borrower copies are treated as stale duplicates (`copiedLoanNumber`, never imported; officer gets a review card); cross-borrower collisions → `ambiguous` (`:594-630`). Placeholder loan numbers ("TBD", zeros…) are structurally excluded (`transforms.isPlaceholderLoanNumber`, `transforms.js:271-294`).
4. Identity scan of the borrower's **unlinked** RTL apps on normalized address / loan number — exactly one strong match links; multiple → `ambiguous` (`:631-655`).
5. Delete+recreate heal: same borrower + same normalized (or Google-place-canonical) address linked to a task confirmed 404 → relink (`relinked_dead_task`); live sibling with an active (non-terminal) deal → `duplicate_pending` defer (never create a same-address twin) (`:657-747`).
6. Else create (gated by `CLICKUP_INBOUND_CREATE_FILES` / forceCreate).

Outcomes are recorded per task in `clickup_task_index.match_status` (`linked_task|linked_stamp|linked_identity|linked_loannum|relinked_dead_task|created|ambiguous|duplicate_pending|data_only|descoped|skipped`, `db/045:63-65`), and **every non-materialized state produces a visible review row with actions** (`ingest.js:1064-1114`) that auto-closes on natural recovery.

## 9. Error handling philosophy

- **Enrichment is best-effort; the core pull never breaks**: heals, co-borrower resolution, checklist mirror, audits are all try/caught with `console.warn` (dozens of sites, e.g. `ingest.js:798, 914, 962, 1048`). The core upserts (task index, application COALESCE update) are the only must-succeed steps.
- **Nothing stuck is silent** (owner rule): ambiguous/duplicate-pending tasks, dead-lettered pushes, orphaned files, unlinked files past the recovery window, shared emails — each becomes a `sync_review_queue` card with concrete resolution actions, auto-closing on recovery and with sticky dismissals (`suppressIfRejected`).
- **Circuit breakers**: (a) outbound volume breaker — >300 field writes/10 min opens the circuit; further pushes throw `CLICKUP_CIRCUIT_OPEN` and the queue retries later (`orchestrator.js:479-503`); (b) orphan-resolution breaker — if 404s exceed max(5, 50% of checked) or **no** task resolved live, treat as an API/token outage and skip archiving entirely (`clickup-sync.js:913-917`); (c) `isTaskDeletedError` accepts only a hard 404 (or a narrowly-matched 401 "not found" body) — never blanket 401, which would classify the whole portfolio as deleted on a bad token (`:763-773`).
- **Structural write-safety hard stops** at the single HTTP chokepoint (`client.js:15-99`): no task DELETE ever; no empty/null/NaN field value (a clear); task updates are a status-only allowlist. These are what "architecturally incapable" looks like in this codebase — enforced where every request funnels, not at call sites.
- Boot one-shots are bounded and paced (LIMITs per boot, 400 ms pacing in `retryStuckTasksOnce`, oldest-first rotation so a backlog wider than one boot's cap doesn't starve the tail, `clickup-sync.js:602-629`).

## 11. Recommended reusable skeleton — Encompass PULL-ONLY sync worker

The ClickUp stack already contains a proven pull architecture; strip the push half and you have the Encompass Phase-1 shape. Recommended module layout (mirroring names 1:1 so the patterns transfer):

```
src/encompass/
  client.js      — read-only REST client (see hard stops below)
  fields.js      — canonical field-ID registry (Encompass field IDs / canonical names)
  crosswalk.js   — enum label maps (milestones, loan folders, statuses)
  status.js      — Encompass milestone/status → portal status derivation (+ keyword fallback, known-only for gates)
  identity.js    — reuse src/clickup/identity.js patterns (≥2-field match, HMAC ssnHash, corroborated email)
  ingest.js      — encompassLoan → portal upsert (COALESCE fill-only, masked snapshot, review cards)
src/sync/encompass-sync.js — loops: inbox drain (if webhooks), reconcile poll w/ durable watermark, backfill, audits
db/1NN_encompass_sync_core.sql — encompass_loan_index (loanGuid PK, snapshot jsonb, match_status), watermark keys in sync_runtime_state
```

**1. Read-only enforced structurally, not by convention.** Copy the `client.js` hard-stop pattern but invert it into an **allowlist of GET (and the few read-semantic POSTs)**: `call()` throws on any method other than GET except an explicit allowlist of read-query POSTs — `POST /encompass/v3/loanPipeline` (pipeline query), `POST /encompass/v3/loans/{id}/fieldReader` (batch field read), `POST /oauth2/v1/token` — mirroring `guardNoTaskDeletion` (`client.js:29-39`). Do not even implement PUT/PATCH/DELETE helpers (contrast: the ClickUp client had to keep `setField` and guard it; Encompass Phase 1 needs none). The existing `sync_queue.target='encompass'` CHECK already exists (`db/schema.sql:305`) — but Phase 1 should enqueue **no** push jobs; add a worker-side scope filter like `queue.js:44-57` so a future push worker can never accidentally drain pull-era rows.

**2. Watermark poll as the primary loop (webhooks optional).** Encompass Developer Connect has webhook subscriptions (`POST /webhook/v1/subscriptions` — see postman-endpoint-index.md lines 151-154), but start exactly the way ClickUp treats webhooks: *trigger only, re-fetch authoritative data*. The reconcile loop is the backbone:
   - Query `POST /encompass/v3/loanPipeline` filtered on `Loan.LastModified > watermark` (canonical field), paged `start/limit` (index lines 343-350), then `GET /encompass/v3/loans/{loanId}` (or fieldReader for a fixed field list) per changed loan.
   - Reuse `reconcileSince`/`nextWatermark` **verbatim** — they are pure, exported, and tested (`clickup-sync.js:722-737`); store `encompass_reconcile_watermark` in the existing `sync_runtime_state` table (no new infra). Keep: pre-query capture, advance-only-on-clean-pass, 2-min overlap, 24 h default / 72 h clamp.
   - If webhooks are added: raw-body route outside the JSON parser, signature check, `sha256(raw)` dedup into an `encompass_webhook_inbox` clone of `db/042`, drain loop with claim-time reclaim (`processing_started_at`, db/080 lesson).

**3. Idempotent ingest keyed on `loanGuid`.** Clone `ingestTask`'s spine: map → resolve borrower (reuse `identity.js` as-is — it is already source-agnostic) → link application (stamp equivalent = `applications.encompass_loan_guid`, natural key = loan number, then ≥2-field identity, then defer/ambiguous with review cards) → **COALESCE fill-only** column update (blank never clears) → masked snapshot upsert into `encompass_loan_index (loan_guid PK, snapshot, snapshot_at, match_status, last_seen)`. Keep the `unmapped` bucket in the snapshot — for Encompass's thousands of fields it is the cheapest possible "what are we not capturing" audit.

**4. The rules layer reads the snapshot, not the API.** The Phase-1 business rules ("a portal condition can't clear unless Encompass agrees"; "no portal CTC unless Encompass shows CTC") should be gates over `encompass_loan_index.snapshot` + `applications.encompass_*` mirror columns with a freshness check (`snapshot_at` within N minutes; stale → refuse-and-refresh), exactly like `applyChecklistStatuses`' no-downgrade model (`checklist.js:93-101`) — and like it, the gate module must **physically contain no client calls that write** (the "loopback guard by absence" pattern, `ingest.js:757-761`).

**5. Retry/backoff.** Port `client.js`'s tagged-error contract unchanged (token bucket sized to Encompass rate limits; 3 in-call tries; Retry-After; `e.retryable/status`; value-free messages) and the two-tier queue budget (fast exponential for normal failures, patient fixed-interval for outage-class). OAuth2 client-credentials token caching with early refresh belongs in `client.js`; a 401 should refresh-once-then-fail, never blanket-retry (the `isTaskDeletedError` lesson: never let auth failure be classified as data absence — gate any "loan deleted/archived" conclusion behind a live-fraction circuit breaker like `clickup-sync.js:913-917`).

**6. Visibility from day one.** Reuse `sync_review_queue` producers for: unmatched Encompass loans, ambiguous matches, identity mismatches (portal vs. snapshot audit, zero extra API calls — `auditIdentityMismatchesOnce` pattern), and stale-watermark alarms.
