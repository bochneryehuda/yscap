## 1. PII guarding and redaction — three distinct layers

### 1.1 Chat PII interception (`src/lib/pii-guard.js`)
- Detects SSNs (3-2-4 with separators, or bare 9-digit runs **only when nearby context** like "ssn/social/tax id" appears — avoids flagging phone/loan numbers) and card-like 13–19 digit runs validated with **Luhn** (`pii-guard.js:19-35`).
- Policy (`pii-guard.js:1-15`): **borrower sends → BLOCK the send** entirely (message never persists anywhere) with a redirect to the secure upload; **staff sends → REDACT in place** keeping last 4 (`[SSN ending 1234]`) and flag it. Detection runs **before persist/broadcast** — explicitly to stop the classic leak of a raw value landing in push/email payloads even when the stored copy was masked.
- Redaction events are audited (`pii_redacted` action, `src/lib/audit-actions.js:21`).

### 1.2 Storage-side redaction of JSON blobs (`src/lib/redact.js`)
- `redactPII(obj)` recursively **drops any SSN-ish key** (`(^|_)ssn$|social.?security|(^|b\d)ssn`, `redact.js:7`) from objects before they are stored as `raw_intake`/draft-snapshot jsonb. The SSN lives **only** in `borrowers.ssn_encrypted`; it must not survive in a jsonb blob.
- Applied at every intake write path: `src/routes/borrower.js:398, 2719`, `src/routes/leads.js:171`, `src/routes/intake.js:118`.

### 1.3 Borrower-facing text scrubbing (`src/lib/borrower-safe.js` + `test/borrower-safe.test.js`)
- Capital-partner / note-buyer names (BlueLake, Temple View, RCN, Churchill, Fidelis) may **never** reach a borrower surface; borrower copy says "Gold Standard program" (`borrower-safe.js:23-33`).
- Defense-in-depth in three layers (`borrower-safe.js:10-21`): on **write** (condition authoring), on **output** (borrower render), and at the **notify chokepoint** (`notifyBorrower`) so every borrower email/in-app alert is scrubbed regardless of who built the text.
- `scrubTextExcept(text, protect)` protects legitimate collisions ("12 Churchill Lane" addresses) via private-use-codepoint sentinels (`borrower-safe.js:52-86`). Staff surfaces are never scrubbed.
- Standalone assertion test `test/borrower-safe.test.js` (no test runner in repo — run with `node test/borrower-safe.test.js`).

**Pattern to copy for Encompass:** the portal has an established doctrine of (a) chokepoint filtering before persistence, (b) key-based redaction of any raw payload jsonb, and (c) borrower-facing output scrubbing. Raw Encompass loan JSON is exactly the kind of blob layer (b) exists for.

## 2. Encryption at rest and crypto primitives (`src/lib/crypto.js`)

Zero-dependency (Node built-in `crypto` only; only `express` + `pg` are installed).

| Concern | Mechanism | Location |
|---|---|---|
| Passwords | scrypt N=16384,r=8,p=1, random 16-byte salt, async (threadpool), timing-safe verify | `crypto.js:14-39` |
| Sessions | Hand-rolled HS256 JWT, timing-safe signature compare, `exp` enforced | `crypto.js:41-63` |
| MFA | RFC 6238 TOTP (SHA1/6-digit/30s), ±1 window | `crypto.js:65-106` |
| **SSN at rest** | **AES-256-GCM**; key = SHA-256 of `cfg.ssnKey`; random 12-byte IV; stored as `iv‖tag‖ciphertext` in `borrowers.ssn_encrypted bytea` | `crypto.js:108-126`; `db/schema.sql:57` |
| SSN write chokepoint | `ssnForStorage(raw)` — the **single** sanctioned path to persist an SSN: normalizes to 9 digits, returns `{encrypted, last4, digits}` or null; canonical ciphertext regardless of input formatting (fix for audit #234) | `crypto.js:128-139` |
| SSN matching | `ssn_hash` = HMAC-SHA-256 with `SSN_MATCH_KEY` (derived from `SSN_ENCRYPTION_KEY` by default) — equality matching without decryption; indexed | `src/config.js:88-90`; `db/044_borrower_history_backfill.sql:10-12` |
| Display | `ssn_last4 char(4)` only (`db/schema.sql:58`) |
| Appraisal card | Card JSON AES-encrypted with the same `decryptSSN` machinery; reveal audited (`view_appraisal_card`) | `src/routes/staff.js:2838` |
| MFA backup codes | 10 one-time codes, stored **sha256-hashed**, plaintext shown once | `crypto.js:144-158`; `db/088_mfa_backup_codes.sql` |
| Password policy | ≥10 chars, upper/lower/digit/symbol, denylist, identity-echo rejection; "HIPAA-aware / NYDFS 23 NYCRR 500" hardening comment | `crypto.js:160-200` |

**SSN read path is gated + audited:** `GET /api/staff/borrowers/:id/ssn` checks `canSeeBorrower` (horizontal authorization) and writes a `view_ssn` audit row **before** returning the decrypted value (`src/routes/staff.js:3441-3448`).

## 3. Permission / role model

### 3.1 Capability system (`src/lib/permissions.js`)
- Roles: `super_admin, admin, underwriter, loan_officer, loan_coordinator, processor, software_setup` (`permissions.js:16-24`). Not a strict hierarchy — **capabilities are the real authority**; `super_admin` implicitly has all (`permissions.js:74`).
- 11 capabilities (`permissions.js:28-44`): `see_all_files`, `review_conditions` vs `sign_off_conditions` (two-tier condition workflow — LOs stamp "reviewed", processors/underwriters sign off), `manage_conditions`, `manage_pricing`, `waive_conditions`, `delete_files`, `manage_vendors`, `manage_team`, `platform_setup`, `view_audit_log`.
- Effective perms = role defaults (`ROLE_DEFAULTS`, `permissions.js:49-61`) + per-user jsonb overrides in `staff_users.permissions` (`effectivePermissions`, `permissions.js:73-84`); gates call `can(actor, cap)` (`permissions.js:87-93`).
- `sanitizeOverrides` whitelists only known capability keys from client payloads (`permissions.js:96-101`).

### 3.2 File / borrower scoping (`src/routes/staff.js`)
- `seesAll = can(actor,'see_all_files')` (`staff.js:39`); processors additionally see all borrowers (`seesAllBorrowers`, `staff.js:45`).
- The single access chokepoint is `VISIBLE_OFFICERS_SQL(alias, p)` (`staff.js:136-140`): actor is primary LO/processor, OR on the primary's `visible_officer_ids` delegation list, OR an active row in `application_assignees` (assistants, `db/103`). Reused by the `/applications/:id` path middleware, `canTouchApp`, `canSeeBorrowerId` (SSN), `canSeeDocument` (downloads), pipeline/search scopes. Chat modules use the shared `assigneeExistsSql` from `permissions.js:108-110`.
- `canSeeBorrowerId` (`staff.js:2812-2826`) is the documented "GLBA / PII horizontal-authorization gate": a non-privileged staffer may see a borrower only if assigned to one of that borrower's files (co-borrower counted).
- Borrower vs staff data separation: `borrowers` (PII) is a separate table from `borrower_auth` (login) "to reduce blast radius" (CLAUDE.md data-model note). Conditions carry internal `label/hint` vs `borrower_label/borrower_hint`; audience flags (`borrower/staff/both`) drive what borrowers see.

### 3.3 Session security
- JWT carries `token_version` checked against the DB **on every authenticated request** — instant revocation on logout/password reset (`src/auth/index.js:56-76`).
- A pending-MFA challenge JWT carries `mfa:true` and is rejected as an access token (`auth/index.js:54-55`).
- Brute-force lockout on both staff and borrower logins (`failed_attempts`/`locked_until`, `auth/index.js:235-274`); staff lockout added by `db/084_staff_login_security.sql` (staff "can see every borrower's PII and decrypted SSNs" — the migration comment).
- Email-code brute-force cap: `email_tokens.code_attempts` (`db/118_email_token_attempts.sql`) lets the verify handler retire a token after a few wrong 6-digit guesses.

## 4. Audit logging

### 4.1 Store
`audit_log` (`db/schema.sql:322-335`): `actor_kind ('borrower'|'staff'|'system')`, `actor_id`, `action` (snake_case code), `entity_type/entity_id`, `ip_address inet`, `user_agent`, `detail jsonb`, `created_at`. Indexed by actor and entity; `db/079_audit_log_index.sql` adds `(created_at DESC, id DESC)` + `(action)` for the paginated company-wide screen.

### 4.2 Writers
Per-router `audit(req, action, entity_type, entity_id, detail)` helpers insert with actor + IP + UA (e.g. `src/routes/staff.js:120-125`; also `borrower.js:49`, `admin-conditions.js:23`, `staff-chat.js:23`; system actors in `src/sync/clickup-sync.js:776`, `src/lib/conditions/engine.js:162`, `src/lib/sharepoint-backup.js:910`, `src/lib/sync-file-review.js:84`).

### 4.3 Rendering / taxonomy
- `src/lib/audit-actions.js` maps ~100 action codes to labels + categories for the company-wide log (`GET /api/staff/audit-log`, gated by `view_audit_log`). First category is **"Access to PII (GLBA sensitive)"**: `view_ssn`, `view_appraisal_card`, `download_document`, `pii_redacted` (`audit-actions.js:17-22`). There is already a `sync` category ("ClickUp / system", `audit-actions.js:129-134`) — Encompass actions belong in a sibling category.
- Unknown/future codes render via `describeAction()` humanization fallback (`audit-actions.js:159-163`) — new Encompass action codes render without a code change.
- `src/lib/activity.js` merges domain tables + audit_log into a per-file feed; every row is tagged `borrower_safe` and the borrower feed shows **only events the borrower themselves performed** (`activity.js:170-173, 201-204`). Any Encompass-sourced audit rows would be staff-only unless explicitly whitelisted in `AUDIT_RENDER` with `borrowerSafe:true`.

### 4.4 Integration-write journaling precedent (the bar for Encompass)
The ClickUp sync journals **every outbound field write** (before + after values, changed/blocked flags, source) to `clickup_write_log` (`db/107_clickup_write_log.sql`), with **SSN/card values masked before landing in the journal** (`src/clickup/orchestrator.js:509-527` — card lines are parsed so the CVV never persists even masked). Phase 1 Encompass is read-only so there are no writes to journal — but the equivalent expectation is a **fetch/pull journal**.

## 5. Transport & HTTP hardening

- `src/lib/security.js` (applied to every response, `src/server.js:21`): `X-Content-Type-Options: nosniff`, `X-Frame-Options: SAMEORIGIN` (tool iframes), `Referrer-Policy: strict-origin-when-cross-origin`, `Permissions-Policy` lockdown, HSTS (prod + TLS-terminated only). No global CSP (marketing site inline scripts) — but **document downloads carry their own strict CSP + sandbox** and an inline-render allowlist (PDF/images/plain text only; SVG/HTML forced to download to prevent stored XSS reading the localStorage token) in `src/lib/serve-document.js:31-44`.
- `src/lib/safe-router.js`: every router is created through a factory that forwards async rejections to the JSON error middleware — no hung requests/502s; an Encompass router must be created the same way.
- `trust proxy` set; JSON body limit sized for base64 uploads (`server.js:19-37`).

## 6. Rate limiting (`src/lib/rate-limit.js` + `src/server.js:44-49`)

- Zero-dependency in-memory sliding window keyed `bucket:ip` (correct because the service is deliberately single-instance — it mounts a disk). Emits `RateLimit-*` headers and 429 + `Retry-After`.
- Buckets: `/auth` 30/min, `/api/intake` 20/min, `/api/leads` 20/min, `/api/guest` 90/min, `/api/address` 120/min. Purpose: cross-account credential stuffing, unauthenticated-surface flooding, and event-loop protection (scrypt on login).
- Note: this limits **inbound** abuse only. There is no generic **outbound** third-party throttle; the ClickUp sync built its own outbound circuit breaker (`CLICKUP_MAX_FIELD_WRITES_10MIN`, default 300 — CLAUDE.md write-safety rules). An Encompass client needs its own outbound rate/concurrency budget to respect Developer Connect API limits.

## 7. How API secrets are stored / loaded today

- **Everything comes from env vars, centralized in `src/config.js`** (`config.js:1`). A zero-dependency `.env` loader reads a bundled root `.env` for local runs, never overriding real env, never throwing (`config.js:8-26`). `.env` is git-ignored; `.env.example` documents the shape with placeholder values only.
- **`resolveSecret(name)`** (`config.js:34-50`) refuses the dev placeholder in production: if `JWT_SECRET` / `SSN_ENCRYPTION_KEY` is missing, it generates a strong ephemeral value, logs a loud SECURITY warning, and exposes `jwtSecretGenerated`/`ssnKeyGenerated` on `/api/health` (`config.js:92-94`) so the operator sees sessions/SSNs won't survive a restart.
- Existing third-party credential blocks all follow the same pattern — plain env consumed only server-side, integration dormant until configured: ClickUp (`clickupToken`, master switches default **off**, `config.js:208-232`), Microsoft Graph/SharePoint (client secret or preferred **certificate** auth `MS_CLIENT_CERT_PEM(_B64)`, master switch `SHAREPOINT_BACKUP_ENABLED=0` default, `config.js:166-206`), DocuSign JWT-grant (RSA private key PEM in env, `config.js:253-260`), Plaid, Xactus (`config.js:262-273`), Resend webhook Svix secret (`config.js:149-154`), shared-secret `INTAKE_API_KEY` checked via `x-intake-key` header (`src/routes/intake.js:21-24`). `src/lib/integrations/index.js` aggregates a `configured()` status per provider for the admin setup screen.
- Standing owner rule (CLAUDE.md): never hardcode/commit secrets; a credential pasted in chat is considered compromised and must be rotated; secrets are set in the Render dashboard.
- Frontend never sees provider keys — e.g. address autocomplete is proxied server-side precisely so "any real key lives only here, never in the public site bundle" (`config.js:236-241`).

## 8. Existing integration guard-rail precedents relevant to "architecturally read-only"

- `sync_queue` already models Encompass as a target: `target IN ('clickup','encompass','graph')` with `direction IN ('push','pull')` (`db/schema.sql:302-314`) — a **read-only** Phase 1 means only `direction='pull'` rows for `target='encompass'` may ever be enqueued, and ideally a CHECK/guard makes `('encompass','push')` unrepresentable.
- ClickUp already stores a free-text `encompass_status` on applications pulled from a ClickUp dropdown (`src/clickup/fields.js:74`, `src/clickup/mapper.js:122` — note `dir: 'pull'`), so "Encompass status" as a concept already exists in the portal data model via ClickUp.
- The SharePoint client proves the house style for structural incapability: `sharepoint.remove()` is a **permanent throwing no-op**; the HTTP chokepoint (`src/clickup/client.js`) refuses destructive payload shapes regardless of caller. The Encompass client should follow the same pattern: a client module that only implements GET (and the POST-only-because-HTTP token/cursor endpoints), with anything else throwing.

## 9. CHECKLIST — what the read-only Encompass integration must satisfy

### A. Secret storage & auth
1. All Encompass credentials (client id, client secret, instance id `BE11397907`, any API-user password) live **only** in env vars surfaced through `src/config.js` (an `encompass: {...}` block like `docusign`/`xactus`); never in source, docs, commits, or the frontend bundle. `.env.example` gets empty placeholders only.
2. A master switch defaulting **off** (`ENCOMPASS_ENABLED=0` style, mirroring `SHAREPOINT_BACKUP_ENABLED`/`CLICKUP_SYNC_ENABLED`) — nothing touches Encompass until explicitly enabled; expose a `configured()` status in `src/lib/integrations/index.js`.
3. Prefer the **Client Credentials (API User)** OAuth2 grant (`POST /oauth2/v1/token` per the Postman index) scoped to a **read-only Encompass persona/API user** — least privilege enforced on the Encompass side too, so even a bug in our code cannot write. Never the Resource Owner Password grant with a human's credentials.
4. Access tokens are held in memory only — never persisted to the DB, logs, or audit `detail` jsonb; token-refresh failures must degrade gracefully (integration reports unhealthy, portal keeps working — the `notify.js` "failed send never breaks the request" doctrine).

### B. Architecturally read-only
5. The Encompass HTTP client module exposes **GET-only** methods (plus the token POST); any PUT/PATCH/DELETE (and business POSTs) throw structurally, following `sharepoint.remove()` / `guardTaskUpdatePayload` precedent — no caller, present or future, can bypass.
6. `sync_queue` rows for `target='encompass'` are `direction='pull'` only; add a guard (app-level, ideally also a DB CHECK in a new migration) making an encompass push row unrepresentable.
7. Portal rules layered on Encompass data (condition-clear parity, clear-to-close gating) are enforced portal-side like `signOffGate` — they read Encompass-sourced fields, never call back out.

### C. PII handling (SSN / DOB / financials pulled from Encompass)
8. Any SSN arriving in Encompass loan JSON goes through the **single chokepoint `ssnForStorage()`** (`crypto.js:135-139`) → `ssn_encrypted` + `ssn_last4` + `ssn_hash`; the raw digits must never persist elsewhere.
9. Every stored raw Encompass payload/snapshot jsonb is passed through `redactPII()` first (extend `SSN_KEY` in `src/lib/redact.js` to match Encompass field spellings, e.g. `taxIdentificationNumber`, `ssn` field IDs like 65/97/HMDA variants) — mirroring how `raw_intake` is scrubbed today.
10. DOB values are date-only `'YYYY-MM-DD'` strings end-to-end (the CLAUDE.md date rules; pg OID 1082 parser) and pass `sanitizeDob` plausibility; conflicting DOB/SSN values between Encompass and the portal must go to a **review queue**, never silently overwrite (the `sync_review_queue` / `decideDob()` precedent — a second external system makes this rule more important, not less).
11. Encompass values never enter logs, error messages, or the write/fetch journal unmasked — reuse the `maskSSN`/`maskCard` journal masking pattern (`orchestrator.js:509-527`).
12. Borrower-facing surfaces rendering Encompass-derived text must run through `borrower-safe.scrubText` (investor/note-buyer names can appear inside Encompass loan data).

### D. Audit trail of every Encompass fetch
13. **Every fetch is audited**: an `audit_log` row (`actor_kind='system'` for scheduled pulls; `'staff'` + real actor id when a staffer triggers a refresh) with action codes in a new `encompass` (or reuse `sync`) category — e.g. `encompass_pull_loan`, `encompass_pull_milestones`, `encompass_ctc_check` — carrying loan GUID, endpoint class, and result in `detail` (never PII). Register labels in `src/lib/audit-actions.js` (fallback already renders unknown codes).
14. A dedicated append-only fetch journal (an `encompass_pull_log` table modeled on `clickup_write_log`, `db/107`) recording what field values were adopted into the portal (before/after, masked), so cross-system history "never goes dark" (the ClickUp incident rule). New numbered idempotent migration; add indexes like `db/079`.
15. Field changes adopted from Encompass surface in the per-file activity feed via `activity.js` as **staff-only** rows (borrowerSafe:false) unless deliberately whitelisted.

### E. Who can see Encompass-sourced data
16. Encompass-sourced fields inherit the existing scoping: reads route through `VISIBLE_OFFICERS_SQL` / `canSeeBorrowerId` / `seesAll` — no new parallel access path. If Encompass data is queryable per-application, it must sit behind the `/applications/:id` scoped middleware.
17. Encompass-sourced **SSN/DOB** obey the same reveal rules as portal-native: masked by default (last4), full reveal only via the audited `view_ssn`-style endpoint gated by `canSeeBorrower`.
18. Borrowers see only derived, borrower-safe outcomes (e.g. "condition cleared", "clear to close") — never raw Encompass payloads, internal condition titles, investor names, or underwriting commentary (the `borrower_title`/audience model, `activity.js:161-163`).
19. Admin-only management: Encompass connection setup/status behind `platform_setup`; the fetch journal viewable under `view_audit_log`.

### F. Operational hygiene
20. New routes built via `safeRouter` (`src/lib/safe-router.js`); Encompass health surfaced on `/api/health` like `storageWritable`/`jwtStable`.
21. Outbound throttle + circuit breaker for Encompass API limits (mirror `CLICKUP_MAX_FIELD_WRITES_10MIN` breaker), and inbound rate limiting on any new unauthenticated webhook endpoint (none expected in Phase 1; if Encompass webhooks are ever added, verify signatures like the Resend Svix flow, `config.js:149-154`).
22. Any Encompass credential ever pasted into chat/transcripts is treated as compromised and rotated before use (standing owner rule).
23. Every schema addition is a new numbered idempotent `db/NNN_*.sql`; backfills are deterministic and idempotent ("previous AND future files" rule).
